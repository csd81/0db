# 8. Speciális adattípusok

## Gráfmodellezés

A hagyományos adatbázis-modellezésben az UML-osztálymodell vagy az egyed-kapcsolat modell (ERM) az alábbi módon alakítható át relációs modellre:

- egy-egy kapcsolatok egy-egy asszociációkra és öröklésre (szülő/gyermek specializáció)
- egy-több kapcsolatok egy-több asszociációkra, aggregációra és kompozícióra
- több-több kapcsolatok több-több asszociációkra

Ezután az egy-egy és egy-több kapcsolatok idegen kulcsokkal valósíthatók meg a relációs modellben, a több-több kapcsolatokat pedig összekötő táblák (linking tables) reprezentálják. Létezik azonban egy természetesebb és kevésbé korlátozott mód az összes UML/ERM-tulajdonság modellezésére és megvalósítására: a gráfkoncepció, amely a sémában csak csomóponttípusokat és éltípusokat használ azok attribútumaival, az adatbázisban pedig csomópont- és élpéldányokat. Valójában a gráf-technológia egy régi adatbázis-struktúra, a hálózati adatbázismodell újraéledése.

## Gráftáblák SQL Serveren

Az MS SQL Server 2022-ben a gráf- és éltípusú táblák természetesebb módon fejezik ki a több-több kapcsolatokat, mint a hagyományos összekötő táblák a relációs modellben. A gráftáblák alkalmasak összetett, többféle csomóponttípust tartalmazó hálózatok modellezésére.

- Egy „gráf" gráf- és éltáblák gyűjteménye. Adatbázisonként csak egy gráfunk lehet.
- A csomóponttáblák tartalmazzák az összekapcsolni kívánt entitásokat. Minden rekordhoz automatikusan generálódik egy rejtett `$node_id`.
- Az éltáblák tartalmazzák a csomópontokat összekötő éleket. Jó gyakorlat, hogy a bizonyos csomóponttípusokat összekötő éleket külön éltáblákban tároljuk a másféléket összekötő élektől. Az élek a megengedett céltípusok tekintetében is megszoríthatók, így a hálózatnak egyfajta „gráfsémája" lehet. Egy élnek lehetnek attribútumai. Minden rekordhoz automatikusan generálódik egy rejtett `$edge_id`. Egy él mindig irányított: `$from_id`-ből `$to_id`-be.
- Az SQL-lekérdezések a `MATCH` kulcsszó segítségével kapcsolati mintákat adhatnak meg, így olyan csomópontokat vagy részgráfokat kérhetünk le, amelyek bizonyos kapcsolati tulajdonságokkal rendelkeznek. Egy minta egy részgráfot ad vissza. Két részgráf metszete a mintaspecifikációban `AND`-del választható ki.
- Bár a gráftáblákon nem kötelező az elsődleges kulcs, a lekérdezések felgyorsítására automatikusan létrejön egy egyedi, nem fürtözött index a nullable `$node_id`, `$edge_id` oszlopokon.
- A `MATCH` minták tartalmazhatnak `SHORTEST_PATH` konstrukciót, amely a csomópontok közötti legrövidebb utat adja vissza.

```sql
from person, likes, restaurant, livesin, city, locatedin
where match (person-(likes)->restaurant-(locatedin)->city and person-(livesin)->city);

-- azok a személyek, akiknek legalább egy barátja olyan éttermet kedvel, ami ugyanabban a városban van, ahol a barát él
select distinct p1.id, p1.name
from person p1, person p2, friendof, likes, restaurant, livesin, city, locatedin
where match (p1-(friendof)->p2 and p2-(likes)->restaurant-(locatedin)->city and p2-(livesin)->city);

-- 2 olyan személy megkeresése, akiknek közös barátjuk van
-- a friendof IRÁNYÍTOTT él, noha a kapcsolat irányítatlan (kölcsönös)
-- az irányítatlan kapcsolatok kifejezésére csak két él használatával van mód
select p0.name person, p1.name Friend1, p2.name Friend2
from person p1, friendof f1, person p2, friendof f2, person p0
-- where match(p1-(f1)->p0<-(f2)-p2) and p1.id <> p2.id
where match(p1-(f1)->p0 and p2-(f2)->p0) and p1.id <> p2.id; -- egyenértékű

-- SHORTEST_PATH ----------------------------------------------
-- barátok: az összes legrövidebb út Johntól
select p1.name,
    p1.name + '->' + string_agg(p2.name, '->') within group (graph path) paths,
    last_value(p2.Name) within group (graph path) last_name,
    count(p2.name) within group (graph path) depth
from person p1, person for path as p2,
    friendof for path as friend
-- where match(shortest_path(p1(-(friend)->p2) {1,2})) and p1.name='john' -- nem tartalmazza a 3 hosszú john->mary->alice->tom utat
where match(shortest_path(p1(-(friend)->p2) +)) and p1.name = 'john'; -- mindet tartalmazza

-- barátok: a legrövidebb út John és Jacob között
select name, paths, depth
from (
    select p1.name,
        p1.name + '->' + string_agg(p2.name, '->') within group (graph path) paths,
        last_value(p2.Name) within group (graph path) last_name,
        count(p2.name) within group (graph path) depth
    from person p1, person for path as p2,
        friendof for path as friend
    where match(shortest_path(p1(-(friend)->p2)+)) and p1.name = 'john'
    -- and p2.name='jacob' -- HIBA: FOR PATH táblákból nem lehet oszlopokat szelektálni
) a where last_name = 'jacob';
```

## Egy natív gráfadatbázis: neo4j

### Technikai kérdések

Konfiguráció, memória bővítése:

- Állítsd le biztonságosan a virtuális gépet
- Növeld a memóriát 8 GB-ra
- Indítsd el a gépet

Konfiguráció, jogosultságok hozzáadása:

- **1. lépés:** Menj a `c:\Program Files\Neo4j Desktop` mappához.
- **2. lépés:** Jobb klikk a Neo4j Desktopon, válaszd a Tulajdonságokat.
- **3. lépés:** Válaszd a „Security" (Biztonság) fület. Ott válaszd a USERS-t.
- **4. lépés:** Kattints az Edit (Szerkesztés) gombra, és válaszd a USERS-t.
- **5. lépés:** A Permission (Jogosultságok) résznél pipáld be az Allow Full Control mezőt.
- **6. lépés:** Kattints az Apply (Alkalmaz) gombra alul.

### Bevezetés

A Neo4j a legnépszerűbb, nyílt forráskódú, Java alapú gráfadatbázis-technológia, mely asztali és felhőalapú adatbázis-szervereket egyaránt kínál. Az adatbázisfájlok minden példányhoz külön mappában tárolódnak a `C:\Users\db\.Neo4jDesktop\relate-data\dbmss` mappán belül. Az adatbázismappán belül a perzisztált adatfájlok a `data\databases\neo4j` mappában találhatók. Az adatok láncolt listákban vannak tárolva, az alábbi adatfájlokban:

- `Neostore.nodestore.*` fájlok: a csomópontok. A csomópontok azonosítására legyen egy egyedi tulajdonság, pl. 'name'.
- `Neostore.label.*` fájlok: a csomópontokhoz címkék (labels) rendelhetők, amelyek csoportosítják őket; pl. az összes rendelést képviselő csomópont címkéje lehet „Orders". Ez hasonlít egy csomóponttípusra, de egy csomóponthoz több címke is tartozhat.
- `Neostore.relationship.*` fájlok: ezek a fájlok tárolják az éleket (másnéven kapcsolatokat).
  - Minden kapcsolatot irányítottan kell létrehoznunk, de lekérdezéskor az él iránya figyelmen kívül hagyható.
  - A kapcsolatokon indexelhetünk.
  - Egy kapcsolatnak egyetlen típusa van (= a kapcsolat egyetlen címkéje).
- `Neostore.property.*` fájlok: a csomópontokhoz és élekhez rendelt tulajdonságok (attribútumok). A tulajdonságok kulcs-érték párok. Egy élnek vagy csomópontnak több tulajdonsága is lehet.

További hivatkozások:

- https://neo4j.com/docs/getting-started/
- https://neo4j.com/developer/kb/understanding-data-on-disk/

Az attribútum típusazonosítója minden attribútumnál a tényleges adat előtt, egy attribútum-adatblokkban tárolódik. Egy adatblokk 8 bájt, az attribútumrekord pedig 4 blokkot foglal. A natív adattípusok: `bool`, `float`, `integer`, `point`, `string`, `date/time`, valamint ezek listái (tömbjei). A `PATH` adattípus csomópontok és kapcsolatok váltakozó sorozata. Bármely típus felveheti a `NULL` értéket. A gráfnak „opcionális" sémája van, azaz minden csomópontnak és élnek lehetnek eltérő tulajdonságai, de megszorításokkal rákényszeríthetünk egy tárgyterületi modellt.

## A Cypher lekérdezőnyelv

Az adatbázist a Cypher lekérdezőnyelvben kezeljük. Az alábbi példák egy Movies nevű demó adatbázist használnak.

További hivatkozás: https://neo4j.com/docs/cypher-manual/current/introduction/

A Cypher mintákra épül.

- Példa csomópontmintára: `(:Movie {title: 'Forrest Gump', released: 1994})`
- Példa kapcsolatmintára: `-[:ACTED_IN {roles: ['Forrest']}]->`

Néhány szabály:

- A kettőspont előtt megadható egy alias (referenciaváltozó) a csomópont- vagy kapcsolatspecifikációban, pl. `"p:Person"`.
- Irányítatlan kapcsolatot használó lekérdezésben a nyílnak nincs hegye.
- A kapcsolattípusok és tulajdonságértékek kisbetű-nagybetű érzékenyek, bár maga a Cypher nyelv nem az.
- A minták kombinálhatók lekérdezésekben.

Néhány példa:

```cypher
// minden csomópont, aminek Movie címkéje van
MATCH (films:Movie) RETURN films.title

// azok a filmek, amelyekben T.H. játszott
MATCH (actor:Person)-[:ACTED_IN]-(film:Movie)
WHERE actor.name = 'Tom Hanks'
RETURN actor.name, film.title;

// mint fent, a rendezővel együtt
MATCH (actor:Person)-[:ACTED_IN]-(film:Movie),
      (director:Person)-[:DIRECTED]-(film:Movie)
WHERE actor.name = 'Tom Hanks'
RETURN actor.name, film.title, director.name;
```

A `RETURN` egy fő utasítás. Tipikus fő utasítások:

- `RETURN`: megkeresi és visszaadja az illeszkedő részgráfot
- `DELETE`: csomópontok, kapcsolatok vagy utak törlése
- `REMOVE`: tulajdonságok eltávolítása csomópontokról és kapcsolatokról, címkék eltávolítása csomópontokról
- `SET`: címkék frissítése csomópontokon, és tulajdonságok frissítése csomópontokon és kapcsolatokon
- `CREATE`: csomópontok és kapcsolatok hozzáadása
- `MERGE`: megpróbál illeszteni egy mintát (amely tartalmazhat tulajdonságértékeket is); ha nem talál pontos egyezést, létrehozza a mintát. Kombinálható `RETURN`-nel.

`MERGE` és `CREATE` példák:

```cypher
CREATE (m:Movie {name: 'old movie'}) RETURN *;
// visszaadja az új csomópontot

MERGE (m:Movie {name: 'old movie'}) RETURN *;
// nem jön létre új csomópont, mert pontos egyezés volt

CREATE (m:Movie {name: 'old movie'}) RETURN *;
// most két 'old movie' csomópontunk van

MATCH (m:Movie {name: 'old movie'}) DELETE m;
// mindkettőt töröltük
```

Az alábbi példában létrehozunk egy új Persont, és egy új `WROTE` kapcsolattal kötjük össze egy meglévő filmmel:

```cypher
MERGE (p:Person {name: 'Henry Hukk'}) RETURN p;

MATCH (p:Person {name: 'Henry Hukk'}), (m:Movie {title: 'Top Gun'})
MERGE (p)-[:WROTE]->(m)
RETURN p, m;
```

Egy `CREATE` példa:

```cypher
CREATE (:Person:Actor {name: 'Tom Hanks', born: 1956})   // ez egy 2 címkés csomópont
       -[:ACTED_IN {roles: ['Forrest']}]->               // a szerepek egy tömbben vannak
       (:Movie {title: 'Forrest Gump', released: 1994})
       <-[:DIRECTED]-                                    // figyeld a nyíl irányát
       (:Person {name: 'Robert Zemeckis', born: 1951})
```

Így állítunk be vagy hozunk létre új tulajdonságot egy csomóponton:

```cypher
MATCH (p:Person {name: 'Paul Blythe'})
SET p.birthdate = date({year: 2018})   // felülírja, ha már létezett
RETURN p
```

Példa tulajdonságérték-alapú szűrésre (hasonló értékelések keresése):

```cypher
MATCH (p1:Person)-[r1:REVIEWED]->(m1:Movie),
      (p2:Person)-[r2:REVIEWED]->(m2:Movie)
WHERE r1.rating = r2.rating
RETURN p1.name, r1.rating, m1.title, p2.name, r2.rating, m2.title;
```

Legrövidebb út keresése (lásd https://neo4j.com/docs/cypher-manual/current/patterns/shortest-paths/):

```cypher
MATCH p = shortestPath((p1:Person)-[:FOLLOWS*]-(p2:Person))
// p egy Path. Figyeld meg a *-ot
WHERE p1.name = 'James Thompson' AND p2.name = 'Paul Blythe'
RETURN [n IN nodes(p) | n.name] AS stops
// visszaadja: ["James Thompson", "Jessica Thompson", "Angela Scope", "Paul Blythe"]
```

A gráf átmérője irányítatlan `FOLLOWS` dimenzió mentén:

```cypher
MATCH (p1:Person), (p2:Person)
WHERE id(p1) <> id(p2)   // a kezdő és a végpont nem lehet ugyanaz
MATCH p = shortestPath((p1:Person)-[:FOLLOWS*]-(p2:Person))
RETURN p, length(p) AS length
ORDER BY length DESC LIMIT 1
```

## Adatok importálása Neo4j-be

A legegyszerűbb mód arra, hogy egy gráfot SQL Serverről Neo4j-be migráljunk, a CSV-fájlokon keresztüli út. Ez egy példa a központi Northwind táblák gráfváltozatára.

1. Hozz létre megfelelő nézeteket, amelyek csak a gráfhoz szükséges adatokat tartalmazzák:

```sql
use northwind

create view vi_products as
select productid, productname from products;
go

create view vi_orders as
select orderid, cast(orderdate as date) odate from orders;
go

create view vi_orders_products as
select o.orderid, p.productid, od.Quantity quantity,
    od.Quantity * od.UnitPrice * (1 - od.Discount) price
from orders o join [Order Details] od on o.orderid = od.OrderID
    join products p on od.ProductID = p.productid;
go
```

2. Exportáld a nézeteket CSV-fájlokba.

3. Másold a fájlokat a Neo4j adatbázis import mappájába: `C:\Users\db\.Neo4jDesktop\relate-data\dbmss\[DB id]\import`.

4. Töltsd be a rendeléseket és termékeket:

```cypher
LOAD CSV WITH HEADERS FROM 'file:///products.csv' AS line
CREATE (p:product {id: line.productid, name: line.productname})
RETURN p.id, p.name;

LOAD CSV WITH HEADERS FROM 'file:///orders.csv' AS line
CREATE (o:orders {id: line.orderid, odate: date(line.odate)})
RETURN o.id, o.odate;

LOAD CSV WITH HEADERS FROM 'file:///order_products.csv' AS line
MATCH (p:product {id: line.productid}), (o:orders {id: line.orderid})
MERGE (o)-[c:CONTAINS {quantity: toInteger(line.quantity), price: toFloat(line.price)}]->(p)
RETURN c;
```

## Képek és BLOB-ok tárolása

- Mérsékelt méretű egyedi képekhez (mint például egy webshopban található áru képei) a `VARBINARY(MAX)` adattípus ajánlott:

```sql
insert my_table(image_column)
select * from openrowset(bulk 'c:\my_path\my_photo.png', single_blob);
```

- Nagy méretű videó- és egyéb BLOB-fájlokhoz az SQL Serveren a FileTables technológia ajánlott. A fájltáblák mind az adatbázisból, mind más alkalmazásokból elérhetők, ugyanúgy, mint a fájlrendszerben tárolt normál fájlok.

További hivatkozás: https://docs.microsoft.com/en-us/sql/relational-databases/blob/load-files-into-filetables?view=sql-server-2017
