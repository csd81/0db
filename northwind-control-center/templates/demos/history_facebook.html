The News Feed Query That Melted MySQL

What loading ONE person's News Feed required:
  1. Find User A's friends: SELECT friend_id FROM friendships WHERE user_id = A (could be 500 rows)
  2. Find all posts by those 500 friends in last 24h: SELECT * FROM posts WHERE user_id IN (500 IDs) AND created_at > NOW()-1DAY — a massive IN clause
  3. Filter by privacy: for each post, check if User A is in the allowed audience list
  4. Check engagement: did User A already like each post? Another JOIN on the likes table
  5. Rank by relevance: apply EdgeRank scoring algorithm across all candidates
  6. Paginate: return top 25
This could require 5-10 SQL JOINs across tables with billions of rows. Repeated for 300M users simultaneously.
Why JOINs Destroy at Scale
A SQL JOIN is O(N×M) for naive implementations. A JOIN between a 300M-row friends table and a 10B-row posts table, filtered and re-JOINed for privacy — requires the database to touch potentially trillions of row combinations. Database query optimizers help but have hard limits.
The "Static Profile" vs "Dynamic Feed" Shift
MySpace era: you visited a user's profile page (one simple query: SELECT * FROM users WHERE id=?). Facebook's News Feed: one page load = queries about 500 people's activities, each with complex social graph traversal. The read patterns changed fundamentally, but the database didn't.
Phase 1: Memcached at Absurd Scale (2008)

Basic Memcached: A simple in-memory key-value store. set("user:12345", serialized_user_object, ttl=300). Application checks cache first; on miss, queries database and populates cache. Created in 2003 by Brad Fitzpatrick for LiveJournal to reduce database load.

What Facebook did to it: They scaled Memcached to thousands of servers, modified the protocol, built a custom client library (McRouter), and developed techniques for handling cache invalidation at unprecedented scale. Facebook became Memcached's de facto steward, contributing more improvements than anyone else.

The "look-aside cache" pattern: MySQL stores the source of truth. Memcached stores serialized copies of hot data. Application code checks cache → if miss, read from MySQL, write to cache. 99% of requests served from RAM. MySQL only handles 1% of traffic (writes + cache misses).

The stampede problem: When a popular cache key expires, thousands of requests simultaneously miss and all try to query MySQL at once — a "thundering herd." Facebook implemented "lease" tokens: on a miss, the first client gets a lease (a token saying "I'm rebuilding this"), other clients get told to wait briefly. Only one MySQL query happens.

# The Facebook look-aside cache pattern
def get_news_feed(user_id):
    cache_key = f"feed:{user_id}"

    # Try Memcached first (RAM, ~0.1ms)
    feed = memcached.get(cache_key)
    if feed:
        return deserialize(feed)  # 99% path

    # Cache miss — request lease token
    lease_token = memcached.get_lease(cache_key)
    if not lease_token:
        # Another request is rebuilding it
        time.sleep(0.01)
        return get_news_feed(user_id)  # retry

    # 1% path: query MySQL (10-50ms)
    friends = mysql.query(
        "SELECT friend_id FROM friendships WHERE user_id=%s",
        user_id
    )
    posts = mysql.query(
        "SELECT * FROM posts WHERE user_id IN %s "
        "AND created_at > NOW()-INTERVAL 1 DAY",
        friends
    )
    feed = rank_and_filter(posts, user_id)

    # Populate cache, release lease
    memcached.set(cache_key, serialize(feed),
                  ttl=60, lease=lease_token)
    return feed
Phase 2: TAO — The Associations and Objects (2013)
The Core Insight
Facebook's entire social graph can be modeled as just two primitive types:
OBJECT (node)
A typed entity: User, Post, Comment, Photo, Page, Event. Has an ID and a property map.
obj_get(type="User", id=12345) → {name, birthday, ...}
ASSOCIATION (edge)
A typed relationship between two objects, with an optional timestamp and data payload.
assoc_get(id1=12345, type=LIKES, id2=post_67890)
assoc_range(id1=12345, type=FOLLOWS, 0, 100)
Every social action becomes a simple TAO call. News Feed assembly queries TAO's RAM, not MySQL. TAO handles caching, invalidation, and replication internally. MySQL is TAO's durable backing store — written to asynchronously, queried only by TAO internals.

🏗️ TAO's Two-Tier Architecture: Leaders and Followers
Leader Tier (one per region)
A small cluster of TAO servers that own the authoritative cached state for a shard of the graph. All writes go to the Leader. Leader talks to MySQL for persistence. Leader fan-outs invalidations to all Followers. Each Leader handles ~1M objects in memory.
Follower Tier (many per region)
Dozens of Follower servers per region handle read traffic. On a cache miss, Followers query their regional Leader (not MySQL). Followers serve >95% of all TAO requests. A Follower failure is invisible — request routes to another Follower. Geographic replication: each datacenter has its own Leader+Follower cluster, replicating asynchronously.
Invalidation: When a write occurs (user likes a post), the Leader updates MySQL and invalidates the affected cache keys in all Followers. Followers receive invalidate(key) messages and delete stale entries. Next read misses cache → fetches fresh from Leader. Invalidation is fire-and-forget — it's OK if a Follower briefly serves stale data (a few milliseconds) before the invalidation arrives.

The Lasting Verdict

Facebook published the TAO paper at USENIX ATC 2013. The architectural principle — model social data as a graph of objects and associations, cache it entirely in RAM, and use a database only for durable persistence — became the blueprint for social network engineering. LinkedIn built a similar system (Espresso). Twitter's "flock" social graph service used comparable ideas. Memcached's lease pattern, thundering herd prevention, and invalidation-at-scale techniques are now standard knowledge in backend engineering. Facebook also donated Apache Cassandra (a distributed column-store for message storage) from this same era, which became one of the most deployed databases in history.

🔢 The 99%/1% Rule
Social platforms are 99% reads (loading feeds, profiles, photos) and 1% writes (posts, likes, comments). Architecture must be optimized for the 99% case. Putting every read through a write-optimized relational DB (which prioritizes consistency) is backwards. TAO inverts this: reads always hit RAM, writes are handled asynchronously by the slower but durable MySQL layer.
🌐 Why Not a Graph Database?
Neo4j and similar graph databases exist specifically for this use case. Facebook evaluated them — at their scale, no off-the-shelf graph DB could handle 1B requests/second. TAO is purpose-built for Facebook's specific access patterns: point lookups by ID, range queries over associations sorted by time. A general-purpose graph DB optimizes for graph traversals (shortest path) that Facebook rarely needs.
