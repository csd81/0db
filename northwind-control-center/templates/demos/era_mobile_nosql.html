⚠ The Breaking Point: ACID Meets Mobile Reality

Relational databases are designed for structured, predictable data. A mobile social app generates heterogeneous, unpredictable, user-shaped data — a tweet has different fields than a photo post, a check-in, or a product review. Forcing all of it into rigid tables required massive up-front schema design and constant ALTER TABLE migrations.

Schema Rigidity

Adding a new field to a 100M-row MySQL table required a full table rewrite — sometimes hours of downtime. Startup iteration velocity demanded schema changes daily. Relational databases made this operationally painful.

Traffic Unpredictability

A viral tweet at 2am could spike writes 1000× in seconds. MySQL master-slave replication lagged under write storms. The master became a single point of failure and a write bottleneck simultaneously.

The CAP Theorem Becomes Mainstream

Brewer's CAP theorem (2000, formalized 2002) stated: a distributed database can only guarantee two of three properties simultaneously. The NoSQL movement forced every team to consciously choose.

C
Consistency

Every read gets the most recent write or an error. No stale data.

A
Availability

Every request receives a response. No errors, even if the data might be slightly stale.

P
Partition Tolerance

The system continues operating even when network partitions split nodes. Required for any real distributed system.

The Practical Choice: Since network partitions are inevitable in distributed systems, P is non-negotiable. The real choice is C vs A. MongoDB chose CP (strong consistency, may reject writes if no quorum). Cassandra chose AP (always accepts writes, eventual consistency). Neither is wrong — they fit different workloads.

✦ MongoDB: Abandon the Schema, Ship Faster

MongoDB stored data as BSON documents (binary JSON) with no required schema. A product document could have 3 fields or 300 fields. No migration needed. Developers could iterate on data models without DBA approval or ALTER TABLE downtime.

// MongoDB: schemaless document — different products, different shapes
// No CREATE TABLE, no ALTER TABLE, no migration scripts

db.products.insertMany([
  { // Electronics product
    name: "iPhone 6", price: 649, category: "phone",
    specs: { ram_gb: 1, storage_gb: 64, display_inch: 4.7, nfc: true },
    images: ["/img/iphone6-front.jpg", "/img/iphone6-back.jpg"]
  },
  { // Book — completely different shape, same collection
    name: "Clean Code", price: 35, category: "book",
    author: "Robert C. Martin", isbn: "978-0132350884", pages: 464
  }
]);

// Query with rich indexing — no joins needed for embedded documents
db.products.find({ "specs.nfc": true, price: { $lt: 700 } })
           .sort({ price: -1 })
           .limit(10);

✦ Cassandra: Always Write, Never Block

Apache Cassandra (born at Facebook for the Inbox feature, 2008) applied Amazon Dynamo's consistent hashing to a wide-column store. Every node is equal — no master. Writes are accepted by any node in any data center. Data propagates asynchronously. A like count might be "2 seconds stale" on a replica — that's an acceptable trade-off for 100% write availability across 5 global data centers.

-- Cassandra: wide-column model optimized for time-series social data
-- Primary key design is everything — determines partition + sort order

CREATE TABLE user_activity (
    user_id   UUID,
    event_ts  TIMESTAMP,
    action    TEXT,
    target_id UUID,
    metadata  MAP<TEXT, TEXT>,
    PRIMARY KEY (user_id, event_ts)       -- partition by user, cluster by time
) WITH CLUSTERING ORDER BY (event_ts DESC)
  AND default_time_to_live = 7776000;   -- auto-expire after 90 days

-- Write: goes to any node (coordinator), replicated to N replicas async
-- Consistency level QUORUM: wait for majority (2 of 3 replicas) to confirm
-- Consistency level ONE: return immediately after 1 replica write
-- Cassandra never rejects a write due to node failure — hinted handoff queues it
Era 2 Timeline
Jan 2007 — Apple launches iPhone. Mobile web traffic begins exponential growth. 24/7 write patterns emerge.
2007 — Amazon publishes the Dynamo paper. Consistent hashing + eventual consistency enters mainstream vocabulary.
2008 — Facebook open-sources Cassandra. NoSQL's CP vs AP dichotomy becomes a daily engineering decision.
2009 — MongoDB 1.0 released. "Schemaless" becomes a selling point rather than a red flag.
2010 — Twitter migrates from MySQL to Cassandra for tweet storage. 100M+ users, 340M tweets/day.
2012 — Netflix migrates to Cassandra for all streaming metadata. "NoSQL at Netflix scale" becomes a conference circuit staple.

The Verdict

Era 2 didn't kill SQL — it forced engineers to choose consciously. The NoSQL movement's lasting contribution wasn't MongoDB or Cassandra themselves, but the realization that there is no universal data model. ACID is the right answer for banking. Eventual consistency is the right answer for like counts. The question "what are your consistency requirements?" became a first-class design concern.
