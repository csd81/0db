Why SQL LIKE Falls Short
-- Find products containing "chai"
SELECT ProductName FROM Products
WHERE ProductName LIKE '%chai%';
LimitationExplanation
No relevance rankingAll matching rows are equally "good" — no score
No index useLIKE '%chai%' cannot use B-tree — scans every row
No tokenization'chai tea' ≠ 'tea chai' without extra wildcards
No fuzziness'Chaai' (typo) returns nothing
Language-unawarePlurals, stemming, synonyms need manual workarounds

Analyzer Pipeline

When a document is indexed, Elasticsearch runs the field value through an analyzer pipeline:

"Grandma's Boysenberry Spread"
↓ tokenizer (standard)
["Grandma's", "Boysenberry", "Spread"]
↓ token filters (lowercase, stop words, stemmer)
["grandma", "boysenberri", "spread"]
↓ inverted index
grandma    → [doc 7]
boysenberri → [doc 7]
spread     → [doc 7, doc 12, doc 31]

At query time, the same analyzer runs on the query text. Engine looks up matching terms in the inverted index in O(1) — no table scan required.

BM25 Scoring Formula
score(t, d) = IDF(t) × (tf(t,d) × (k1 + 1)) / (tf(t,d) + k1 × (1 − b + b × |d|/avgdl))

Practical Implications
Example: Why "Chai" Outranks a Description
Product name = "Chai" → high IDF × short doc → high score
Product description mentioning "chai" 5 times in a long paragraph → lower score due to length normalization + TF saturation
Rules of Thumb
  • High IDF = rare term = more discriminating = higher weight
  • TF saturation: 1 occurrence ≫ 0, but 10 occurrences ≈ 3
  • Short documents with exact matches win over long ones with many mentions
  • Adding more search terms narrows results (all terms must match with operator: and)
BM25 has been the industry standard for full-text ranking since the 1990s — used in Lucene, Solr, Elasticsearch, and many other systems.
Aggregations

ES aggregations run in parallel with search — no second query needed. Two main types:

Bucket Aggregations (≈ GROUP BY)
{"aggs": {
  "price_ranges": {
    "range": {
      "field": "unit_price",
      "ranges": [
        {"key": "$0–$10",  "from": 0,  "to": 10},
        {"key": "$10–$25", "from": 10, "to": 25},
        {"key": "$25+",    "from": 25}
      ]
    }
  }
}}
Metric + Nested Aggregation (≈ AVG per GROUP)
{"aggs": {
  "categories": {
    "terms": {"field": "category_name.keyword"},
    "aggs": {
      "avg_price": {"avg": {"field": "unit_price"}}
    }
  }
}}

Dual-Database Architecture Pattern
Client App
│
├─ WRITE → SQL Server / PostgreSQL (authoritative)
│     └─ trigger / message queue event
│         ↓
│         Elasticsearch index (read-optimized projection)
│
└─ SEARCH → Elasticsearch (full-text, fuzzy, relevance)
  • SQL owns authoritative data and ACID transactions
  • ES owns the search projection — a denormalized read view
  • Sync via trigger, CDC, or message queue (e.g. Debezium → Kafka → ES)
  • If ES is temporarily down: fall back to SQL LIKE (degraded UX, not data loss)
Used by e-commerce platforms, CMS systems, and SaaS applications at scale worldwide.
