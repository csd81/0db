tr_demo_bad — what not to do
CREATE TRIGGER tr_demo_bad ON Orders FOR INSERT AS DECLARE @orderid int SELECT @orderid = OrderID FROM inserted PRINT 'New order ID: ' + CAST(@orderid AS varchar) WAITFOR DELAY '00:00:10' -- 10 seconds of blocking! SELECT 1/0 -- divide-by-zero error
Why this kills the trading system:
The trigger runs inside the same transaction as the INSERT Orders. All table locks are held for the full 10 seconds. If the trigger raises an error, SQL Server automatically rolls back the entire INSERT — the order disappears.
Three tests from the chapter:
Test 1 — no WAITFOR, no error → ORDER saved instantly ✓
Test 2 — WAITFOR only → ORDER saved, but 10s delay ✗
Test 3 — WAITFOR + error → ORDER lost, system crashes ✗✗
What happens (Test 3)
T+0
Trading App: INSERT INTO Orders (CustomerID='AROUT', …)
T+0
tr_demo_bad fires — runs inside the INSERT transaction. Table row is locked.
T+0
WAITFOR DELAY '00:00:10' — trigger sleeps 10 seconds. The INSERT is blocked. All locks held. Any concurrent SELECT on Orders is blocked too.
T+10
SELECT 1/0 → Msg 8134: Divide by zero error. Trigger raises exception.
T+10
ROLLBACK — the INSERT Orders statement is undone. The order does not exist.
T+10
Trading App receives the error. The operator sees a crash. The order must be re-entered manually — or is lost entirely.

Tight coupling occurs when a transaction synchronously calls an external service — if that service is slow or unavailable, the INSERT blocks or rolls back, cascading the failure to every caller up the stack. The loose coupling pattern replaces the direct call with an append to a durable event log: the INSERT completes immediately, and a separate process consumes the log at its own pace, retrying failures without ever touching the original transaction. The demo implements both patterns using a SQL Server AFTER trigger writing to order_log and a SQL Agent job processing it asynchronously, then deliberately breaks the downstream service to show that the tight version fails the INSERT while the loose version never notices. This event-log pattern — write durably, process independently — is the design principle behind Kafka, SQS, and every resilient distributed system.

sp_order_process — the job's stored procedure
CREATE PROC sp_order_process AS DECLARE @event_id int, @event_type varchar(50), @order_id int, @result int DECLARE cursor_events CURSOR FORWARD_ONLY STATIC FOR SELECT event_id, event_type, order_id FROM order_log WHERE status=0 -- only unprocessed events SET XACT_ABORT ON; SET NOCOUNT ON OPEN cursor_events FETCH NEXT FROM cursor_events INTO @event_id, @event_type, @order_id WHILE @@fetch_status = 0 BEGIN UPDATE order_log SET time_process_begin=GETDATE() WHERE event_id=@event_id BEGIN TRAN IF @event_type = 'new order' EXEC sp_commit_new_order_to_inventory @order_id, @result OUTPUT ELSE IF @event_type = 'address changed' ... -- simulated IF @result=0 COMMIT TRAN ELSE ROLLBACK TRAN UPDATE order_log SET time_process_end=GETDATE(), status=CASE WHEN @result=0 THEN 2 ELSE 1 END WHERE event_id=@event_id FETCH NEXT FROM cursor_events INTO @event_id, @event_type, @order_id END CLOSE cursor_events; DEALLOCATE cursor_events
Design notes: The cursor is necessary here because completely different actions are needed per event type. Each event gets its own BEGIN TRAN / COMMIT|ROLLBACK. Failed events get status=1 and can be retried by resetting them to status=0. The FORWARD_ONLY STATIC cursor takes a snapshot so new events added during processing are not included in this run.
