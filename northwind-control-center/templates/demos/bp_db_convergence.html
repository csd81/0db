⚠ The Split-Brain Problem

In late 2022 and 2023, engineering teams scrambled to add AI to their products. The "obvious" move was to add a standalone vector database (Pinecone, Weaviate, Qdrant) alongside the existing PostgreSQL. This created a dangerous dual-write architecture:

→ row deleted from DB
OUT OF
SYNC ⚠
Pinecone (Vector DB)
Customer embedding still
exists → AI still "knows" them
GDPR Violation

A user exercises their "right to be forgotten" and deletes their account. The SQL row is deleted. But the vector embedding — which encodes their profile, preferences, and behavior — remains in Pinecone indefinitely. This is a textbook GDPR Article 17 violation.

Consistency Nightmare

Price changes, product updates, and content moderation actions in SQL aren't reflected in the vector DB until the next sync job runs. AI searches return results based on stale data. A product pulled for safety violations still appears in AI recommendations.

✦ pgvector: Move AI to the Data, Not Data to AI

The pgvector PostgreSQL extension adds a native vector column type and HNSW/IVFFlat indexes. Vector similarity search runs inside the same SQL transaction as your business logic — no separate database, no sync job, no consistency gap. DELETE CASCADE on a user row deletes their embedding atomically.

-- pgvector: AI-native Postgres — no separate vector database needed

-- 1. Enable extension and add vector column to existing table
CREATE EXTENSION IF NOT EXISTS vector;
ALTER TABLE products ADD COLUMN embedding vector(1536);  -- OpenAI ada-002 dim
CREATE INDEX ON products USING hnsw (embedding vector_cosine_ops);

-- 2. Hybrid query: traditional SQL filters + vector semantic search IN ONE TRANSACTION
SELECT
    p.product_id,
    p.product_name,
    p.unit_price,
    p.units_in_stock,
    1 - (p.embedding <=> $1) AS semantic_similarity
FROM products p
WHERE
    p.discontinued = 0                  -- hard filter (relational)
    AND p.unit_price BETWEEN 10 AND 100  -- price range filter (relational)
    AND p.units_in_stock > 0            -- stock check (relational)
ORDER BY p.embedding <=> $1            -- semantic similarity (vector ANN)
LIMIT 10;
-- $1 = OpenAI embedding of user's search query: "light refreshing summer drink"
-- Returns products semantically similar to "summer drink" + in stock + right price
-- ALL in one query. Zero external API. Full ACID. No sync lag.

-- 3. DELETE is fully consistent — no orphaned embeddings
DELETE FROM products WHERE product_id = 77;  -- embedding deleted atomically

✦ The Lakehouse: ACID Transactions on a Petabyte Data Lake

The decade-long war between Data Warehouses (structured, ACID, expensive) and Data Lakes (unstructured, cheap, no guarantees) ended with the invention of open table formats — Apache Iceberg and Delta Lake — that apply a metadata + transaction log layer on top of ordinary Parquet files in S3, giving them full ACID properties.

-- Apache Iceberg: ACID SQL on a data lake with PDFs, images, raw logs
-- The table lives in S3 as Parquet files + Iceberg metadata JSON
-- Any engine (Spark, Trino, Flink, DuckDB, Snowflake) reads the same format

-- MERGE: upsert 10M rows into a 500TB Iceberg table — fully ACID
MERGE INTO lakehouse.orders AS target
USING staging.order_updates AS source
   ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET
    target.status       = source.status,
    target.shipped_date = source.shipped_date
WHEN NOT MATCHED THEN INSERT (order_id, customer_id, total, status)
    VALUES (source.order_id, source.customer_id, source.total, source.status);
-- Concurrent writers? Iceberg uses optimistic concurrency + snapshot isolation
-- Time Travel: query as-of any past snapshot for free (metadata pointer only)

-- Schema evolution without rewrite: add a column to a 500TB table in milliseconds
ALTER TABLE lakehouse.orders ADD COLUMN ai_category VARCHAR;
-- Old Parquet files unchanged — Iceberg schema evolution tracks this in metadata
Why Iceberg Won: Apache Iceberg is engine-agnostic. The same table can be queried by Spark (batch training), Flink (streaming), Trino (ad-hoc SQL), DuckDB (local analysis), and Snowflake/BigQuery (BI) simultaneously — all reading the same Parquet files in S3 with full consistency. Delta Lake (Databricks) solves the same problem but requires the Delta Lake client library. Iceberg's open spec means no vendor lock-in.

The Verdict

The database convergence era proved that standalone vector databases were a transitional technology — necessary when no existing database supported vectors, quickly superseded once they did. The deeper shift was the Lakehouse: finally unifying structured and unstructured data under ACID guarantees and an open format readable by any engine. The question "where should I store my data?" has been replaced by "what engine should I query this universal table with?"
