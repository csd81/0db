dbt Model: Raw → Staging → Mart (SQL Transformation Layers)
Layer 1: stg_orders.sql (Staging)
SELECT id AS order_id, created_at AS ordered_at, customer_id, status, amount / 100 AS amount_usd, LOWER(currency) AS currency FROM {{ '{{' }} source('postgres', 'raw_orders') {{ '}}' }} WHERE created_at >= '2020-01-01'
Layer 2: fct_orders.sql (Fact Table)
SELECT o.order_id, o.ordered_at, o.customer_id, c.country, o.amount_usd, DATEDIFF(day,o.ordered_at, s.shipped_at) AS days_to_ship FROM {{ '{{' }} ref('stg_orders') {{ '}}' }} o LEFT JOIN {{ '{{' }} ref('stg_customers') {{ '}}' }} c ON o.customer_id = c.customer_id LEFT JOIN {{ '{{' }} ref('stg_shipments') {{ '}}' }} s ON o.order_id = s.order_id
Layer 3: revenue_by_country.sql (Mart)
SELECT DATE_TRUNC('month', ordered_at) AS month, country, SUM(amount_usd) AS revenue, COUNT(*) AS order_count, AVG(amount_usd) AS avg_order_value, AVG(days_to_ship) AS avg_ship_days FROM {{ '{{' }} ref('fct_orders') {{ '}}' }} WHERE status = 'completed' GROUP BY 1, 2
dbt compiles these SQL files into a DAG (Directed Acyclic Graph). Dependencies resolved via {{ ref() }} macro. Runs in topological order. dbt test validates: not-null, unique, referential integrity, custom assertions. dbt docs generate creates a searchable lineage graph of all models, tests, and source systems.
Open Table Formats: Delta Lake vs Iceberg vs Hudi
Apache Iceberg
Netflix, Apple, LinkedIn
Metadata tree: snapshot → manifest list → manifest files → data files. ACID transactions via snapshot isolation. Hidden partitioning: query optimizer uses partition stats without exposing partition columns to SQL users. Schema evolution: add/rename/drop columns without rewriting data. Time travel: SELECT * FROM orders FOR SYSTEM_TIME AS OF '2025-01-01'. Engine-agnostic (Spark, Trino, Flink, Dremio).
Delta Lake
Databricks, Uber, Shopify
Transaction log as JSON file sequence (_delta_log). ACID via optimistic concurrency on log files. MERGE INTO for upserts (GDPR right-to-be-forgotten). Z-ordering: co-locate related data (by user_id) to skip entire files during scan. Vacuum: remove old file versions. Auto-optimize: automatically compacts small files. Native Spark integration — the Databricks default format.
Apache Hudi
Uber, Amazon, ByteDance
Record-level upserts — designed for streaming CDC (Change Data Capture). COW (Copy-on-Write): rewrites Parquet files on update (faster reads). MOR (Merge-on-Read): appends delta logs, merges at read time (faster writes). Incremental queries: readType=incremental, beginInstantTime=... — replicate only changes since last read. Used by Uber for their streaming ELT pipeline (100M+ events/day).
Apache Parquet
Universal columnar storage
Columnar: all values for column X stored together → Predicate pushdown: read only columns needed. Row group statistics (min/max) → skip entire row groups (data skipping). Nested types: struct, list, map. Encoding: dictionary, RLE, bit-packing. Compression: Snappy, Gzip, Zstd. Standard format across Spark, Pandas, Arrow, BigQuery, Athena — interoperability without conversion.
Snowflake
Cloud-only data warehouse
Compute/storage separation: Virtual Warehouses (clusters) auto-suspend when idle. Micro-partitions: automatic, 50–500 MB compressed. Automatic clustering: Snowflake rewrites partitions to optimize for common filter columns. Zero-copy cloning: instant database/table clone for dev environments (shares underlying storage). Time Travel: 90-day data history. Multi-cluster for concurrency.
Google BigQuery
Serverless columnar warehouse
Dremel engine: petabyte queries in seconds. Slot-based compute: each query gets allocated "slots" (virtual CPUs) automatically. Nested repeated fields (STRUCT, ARRAY) without joins. BI Engine: in-memory analysis for sub-second queries on cached results. Storage API for Arrow-based reads. BigQuery ML: train models with SQL. Standard SQL + JSON + ML all in one query language.
🗄️ Storage Layers
Data Warehouses (OLAP)
Snowflake, BigQuery, Amazon Redshift, ClickHouse. Columnar storage, MPP (Massively Parallel Processing) execution. Optimized for aggregations over billions of rows. NOT for OLTP (too slow for single-row lookups). The serving layer for analytics workloads.
Data Lakes (Raw Storage)
AWS S3, Azure Data Lake Storage, Google Cloud Storage. Object storage — infinite scale, cheap (~$23/TB/month). Stores raw files: CSV, JSON, Parquet, Avro. Accessed by Spark, Athena, Presto, Databricks. "Schema on read" vs warehouse's "schema on write".
Lakehouse (Best of Both)
Delta Lake / Iceberg / Hudi on top of object storage → ACID transactions, time travel, schema enforcement on cheap lake storage. Databricks Delta Lakehouse: eliminates the need for a separate warehouse for many workloads. Airbnb's Data Platform moved entirely to Lakehouse architecture in 2022.
Feature Store (ML)
Feast, Tecton: bridges analytics (batch features computed by Spark) and ML serving (real-time feature lookup). Historical features for training + live features for inference via same API. Uber's Michelangelo platform: feature computation, storage, and serving at 1M+ predictions/second.
⚙️ Compute & Processing
Apache Spark
Distributed in-memory computation. RDDs → DataFrames → Datasets API. Catalyst optimizer rewrites SQL into efficient physical plans. Adaptive Query Execution (AQE): re-optimizes at runtime based on data statistics. PySpark for data scientists, Scala for performance. Processes petabytes across 1000s of nodes. Uber processes 15 PB/day with Spark.
dbt (Data Build Tool)
SQL-first transformation framework. Models = SELECT statements. ref() and source() macros build a DAG. dbt run executes models in dependency order. dbt test validates data quality. dbt docs generate builds lineage documentation. Adopted by 30,000+ companies. Democratizes data transformation — data analysts can write SQL, not Python jobs.
Fivetran / Airbyte
Managed ELT connectors. 500+ pre-built connectors (Salesforce, MySQL, Stripe, Google Ads). CDC (Change Data Capture) via database replication slots or log-based capture. Incremental syncs: only pull rows changed since last sync. Normalization option: auto-creates staging tables. Fivetran charges per Monthly Active Row (MAR) — rows that changed.
Apache Airflow
Pipeline orchestration. DAGs defined in Python: dependency graph of tasks (operators). Scheduled or triggered. Backfilling: re-run historical date ranges. Sensor operators: wait for external conditions (file arrived in S3, upstream DAG completed). Retry with exponential backoff. Airbnb originated Airflow in 2014; now Apache project with 15M+ downloads/month.
🌐 Query Engines & BI
Presto / Trino
Distributed SQL query engine — queries data in-place across multiple sources (S3, Cassandra, MySQL, Redis) without ETL. Facebook originated Presto; Trino is the community fork. Workers pull data from connectors in parallel. Used by Meta, Netflix, Airbnb for ad-hoc exploration without moving data to a warehouse.
Tableau / Looker / PowerBI
BI visualization layer. Looker: defines a "LookML" semantic layer — business logic defined once, used everywhere. Connects to warehouse via JDBC. Live queries vs extracts. Looker Studio (free Google version). PowerBI: Microsoft ecosystem, DAX formula language. Tableau: drag-and-drop, strong visualization, VizQL query language generated from UI interactions.
Metabase / Superset (OSS)
Open-source alternatives. Apache Superset: Python-based, connects to any SQL source, Druid for real-time, Snowflake for batch. Used by Airbnb (who donated it to Apache). Metabase: simpler no-code interface, good for non-technical users. Both free, self-hosted. Superset processes billions of rows via pushdown to warehouse.
ClickHouse (OLAP at sub-second)
Column-oriented DBMS from Yandex. Vectorized query execution (SIMD). 10B rows/second scan throughput on single node. Used for real-time analytics dashboards (sub-100ms query on 1B rows). Cloudflare uses ClickHouse for their analytics (20T rows). Alternative to full Snowflake for real-time OLAP use cases.
Workflow Deep Dive
1
ELT: Extract from Source Systems
Fivetran connects to production PostgreSQL via logical replication slot (read WAL log). Change Data Capture: only rows with updated_at > last_sync_time extracted (incremental). Full refresh for small lookup tables. Source columns preserved exactly ("load first, transform later" vs ETL's "transform before load"). New columns in source appear automatically in warehouse staging tables without code changes.
2
Transform: dbt Runs SQL Models
dbt run compiles Jinja-templated SQL into plain SQL, submits to warehouse (Snowflake/BigQuery) which runs it at MPP scale. Three layers: Staging (1:1 source clean-up), Intermediate (business logic joins), Marts (aggregated tables for BI consumption). dbt lineage: every column traced back to its source. dbt test: assert no NULLs in order_id, assert all foreign keys valid, custom: assert revenue non-negative.
3
Warehouse Query: MPP Execution
Snowflake: query parsed → Catalyst-like optimizer generates physical plan → plan distributed across N worker nodes (each reads different Snowflake micro-partitions in parallel from S3). Partition pruning: if WHERE revenue_month = '2025-01' → only read partitions containing that month. Column projection: SELECT only 3 columns → read only those 3 column files. A 10-node cluster processes 1 TB query in ~30 seconds.
4
Spark for Large-Scale Feature Engineering
When transformation is too complex for SQL (graph algorithms, iterative ML training, custom Python libraries): Databricks Spark cluster. spark.read.parquet("s3://data-lake/orders/") → distributed DataFrame. Window functions: .withColumn("7day_avg", avg("revenue").over(Window.orderBy("date").rowsBetween(-6,0))). Wide transformations (shuffle): groupBy().agg() → data redistribution across cluster nodes. AQE: adjusts partition count after seeing actual data distribution.
📦 ETL vs ELT
ETL (old way): Extract from source → Transform on intermediate server → Load to warehouse. Problem: transformation code lives outside the warehouse, separate infra, can't use warehouse MPP for transformation. ELT (modern): Extract → Load raw → Transform IN the warehouse using SQL/dbt. Warehouse compute handles transforms. No separate ETL cluster.
⚡ Columnar Advantage
Row store: SELECT SUM(revenue) FROM orders reads ALL columns (id, user_id, status, created_at, revenue...) just to sum one. Columnar: only reads the revenue column file. 10 columns: 10× less I/O. Plus: same column = high compression ratio (RLE, dict encoding). Snowflake query on 1B rows, 50 columns, SELECT 2: reads 2/50 = 4% of data. 25× faster than row store.
🎯 CAP Position
Eventual consistency, sacrifices availability for correctness. Analytical data is inherently historical — a dashboard showing yesterday's revenue doesn't need real-time consistency. Snowflake/BigQuery: CP at write time (ACID transactions on table updates via Iceberg/internal formats), AP for reads (queries may see slightly stale materialized views). Refresh lag <1 minute acceptable for executive BI.
