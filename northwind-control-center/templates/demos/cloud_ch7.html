The Big Data Definition

The age of analogue data storage (paper, magnetic tape, vinyl) is over. Digital data generation is now mainstream and continuous — manufacturing, public administration, social networks, IoT — all generating raw data that must be processed to extract information.

A system is "Big Data" if:
It consists of datasets large enough that typical systems are incapable of handling
It requires new processing methods to cope with the data volume in a timely manner
The Three Vs
Volume
Orders of magnitude larger than conventional systems
Velocity
Speed of data movement significantly higher than traditional systems
Variety
Wide range of sources; varying quality and types of data
Big Data vs HPC
Aspect HPC Big Data
PurposeScientific/engineering simulationData processing & analysis
AlgorithmsUnique, optimised per problemStandard pipeline steps
UsersScientists (exclusive toy)All industries (mainstream)
Big Data Life Cycle — 4 Stages

Data processing is different from running a SQL query — it follows a structured lifecycle before raw data becomes actionable insight.

1
Ingest / Collect
Extract, sort, label, and transform data from diverse sources. Raw sensor readings, log files, transaction records, API streams. This stage defines data quality for everything downstream.
2
Persist — Long-Term Storage
Store in a distributed filesystem (HDFS, GCS) or distributed database. Must survive node failures — replication across multiple nodes. Storage is the foundation for repeated analysis runs.
3
Process and Analyse
Batch vs real-time processing (discussed next tab). Query processing, statistical algorithms, ML, feature extraction. The compute-intensive stage — MapReduce, Spark, Flink, Beam run here.
4
Visualise Results
Detect trends, surface changes, deliver insights to decision-makers. Dashboards, reports, alerts, recommendation feeds. Information extracted from raw data becomes business value here.
Data Processing Frameworks Covered (ch. 7)

Batch Processing — Bounded Input

The traditional model: read all input data into memory, process it, write results. The entire dataset must be available before execution begins.

Batch model:
Input data (finite, static, persistent on disk)
  → Read ALL data into memory
    → Process entire dataset
      → Write results to output files/DB
Execution is staged: input → processing → output
Batch Characteristics
Input is bounded (has a start and end)
Data is static and persistent before processing begins
Program has access to the complete dataset
Staging: read → process → write (sequential phases)
Latency: results only available after entire run finishes
Cannot process infinite / continuously arriving data
Batch Use Cases
End-of-day financial reporting · Monthly billing runs · Nightly ETL pipelines · Scientific simulations on fixed datasets · Training ML models on historical data · WordCount on a log archive
Stream Processing — Unbounded Input

In many modern applications, data is generated continuously in real time. A continuous stream will never finish — you cannot wait for all input to arrive before processing.

Stream model:
Data stream (infinite, arrives continuously)
  → Record arrives (single event/chunk)
    → Process immediately (no buffering)
      → Emit result for this record
  → Next record arrives → repeat indefinitely
Stream Advantages
Processes data as it arrives — no waiting for complete dataset
Lower memory: only a small portion of data in memory at once
Lower latency: results available immediately per record
Can be used even when the whole dataset is available (still more efficient)
Better system utilisation — resources shared across time
Stream Use Cases
Live media (video, audio, text, messaging)
IoT — traffic data (autonomous driving, jams)
Manufacturing — process & assembly logs
Social networks — live feed, security event detection
Financial — fraud detection, real-time trading signals
Hadoop/MapReduce — Components
HDFS
Hadoop Distributed File System. Fault-tolerant, replicated across commodity nodes. Input data lives here; job output written here.
YARN
Cluster coordinator and scheduler. Manages resource allocation; routes jobs to available nodes.
MapReduce
The batch processing engine. Divides data into key-value pairs, distributes Map tasks, collects Reduce results.
Hadoop Common
Shared utilities and libraries used by HDFS, YARN, and MapReduce subsystems.
YARN Architecture
ResourceManager (1 per cluster — master)
  → Receives JAR files from clients
  → Schedules subtasks across nodes
  → Provides monitoring & feedback

NodeManager (1 per cluster node)
  → Reports resource usage to ResourceManager
  → Launches/monitors task containers

MRAppMaster (1 per application)
  → Coordinates execution of one specific job
Fault Tolerance Philosophy
"In a distributed system, failure is not the exception but the norm." With thousands of servers, there will always be at least one failing at any moment. Hadoop provides application-level fault tolerance — failed tasks are automatically rescheduled on healthy nodes. Compute and storage nodes are typically co-located to minimise data movement.
Key-Value Pipeline & Execution Phases
Key-Value transformation chain:
(input) <k1,v1> → map → <k2,v2> → combine → <k2,v2> → reduce → <k3,v3> (output)
All keys and values must be serializable.
6 Execution Phases
1
Input — Reads and parses input data from HDFS; forwards key-value pairs to Mapper
2
Map — User-defined function processes incoming KV pairs, emits intermediate KV pairs. Runs in parallel across all nodes.
3
Combiner — Optional local reduce within a single Mapper node. Reduces data transferred during Shuffle, improving performance.
4
Shuffle & Sort — Redistributes intermediate KV pairs by key across Reducers; sorts by key for efficient aggregation.
5
Reducer — User-defined aggregation, filtering, or combining on all values for a given key. Outputs zero or more KV pairs.
6
Output — Final KV results saved to HDFS in the required format.
Java — Hadoop WordCount (Mapper + Reducer + Driver)
public class WordCount {

  // Mapper: splits each line into words, emits (word, 1)
  public static class TokenizerMapper
      extends Mapper<Object, Text, Text, IntWritable> {
    private final static IntWritable one = new IntWritable(1);
    private Text word = new Text();
    public void map(Object key, Text value, Context ctx)
        throws IOException, InterruptedException {
      StringTokenizer itr = new StringTokenizer(value.toString());
      while (itr.hasMoreTokens()) {
        word.set(itr.nextToken());
        ctx.write(word, one);    // emit ("apple", 1)
      }
    }
  }

  // Reducer: sums counts for each word
  public static class IntSumReducer
      extends Reducer<Text, IntWritable, Text, IntWritable> {
    private IntWritable result = new IntWritable();
    public void reduce(Text key, Iterable<IntWritable> values,
        Context ctx) throws IOException, InterruptedException {
      int sum = 0;
      for (IntWritable val : values) { sum += val.get(); }
      result.set(sum);
      ctx.write(key, result);    // emit ("apple", 2)
    }
  }

  public static void main(String[] args) throws Exception {
    Configuration conf = new Configuration();
    Job job = Job.getInstance(conf, "word count");
    job.setJarByClass(WordCount.class);
    job.setMapperClass(TokenizerMapper.class);
    job.setCombinerClass(IntSumReducer.class);  // local reduce
    job.setReducerClass(IntSumReducer.class);
    job.setOutputKeyClass(Text.class);
    job.setOutputValueClass(IntWritable.class);
    FileInputFormat.addInputPath(job, new Path(args[0]));
    FileOutputFormat.setOutputPath(job, new Path(args[1]));
    System.exit(job.waitForCompletion(true) ? 0 : 1);
  }
}
Why Spark — The Hadoop Bottleneck

Hadoop uses the file system for intermediate data between Map and Reduce phases. Disk I/O bandwidth is orders of magnitude lower than memory bandwidth — this becomes a prohibitive bottleneck at scale.

Hadoop intermediate data flow (slow):
Map → write to disk → Shuffle → read from disk → Reduce
Reduce → write to disk → next Map → read from disk…
Spark RDD flow (fast):
Map → RDD in memory → transform → RDD in memory → Reduce
Disk only at: input read and final output write
Spark Performance
100×
faster (in-memory operations)
10×
faster (disk operations — fewer I/O passes)
Spark Execution Modes
Standalone Spark runs as its own independent cluster environment — no external cluster manager needed.
Hadoop YARN Spark runs on top of YARN — shares the Hadoop cluster with other YARN applications.
MR Mode Spark jobs executed as Hadoop/MapReduce jobs. Integrates legacy MR programs with Spark.
Spark Capabilities (beyond Map/Reduce)
SQL queries Machine Learning (MLlib) Graph Analytics (GraphX) Stream Processing Python / Scala / R / Java APIs
Spark WordCount — RDD Pipeline

Spark's core abstraction is the Resilient Distributed Dataset (RDD) — a distributed, immutable, fault-tolerant in-memory collection that Spark operates on in parallel across all nodes. Transformations (flatMap, mapToPair) are lazy; only collect() triggers actual execution.

Java — Spark WordCount (RDD API)
package org.apache.spark.examples;
import scala.Tuple2;
import org.apache.spark.api.java.JavaPairRDD;
import org.apache.spark.api.java.JavaRDD;
import org.apache.spark.sql.SparkSession;
import java.util.Arrays;
import java.util.List;
import java.util.regex.Pattern;

public final class JavaWordCount {
  private static final Pattern SPACE = Pattern.compile(" ");

  public static void main(String[] args) throws Exception {
    // 1. Create Spark session (entry point to all Spark functionality)
    SparkSession spark = SparkSession.builder()
        .appName("JavaWordCount")
        .getOrCreate();

    // 2. Read input file → RDD of lines (lazy)
    JavaRDD<String> lines =
        spark.read().textFile(args[0]).javaRDD();

    // 3. Split lines into words (flatMap — 1 line → many words)
    JavaRDD<String> words = lines.flatMap(
        s -> Arrays.asList(SPACE.split(s)).iterator());

    // 4. Map each word to (word, 1) pair
    JavaPairRDD<String, Integer> ones =
        words.mapToPair(s -> new Tuple2<>(s, 1));

    // 5. Reduce by key (sum all 1s for each word)
    JavaPairRDD<String, Integer> counts =
        ones.reduceByKey((i1, i2) -> i1 + i2);

    // 6. collect() triggers execution; print results
    List<Tuple2<String, Integer>> output = counts.collect();
    for (Tuple2<?,?> tuple : output) {
      System.out.println(tuple._1() + ": " + tuple._2());
    }
    spark.stop();
  }
}
Hadoop vs Spark — WordCount Pipeline Comparison
Step Hadoop MR (disk) Spark (RDD)
Input readHDFS → memoryHDFS → RDD (lazy)
After MapWritten to local diskStays in memory (RDD)
ShuffleDisk read + networkNetwork only (in-memory)
After ReduceWritten to HDFSIn-memory until collect()
Final outputHDFS writeHDFS write (or collect to driver)
GCP Dataproc — Managed Hadoop & Spark

Creating a Hadoop/Spark cluster from scratch requires provisioning VMs, installing software, configuring networking, and setting up a job scheduler. Dataproc automates all of this — cluster creation takes seconds instead of hours.

Dataproc Integrations
BigQuery Cloud Storage Cloud Bigtable Cloud Logging Cloud Monitoring
4 Ways to Access Dataproc
① REST API — programmatic access from any language
② Cloud SDK — gcloud dataproc commands
③ Cloud Dataproc UI — web console
④ Cloud Client Libraries — Java, Python, Go, etc.

# Enable Stackdriver logging on cluster create:
--properties \
  dataproc:dataproc.logging.stackdriver.enable=true

# Enable YARN container log collection:
--properties \
