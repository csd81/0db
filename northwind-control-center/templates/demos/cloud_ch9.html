Function-as-a-Service — The Serverless Paradigm
What is FaaS?
Submit code directly to the cloud without managing any runtime infrastructure. The cloud provider handles provisioning, scaling, and maintenance automatically.
Key Properties
✓ No server management (provision/upgrade/manage)
✓ Automatic scaling with load
✓ Integrated monitoring, logging, debugging
✓ Role-based & per-function security
✓ Networking to access other cloud components
The Historical Vision
Cloud Functions unify in a practical way the once-utopian visions of past computing systems:
Grid Computing — seamless execution over geographically distributed resources
SOA — a set of public/private services composable on demand
Cloud Execution Model Comparison
Dimension IaaS (EC2, Compute Engine) PaaS (Beanstalk, App Engine) FaaS (Cloud Functions, Lambda)
Unit of Deployment Virtual Machine Application Function
Server Management User manages OS & runtime Platform manages runtime None — fully abstracted
Scaling Manual / Autoscaling groups Platform scales app Automatic, per-invocation
Billing Per hour (running VM) Per instance-hour Per invocation + execution time
State Stateful allowed Stateful (session support) Must be stateless

HTTP Functions
How HTTP Functions Work
Invoked from standard HTTP requests. The caller waits synchronously for the function to complete and the response to be received.
GET POST PUT DELETE OPTIONS
Java Interface
Implement com.google.cloud.functions.HttpFunction — a functional interface with a single service() method.
Similar in style to Java Servlets in the Enterprise Platform: same HttpRequest / HttpResponse pattern.
Java — HelloHttp
public class HelloHttp implements HttpFunction {
    private static final Logger logger = Logger.getLogger(HelloHttp.class.getName());
    private static final Gson gson = new Gson();

    @Override
    public void service(HttpRequest request, HttpResponse response) throws IOException {

        // 1. Check URL query parameter "name" (default: "world")
        String name = request.getFirstQueryParameter("name").orElse("world");

        // 2. Parse JSON body and look for "name" field
        try {
            JsonElement requestParsed = gson.fromJson(request.getReader(), JsonElement.class);
            JsonObject requestJson = null;
            if (requestParsed != null && requestParsed.isJsonObject()) {
                requestJson = requestParsed.getAsJsonObject();
            }
            if (requestJson != null && requestJson.has("name")) {
                name = requestJson.get("name").getAsString();
            }
        } catch (JsonParseException e) {
            logger.severe("Error parsing JSON: " + e.getMessage());
        }

        // 3. Write response
        var writer = new PrintWriter(response.getWriter());
        writer.printf("Hello %s!", name);
    }
}
Node.js — Same Function (1 line!)
const escapeHtml = require('escape-html');

/**
 * HTTP Cloud Function.
 * @param {Object} req Cloud Function request context.
 * @param {Object} res Cloud Function response context.
 */
exports.helloHttp = (req, res) => {
    res.send(`Hello ${escapeHtml(
        req.query.name || req.body.name || 'World'
    )}!`);
};
The compactness of Node.js vs Java is striking — same logic in 3 lines vs 30 lines.
Request Processing Flow
1. URL Param
?name=Alice → query parameter
2. JSON Body
{"name":"Bob"} → parsed with Gson
3. Default
Neither provided → "world"
4. Response
printf("Hello %s!", name) → HTTP 200

Background Functions — Event-Driven Execution
How Background Functions Work
Designed to handle events originating from the cloud infrastructure — e.g. a change in a Cloud Storage bucket or a message arriving to a Cloud Pub/Sub topic.
The function is not called directly by a user — it is triggered automatically by a cloud event.
Java Interface
Implement com.google.cloud.functions.BackgroundFunction<T>. The generic parameter <T> determines which event type the function responds to.
Key method: accept(T event, Context context) — invoked upon the event of interest.
Java — HelloGcs (Cloud Storage Trigger)
import com.google.cloud.functions.BackgroundFunction;
import com.google.cloud.functions.Context;
import functions.eventpojos.GcsEvent;
import java.util.logging.Logger;

// Generic type <GcsEvent> tells GCP: invoke this when a GCS event occurs
public class HelloGcs implements BackgroundFunction<GcsEvent> {
    private static final Logger logger = Logger.getLogger(HelloGcs.class.getName());

    @Override
    public void accept(GcsEvent event, Context context) {
        logger.info("Event: "          + context.eventId());
        logger.info("Event Type: "     + context.eventType());
        logger.info("Bucket: "         + event.getBucket());
        logger.info("File: "           + event.getName());
        logger.info("Metageneration: " + event.getMetageneration());
        logger.info("Created: "        + event.getTimeCreated());
        logger.info("Updated: "        + event.getUpdated());
    }
}

Function Execution Model
🔒
Isolated & Secure
Every Function executes in an isolated, secure runtime context. No shared state between instances.
📊
One Request per Instance
Each Function instance services one client/request at a time. Multiple instances are launched automatically when needed.
⚠️
Must Be Stateless
Functions must be stateless. If state is required, persist it to Cloud Storage or Datastore between invocations.
Delivery Guarantees
HTTP Functions At-Most-Once
Each HTTP request triggers the function at most one time. If the invocation fails, there is no automatic retry. The caller is responsible for retrying.
Background Functions At-Least-Once
Background functions may be invoked more than once for the same event (retries on failure). Functions should be designed to be idempotent when possible.
Comparison with Stream Frameworks
Same delivery semantics as Ch. 8 stream systems: HTTP = At-Most-Once (Storm-like), Background = At-Least-Once (Kafka-like). "Exactly-Once" is not natively provided — must be implemented at application level.
Trigger Types Summary
HTTP
HTTP Triggers — synchronous request-response. Supports all HTTP verbs.
Pub/Sub
Cloud Pub/Sub Triggers — event-driven; invoked when a message arrives on a subscribed topic.
GCS
Cloud Storage Triggers — fired on object finalize, delete, archive or metadata updates in a bucket.
Firestore
Cloud Firestore Triggers — fired on document create, update, delete, or write events.
Firebase
Firebase Triggers — Realtime Database events, Firebase Auth events (user create/delete), Analytics conversion events.
Direct
Direct Triggers — direct invocation for testing and internal calls without an HTTP endpoint.
Deploying Functions

Deployment uploads an archive of the function's source code to a Google Cloud Storage bucket. Three deployment paths are available.

1
Deploy from Local Machine
gcloud functions deploy NAME \
    --runtime RUNTIME \
    TRIGGER \
    [FLAGS...]
Substitute NAME (function name), RUNTIME (e.g. java11), and TRIGGER (e.g. --trigger-http).
2
Deploy from Source Control
gcloud functions deploy NAME \
    --source https://source.developers.google.com/ \
      projects/PROJECT_ID/ \
      repos/REPOSITORY_ID/ \
      moveable-aliases/master/ \
      paths/SOURCE \
    --runtime RUNTIME \
    TRIGGER... [FLAGS...]
3
Deploy from GCP Console — Step-by-Step
1
Open the Functions Overview page in the Cloud Console.
2
Confirm the correct project (with Cloud Functions enabled) is selected.
3
Click Create function.
4
Name your function.
5
In the Trigger field, select HTTP.
6
In Authentication, select Allow unauthenticated invocations.
7
Click Save to save changes, then click Next.
8
In Source code, select Inline editor (or upload a ZIP).
9
Use the Runtime dropdown to select the desired Java runtime (e.g. java11).
10
Click Deploy at the bottom of the page.
11
Console redirects to the Functions Overview page while deploying.
12
Once deployed → display function menu → click Test function.
Monitoring & Logs
📋
Automatic Log Entries
Each function invocation generates log entries automatically. View them on the Functions Overview page → display function menu → View logs.
🔍
Integrated Monitoring
Built-in GCP monitoring (Cloud Monitoring) tracks invocations, execution times, errors, and memory use. No setup required.
🐛
Debugging
Cloud Debugger lets you inspect application state in live functions without stopping or slowing them. Set snapshots and logpoints without redeployment.
