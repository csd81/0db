BigQuery ML moves model training inside the database engine, eliminating the ETL pipeline that normally exports data to Python before fitting. A linear regression model trains with CREATE MODEL … OPTIONS(model_type='linear_reg') and makes predictions with a plain SELECT … FROM ML.PREDICT — both written in standard SQL. The demo covers regression, binary classification, and k-means clustering, each expressed as a SQL statement against Northwind sales data with no Python required. This serverless ML approach lets data analysts build and deploy predictive models without leaving their SQL environment or learning a new framework.

Cloud DB Advantages
● High availability — geographically distributed infrastructure, typically better than corporate servers
● Auto scalability — on-demand, load-proportional billing; no capacity planning
● Ecosystem integration — NLP, image processing, ML, Pub/Sub, Dataflow all in-cloud and latency-free to the DB
● No operational overhead — patching, backups, HA failover managed by provider
Cloud DB Disadvantages
● Reduced functionality — cloud DBs often lack features of on-prem; workarounds required
● Fixed + variable costs — storage always billed even when idle; total cost must be modelled
● Data sovereignty — sensitive data (healthcare records) physically leaving the network is a compliance risk
● Internet dependency — critical production systems should not depend on Internet (→ private cloud)

BigQuery — Dremel Columnar Storage
BigQuery stores data in Dremel — a compressed, optionally-partitioned column store. All data encrypted at rest. Queries are read-only OLAP — no UPDATE/DELETE on live tables.
Logical model: project.dataset.table — datasets contain related tables with a fixed schema.
For best performance, external data (Bigtable, Cloud Storage, Google Drive) should be first imported into Dremel.
Denormalized data preferred — joins across huge tables are expensive; BigQuery expects pre-flattened fact tables.
Reference: Panoply BigQuery Architecture, Google BigQuery Explained Storage Overview

BigQuery ML — CREATE MODEL 
-- Step 1: Denormalized view → CSV → BigQuery table 'northwind.nw'
-- Step 2: Create linear regression model
CREATE MODEL `northwind.nw_model`
OPTIONS (
    model_type='linear_reg',
    input_label_cols=['value_numeric']
) AS
SELECT value_numeric, country, categoryid,
       p_unitprice, discount, pyear
FROM `northwind.nw`
WHERE value_numeric IS NOT NULL

-- Evaluate: MAE ≈ 342, explained variance ≈ 53%
SELECT * FROM ML.EVALUATE(MODEL `northwind.nw_model`, (
  SELECT value_numeric, country, categoryid,
         p_unitprice, discount, pyear
  FROM `northwind.nw`
))

-- Predict: only rows where abs error < 40%
SELECT * FROM ML.PREDICT(MODEL `northwind.nw_model`, (
  SELECT value_numeric, country, categoryid,
         p_unitprice, discount, pyear
  FROM `northwind.nw`
))
WHERE ABS((predicted_value_numeric - value_numeric))
      / value_numeric < 0.4

Interpretation
A logistic regression classifier with OneHotEncoded country + categoryid mirrors the BigQuery LOGISTIC_REG model type.
The BOOSTED_TREE_CLASSIFIER in BigQuery (gradient boosted decision trees) typically achieves higher accuracy than logistic regression — try it on BigQuery to compare.
The value_nominal threshold split (200/1200) is a business rule — an expert choice about what constitutes "low", "medium", "high" order value.
BigQuery ML — Natality Model (public dataset) 
-- Public table: bigquery-public-data.samples.natality
-- ~137M birth records in the USA
-- Sample with RAND() < 0.0001 (~13,700 rows)

CREATE MODEL `us_dataset.natality_model`
OPTIONS (
    model_type='linear_reg',
    input_label_cols=['weight_pounds']
) AS
SELECT
    weight_pounds,
    is_male,
    gestation_weeks,
    mother_age,
    CAST(mother_race AS STRING) AS mother_race
FROM `bigquery-public-data.samples.natality`
WHERE weight_pounds IS NOT NULL
  AND RAND() < 0.0001

-- Training info
SELECT * FROM ML.TRAINING_INFO(
    MODEL `us_dataset.natality_model`)

-- Single prediction
SELECT * FROM ML.PREDICT(
    MODEL `us_dataset.natality_model`,
    (SELECT TRUE AS is_male,
            40   AS gestation_weeks,
            30   AS mother_age,
            '38' AS mother_race))
Dataset Notes
Dataset location matters: the natality public table is in the US multi-region. Your BigQuery dataset must also be in US, otherwise you must set More → Query settings → Advanced → Data location to US.
RAND() sampling: the table has 137M rows. WHERE RAND() < 0.0001 samples approximately 13,700 rows for training — keeps cost and time low.
Mother race codes (NCHS): 1=White 2=Black 3=Am.Indian 6=Asian/PI 38=Korean
