02_migrate_from_sql.py — ETL reference reads SQL Server graph tables → writes to Neo4j
{% raw %} 
from neo4j import GraphDatabase
import pyodbc

driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))
sql = pyodbc.connect(SQL_CONN_STR)

with driver.session() as neo:
    neo.run("MATCH (n) DETACH DELETE n")          # clear

    # Load nodes from SQL Server graph tables
    for table, label in [('person','Person'),('restaurant','Restaurant'),('city','City')]:
        rows = sql.execute(f"SELECT id, name FROM {table}").fetchall()
        for row in rows:
            neo.run(f"CREATE (:{label} {{id:$id, name:$name}})", id=row[0], name=row[1])

    # Load FRIEND_OF edges using $node_id / $from_id / $to_id
    edges = sql.execute("""
        SELECT p1.id, p2.id FROM friendof f
        JOIN person p1 ON p1.$node_id = f.$from_id
        JOIN person p2 ON p2.$node_id = f.$to_id
    """).fetchall()
    for e in edges:
        neo.run("MATCH (p1:Person {id:$id1}),(p2:Person {id:$id2}) CREATE (p1)-[:FRIEND_OF]->(p2)",
                id1=e[0], id2=e[1])

ConceptSQL Server GraphNeo4j / Cypher

Edge identity$edge_id (internal JSON)id(r) function

Relationship typesEdge table name:LIKES, :FRIEND_OF — exactly one per relationship

              
PropertiesRegular columnsKey-value map on node or relationship

            
          

        

      


      
      

        

          DDL
        

        

          

            
OperationSQL ServerCypher

            
              
Create node type
                  CREATE TABLE person (id INT PRIMARY KEY, name VARCHAR(100)) AS NODE
                  No DDL needed — just CREATE

              
Create edge type
                  CREATE TABLE likes (rating INT) AS EDGE
                  No DDL needed

              
Add edge constraint
                  ALTER TABLE friendof ADD CONSTRAINT ec_1 CONNECTION (person TO person)
                  No direct equivalent (APOC or application logic)

              
Create index
                  CREATE INDEX ON person(name)
                  CREATE INDEX FOR (p:Person) ON (p.name)

            
          

        

      


      
      

        

          DML
        

        

          

            
OperationSQL ServerCypher

            
              
Insert node
                  INSERT INTO person VALUES (1, 'john')
                  CREATE (:Person {id:1, name:'john'})

              
Insert edge
                  INSERT INTO likes VALUES ((SELECT $node_id FROM person WHERE id=1), (SELECT $node_id FROM restaurant WHERE id=1), 9)
                  MATCH (p:Person {id:1}), (r:Restaurant {id:1}) CREATE (p)-[:LIKES {rating:9}]->(r)

              
Update node
                  UPDATE person SET name='johnny' WHERE id=1
                  MATCH (p:Person {id:1}) SET p.name='johnny'

              
Delete node+edges
                  DELETE FROM person WHERE id=1 (must delete edges first)
                  MATCH (p:Person {id:1}) DETACH DELETE p

            
          

        

      


      
      

        

          Pattern Matching
        

        

          

            
QuerySQL Server (T-SQL MATCH)Cypher

            
              
Direct friends
                  ... WHERE MATCH(p1-(friendof)->p2) AND p1.name='john'
                  MATCH (:Person {name:'john'})-[:FRIEND_OF]->(p2:Person) RETURN p2.name

              
Two-hop path
                  MATCH(p1-(friendof)->p2-(likes)->restaurant)
                  MATCH (:Person {name:'john'})-[:FRIEND_OF]->()-[:LIKES]->(r:Restaurant)

              
AND pattern
                  MATCH(p-(likes)->r-(locatedin)->c AND p-(livesin)->c)
                  MATCH (p)-[:LIKES]->(r)-[:LOCATED_IN]->(c), (p)-[:LIVES_IN]->(c)

              
Shortest path
                  MATCH(SHORTEST_PATH(p1(-(f)->p2)+)) WHERE p1.name='john'
                  MATCH path=shortestPath((:Person {name:'john'})-[:FRIEND_OF*]->(dest))

              
Variable length
                  SHORTEST_PATH(p1(-(f)->p2){1,2})
                  MATCH (p1)-[:FRIEND_OF*1..2]->(p2)

            
          

        

      


      
      

        

          Key Differences
        

        

          

            
AspectSQL Server GraphNeo4j

            
              
Storage
                  Relational tables + hidden graph columns
                  Native property graph store (pointer chains)

              
Query language
                  T-SQL + MATCH extension
                  Cypher — declarative, pattern-first

              
Multi-label nodes
                  Not supported (one table = one type)
                  Supported (:Person:Employee)

              
Schema enforcement
                  Via table definitions + FK constraints
                  Optional (constraints via CREATE CONSTRAINT)

              
Graph algorithms
                  Limited (SHORTEST_PATH only)
                  Rich GDS plugin: PageRank, Louvain, BFS, betweenness…

              
Visualization
                  Requires external tools
                  Built-in Neo4j Browser (graph visual + table)

              
Horizontal scale
                  SQL Server clustering
                  Neo4j Fabric / Aura clusters

              
Transactions
                  Full ACID
                  Full ACID (MVCC)

              
Best for
                  Graphs embedded in existing relational schema
                  Large, deeply connected graphs — complex traversals
