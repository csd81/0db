Polyglot Persistence

Seven database categories · Choose the right engine for the right job · Click a node to explore

Polyglot persistence describes architectures where each component chooses the database most suited to its data model and access patterns rather than forcing every workload into a single general-purpose system. An e-commerce platform might use SQL Server for transactional orders, Redis for sub-millisecond cart state, Elasticsearch for typo-tolerant product search, MongoDB for flexible product catalogs, and a graph database for logistics routing — each database doing exactly what it excels at. The interactive hub diagram lets you click each database category to see its native query syntax, consistency model, latency profile, and the real companies running it at scale. This architecture pattern is the default choice for any system operating at Amazon, Netflix, or Uber scale.

    Right Tool, Right Job

    Polyglot persistence is not complexity for its own sake. A billing system needs ACID guarantees;
    a shopping cart needs microsecond latency; a recommendation engine needs graph traversal.
    Forcing all three through a single RDBMS incurs painful trade-offs on all fronts.
    Each engine above excels at one thing precisely because it gave up the others.
  

  

    CAP Per Subsystem

    Each category makes different CAP trade-offs. RDBMS favour Consistency (CP).
    Cassandra/DynamoDB favour Availability (AP). Redis is CP but RAM-bound.
    Polyglot architecture lets you choose the correct CAP posture per subsystem
    rather than accepting one universal compromise across your entire data layer.
  

  

    Operational Cost

    Complexity scales with the number of systems: separate backups, monitoring, expertise,
    and failure modes for each. The answer is managed cloud services (RDS, DynamoDB,
    ElastiCache, Neptune, Redshift, Pinecone) — letting engineering teams focus on
    data modelling rather than cluster management.
