Why Streams Require Different Algorithms
Batch Processing
Entire dataset in memory or file. Multi-pass algorithms are possible. Processing ends when input is exhausted.
Input → Process → Output
Stream Processing
Continuous, unbound incoming data. Algorithms process elements as they arrive — no access to the full dataset.
∞ Stream → Process continuously
Benefits
✓ Decreased latency — real-time results
✓ Reduced memory — only current window in RAM
✓ 24/7 cloud-native operation
Two-Pass Variance — Correct

Pass 1: compute mean. Pass 2: compute variance. Requires entire array. Numerically stable.

double varianceTwoPass(double[] data) {
    // Pass 1: mean
    double mean = 0;
    for (int i = 0; i < data.length; i++)
        mean += data[i];
    mean /= data.length;
    // Pass 2: variance
    double variance = 0.0;
    for (int i = 0; i < data.length; i++)
        variance += (data[i]-mean)*(data[i]-mean);
    return variance / data.length;
}
✓ Numerically correct — safe for production
Single-Pass Naïve — NEVER USE

Uses Var(X) = E[X²] − E[X]². One loop, but suffers catastrophic cancellation — results are wrong when sumOfSquares ≈ (sum²/n).

double varianceSinglePass(double[] data) {
    double sum = 0, sumOfSquares = 0;
    for (int i = 0; i < data.length; i++) {
        sum += data[i];
        sumOfSquares += data[i] * data[i];
    }
    // catastrophic cancellation here ↓
    return (sumOfSquares - (sum*sum)/data.length)
           / data.length;
}
✗ Catastrophic cancellation — incorrect results, never use
Welford's Online Algorithm

Numerically stable one-pass algorithm. Works on streams of unknown length. Only current value + running state needed — no stored data elements.

Running Mean
mk = mk-1 + (xn − mk-1) / n
Old mean corrected by scaled deviation of the new sample
Running Variance
Sn = Sn-1 + (xn − mn)(xn − mn-1)
Var(X) = Sn / n
Uses both new and previous mean — no cancellation
Java Consumer via ArrayBlockingQueue
private class Consumer extends Thread {
    private ArrayBlockingQueue data;
    private long n;
    private double mean, sn, variance;

    public Consumer(ArrayBlockingQueue data) { this.data = data; }

    public void run() {
        while (true) {
            try {
                int x = data.take();              // blocks until producer sends
                n++;
                double oldMean = mean;
                mean = mean + (x - mean) / n;          // Welford mean update
                sn   = sn   + (x - mean) * (x - oldMean); // Welford S update
                variance = sn / n;
            } catch (InterruptedException ex) {
                ex.printStackTrace();
            }
        }
    }
}

SparkConf sparkConf = new SparkConf().setAppName("JavaNetworkWordCount");

JavaDStream<String> words = lines.flatMap(

Apache Flink — Unified Bounded & Unbounded Streams

Flink's key insight: batch data is a bounded stream. One codebase handles both — no duplicate batch and stream implementations. Developers write one algorithm; Flink runs it on either.

Unbounded Stream (Live Data)
… ∞
Has a start but no end. Represents live streaming data. Requires continuous processing without a final output.
Bounded Stream (= Batch)
Well-defined start and end. Treated as a bounded stream — same code runs on both. No separate batch implementation needed.
Stateful Processing

Most distributed systems prefer stateless design. Flink makes state a first-class citizen: managed in-memory, checkpointed to pluggable backends. Supports terabyte-scale state.

ValueState
Single atomic value per key
ListState
List of values per key — append operations
MapState
Map of KV pairs per key — random access
State checkpointing provides exactly-once fault tolerance even across node failures. Flink supports in-memory and on-disk (RocksDB) state backends.
Time Semantics
Event Time
Events processed based on their embedded timestamp. Results are consistent regardless of processing duration or late arrivals. Supports watermarks and late event handling.
Processing Time
Events processed based on wall-clock time of processing. Lowest latency but may miss late events. Suitable for real-time applications with defined latency requirements.
Common Time-Based Operations
Window Aggregations Sessionisation Pattern Detection Time-Based Joins

Apache Storm — Topology Model
🌀
Topology
The computation graph. Runs continuously — unbounded. Represents the full dataflow from source to result.
💧
Spout
Data source at topology edge. Reads from APIs, queues, etc. Emits tuples via nextTuple().
⚡
Bolt
Processing node. Consumes streams, applies operations, emits results. Bolts chain together. Key method: execute().
📦
Tuple
Data element on connecting edges. Named fields. E.g. Values("word") or Values("word", count).

2. SplitSentence Bolt
public void execute(Tuple t,
    BasicOutputCollector out) {
    String sentence =
        t.getStringByField("sentence");
    String[] words = sentence.split(" ");
    for (String w : words)
        out.emit(new Values(w));
}
// declareOutputFields → "word"
3. WordCount Bolt
Map<String,Integer> counts = new HashMap<>();
public void execute(Tuple t,
    BasicOutputCollector out) {
    String word = t.getString(0);
    Integer count = counts.get(word);
    if (count == null) count = 0;
    counts.put(word, ++count);
    // emits running count per word:
    out.emit(new Values(word, count));
}
// declareOutputFields → "word","count"

Storm Cluster Architecture
Nimbus — cluster entry point (analogous to Hadoop JobTracker)
Zookeeper — cluster manager: schedules and monitors Workers
Workers + Supervisors — execute topology tasks with local resource management
Batch vs Stream Key Distinction
In batch: WordCount produces one final count per word. In Storm: the bolt emits a running count — every incoming tuple triggers output, reflecting the partial state of processing at that moment.
Apache Kafka — High-Performance Pub-Sub Message Broker
Core Concepts
Record: key + value + timestamp
Topic: named category of records
Partition: ordered queue within a topic
Broker: cluster node managing partitions
Offset: position marker in a partition
Leader / Follower Replication
Each partition has one leader and N followers replicating the leader. On leader failure, a follower is automatically promoted — zero message loss.
Broker 1: leader A/0, replica A/1
Broker 2: leader A/1, replica A/0
Why Pub-Sub Matters
Unlike point-to-point queues, Kafka guarantees all subscribers receive the events they subscribed to. Multiple producers and consumers operate independently.
Fault-tolerant Auto-partition High-throughput

Apache Beam — Unified Batch & Streaming SDK

Beam is the programming model used by Google Dataflow, the GCP stream processing engine. Write once, run on any backend runner (Dataflow, Spark, Flink).

🔗
Pipeline
Entire program: input → transforms → output. Executed by a backend Runner (Dataflow, Spark, Flink).
📂
PCollection
Distributed dataset. Bounded (batch) or unbounded (stream). Immutable — transforms always create new PCollections.
⚙️
PTransform
One processing step. Takes PCollection(s) as input, returns PCollection(s) as output. Applied via .apply().
Pipeline Composition Pattern
[Output PCollection] =
  [Input PCollection]
    .apply([First Transform])
    .apply([Second Transform])
    .apply([Third Transform]);
Each .apply() returns a new PCollection — chaining builds the execution DAG.
Core Built-in Transforms
ParDo GroupByKey CoGroupByKey Combine Flatten Partition TextIO.Read TextIO.Write
ParDo — parallel per-element transform. Analogous to Map. Most commonly used.
Beam WordCount — 4-Step Pipeline
Step 1
TextIO.Read from GCS
→ PCollection<String> (lines)
Step 2
FlatMapElements → words
→ PCollection<String> (words)
Step 3
Count.perElement()
→ PCollection<KV<String,Long>>
Step 4
TextIO.Write to GCS
"word: count" strings
public class MinimalWordCount {
    public static void main(String[] args) {
        PipelineOptions options = PipelineOptionsFactory.create();
        Pipeline p = Pipeline.create(options);

        p.apply(TextIO.read().from("gs://apache-beam-samples/shakespeare/*"))
         .apply(FlatMapElements.into(TypeDescriptors.strings())
                .via((String line) -> Arrays.asList(line.split("[^\\p{L}]+"))))
         .apply(Filter.by((String word) -> !word.isEmpty()))
         .apply(Count.perElement())
         .apply(MapElements.into(TypeDescriptors.strings())
                .via((KV<String, Long> wordCount) ->
                    wordCount.getKey() + ": " + wordCount.getValue()))
         .apply(TextIO.write().to("wordcounts"));

        p.run().waitUntilFinish();
    }
}
