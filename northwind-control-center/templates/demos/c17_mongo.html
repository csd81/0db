Chapter 17 — MongoDB Schema Validation, CQRS & pgvector

$jsonSchema Validation · CQRS Read/Write Split · Semantic Search with pgvector

MongoDB schema validation enforces data contracts on the document store using JSON Schema rules, with validationLevel controlling strictness — strict rejects all invalid writes; moderate only checks existing valid documents during updates. CQRS — Command Query Responsibility Segregation — maintains separate write and read models so that heavy analytics queries never contend for the same index pages as transactional INSERT and UPDATE paths. The pgvector section demonstrates PostgreSQL's vector extension enabling cosine-similarity search over embedding vectors, bridging relational queries and AI retrieval-augmented generation in a single database. These three patterns together represent the state of the art in modern polyglot persistence.

The Schema-less Footgun

In a relational database, inserting a row without a required column is a compile-time error. In MongoDB, inserting a document without unit_price succeeds silently and corrupts every aggregation that depends on that field.

✗ Silent failure (MongoDB without validator)
db.products.insertOne({
  product_name: "Chai"
  // unit_price missing — no error!
})
// aggregations on unit_price now return NaN
✓ Explicit rejection (with $jsonSchema)
db.products.insertOne({
  product_name: "Chai"
  // unit_price missing
})
// WriteError: Document failed validation
The $jsonSchema Validator 
db.runCommand({
  collMod: 'products',
  validator: {
    $jsonSchema: {
      bsonType: 'object',
      required: ['product_id', 'product_name', 'unit_price'],
      properties: {
        product_id:     { bsonType: 'int' },
        product_name:   { bsonType: 'string' },
        unit_price:     { bsonType: 'double', minimum: 0 },
        units_in_stock: { bsonType: 'int' },
        discontinued:   { bsonType: 'bool' }
      }
    }
  },
  validationLevel:  'moderate',
  validationAction: 'warn'
})

validationLevel — moderate vs strict
LevelBehavior
strict (default) All inserts AND updates must satisfy the schema.
moderate Only new documents and updates to already-valid documents are checked. Existing invalid documents untouched.
Use moderate when retrofitting a validator onto an existing collection — avoids blocking legitimate update operations on legacy documents.
validationAction — error vs warn
ActionBehavior
error Write is rejected; driver receives a WriteError.
warn Write is allowed but violation is logged to MongoDB server log.
warn is right during migration/audit phase — scan log to find which paths produce invalid data before switching to error.

The Contention Problem

A single MongoDB replica set node serving both the Northwind web shop and the analytics dashboard creates contention. An analyst running a 30-second aggregation over three million order documents saturates CPU and I/O — degrading checkout response times.

Single node:
  checkout write (OLTP)  ←── contends ──→  analytics query (OLAP)
  low latency needed          high throughput needed
  small indexed reads         sequential full-collection scans
CQRS — Command Query Responsibility Segregation
Commands
Writes: insert, update, delete
→ always to Primary node
Ensures write durability & immediate consistency
Queries
Reads: find, aggregate
→ to Secondary (secondaryPreferred)
Slightly stale data — acceptable for analytics
# Command path — write to primary
primary_client = MongoClient(uri, readPreference='primary')
primary_client['northwind']['orders'].insert_one(order_doc)

# Query path — read from replica
replica_client = MongoClient(uri, readPreference='secondaryPreferred')
results = replica_client['northwind']['orders'].aggregate(pipeline)

Limits of Traditional Indexing

A B-tree index on product_name supports exact matches and prefix searches efficiently. LIKE '%beverage%' forces a sequential scan. Neither approach captures meaning.

A user searching for "chai" should find Chai, Chang, and Ipoh Coffee — all beverages — not only documents that literally contain "chai".
Embeddings & Vector Indexes

An embedding model converts text into a high-dimensional vector of floating-point numbers. Similar meanings map to geometrically close points.

The embedding for "chai" and "Ipoh Coffee" have high cosine similarity — both appear in similar contexts in the training corpus.

pgvector Setup & Semantic Query 
-- 1. Install extension
CREATE EXTENSION vector;

-- 2. Add embedding column (1536-dim for OpenAI, 384 for MiniLM)
ALTER TABLE products
  ADD COLUMN embedding vector(1536);

-- 3. Create HNSW index for approximate nearest-neighbor
CREATE INDEX ON products
  USING hnsw (embedding vector_cosine_ops);
-- IVFFlat alternative:
-- CREATE INDEX ON products USING ivfflat (embedding vector_cosine_ops)
--   WITH (lists = 100);
-- Semantic search: "chai" → finds beverages by meaning
-- query_embedding is the embedding of the search term
SELECT product_name, category_name,
       1 - (embedding <=> query_embedding) AS similarity
FROM   products
ORDER  BY embedding <=> query_embedding
LIMIT  5;

-- Result for query="chai":
-- product_name         category_name  similarity
-- ──────────────────── ────────────── ──────────
-- Chai                 Beverages      0.98
-- Chang                Beverages      0.87
-- Ipoh Coffee          Beverages      0.83
-- Chartreuse verte     Beverages      0.79
-- Steeleye Stout       Beverages      0.77
-- All top results are beverages -- none contain "chai"!
-- Backfill embeddings using sentence-transformers (Python)
from sentence_transformers import SentenceTransformer
import psycopg2, json

model = SentenceTransformer('all-MiniLM-L6-v2')  # 384-dim
conn  = psycopg2.connect(DSN)
cur   = conn.cursor()

cur.execute("SELECT product_id, product_name FROM products")
products = cur.fetchall()

for pid, name in products:
    embedding = model.encode(name).tolist()
    cur.execute(
        "UPDATE products SET embedding = %s WHERE product_id = %s",
        (json.dumps(embedding), pid)
    )

conn.commit()
print(f"Backfilled {len(products)} product embeddings")
