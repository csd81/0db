CockroachDB

Distributed SQL · Raft consensus · Serializable isolation · PostgreSQL-compatible

CockroachDB was founded in 2015 by ex-Googlers who built Spanner. The name reflects its design goal: like a cockroach, it survives anything — node failures, region outages, even an entire cloud provider going down. It implements the full PostgreSQL wire protocol, so your existing Postgres drivers work without changes, but data is transparently sharded across a cluster of nodes using Raft consensus, giving you global active-active distributed SQL with serializable isolation.

Survivor by Design
  • Raft consensus: majority of replicas must agree
  • Lose 1 of 3 replicas → still fully functional
  • Lose 2 of 5 replicas → still fully functional
  • Full region failure → traffic reroutes to surviving regions
  • No replica ever returns stale reads to clients
  • Automatic range rebalancing when nodes join/leave
  • Rolling upgrades with zero downtime

Relational + Distribution 
-- Standard PostgreSQL DDL
CREATE TABLE orders (
  order_id   UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  customer   STRING NOT NULL,
  region     STRING NOT NULL,
  total      DECIMAL(12,2),
  created_at TIMESTAMPTZ DEFAULT now(),
  INDEX idx_customer (customer),
  INDEX idx_region_created (region, created_at)
);

-- Geo-partitioning: data stays in its region
ALTER TABLE orders PARTITION BY LIST (region) (
  PARTITION us_east VALUES IN ('us-east-1'),
  PARTITION us_west VALUES IN ('us-west-2'),
  PARTITION eu      VALUES IN ('eu-west-1')
);

-- Pin EU data to EU nodes (GDPR compliance!)
ALTER PARTITION eu OF INDEX orders@primary
  CONFIGURE ZONE USING
    num_replicas = 3,
    constraints = '[+region=eu-west-1]',
    lease_preferences = '[[+region=eu-west-1]]';
Multi-Region Patterns 
-- Multi-region database setup
ALTER DATABASE ecommerce
  SET PRIMARY REGION "us-east-1";
ALTER DATABASE ecommerce
  ADD REGION "eu-west-1";
ALTER DATABASE ecommerce
  ADD REGION "ap-northeast-1";

-- Global table: replicated to all regions
-- Fast reads everywhere, slower writes
ALTER TABLE product_catalog SET locality GLOBAL;

-- Regional table: home region is leaseholder
-- Low latency reads/writes in home region
ALTER TABLE orders
  SET LOCALITY REGIONAL BY ROW AS region;
-- CockroachDB automatically routes based on
-- the 'region' column value

-- Follower reads: read slightly stale data
-- from local replica (no cross-region round trip)
SELECT * FROM orders
AS OF SYSTEM TIME follower_read_timestamp()
WHERE region = 'eu-west-1' AND customer_id = $1;
PostgreSQL-Compatible SQL 
-- Serializable transaction (ACID)
BEGIN;
  SELECT balance FROM accounts WHERE id = $1 FOR UPDATE;
  UPDATE accounts SET balance = balance - 100 WHERE id = $1;
  UPDATE accounts SET balance = balance + 100 WHERE id = $2;
COMMIT;
-- If concurrent txn touches same rows: one will retry

-- SAVEPOINT for nested logic
BEGIN;
  SAVEPOINT cockroach_restart;
  -- ... operations ...
  RELEASE SAVEPOINT cockroach_restart;
COMMIT;

-- AS OF SYSTEM TIME: time-travel reads
SELECT * FROM orders
AS OF SYSTEM TIME '-10m'
WHERE region = 'us-east-1';

-- SHOW RANGES: see how data is distributed
SHOW RANGES FROM TABLE orders;
-- Shows: start/end key, replicas, leaseholder node

-- Distributed query plan
EXPLAIN (DISTSQL)
SELECT region, COUNT(*), SUM(total)
FROM orders
WHERE created_at > now() - INTERVAL '30d'
GROUP BY region;

-- Output shows:
-- TableReader → distributed across nodes
-- Aggregator → partial agg on each node
-- Final Aggregator → merge on gateway
-- Each node processes its local ranges

-- Force index usage
SELECT * FROM orders@idx_region_created
WHERE region = 'eu-west-1'
  AND created_at > now() - INTERVAL '7d'
ORDER BY created_at DESC LIMIT 100;
Real-World Deployments
Bose: Migrated from MySQL to CockroachDB for global inventory management. Active-active in 3 regions — retail systems in US, EU, and Asia all write to their nearest region with no cross-region latency penalty.
Freddie Mac: Financial transactions require serializable isolation. CockroachDB's guarantee that no transaction reads stale data is critical for mortgage processing correctness.
Shipt: Kubernetes-native deployment. CockroachDB's Operator for k8s provides declarative cluster management, rolling upgrades, and automated backups in containerized environments.

Raft + HLC

Raft consensus: Each 512MB range has 3 or 5 replicas. The leaseholder replica receives all reads/writes for that range. It proposes changes to followers, which acknowledge; once a majority acknowledges, the change is committed. Losing the leaseholder triggers automatic election.

Hybrid Logical Clock (HLC): CockroachDB uses HLC (wall clock + logical component) to establish a total order of events across nodes without requiring synchronized clocks. This enables serializability across a geographically distributed cluster.

RocksDB storage: Each node stores its replica data in RocksDB (an LSM-based store, C++ version of LevelDB). SSTs, compaction, and Bloom filters handle the storage layer efficiently.

Operations & Monitoring 
-- Cluster status
SELECT node_id, address, is_live, replicas_leaders
FROM crdb_internal.kv_node_status;

-- Active transactions
SELECT * FROM [SHOW TRANSACTIONS];

-- Find hot ranges (high QPS)
SELECT range_id, store_id, lease_holder_node_id,
  queries_per_second
FROM crdb_internal.ranges_no_leases
ORDER BY queries_per_second DESC LIMIT 10;

-- Zone config: control replication
ALTER TABLE orders CONFIGURE ZONE USING
  num_replicas = 5,
  gc.ttlseconds = 90000;

-- Backup to S3 (Enterprise)
BACKUP DATABASE ecommerce
  INTO 's3://my-bucket/crdb-backup?AUTH=implicit'
  AS OF SYSTEM TIME '-10s';
