TimescaleDB

PostgreSQL extension · Hypertables · Full SQL for time-series

Founded in 2017, Timescale Inc. built TimescaleDB as a PostgreSQL extension — not a fork, not a replacement. Every Postgres tool (pg_dump, JDBC drivers, PostGIS, foreign data wrappers) works unchanged. Hypertables automatically partition time-series data into chunks while exposing a single table interface. Continuous aggregates provide real-time materialized views refreshed automatically. Native compression achieves 90–95% storage reduction.

The "Full SQL for Time-Series" Bet

Purpose-built time-series databases (InfluxDB, Prometheus) sacrifice SQL for speed. TimescaleDB makes the opposite bet: time-series IS relational. IoT readings JOIN to device metadata tables. Financial ticks JOIN to instrument reference tables. Log events JOIN to user tables. A custom query language breaks that.

Hypertables look like regular PostgreSQL tables but are automatically partitioned into chunks by time interval. Each chunk is a real PostgreSQL table — this means chunk exclusion via table constraint exclusion, no special query planner needed. Grafana Cloud uses TimescaleDB as its metrics backend for millions of series. 

-- Create hypertable from existing table
CREATE TABLE metrics (
    time        TIMESTAMPTZ NOT NULL,
    device_id   TEXT NOT NULL,
    metric_name TEXT NOT NULL,
    value       DOUBLE PRECISION
);

-- Convert to hypertable, partition by 1 day
SELECT create_hypertable('metrics', 'time',
    chunk_time_interval => INTERVAL '1 day');

-- Add space partitioning by device_id (8 hash buckets)
SELECT add_dimension('metrics', 'device_id', number_partitions => 8);

-- Query exactly like regular PostgreSQL
SELECT time, value
FROM metrics
WHERE device_id = 'sensor-42'
  AND time > NOW() - INTERVAL '24 hours'
ORDER BY time DESC;

Hypertable Chunk Architecture 
-- Hypertable: single logical table, many physical chunks
SELECT show_chunks('metrics');
-- _timescaledb_internal._hyper_1_1_chunk  (2024-01-01 to 2024-01-02)
-- _timescaledb_internal._hyper_1_2_chunk  (2024-01-02 to 2024-01-03)
-- _timescaledb_internal._hyper_1_3_chunk  (2024-01-03 to 2024-01-04)

-- Each chunk IS a regular PostgreSQL table
\d _timescaledb_internal._hyper_1_1_chunk
-- Same columns as metrics table + CHECK constraint on time range

-- Chunk exclusion: query only touches relevant chunks
EXPLAIN SELECT * FROM metrics
WHERE time BETWEEN '2024-01-02' AND '2024-01-03';
-- → Scans ONLY _hyper_1_2_chunk (one chunk, not all 365)

-- Continuous Aggregates (real-time materialized views)
CREATE MATERIALIZED VIEW metrics_hourly
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 hour', time) AS bucket,
    device_id,
    AVG(value)   AS avg_value,
    MAX(value)   AS max_value,
    MIN(value)   AS min_value,
    COUNT(*)     AS sample_count
FROM metrics
GROUP BY bucket, device_id;

-- Auto-refresh policy: refresh every 30 min, lag 1h
SELECT add_continuous_aggregate_policy('metrics_hourly',
    start_offset => INTERVAL '3 hours',
    end_offset   => INTERVAL '1 hour',
    schedule_interval => INTERVAL '30 minutes');
Native Compression & Retention 
-- Enable compression on hypertable
ALTER TABLE metrics SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_id',
    timescaledb.compress_orderby   = 'time DESC'
);

-- Auto-compress chunks older than 7 days
SELECT add_compression_policy('metrics',
    compress_after => INTERVAL '7 days');

-- Check compression ratio
SELECT pg_size_pretty(before_compression_total_bytes) AS before,
       pg_size_pretty(after_compression_total_bytes)  AS after,
       ROUND(compression_ratio::numeric, 2)           AS ratio
FROM hypertable_compression_stats('metrics');
-- before: 15 GB  |  after: 780 MB  |  ratio: 20.5x

-- Retention policy: drop data older than 90 days
SELECT add_retention_policy('metrics',
    drop_after => INTERVAL '90 days');

-- Tiered storage (Timescale Cloud): move old chunks to S3
SELECT add_tiering_policy('metrics',
    tiering_after => INTERVAL '30 days');
Time-Series SQL Hyperfunctions 
-- ── 1. time_bucket() — equivalent to PromQL's step
SELECT time_bucket('5 minutes', time) AS bucket,
       device_id,
       AVG(value) AS avg_temp
FROM metrics
WHERE metric_name = 'temperature'
  AND time > NOW() - INTERVAL '24 hours'
GROUP BY bucket, device_id
ORDER BY bucket DESC;

-- ── 2. first() / last() — first/last value in a group
SELECT time_bucket('1 hour', time) AS bucket,
       device_id,
       first(value, time) AS open,   -- first value chronologically
       last(value, time)  AS close   -- last value chronologically
FROM metrics
WHERE metric_name = 'price'
GROUP BY bucket, device_id;

-- ── 3. interpolate() — fill gaps in sparse series
SELECT time_bucket_gapfill('1 hour', time) AS bucket,
       device_id,
       interpolate(AVG(value))              AS interpolated_temp
FROM metrics
WHERE device_id = 'sensor-1'
  AND time BETWEEN '2024-01-01' AND '2024-01-02'
GROUP BY bucket, device_id;

-- ── 4. locf() — last observation carried forward
SELECT time_bucket_gapfill('15 minutes', time) AS bucket,
       device_id,
       locf(last(value, time)) AS carried_forward
FROM metrics
WHERE device_id = 'sensor-1'
GROUP BY bucket, device_id;

-- ── 5. histogram() — value distribution
SELECT device_id,
       histogram(value, 0, 100, 10) AS temp_distribution
FROM metrics
WHERE metric_name = 'cpu_percent'
  AND time > NOW() - INTERVAL '1 hour'
GROUP BY device_id;

-- ── 6. JOIN with relational metadata (TimescaleDB's killer advantage)
SELECT m.time,
       m.value AS temperature,
       d.location,
       d.owner,
       d.firmware_version
FROM metrics m
JOIN devices d ON m.device_id = d.id
JOIN device_groups g ON d.group_id = g.id
WHERE g.name = 'manufacturing-floor-a'
  AND m.metric_name = 'temperature'
  AND m.time > NOW() - INTERVAL '6 hours'
  AND m.value > d.temperature_threshold   -- alert threshold from metadata
ORDER BY m.time DESC;

-- ── 7. Approximate percentile with tdigest
SELECT device_id,
       approx_percentile(0.50, percentile_agg(value)) AS p50,
       approx_percentile(0.95, percentile_agg(value)) AS p95,
       approx_percentile(0.99, percentile_agg(value)) AS p99
FROM metrics
WHERE metric_name = 'response_time_ms'
  AND time > NOW() - INTERVAL '1 day'
GROUP BY device_id;

-- ── 8. Moving average with window functions (standard SQL)
SELECT time,
       device_id,
       value,
       AVG(value) OVER (
           PARTITION BY device_id
           ORDER BY time
           ROWS BETWEEN 11 PRECEDING AND CURRENT ROW
       ) AS moving_avg_12
FROM metrics
WHERE device_id = 'sensor-42'
ORDER BY time DESC;
Killer Use Cases
Infrastructure Metrics (Grafana Cloud)

Grafana Cloud uses TimescaleDB as its metrics backend. Millions of metric series ingested via Prometheus remote_write. SQL lets analysts write custom dashboards impossible in PromQL.

Financial Tick Data with Relational Joins

Trading systems store tick-by-tick OHLCV data and JOIN to instrument metadata, corporate actions, and index memberships — all in one SQL query. InfluxDB or Prometheus can't do this.

IoT & Industrial Monitoring

Factory floors: sensor readings JOIN to machine specs, maintenance schedules, and production orders. Outscale (European cloud) uses TimescaleDB for infrastructure KPI tracking.

# Because chunks are real Postgres tables:
- pg_dump / pg_restore works on individual chunks
- Logical replication replicates chunks
- pg_partman / Citus can co-exist
- All pg extensions work (PostGIS, pgvector, etc.)
Timescale Cloud: Tiered Storage 
# Tiered storage: hot → cool → archive
hot tier  (SSD, recent data)    ← default storage
cool tier (S3, compressed)      ← tiering policy
archive   (Glacier/Deep Archive) ← lifecycle rules

# Data automatically moves based on age
# Queries transparently span both tiers:
SELECT avg(value)
FROM metrics
WHERE time > NOW() - INTERVAL '365 days'  -- spans hot+cool
GROUP BY time_bucket('1 day', time);
-- TimescaleDB fetches S3 chunks as needed

# Distributed hypertables (multi-node)

SELECT create_distributed_hypertable(

);
