🎵 Spotify on GCP — Cloud Workflow

Spotify migrated from AWS to GCP in 2016. Every play, skip, and like is a streaming event that flows through Cloud Pub/Sub → Dataflow → BigQuery → Vertex AI. Audio is served from Cloud Storage via CDN Edge nodes near the listener. Click a scenario to see the data flow animated live.

Cloud Pub/Sub — Event Bus
Fully managed, serverless message broker. Publishers push messages to a topic; subscribers pull from a subscription. Decouples producers from consumers — the play-events service doesn't know or care who processes the events downstream. Delivers at-least-once; deduplication via message_id on the consumer. Spotify ingests ~1.4 billion events per day through Pub/Sub topics.
Cloud Dataflow (Apache Beam)
Unified stream and batch processing engine. A Beam pipeline is a DAG of transforms (ParDo, GroupByKey, Combine). The Dataflow runner auto-scales worker VMs, handles watermarks for late-arriving events, and writes results to BigQuery via streaming inserts. Spotify uses Dataflow for real-time play aggregation and Discover Weekly batch jobs.
BigQuery — Analytics Warehouse
Serverless columnar data warehouse. Streaming inserts land in a buffer; within seconds they are queryable. Spotify stores the full play history (petabyte-scale) in BigQuery. Columnstore + partition pruning (by date) let analysts run aggregate queries over billions of rows in seconds. Powers both real-time dashboards and weekly ML batch jobs.
Vertex AI — ML Platform
Manages the full ML lifecycle: data prep, training (on TPUs/GPUs), experiment tracking, model registry, and online prediction endpoints. Spotify trains collaborative-filtering and two-tower retrieval models for Discover Weekly on Vertex AI Training jobs, then deploys prediction endpoints that Cloud Run calls at recommendation time. Inference latency target: <50ms p99.
Cloud Storage + Cloud CDN — Audio Delivery
Audio files are stored in GCS buckets in multiple formats (OGG 320kbps, AAC 256kbps, etc.) transcoded by Cloud Functions on upload. Cloud CDN caches popular tracks at Points of Presence (PoPs) near listeners — 96%+ of playback requests are CDN cache hits. On an album drop, Spotify pre-warms CDN edges with the new tracks before release time to avoid a "thundering herd" against GCS origin servers.
Cloud Bigtable — User Profiles
Wide-column NoSQL store designed for sub-10ms reads at any scale. Spotify uses Bigtable to store per-user feature vectors (listening history summaries, genre affinity, recent skips). When Cloud Run needs to score recommendations, it reads the user's feature vector from Bigtable (single row key lookup) in <5ms, then calls the Vertex AI endpoint with that vector to get ranked track IDs.
Cloud Run — Recommendations API
Stateless containerised service that scales to zero between requests. Receives user context → reads Bigtable profile → calls Vertex AI online prediction → filters/ranks results → returns track list in JSON. Cloud Run auto-scales horizontally in <1s. No VMs to manage, billed per 100ms of CPU. The polyglot pattern: one API call orchestrates four different storage/compute systems.
Discover Weekly — Weekly Batch Pipeline
Every Monday, a Cloud Composer (Airflow) DAG orchestrates:
1. BigQuery export of the week's play history → GCS
2. Dataflow batch job: compute user–song interaction matrix
3. Vertex AI Training: retrain collaborative-filter model
4. Batch prediction: score all (user, track) pairs → 30 tracks per user
5. Write playlists to Bigtable → served by Cloud Run next time the user opens the app
