Tencent TDSQL

Distributed MySQL · WeChat Pay · Financial-grade synchronous replication

TDSQL (Tencent Distributed SQL) was built internally at Tencent in 2012 (VLDB 2022 paper), initially to power WeChat Pay's explosive growth after the 2013 WeChat red packet launch. It became a commercial product on Tencent Cloud in 2019. TDSQL extends MySQL with transparent horizontal sharding, distributed transactions, and MAR (Multi-source Asynchronous Replication) — Tencent's synchronous replication protocol that provides zero-data-loss durability without the latency of Paxos. TDSQL-C is the serverless variant (MySQL and PostgreSQL-compatible, scales to zero).

Born from WeChat Red Packets

In February 2014 (Chinese New Year), WeChat launched digital red packets (hongbao). 800 million red packets were sent in 24 hours — a traffic spike that would have destroyed traditional MySQL. TDSQL's transparent sharding absorbed the load seamlessly. The system processed peak 1.58 million transactions per second during the 2023 Spring Festival.

TDSQL's key innovation is its MAR replication protocol — a synchronous multi-copy replication that guarantees zero data loss (RPO=0) while maintaining near-synchronous latency by pipelining log writes across replicas. This is stricter than MySQL's semisynchronous replication but faster than Paxos consensus.

Transparent Sharding via Proxy 
-- Application connects to TDSQL Proxy (looks like MySQL)
-- Proxy transparently routes to correct shard

-- Shard table (configured via Tencent Cloud console or CLI):
-- Table: payments
-- Shard key: user_id
-- Shard count: 8 (physical MySQL InnoDB instances)

-- Application code: standard MySQL — no sharding awareness
import mysql.connector

conn = mysql.connector.connect(
    host='tdsql-proxy.example.tencent.com',
    port=3306,
    user='appuser', password='...', database='wechat_pay'
)

cursor = conn.cursor()

# Standard MySQL INSERT — proxy routes to shard by user_id hash
cursor.execute("""
    INSERT INTO payments (payment_id, user_id, amount, currency, status)
    VALUES (%s, %s, %s, %s, %s)
""", ('pay-001', 10086, 88.00, 'CNY', 'pending'))

# Standard MySQL SELECT — proxy routes to correct shard
cursor.execute("""
    SELECT payment_id, amount, status, created_at
    FROM payments
    WHERE user_id = %s
    ORDER BY created_at DESC LIMIT 20
""", (10086,))

# Cross-shard query — proxy fans out to all shards, merges results
cursor.execute("""
    SELECT status, COUNT(*), SUM(amount)
    FROM payments
    WHERE created_at > DATE_SUB(NOW(), INTERVAL 24 HOUR)
    GROUP BY status
""")
# Proxy: fans out to 8 shards, aggregates 8 result sets
Global Unique Sequences & XA Transactions 
-- ── Global Unique Sequence (across shards)
-- Standard AUTO_INCREMENT only unique within one shard
-- TDSQL provides globally unique IDs via sequence service

CREATE SEQUENCE payment_seq
    START WITH 1
    INCREMENT BY 1
    MINVALUE 1
    CACHE 1000;  -- pre-allocate 1000 IDs to reduce coordination

-- Use in application:
INSERT INTO payments (payment_id, user_id, amount)
VALUES (NEXT VALUE FOR payment_seq, 10086, 88.00);

-- ── Distributed Transaction (XA compatible)
-- Single-shard transactions: standard MySQL ACID (no overhead)
-- Cross-shard transactions: TDSQL's enhanced 2PC (XA + MAR)

XA START 'txn-001';
-- Shard 1: deduct from sender's account (user_id=10086)
UPDATE accounts SET balance = balance - 100 WHERE user_id = 10086;
-- Shard 5: add to receiver's account (user_id=99999)
UPDATE accounts SET balance = balance + 100 WHERE user_id = 99999;
XA END 'txn-001';
XA PREPARE 'txn-001';
-- MAR: all replicas ack prepare before COMMIT
XA COMMIT 'txn-001';

-- ── Global binlog (for downstream replication/CDC)
-- TDSQL aggregates per-shard binlogs into global binlog stream
-- Applications can subscribe to global binlog via DTS (Data Transfer Service)
-- Used for: ElasticSearch sync, Redis cache invalidation, audit logging
MySQL-Compatible SQL — Financial Use Patterns 
-- ── 1. Idempotent INSERT (critical for payment systems)
INSERT INTO payments (payment_id, user_id, amount, status)
VALUES ('pay-uuid-001', 10086, 88.00, 'pending')
ON DUPLICATE KEY UPDATE
    status = IF(status = 'pending', 'pending', status);
    -- Prevents double-payment: if payment_id exists, no-op

-- ── 2. Payment state machine with optimistic locking
-- version column prevents concurrent state transitions
UPDATE payments
SET status = 'completed',
    completed_at = NOW(),
    version = version + 1
WHERE payment_id = 'pay-uuid-001'
  AND status = 'pending'
  AND version = 3;
-- Check rows affected: 0 = conflict (someone else updated), retry

-- ── 3. Read-write split (TDSQL proxy handles automatically)
-- Writes → primary shard (strong consistency)
-- Reads with hint → replica (eventual consistency, lower latency)
SELECT /*+ TDSQL_SLAVE() */ balance FROM accounts WHERE user_id = 10086;
-- Routes to replica — balance may be 100ms stale (acceptable for display)

-- ── 4. Window functions for financial analytics
SELECT user_id,
       payment_date,
       amount,
       SUM(amount)  OVER (PARTITION BY user_id ORDER BY payment_date) AS running_total,
       AVG(amount)  OVER (PARTITION BY user_id ORDER BY payment_date ROWS 6 PRECEDING) AS weekly_avg,
       ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY amount DESC) AS spend_rank
FROM payments
WHERE payment_date >= DATE_SUB(CURDATE(), INTERVAL 30 DAY);

-- ── 5. JSON for flexible transaction metadata
INSERT INTO transactions (txn_id, user_id, metadata) VALUES (
    'txn-001', 10086,
    JSON_OBJECT(
        'merchant', 'Starbucks',
        'category', 'food',
        'location', JSON_OBJECT('city', 'Shenzhen', 'lat', 22.54, 'lng', 114.06),
        'tags', JSON_ARRAY('coffee', 'breakfast')
    )
);
SELECT JSON_UNQUOTE(JSON_EXTRACT(metadata, '$.merchant')) AS merchant
FROM transactions
WHERE user_id = 10086
  AND JSON_EXTRACT(metadata, '$.category') = '"food'";
Killer Use Cases
WeChat Pay (1.2B users)

All WeChat Pay transactions run on TDSQL. Spring Festival 2023: 1.58M TPS peak, 10B+ red packets in one day. Zero data loss across primary/replica failover events.

Postal Savings Bank of China

PSB (600M customers, largest retail bank by customer count) uses TDSQL for core banking system. Replaced Oracle RAC with TDSQL across hundreds of provincial branches.

China Merchants Bank

CMB uses TDSQL for credit card processing and online banking. Key requirement: synchronous replication — no data loss acceptable for financial transactions. TDSQL MAR delivers RPO=0.
