The Distributed Consensus Problem

A single-node database is simple: one writer, one truth. The moment you add a second node for fault tolerance, every node must agree on the same sequence of writes — even when messages are lost, delayed, or nodes crash.

Node A writesNode B readsResult without consensus
INSERT order 999 SELECT * FROM orders May return 0 or 1 rows depending on timing
Two nodes accept conflicting updates — Split brain — two different "truths"

What Can Go Wrong Without Consensus
Split Brain
Two nodes become leaders simultaneously. Both accept writes. Data diverges permanently.
Phantom Read
A committed write appears on one node but not others. Read returns stale answer.
Write Loss
Leader fails after ack but before majority replication. Data disappears after failover.

Proof-of-Work Mechanism
  1. Node that wants to add data must solve a cryptographic puzzle: find a nonce such that SHA256(payload + nonce) starts with N zeros
  2. First solver broadcasts the solution
  3. All nodes verify (cheap) and accept the block
Mining latency is a security feature for Bitcoin. For OLTP it is catastrophic. Ch.4 simulation showed VMs racing to find the nonce — that latency is your write latency in PoW.
Raft Core Guarantee

Raft (Ongaro & Ousterhout, 2014) was designed to be understandable. It solves consensus for crash-fault-tolerant clusters of known peers.

"If a log entry is committed, every future leader will also have that entry."
This prevents the "revert accepted write" failure mode
Fault Tolerance Formula
With N nodes, Raft tolerates up to ⌊(N−1)/2⌋ simultaneous failures.
3 nodes → 1 failure  |  5 nodes → 2 failures  |  7 nodes → 3 failures

There is exactly one leader at a time, or zero during election.
Leader Election
  1. Each follower has a randomized election timeout (150–300 ms)
  2. On timeout: transition to Candidate, increment term, send RequestVote RPCs
  3. A node votes YES if:
    — has not voted in this term AND
    — candidate log ≥ own log
  4. Majority of votes → becomes new leader for this term
  5. Leader sends immediate heartbeat to suppress new elections
Randomized timeouts prevent all followers from calling elections simultaneously and splitting votes forever.

  |          |--Notify------------->|
Write committed once majority of nodes have it in their log — even if some followers are slow or disconnected.
Term Counter — The Logical Clock

Raft uses a monotonically increasing term number as a logical clock. Each term begins with an election. Terms are used to detect stale leaders:

  • If a node receives a message from a lower term → reject
  • If a node discovers a higher term → step down to follower
  • Prevents "zombie leaders" from confusing the cluster after recovering from partition
Term 1: Node1 elected leader
Term 1: Node1 replicates 10 writes
Term 1: Node1 crashes
Term 2: Node2 wins election (timeout)
Term 2: Node2 becomes leader
Term 2: Node1 rejoins as follower
Term 2: Node1 catches up the log

HTTP API Quick Reference
READ (query)
# Strong consistency (leader only):
curl 'http://localhost:4001/db/query?q=SELECT+*+FROM+orders+LIMIT+5'

# Weak consistency (any node, possibly stale):
curl 'http://localhost:4003/db/query?q=SELECT+COUNT(*)+FROM+orders&level=weak'
WRITE (execute)
curl -XPOST http://localhost:4001/db/execute \
  -H 'Content-Type: application/json' \
  -d '["INSERT INTO orders VALUES (10999,\"ALFKI\",\"Germany\",999.99)"]'

# Response:

Flask Integration (Python requests)
{% raw %} 
import requests

class RqliteClient:
    def __init__(self, url):
        self.url = url.rstrip('/')

    def query(self, sql):
        r = requests.get(f'{self.url}/db/query',
                         params={'q': sql, 'level': 'weak'})
        r.raise_for_status()
        result = r.json()['results'][0]
        return result.get('columns', []), result.get('values', []) or []

    def execute(self, sql):
        r = requests.post(f'{self.url}/db/execute', json=[sql])
        r.raise_for_status()
        return r.json()['results'][0]
