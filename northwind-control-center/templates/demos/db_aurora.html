Amazon Aurora

MySQL/PostgreSQL-compatible · 6-copy quorum storage · 5× MySQL throughput

Launched by AWS in 2014 (SIGMOD 2017 paper), Aurora decouples compute from storage. Instead of shipping redo log pages to replicas, Aurora ships only the redo log itself — the storage layer reconstructs pages on demand. This architecture enables instant failover, crash recovery in under 30 seconds, and up to 128 TB of auto-scaling storage without any configuration. Aurora is MySQL or PostgreSQL from the application's perspective, but with fundamentally different internals.

The Log IS the Database

Traditional MySQL ships full data pages to replicas — 7–8x write amplification. Aurora ships only the redo log records to 6 storage nodes across 3 AZs. Storage nodes reconstruct pages themselves. The result: 1/7th the I/O of standard MySQL at equivalent throughput.

Crash recovery doesn't need to replay the full redo log — the storage nodes already have the materialized pages. Restart takes under 30 seconds even from a cold crash. Airbnb migrated away from MySQL clusters to Aurora and cut their DBA overhead dramatically.

-- Pushes down filter/projection to storage nodes

MySQL/PostgreSQL SQL — Zero Application Changes 
-- Aurora MySQL: same SQL as MySQL 8.0
-- ── 1. Standard OLTP
SELECT o.order_id, c.email, o.total
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'pending'
  AND o.created_at > NOW() - INTERVAL 24 HOUR
ORDER BY o.created_at DESC
LIMIT 100;

-- ── 2. Window functions (MySQL 8.0 / Aurora MySQL 3)
SELECT customer_id,
       order_date,
       order_total,
       SUM(order_total) OVER (
           PARTITION BY customer_id
           ORDER BY order_date
           ROWS UNBOUNDED PRECEDING
       ) AS running_total
FROM orders;

-- ── 3. JSON columns
INSERT INTO sessions (session_id, metadata) VALUES (
    'sess-123',
    JSON_OBJECT('ip', '1.2.3.4', 'ua', 'Chrome/120', 'features', JSON_ARRAY('a', 'b'))
);
SELECT JSON_EXTRACT(metadata, '$.ip') FROM sessions WHERE session_id = 'sess-123';

-- ── 4. Aurora PostgreSQL: same as PostgreSQL 16
-- Full CTE, window functions, JSONB, PostGIS, pgvector all work
CREATE EXTENSION IF NOT EXISTS vector;
ALTER TABLE products ADD COLUMN embedding vector(1536);
CREATE INDEX ON products USING ivfflat (embedding vector_cosine_ops);

SELECT name, 1 - (embedding <=> $1::vector) AS similarity
FROM products
ORDER BY embedding <=> $1::vector
LIMIT 10;
When to Choose Aurora
Migrating MySQL/PostgreSQL to AWS

Zero application changes. AWS DMS (Database Migration Service) handles the migration. Airbnb, Pinterest, Netflix all run Aurora as their primary OLTP database.

Variable Traffic (Serverless v2)

SaaS products with unpredictable load patterns. Scales from 0.5 ACU to 128 ACU in milliseconds. Pay only for capacity used, not provisioned.

Multi-Region Active-Active

Aurora Global Database provides cross-region read replicas with <1s replication lag. Failover to secondary region in ~1 minute for DR.

Aurora Storage Architecture 
┌──────────────────────────────────────────────────────────────────┐
│  Compute Layer (EC2 instances)                                   │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────────┐      │
│  │   Writer       │  │  Read Replica  │  │  Read Replica  │      │
│  │  (Primary)     │  │     (AZ-b)     │  │     (AZ-c)     │      │
│  │  MySQL/PG      │  │  MySQL/PG      │  │  MySQL/PG      │      │
│  └────────┬───────┘  └───────┬────────┘  └───────┬────────┘      │
│           │  redo log only   │                    │              │
│           │  (not pages)     └──────────────────┘                │
│           ▼                                                      │
│  Aurora Storage Layer (distributed, shared)                      │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  AZ-1: Storage Node A  ●  Storage Node B                │     │
│  │  AZ-2: Storage Node C  ●  Storage Node D                │     │
│  │  AZ-3: Storage Node E  ●  Storage Node F                │     │
│  │                                                         │     │
│  │  Write quorum: 4/6 nodes must acknowledge               │     │
│  │  Read quorum:  3/6 nodes (can tolerate 2 failures)      │     │
│  │  Protection Group: 10 GB segment, 6-way replicated      │     │
│  └─────────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────────┘

Key insight: Writer sends redo log to 6 storage nodes.
Storage nodes reconstruct pages independently.
→ Writer does NOT wait for page reconstruction (only log ack)
→ Read replicas fetch pages from storage, not from writer
→ No replica lag from page shipping — only metadata lag (~10ms)
