Fan-out Strategy: Write vs Read (The Celebrity Problem)
✅ Fan-out on WRITE (Normal Users)
User with <10K followers. When they post, the fan-out service pushes the post ID into each follower's pre-computed feed cache (Redis list). When a follower opens their app, the feed is instantly available — just read from cache. Very fast reads.
Latency: Read O(1) from cache
Write cost: O(followers) Redis writes on each post
Max followers: ~1M before it becomes too expensive
⚡ Fan-out on READ (Celebrities)
Beyoncé has 300M followers. Pushing her post to 300M Redis entries would take 30 minutes. Instead: her post is stored in the graph (TAO). When any of her followers opens their feed, the aggregator pulls Beyoncé's latest post at read time and merges it into the ranked feed. Slower reads, but write cost is O(1).
Latency: Read requires TAO lookup + merge
Write cost: O(1) — just store the post
Threshold: Meta uses ~1M follower cutoff
🗄️ Storage & Data Layer
TAO (The Associations and Objects) — Meta
Meta's custom graph DB. Every entity (User, Post, Page) is an Object; every relationship (Like, Follow, FriendOf) is an Association. Read-through cache tier using Memcached. Stores 100+ trillion edges. Globally replicated to 20+ data centers.
Apache Cassandra
Timeline/feed storage. Wide-row per user: partition(user_id) → sorted(post_timestamp, post_id). AP system — never refuses a write. Used by Instagram, Twitter. ~1M writes/second at Twitter scale.
RocksDB (embedded KV)
LSM-tree key-value store. Optimized for write-heavy workloads. Used inside Cassandra, Kafka, and as a standalone store at Meta/LinkedIn. Handles billions of tiny updates efficiently.
Memcached + Mcrouter
Meta's distributed cache. Mcrouter is a Memcached proxy that handles routing, replication, and failure. Facebook serves ~70% of requests entirely from Memcached — never touching MySQL.
⚙️ Compute & Processing
Apache Kafka
The nervous system of social platforms. Topics: user-posts, notifications, feed-updates, analytics-events. Partitioned by user_id for ordered per-user processing. LinkedIn originated Kafka; now used by every major social platform.
Apache Storm / Heron
Stream processing. Heron (Twitter's replacement for Storm) processes 10M+ events/second. Counts likes, trending topics (every 30 seconds), and notification coalescing ("X and 47 others liked your post").
Feed Ranking ML (PyTorch / TF)
EdgeRank (Facebook's old algorithm) replaced by deep neural nets. Features: edge weight (engagement history), affinity score (how often you interact), time decay, media type. Re-ranked on every feed open. LinkedIn's PYMK ("People You May Know") uses graph traversal + ML.
Hack / PHP / Erlang
Facebook: Hack (PHP superset). WhatsApp: Erlang/OTP (actor model, 2M connections per server). These language choices drive core architectural decisions about concurrency and state management.
🌐 Infrastructure & Protocols
Custom Private Data Centers (Meta)
Meta's Prineville OR, Forest City NC, and Luleå Sweden DCs. Open Compute Project (OCP) hardware — custom server designs. Gravity-fed cooling. PUE (Power Usage Effectiveness) of 1.07 — near theoretical minimum. All powered by renewable energy.
TPOP (Traffic PoPs)
Meta's custom edge nodes placed in ISPs. Handle TLS termination, HTTP/2 multiplexing, and static asset caching. Similar to Netflix Open Connect but for social content (images, JS bundles).
WebSocket / Long Polling
Real-time updates (likes, comments appearing live) use WebSocket connections. Server pushes notification events. Mobile apps use FCM/APNs push notifications when WebSocket not maintained. "Instant" reactions: <200ms end-to-end.
Global BGP Anycast
Meta and LinkedIn operate their own AS (Autonomous Systems). BGP Anycast routes users to nearest PoP. When Cloudflare had a BGP incident in 2021, Meta used their own backbone — insulation from 3rd-party CDN failures.
Workflow Deep Dive
1
User Posts: Write to MySQL + Event Published
Post written to MySQL (primary source of truth) with ACID transaction. Post ID auto-incremented (Snowflake ID at scale — 64-bit: timestamp + datacenter + sequence). Media (images/video) uploaded separately to CDN. Post metadata published to Kafka topic user-posts with partition key = user_id.
2
Kafka Fan-out: One Message, Many Consumers
Multiple consumer groups read the same Kafka message independently: Notification service (fans out push notifications), Feed update service (pushes to follower caches), Analytics service (tracks engagement), Search indexer (indexes post content). Kafka retains messages for 7 days — consumers can replay.
3
TAO Graph Update: Associations Written
Fan-out service reads follower list from TAO (getAssocRange(user_id, FOLLOWS, 0, 1000) — paginated). For each follower: prepend post_id to their feed timeline in Cassandra. TAO writes new Association: (user_id) → [POSTED] → (post_id). This powers "posts by people you follow" graph traversal.
4
Feed Assembly: ML Ranking
When user opens feed: fetch ~500 candidate posts from Cassandra timeline (mix of fan-out-on-write and celebrity TAO lookups). Run deep learning ranker over candidates (PyTorch model, ~2ms inference). Score = f(affinity, engagement_history, recency, diversity). Return top-50. Diversity: no 2 consecutive posts from same author.
📊 Read:Write Ratio
Social platforms are 99% reads, 1% writes. Every write (post, like, comment) triggers 100–1000 cache updates. This is why Memcached is the dominant cost — Facebook's Memcached fleet processes 1 billion requests/second. This ratio drives every architectural decision.
🔗 Why Graph DB?
"Friend of a friend" queries in SQL: SELECT f2.user_id FROM Friends f1 JOIN Friends f2 ON f1.friend_id=f2.user_id WHERE f1.user_id=? — O(N²) joins at scale. Graph traversal is O(edges traversed). TAO handles 1 billion graph reads/second that would be impossible in MySQL.
🎯 CAP Position
AP system. Stale like counts, slightly delayed notifications — all acceptable. TAO is eventually consistent: replicas may lag by <1 second. This means your friend might not see your comment immediately in another country. Strict consistency would require cross-datacenter coordination on every request — unacceptable latency.
