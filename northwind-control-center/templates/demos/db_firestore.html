Google Cloud Firestore

NoSQL Document · SDK-based queries · Real-time sync · Serverless auto-scaling

Launched in 2017 as the successor to Firebase Realtime Database, Firestore is Google's serverless, globally distributed document database designed specifically for mobile and web applications. Its killer features: real-time listeners (apps update instantly when data changes), offline support (client SDKs cache data locally for offline use), and security rules that let you skip a backend server entirely for simple apps. Firestore auto-scales to zero cost when idle and handles billions of reads per day at Google's own scale. The trade-off for this simplicity: no server-side JOINs, limited query expressiveness, and a document-oriented data model that requires upfront design.

Core Concepts

Collections & Documents — documents live in collections. Every document has a unique ID (auto-generated or custom). Documents can contain subcollections (nested collections).

Field Types — String, Number, Boolean, Timestamp, GeoPoint, Array, Map (nested object), Null, Reference (pointer to another document), Bytes.

1MB document limit — each document is max 1MB. Large arrays or deeply nested maps can hit this. Design with this constraint in mind.

Indexes — single-field indexes created automatically. Composite indexes must be explicitly created for queries on multiple fields. Every query requires an index.

Security Rules — Firestore Security Rules evaluate on the server. Apps can write directly from the browser/mobile with no backend server, using rules to enforce access control.

Security Rules Example 
// firestore.rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Users can read/write only their own document
    match /users/{userId} {
      allow read, write: if request.auth != null
        && request.auth.uid == userId;
    }

    // Orders: owner can read/write, admin can read all
    match /orders/{orderId} {
      allow read: if request.auth.uid == resource.data.userId
        || request.auth.token.admin == true;
      allow create: if request.auth != null
        && request.resource.data.userId == request.auth.uid
        && request.resource.data.total > 0;
      allow update: if request.auth.token.admin == true;
    }

    // Public read for products
    match /products/{productId} {
      allow read: if true;
      allow write: if request.auth.token.admin == true;
    }
  }
}
SDK Query Primer (JavaScript) 
import { getFirestore, collection, doc, addDoc,
  getDoc, getDocs, query, where, orderBy,
  limit, onSnapshot, updateDoc, writeBatch,
  serverTimestamp, increment } from 'firebase/firestore';

const db = getFirestore();

// 1. Add a document (auto-generated ID)
const docRef = await addDoc(collection(db, 'orders'), {
  userId: 'user_42',
  total: 59.99,
  status: 'pending',
  createdAt: serverTimestamp()
});

// 2. Get a single document by ID
const snap = await getDoc(doc(db, 'users', 'user_42'));
if (snap.exists()) console.log(snap.data());

// 3. Query with filters (requires composite index)
const q = query(
  collection(db, 'orders'),
  where('status', '==', 'pending'),
  where('total', '>', 50),
  orderBy('total', 'desc'),
  limit(10)
);
const results = await getDocs(q);
results.forEach(d => console.log(d.id, d.data()));

// 4. Real-time listener — updates push to client
const unsubscribe = onSnapshot(
  query(collection(db, 'orders'),
        where('userId', '==', 'user_42')),
  (snapshot) => {
    snapshot.docChanges().forEach(change => {
      if (change.type === 'added') console.log('New:', change.doc.data());
      if (change.type === 'modified') console.log('Updated:', change.doc.data());
      if (change.type === 'removed') console.log('Deleted:', change.doc.id);
    });
  }
);
// unsubscribe() to stop listening

// 5. Atomic batch write
const batch = writeBatch(db);
batch.set(doc(db, 'orders', 'ord_123'), { status: 'shipped' });
batch.update(doc(db, 'users', 'user_42'), {
  orderCount: increment(1)
});
await batch.commit(); // atomic: all or nothing

// 6. Transaction — read-then-write atomically
import { runTransaction } from 'firebase/firestore';
await runTransaction(db, async (txn) => {
  const userRef = doc(db, 'users', 'user_42');
  const userSnap = await txn.get(userRef);
  const points = userSnap.data().loyaltyPoints;
  txn.update(userRef, { loyaltyPoints: points + 100 });
});
Killer Use Cases

Real-Time Chat & Collaboration — onSnapshot listeners deliver messages to all participants instantly. No polling. Firestore scales this automatically.

Mobile App Backends (no backend server) — Flutter/React Native apps write directly to Firestore with security rules as the authorization layer. Duolingo uses Firebase/Firestore for content delivery.

Gaming State Sync — lobby/matchmaking state, player inventory, leaderboards. Real-time updates across players without a dedicated game server.

Offline-First Apps — the client SDK caches all queries locally. Apps work offline and sync automatically when connectivity resumes. Ideal for field workers.

Limitations

No JOINs — data must be denormalized for query patterns. Server-side JOIN is impossible; client-side multi-collection lookups require multiple reads.

No aggregation queries (historically) — COUNT(*), SUM(), AVG() require client-side accumulation or Cloud Functions triggers (now supported via aggregation queries in newer versions).

Index required for every compound query — if you add a new filter or sort, you must create a composite index first. Can surprise developers used to SQL.

Performance & Pricing

Read latency: <100ms globally

Writes: Strong consistency within region

Free tier: 50K reads, 20K writes/day

Scale: Automatic, no provisioning

Limit: 1 write/sec per document

Internal Architecture

Spanner under the hood — Firestore is built on top of Google Cloud Spanner, inheriting its globally consistent, strongly consistent storage. This gives Firestore its consistency guarantees.

Document storage — documents are stored as structured data in Spanner with indexes maintained automatically. Composite indexes must be explicitly created.

Real-time push — a separate streaming layer maintains WebSocket/gRPC connections to clients and pushes delta updates when documents change. Google manages this at global scale.

Offline cache — client SDKs maintain a local cache (IndexedDB in browsers, SQLite on mobile). Queries run against the cache when offline and sync when online.

Consistency Model

Strong consistency — reads after writes always return the latest data (within a single region/multi-region configuration). This is stronger than DynamoDB's eventual consistency default.

Document-level atomicity — operations on a single document are atomic. Batch writes (up to 500 documents) and transactions provide multi-document atomicity.

Write limit — max 1 sustained write/second per document. Design your data model to avoid "hot documents" (e.g., global counters). Use distributed counters for high-write metrics.
