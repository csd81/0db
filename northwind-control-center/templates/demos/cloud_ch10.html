Chapter 10 — Service Communication: Pub/Sub

Google Pub/Sub · Topics & Subscriptions · Message Flow · Pull/Push Delivery · Java Publish & Subscribe
Why Pub/Sub? Loose Coupling at Scale
Synchronous RPC — Unsuitable
✗ Tight coupling between services
✗ Caller blocks waiting for response
✗ Failure of one service cascades
✗ Does not scale to large distributed apps
Pub/Sub Messaging — The Solution
✓ Loose coupling — services don't know each other
✓ Asynchronous — sender does not block
✓ Fault-tolerant — messages are retained
✓ The only viable strategy for large-scale apps
Google Pub/Sub Characteristics
✓ High-availability, real-time, asynchronous
✓ Low-latency, high-throughput
✓ Available in all Google Cloud regions
✓ Universal messaging & event delivery
Core Components
📢
Publisher
Any app that can make HTTPS requests to pubsub.googleapis.com. Creates topics and sends messages to them.
📌
Topic
A named resource to which publishers send messages. Acts as a logical channel.
🔖
Subscription
A named resource representing the stream of messages from a specific topic to a subscribing app.
📨
Subscriber
Any app that can make HTTPS requests to pubsub.googleapis.com. Receives messages via Pull or Push.
Message
A combination of data and optional attributes that a publisher sends to a topic and that is eventually delivered to subscribers. Attributes are key-value pairs defined by the publisher.

Many-to-One: Subscriber X wants A and B

Same message from C is delivered independently via YC to Y and via ZC to Z.
Pub/Sub supports an N-to-M mapping. Subscribers can subscribe to any number of topics; the same message is delivered to each subscription independently. The mapping is configured via Topic-Subscription configurations.
Message Format

At least one of the first three fields must be present. The last two are added automatically by the Pub/Sub service.

6-Step Message Delivery Flow
1
Publisher creates topic and sends messages with payload + optional attributes.
2
Service retains message on behalf of subscriptions until acknowledged by a subscriber.
3
Pub/Sub forwards message from the topic to all of its subscriptions, individually.
4
Subscriber receives message either by Pub/Sub pushing to an endpoint, or the subscriber pulling from the service.
5
Subscriber sends Ack to Pub/Sub service for each received message.
6
Service removes acknowledged message from the subscription's message queue.
Delivery Semantics & Retention
Default: At-Least-Once
Pub/Sub guarantees that each message is delivered at least once to each subscription. Duplicate deliveries may occur — subscribers must handle idempotency.
7-Day Message Retention
If a message cannot be delivered within 7 days, it is deleted without delivery. If there is no subscription at the time of publication, the message will not be delivered at all.
Exactly-Once with Beam SDK
When Pub/Sub is used with the Apache Beam SDK streaming model, it is possible to achieve Exactly-Once semantics with ordered message delivery.
Background Function Triggered by Pub/Sub

A Cloud Function that receives a Pub/Sub message, Base64-decodes the payload, and logs a personalised greeting. The message data is always Base64-encoded in the PubSubMessage.getData() field.

Java — HelloPubSub
import com.google.cloud.functions.BackgroundFunction;
import com.google.cloud.functions.Context;
import functions.eventpojos.PubSubMessage;
import java.nio.charset.StandardCharsets;
import java.util.Base64;
import java.util.logging.Level;
import java.util.logging.Logger;

public class HelloPubSub implements BackgroundFunction<PubSubMessage> {
    private static final Logger logger = Logger.getLogger(HelloPubSub.class.getName());

    @Override
    public void accept(PubSubMessage message, Context context) {
        String name = "world";
        if (message != null && message.getData() != null) {
            // Message data is Base64-encoded; decode to get the original string
            name = new String(
                Base64.getDecoder().decode(
                    message.getData().getBytes(StandardCharsets.UTF_8)),
                StandardCharsets.UTF_8);
        }
        logger.info(String.format("Hello %s!", name));
        return;
    }
}
Deploy the Function
gcloud functions deploy java-pubsub-function \
    --entry-point functions.HelloPubSub \
    --runtime java11 \
    --memory 512MB \
    --trigger-topic SAMPLE_TOPIC
--trigger-topic creates a subscription to SAMPLE_TOPIC automatically and binds it to the function.
Create the Topic
gcloud pubsub topics create SAMPLE_TOPIC
Publish a Test Message
gcloud pubsub topics publish SAMPLE_TOPIC \
    --message Einstein
Triggers the function → logs "Hello Einstein!". The function is invoked with At-Least-Once semantics.
Base64 Encoding Explained
Pub/Sub encodes all message data as Base64 for safe transport over HTTP. The function must decode the bytes using Base64.getDecoder().decode() and convert to a String using StandardCharsets.UTF_8.
Publishing Messages — Java API with Error Handling
Publishing Steps
1. Retrieve TopicName from project + topic IDs
2. Create Publisher instance bound to topic
3. Build PubsubMessage with ByteString data
4. Call publisher.publish() → returns ApiFuture<String>
5. Register async callback: onSuccess / onFailure
6. Shutdown publisher to free resources
Async Callback Pattern
publish() returns immediately with an ApiFuture<String>. The future resolves to the server-assigned message ID (unique within the topic) when the broker confirms receipt.
Callbacks allow non-blocking error handling: check ApiException.isRetryable() to decide whether to retry on failure.
Java — PublishWithErrorHandlerExample
import com.google.api.core.ApiFuture;
import com.google.api.core.ApiFutureCallback;
import com.google.api.core.ApiFutures;
import com.google.api.gax.rpc.ApiException;
import com.google.cloud.pubsub.v1.Publisher;
import com.google.common.util.concurrent.MoreExecutors;
import com.google.protobuf.ByteString;
import com.google.pubsub.v1.PubsubMessage;
import com.google.pubsub.v1.TopicName;
import java.util.Arrays;
import java.util.List;
import java.util.concurrent.TimeUnit;

public class PublishWithErrorHandlerExample {
    public static void publishWithErrorHandlerExample(String projectId, String topicId)
            throws IOException, InterruptedException {

        TopicName topicName = TopicName.of(projectId, topicId);
        Publisher publisher = null;
        try {
            // Create a publisher bound to the topic
            publisher = Publisher.newBuilder(topicName).build();

            List<String> messages = Arrays.asList("first message", "second message");
            for (final String message : messages) {
                ByteString data = ByteString.copyFromUtf8(message);
                PubsubMessage pubsubMessage =
                    PubsubMessage.newBuilder().setData(data).build();

                // publish() returns immediately; message ID arrives asynchronously
                ApiFuture<String> future = publisher.publish(pubsubMessage);

                // Register success / failure callbacks
                ApiFutures.addCallback(
                    future,
                    new ApiFutureCallback<String>() {
                        @Override
                        public void onFailure(Throwable throwable) {
                            if (throwable instanceof ApiException) {
                                ApiException apiEx = (ApiException) throwable;
                                System.out.println(apiEx.getStatusCode().getCode());
                                System.out.println(apiEx.isRetryable()); // retry hint
                            }
                            System.out.println("Error publishing: " + message);
                        }
                        @Override
                        public void onSuccess(String messageId) {
                            // server-assigned ID unique within the topic
                            System.out.println("Published message ID: " + messageId);
                        }
                    },
                    MoreExecutors.directExecutor());
            }
        } finally {
            if (publisher != null) {
                publisher.shutdown();
                publisher.awaitTermination(1, TimeUnit.MINUTES);
            }
        }
    }
}
Pull vs Push Delivery
Pull Delivery
The subscriber explicitly requests messages from Pub/Sub. The subscriber connects to the service behind the scenes and polls for messages. Suitable for high-throughput processing where the subscriber controls the pace.
Subscriber → request → Pub/Sub
Pub/Sub → message → Subscriber
Subscriber → Ack → Pub/Sub
Push Delivery
Pub/Sub sends the message directly to the subscriber's HTTPS endpoint without waiting for a request. Suitable for event-driven architectures and Cloud Functions triggers.
Pub/Sub → POST to endpoint → Subscriber
Subscriber → HTTP 200 → Pub/Sub (= Ack)
Async Pull Subscriber — Java API
Subscribing Steps
1. Build ProjectSubscriptionName from project + subscription IDs
2. Create MessageReceiver lambda: process message + call consumer.ack()
3. Build Subscriber with subscription name + receiver
4. subscriber.startAsync().awaitRunning() — starts pull loop
5. Run until timeout or error, then stopAsync()
MessageReceiver Lambda
Activated for every received message. Two parameters:
PubsubMessage message — the received message; use getMessageId() and getData().toStringUtf8()
AckReplyConsumer consumer — call consumer.ack() to acknowledge; un-acked messages are redelivered
Java — SubscribeAsyncExample (Pull)
import com.google.cloud.pubsub.v1.AckReplyConsumer;
import com.google.cloud.pubsub.v1.MessageReceiver;
import com.google.cloud.pubsub.v1.Subscriber;
import com.google.pubsub.v1.ProjectSubscriptionName;
import com.google.pubsub.v1.PubsubMessage;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.TimeoutException;

public class SubscribeAsyncExample {
    public static void subscribeAsyncExample(String projectId, String subscriptionId) {
        ProjectSubscriptionName subscriptionName =
            ProjectSubscriptionName.of(projectId, subscriptionId);

        // Lambda-based MessageReceiver: called for each incoming message
        MessageReceiver receiver =
            (PubsubMessage message, AckReplyConsumer consumer) -> {
                System.out.println("Id: "   + message.getMessageId());
                System.out.println("Data: " + message.getData().toStringUtf8());
                consumer.ack();  // acknowledge — removes from subscription queue
            };

        Subscriber subscriber = null;
        try {
            subscriber = Subscriber.newBuilder(subscriptionName, receiver).build();
            subscriber.startAsync().awaitRunning();
            System.out.printf("Listening for messages on %s:\n",
                subscriptionName.toString());
            // Run for 30 seconds (unrecoverable errors exit earlier)
            subscriber.awaitTerminated(30, TimeUnit.SECONDS);
        } catch (TimeoutException timeoutException) {
            // Normal shutdown after the 30s window
            subscriber.stopAsync();
        }
    }
}
Ack Importance
If consumer.ack() is NOT called, Pub/Sub will redeliver the message (At-Least-Once semantics). The message persists in the subscription queue until acknowledged or the 7-day TTL expires.
startAsync Pattern
startAsync().awaitRunning() starts the subscriber as an asynchronous service (Guava ServiceManager pattern). The subscriber connects to Pub/Sub behind the scenes and continuously polls for messages.
