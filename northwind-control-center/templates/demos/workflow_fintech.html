Double-Entry Ledger (Strict Accounting Invariant)
-- Every transaction: debits must equal credits BEGIN TRANSACTION; INSERT INTO ledger_entries (account_id, amount, type) VALUES (acct_customer, -1000.00, 'DEBIT'), -- customer pays (acct_merchant, +970.00, 'CREDIT'), -- merchant receives (acct_stripe_fee, +29.00, 'CREDIT'), -- Stripe fee (acct_tax, +1.00, 'CREDIT'); -- tax -- Invariant check: SUM(debits) + SUM(credits) = 0 IF (SELECT SUM(amount) FROM ledger_entries WHERE txn_id = current_txn_id) != 0 THEN ROLLBACK; RAISE EXCEPTION 'Ledger imbalance'; COMMIT;
This double-entry constraint means money cannot appear or disappear. Every debit has a matching credit. SUM of all accounts across the universe = 0. Required by financial regulators.
FoundationDB (Stripe)
Strict serializability — stronger than snapshot isolation. Every transaction sees a consistent snapshot AND concurrent transactions are totally ordered. Used by Stripe for all financial records. ACID across distributed nodes via optimistic concurrency + multi-version concurrency control.
Google Spanner (Global Fintech)
Externally consistent distributed SQL. Uses TrueTime (atomic clock + GPS receiver in every datacenter) to assign globally ordered commit timestamps. "External consistency" = any transaction that commits after another is guaranteed to have a later timestamp. Used by banks running global operations.
CockroachDB
Postgres-compatible distributed SQL. Geo-partitioned tables: European transactions stay in EU datacenters (GDPR). Serializable isolation default. Used by Nubank (Latin America's largest digital bank) and Coinbase.
🗄️ Storage & Data Layer
FoundationDB
Apple's acquisition (used in iCloud). Stripe's core financial store. Ordered key-value store with multi-key ACID transactions. Key insight: strict serializability means no anomalies even under concurrent transactions. Linearizable reads. Used for Stripe's internal charge and payout ledger.
kdb+ / Q Language (HFT)
Time-series database purpose-built for finance. Column-store optimized for temporal data. Q language: vectorized operations over billions of rows in milliseconds. Every tick from every exchange stored in kdb+. Jane Street, Goldman Sachs, Two Sigma run kdb+. License costs $50K–$500K/year.
InfluxDB / TimescaleDB
Open-source time-series for trading metrics, exchange rate history, and system monitoring. InfluxDB: purpose-built TSM engine, 10M writes/second. TimescaleDB: PostgreSQL extension — familiar SQL + time-series optimizations. Used by crypto exchanges for OHLCV candle data.
Immutable Audit Log
Append-only log (Kafka or custom WAL) where every state change is recorded. Cryptographically chained (each entry includes hash of previous). Required by PCI-DSS, SOX, and banking regulations. Cannot be deleted or modified — only new correcting entries added.
⚙️ Compute & Processing
Apache Kafka (Strictly Ordered)
Financial events require strict ordering: payment initiated → payment authorized → payment captured. Kafka topic partitioned by account_id — all events for one account go to one partition → guaranteed ordering. Cannot use random partitioning (that would interleave events from different transactions).
Apache Flink (Fraud Detection)
Real-time stream processing for anomaly detection. Sliding window: "more than 3 transactions from different countries in 10 minutes → flag". Feature engineering: velocity checks, geo-distance calculations, merchant category risk scoring. ML model served inline with <10ms latency. Stripe Radar ML: 99%+ fraud detection accuracy.
FIX Protocol Engine (HFT)
Financial Information eXchange protocol — the universal language of electronic trading. Ultra-low latency implementations in C++: kernel bypass (DPDK), FPGA-based order routing at nanosecond speeds. Co-location: servers physically placed inside NYSE/NASDAQ data centers (1 microsecond advantage).
CQRS + Event Sourcing
Command Query Responsibility Segregation: write path (ledger entries) separated from read path (account balances, reports). Event sourcing: account balance is derived by replaying all transactions — never stored directly. Allows time-travel queries: "what was this balance on March 15th?" without separate audit tables.
🌐 Security & Compliance
PCI-DSS Compliance
Payment Card Industry Data Security Standard. Card numbers never stored in plaintext — tokenized (Stripe stores; merchants get a token). Network segmentation: cardholder data environment (CDE) isolated from rest of infrastructure. Quarterly penetration tests. Strict access logging.
HSM (Hardware Security Modules)
Physical tamper-evident hardware that holds private keys and performs cryptographic operations (signing transactions). Keys never leave the HSM in plaintext. Used by every major payment processor for signing payment tokens. Thales/Entrust nShield: standard HSMs used in finance.
mTLS (Mutual TLS)
Both client and server present certificates. Internal microservices authenticate each other — prevents a compromised service from calling payment APIs. Certificate rotation every 24 hours via Vault (HashiCorp). All internal traffic encrypted, even within VPC.
Bare Metal Co-location (HFT)
High-frequency trading firms (Jane Street, Two Sigma) rent rack space physically inside exchange data centers. Speed of light: London Equinix LD4 to NYSE Mahwah NJ: ~35ms. Co-location: <1 microsecond to exchange matching engine. Kernel bypass (DPDK) + RDMA eliminates OS overhead entirely.
Workflow Deep Dive
1
API + Idempotency: Charge Exactly Once
Client sends POST /v1/charges with Idempotency-Key: {uuid} header. Stripe stores: SET idempotency:{key} "pending" EX 86400 (24h). If same key arrives again: return stored result without re-processing. This means network timeouts are safe to retry — the charge can never happen twice for the same key. Critical: the UUID must be generated before the request, not after.
2
Parallel: Fraud Detection + Ledger Write
Both run concurrently on receipt of payment event. Fraud ML (Flink): velocity checks, IP reputation, device fingerprint, behavioral biometrics — returns risk score in <10ms. Ledger write: BEGIN transaction in FoundationDB, write debit/credit entries, verify double-entry invariant (SUM=0), COMMIT. If fraud score > threshold → rollback ledger + decline. If ledger fails → fraud result discarded. Both must succeed.
3
Bank Clearing via ACH / SWIFT
Stripe settles with banks via ACH (Automated Clearing House) for US transfers. SWIFT for international (SWIFT gpi: same-day tracking). ACH is batch — processes in 3 batches per business day. Stripe holds funds in pooled bank accounts and settles merchants daily (T+2). Real-time payments (RTP) in US (The Clearing House): 24/7, under 60 seconds. SEPA Instant Credit Transfer in EU: 10 seconds.
🔐 Why Stronger Than ACID?
Standard ACID with "snapshot isolation" can still have anomalies (write skew). Finance needs strict serializability — equivalent to a single-threaded execution. FoundationDB and Spanner provide this. Cost: higher latency (10–100ms vs 1–5ms) and lower throughput. Acceptable for payments where correctness matters more than speed.
⚡ HFT: Nanoseconds Matter
HFT strategies exploit price differences that exist for microseconds. Speed stack: FPGA order routing (100 ns) → kernel bypass (DPDK, 500 ns) → co-location (1 μs to exchange) → custom network cards (RDMA). A 1-millisecond latency advantage over competitors generates ~$100M/year additional profit for top HFT firms.
🎯 CAP Position
CP system, always. A payment system that sacrifices consistency for availability would allow double-spending or ledger imbalances. In a network partition, FinTech systems refuse to process transactions rather than risk inconsistency. "Better to reject a payment than to process it twice." Banks are legally obligated to maintain consistent ledgers.
