Alibaba Cloud PolarDB

MySQL/PostgreSQL-compatible · PolarFS shared storage · HTAP via IMCI

PolarDB (2018, PolarFS VLDB 2018) is Alibaba Cloud's cloud-native database, engineered as a compute-storage separated alternative to standard MySQL and PostgreSQL — similar to Amazon Aurora but built for China's hyperscale infrastructure. PolarFS (Polar File System) uses RDMA-connected SSD pools for shared storage with 1/10th the write amplification of standard MySQL. IMCI (In-Memory Columnar Index) enables HTAP: the same PolarDB instance handles both OLTP row queries and OLAP column scans. PolarDB handled Alibaba's Singles' Day (11.11) peak — 580K TPS.

Alibaba's Oracle Replacement

PolarDB was born out of Alibaba's "de-IOE" initiative — eliminating IBM, Oracle, and EMC from their infrastructure. PolarDB provides MySQL and PostgreSQL compatibility while outperforming them via PolarFS shared storage.

Unlike standard MySQL replication (full binlog shipping), PolarDB primary nodes write only redo log records to PolarFS. Read replicas apply the same redo log from shared storage — replica lag drops from seconds to milliseconds because they share the same physical storage rather than replicating data.

IMCI — In-Memory Columnar Index 
-- IMCI: maintains a columnar replica of row data in memory
-- Synchronized automatically from redo log
-- Queries can hit IMCI without read-replica lag

-- Add IMCI to a table
ALTER TABLE orders ADD COLUMN GROUP(
    order_id, customer_id, order_date,
    product_id, quantity, total
) COMMENT 'imci';

-- PolarDB optimizer decides row vs columnar:
-- Point lookup (WHERE id = X)    → row engine
-- Full-scan aggregation           → IMCI columnar
-- Mixed query                     → partition routing

-- Check which engine was used:
EXPLAIN FORMAT=TREE
SELECT customer_id, SUM(total) FROM orders
WHERE order_date > '2024-01-01'
GROUP BY customer_id;
-- → "Using In-Memory Columnar Index" in EXTRA

-- IMCI on read node: dedicated analytics replica
-- Primary: OLTP only
-- Read node 1: IMCI enabled, handles analytics
-- Read node 2..N: row engine, handles read scaling
PolarDB-X: Distributed SQL Layer 
-- PolarDB-X: adds horizontal sharding on top of PolarDB storage
-- MySQL-compatible, application sees single database

-- Shard by order_id:
CREATE TABLE orders (
    order_id    BIGINT PRIMARY KEY,
    customer_id BIGINT,
    order_date  DATE,
    total       DECIMAL(10,2)
) DBPARTITION BY HASH(order_id) TBPARTITION BY HASH(order_id) TBPARTITIONS 8;

-- Global secondary index (cross-shard lookup by customer_id)
CREATE GLOBAL INDEX gsi_customer
ON orders(customer_id)
COVERING(order_date, total)
DBPARTITION BY HASH(customer_id);

-- Transparent distributed JOIN:
SELECT c.name, COUNT(o.order_id) AS orders, SUM(o.total) AS spent
FROM customers c JOIN orders o ON c.id = o.customer_id
GROUP BY c.name;
-- PolarDB-X Proxy routes to correct shards, aggregates results

-- Distributed transaction:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- different shard
COMMIT;  -- 2PC across shards
MySQL/PostgreSQL-Compatible SQL 
-- ── 1. Standard OLTP (same as MySQL 8.0)
SELECT p.product_name, SUM(oi.quantity * oi.unit_price) AS revenue
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
JOIN orders o ON oi.order_id = o.order_id
WHERE o.order_date BETWEEN '2024-01-01' AND '2024-12-31'
  AND o.status = 'completed'
GROUP BY p.product_name
ORDER BY revenue DESC
LIMIT 20;

-- ── 2. Analytics on IMCI (same SQL, different execution path)
-- PolarDB routes automatically based on query pattern + IMCI hint
SELECT /*+ SET_VAR(polar_optimizer_enable_imci=ON) */
    DATE_FORMAT(order_date, '%Y-%m') AS month,
    customer_segment,
    COUNT(DISTINCT customer_id) AS active_customers,
    SUM(total) AS monthly_revenue,
    AVG(total) AS avg_order_value
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE order_date >= '2023-01-01'
GROUP BY month, customer_segment
ORDER BY month, monthly_revenue DESC;

-- ── 3. Window functions (MySQL 8.0 compatible)
SELECT customer_id, order_date, total,
       SUM(total) OVER (
           PARTITION BY customer_id
           ORDER BY order_date
           ROWS UNBOUNDED PRECEDING
       ) AS cumulative_spend,
       RANK() OVER (PARTITION BY customer_id ORDER BY total DESC) AS order_rank
FROM orders;

-- ── 4. JSON (MySQL 8.0 native JSON)
SELECT customer_id,
       JSON_VALUE(preferences, '$.preferred_category') AS fav_category,
       JSON_LENGTH(preferences->'$.wishlist') AS wishlist_count
FROM customers
WHERE JSON_CONTAINS(preferences->'$.interests', '"sports"');

-- ── 5. Recycle Bin (PolarDB-specific: undrop tables)
FLASHBACK TABLE orders TO BEFORE DROP;  -- recover dropped table
SHOW RECYCLEBIN;                         -- list recoverable objects

PolarFS Shared Storage Architecture 
┌──────────────────────────────────────────────────────────────────────┐
│  Compute Layer                                                       │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐              │
│  │   Primary    │   │  Read Replica│   │  IMCI Node   │              │
│  │  (R/W)       │   │  (RO)        │   │  (Analytics) │              │
│  │  Local buffer│   │  Local buffer│   │  Columnar mem│              │
│  └──────┬───────┘   └──────┬───────┘   └──────┬───────┘              │
│         │ redo log only    │                   │                     │
│         │  (not pages)     │ redo log pull     │ redo log pull       │
│         ▼                  ▼                   ▼                     │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  PolarFS (Polar File System)                                  │   │
│  │  RDMA-connected SSD pool across storage nodes                 │   │
│  │  Parallel redo log write: 3-copy across storage nodes         │   │
│  │  Chunk server: 64KB chunks, distributed metadata             │    │
│  │  Write amplification: ~1.5x vs MySQL's ~10-15x               │    │
│  └───────────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────┘

Key advantages over standard MySQL replication:
1. No page shipping — only redo log (1/10th IO)
2. Read replicas share physical storage (no replica lag from data copy)
3. Replica lag: 1–20ms vs 100ms-1s for standard binlog replication
4. Storage auto-grows: 10GB increments up to 100TB (no pre-allocation)
5. Snapshot backup: point-in-time snapshot of PolarFS chunks (instant)
