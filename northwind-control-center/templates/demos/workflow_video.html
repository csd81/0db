🗄️ Storage & Data Layer
Amazon S3 / Google Cloud Storage
Stores every resolution of every segment. Object names follow /{video_id}/{resolution}/{segment_n}.ts. Lifecycle policies move cold segments to Glacier (S3 IA). YouTube stores ~1 exabyte+ of video.
Vitess (horizontally scaled MySQL)
YouTube's metadata store. Shards millions of rows of video metadata, comments, and view counts across 1000s of MySQL shards. A VTGate proxy makes it look like one MySQL instance to application code.
Apache Cassandra
High-availability watch history and recommendation signals. AP system — tolerates node failures. Wide-row design: (user_id, watch_time) → video_id, duration_watched.
Elasticsearch
Full-text search over video titles, descriptions, transcripts (auto-generated by Whisper). Powers autocomplete with prefix queries. ~5ms P99 search latency at YouTube scale.
⚙️ Compute & Processing
FFmpeg
The universal video transcoding engine. Takes raw upload → outputs H.264 (MP4), VP9, AV1 segments at 240p/360p/480p/720p/1080p/4K. Single 4K 10-minute video: ~50 GB raw, 20 parallel FFmpeg workers.
AWS Elemental MediaConvert
Managed transcoding service. Submits jobs as JSON, outputs HLS/DASH segments to S3. Automatically scales GPU instances. Used by Netflix and streaming services that don't want to manage FFmpeg farms.
Apache Flink (stream processing)
Real-time view count aggregation. Kafka topic view-events → Flink job aggregates by video_id per 10-second window → updates Redis counter. This is why YouTube shows "approximated" counts then updates.
TensorFlow / PyTorch (Recommendations)
Deep learning recommendation models (YouTube DNN, TikTok's "For You" ranker). Two-tower model: user embedding + video embedding → dot product similarity. Retrained daily on watch history signals.
🌐 CDN & Streaming Protocols
Netflix Open Connect (Custom CDN)
Netflix's proprietary CDN appliances physically rack-mounted inside ISP data centers. Serves 99%+ of Netflix traffic from local ISP. Eliminates backbone transit costs. Each appliance: 100 TB SSD + 100 Gbps uplink.
HLS (HTTP Live Streaming)
Apple's protocol. Manifest file (.m3u8) lists all available segment URLs at each quality. Player downloads 2–10 second .ts segments. Supported natively by Safari and iOS.
MPEG-DASH
W3C standard. XML MPD manifest. Codec-agnostic (works with H.264, VP9, AV1). Used by YouTube. Adaptive Quality Selection based on bandwidth probe every 3 seconds.
UDP + QUIC (Live Streaming)
Twitch/live video uses RTMP upload from OBS, then QUIC (HTTP/3) for low-latency delivery. QUIC avoids TCP head-of-line blocking — dropped packet doesn't stall other streams.
Codec & Format Reference
H.264/AVC
Widest device support. ~1 Mbps at 720p30. Patent encumbered. Universal fallback.
VP9
Google's open codec. 50% better compression than H.264. YouTube default on Chrome/Android.
AV1
Royalty-free. 30% better than VP9. 20× slower to encode. Used for catalog (not live). Netflix, YouTube.
HEVC/H.265
2× compression vs H.264. Used for 4K/HDR. Requires hardware decoder. Apple ecosystem.
HLS Segments
2–10 second .ts chunks. .m3u8 playlist. Each segment independently decodable.
DASH Segments
fMP4 fragments. MPD XML manifest. Byte-range requests. Supports live & VOD.
Workflow Deep Dive
1
Upload: Chunked Multipart Transfer
Browser splits large file into 5 MB chunks. Each chunk uploaded independently via HTTPS PUT (resumable upload API). YouTube uses a custom QUIC-based upload protocol. If connection drops, upload resumes from last successful chunk. Raw video stored in regional S3/GCS bucket with no processing yet.
2
Parallel Transcoding: FFmpeg Farm
S3 upload triggers Lambda/Pub/Sub event → enqueues transcoding job. Transcoding workers (EC2 c5.4xlarge or custom GPU instances) split video into segments, then fan out across N workers — each transcode one resolution in parallel. A 1-hour 4K video: ~20 parallel jobs, ~10 minutes total. Output: VP9 + H.264 at 6 quality levels.
3
CDN Distribution: Pre-positioning Popular Content
Top 1% of videos (by predicted view count) pre-pushed to all CDN PoPs (Points of Presence) before first request. Long-tail content served on cache miss: CDN origin-fetches from S3, caches locally for subsequent requests. CDN cache hit rate: ~90% for popular content, ~40% for long-tail.
4
Adaptive Bitrate (ABR) Playback
Player downloads .m3u8 master manifest listing all quality URLs. Probes bandwidth by timing the download of 3 segments. If bandwidth > threshold → step up quality. If buffer < 2 seconds → drop quality. Target: maintain 10-30 second buffer. Switch quality at segment boundary (no glitch). Algorithm: BOLA (Buffer Occupancy based Lyapunov Algorithm).
5
Telemetry → Recommendation Pipeline
Player emits quality-change events, buffer stalls, and watch-time heartbeats every 5 seconds to Kafka. Flink aggregates real-time view counts. Daily Spark job computes user engagement features → feeds recommendation model. YouTube Two-Tower: user vector (from watch history) × video vector (from metadata) = relevance score. Top-100 ranked → re-ranked by diversity and freshness.
📦 Storage Math
500 hours uploaded/minute → 30,000 hours/hour. A 1-hour video at 6 qualities averages ~8 GB total. That's 240 TB/hour of new storage — before replication (3×). YouTube: ~1 exabyte total storage, growing ~2 PB/day.
🌍 Why Custom CDN?
At Netflix scale, paying Akamai/Cloudflare is more expensive than building your own. Open Connect costs Netflix ~$200M to build but saves ~$1B/year in CDN fees. Also gives Netflix full control over routing, cache eviction, and traffic shaping for 15% of all US internet traffic.
🎯 CAP Position
AP system — availability is paramount. Stale view counts are fine; buffering is not. Cassandra (AP) for watch history. Vitess/MySQL (CP) only for critical metadata like video ownership and monetization. CDN cache serves potentially stale thumbnails — acceptable.
