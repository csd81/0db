▶ Batch Analytics SQL Server (row-store) vs pandas/NumPy (columnar) idle

This demo runs the same aggregation query through two engines side by side: SQL Server's row-store B-tree index, which loads full tuples even when only two columns are needed, and a pandas columnar scan, which reads only the relevant columns from contiguous memory. Timings are measured at 2,155 Northwind rows and extrapolated to 10 million to show how the gap widens with row count. The cost difference is not a constant multiplier — it grows with row width because wider rows mean more wasted I/O in the row-store path. This is the core reason every data warehouse uses columnar storage for analytical workloads.

Row-store (SQL Server) — stores one row contiguously. Fast for: INSERT/UPDATE, single-row SELECT by PK (clustered index). Slow for: full-table scans, column aggregations — must read all columns even when only 2 are needed.
Columnar (Dremel / BigQuery / pandas) — stores one column contiguously as a column stripe. Fast for: aggregations, GROUP BY — skips unneeded stripes entirely. Slow for: point lookups — must reconstruct a row from many separate stripes. — Melnik et al., VLDB 2010
Dremel column skipping — Q1 only needs ShipCountry + Revenue. SQL Server must read every column in every row page. Dremel reads 2 stripes, ignores the rest → proportionally less I/O at scale. At 87 TB (Dremel T1), skipping 268 of 270 columns saves >99% of I/O.

Dremel Architecture — Melnik et al., VLDB 2010 the paper behind BigQuery
Nested Columnar Storage
Dremel extends columnar storage to nested records (Protocol Buffers). Each value carries two extra integers:
r — repetition level: which repeated field in the path repeated. Lets the reader know when a new record starts vs. a repeated sub-field.
d — definition level: how many optional/repeated fields in the path are actually present. Encodes NULLs without storing them explicitly.
Example: Name.Language.Code = 'en-us' with r=0, d=2 means: new top-level record, both Name and Language are defined.
Multi-Level Serving Tree
Borrowed from web search. A query is rewritten at each level and partial aggregates flow upward:
① Root server
receives query, determines tablets, rewrites to sub-queries
② Intermediate servers
partial aggregation of leaf results; re-aggregate upward
③ Leaf servers (local disk)
scan column stripes in tablets; emit (key, partial_agg) pairs
Stragglers rescheduled by the query dispatcher. 95% read-ahead cache hit rate on tablets. A 3,000-leaf cluster provides 24,000 execution slots.

Field-Oriented Model
Each attribute → one column. The classical relational approach. Easy to program; all GUI / ORM tools bind columns to widgets automatically. Downside: adding a new attribute requires an ALTER TABLE and updates to every layer of logic built on top.
Record-Oriented Model (EAV)
Attributes stored as rows: (emp_id, attrib_id, attrib_value). Adding a new attribute = INSERT one row into the attribute registry — no schema change. Cost: reconstructing a single entity requires pivoting many rows, needs an adapter layer.
PIVOT
Rotates row values into columns. Syntax: PIVOT (AGG(val) FOR dim IN ([v1],[v2]…)). Even when no aggregation is needed the grammar requires an aggregate function (use SUM or MAX on a single-row group — it's a no-op). Column list must be hard-coded; use dynamic SQL (EXECUTE) when unknown.
UNPIVOT
Reverses a PIVOT: turns column values back into rows. Syntax: UNPIVOT (val FOR dim IN (col1, col2…)). Requires matching data types across all source columns. Aggregation information lost during PIVOT cannot be recovered — UNPIVOT is lossless only when no grouping happened.
Field-Oriented vs Record-Oriented
Dimension Field-oriented Record-oriented
Flexibility Rigid — ALTER TABLE required for new attribute Flexible — INSERT one row in attribute registry
Programmability Easy — ORM / GUI bind columns automatically Hard — pivot adapter required per query
Efficiency NULL cells waste storage Sparse — unknown values simply not stored
Typical use Stable schemas (Orders, Products) Variable attributes (product specs, survey answers)
Northwind example: Employees(LastName, Title, City) as three columns vs employee_record(emp_id, attrib_id, attrib_value) with an attributes registry table — adding a new field (e.g. Phone) only requires inserting (4, 'Phone', 'text') into the registry.

What are window functions?
Analytic / windowing functions compute a value for each row using a window of related rows — without collapsing them via GROUP BY. A single query can use several different windows simultaneously, one per column. Syntax: FUNC([arg]) OVER ([PARTITION BY …] [ORDER BY …] [ROWS …])
PARTITION BY
Splits the result set into independent groups; the function resets for each group. Omitting PARTITION BY treats the entire ordered result set as one window. Example: SUM(value) OVER (PARTITION BY YEAR(OrderDate) ORDER BY OrderDate ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) → cumulative revenue per year, resetting at each year boundary.
ROWS frame specification
Narrows the window to a sliding sub-range of rows relative to the current row:
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — running total
ROWS BETWEEN 3 PRECEDING AND 3 FOLLOWING — centred moving average
ROWS BETWEEN 1 PRECEDING AND CURRENT ROW — compare this row with the previous one
Keywords: CURRENT ROW, n PRECEDING, n FOLLOWING, UNBOUNDED PRECEDING/FOLLOWING
Cursor vs window function
A cursor iterates row-by-row in procedural code. Many problems that seem to require a cursor (running totals, detecting consecutive patterns, rank within a group) can be solved with a single declarative window-function query. Window functions are far faster on large result sets — the engine optimises the frame scan once, rather than executing per-row procedural logic.
Key Window Functions
Function What it returns Typical use
ROW_NUMBER() Sequential integer within partition (no ties) Page results; rank rows within group
RANK() / DENSE_RANK() Rank with / without gaps on ties Leaderboards, top-N per group
LAG(expr, n, default) Value from n rows before current Period-over-period delta, pattern detection
LEAD(expr, n, default) Value from n rows after current Look-ahead comparisons
FIRST_VALUE(expr) Expr from the first row of the window Min/max date per group without subquery
LAST_VALUE(expr) Expr from the last row of the window Most recent value per partition
PERCENT_RANK() Relative rank 0–1 Percentile position
SUM / AVG / MIN / MAX Aggregate over the ROWS frame Running total, moving average, YTD
All standard aggregates accept OVER(…) since SQL Server 2012. PERCENT_RANK = (rank − 1) / (rows − 1) in the partition, always 0 for the first row.

Detect two consecutive small orders (value < 200) — replaces a cursor
SELECT a.OrderDate, a.maxi
FROM (
    SELECT OrderDate,
        MAX(value) OVER (
            ORDER BY OrderDate
            ROWS BETWEEN 1 PRECEDING AND CURRENT ROW) maxi,
        ROW_NUMBER() OVER (ORDER BY OrderDate)          sorsz
    FROM #tmp
) a
WHERE a.maxi < 200 AND sorsz > 1  -- exclude first row (no predecessor)

Row store vs Column store
Traditional rowstore: each page holds all columns of consecutive rows — good for OLTP (single-row lookups, DML). Columnstore: each page holds one column across many rows — analytical queries that touch 2–5 of 100 columns read 95–98% less data.
Compression
Columns have low cardinality (e.g. Gender: only 2 distinct values). Run-length encoding compresses sequences of identical values extremely well. Typical columnstore compression: 5–10×. Less I/O → more rows fit in buffer pool → analytical queries run mostly on cached data.
Batch mode execution
SQL Server processes columnstore data in batches of ~900 rows using SIMD CPU instructions, not row-by-row. Batch mode reduces per-row overhead dramatically. Aggregate queries (SUM/COUNT over millions of rows) are typically 10–100× faster than equivalent rowstore execution.
When to use
• Data warehouse / OLAP fact tables (large, mostly-read)
• Reporting tables with heavy aggregation
• Historical data that rarely changes
⚠ Not suitable for OLTP point lookups or high-frequency single-row updates
Columnstore Index — Syntax & Internals 
-- Non-clustered: keeps rowstore + adds columnar copy
CREATE NONCLUSTERED COLUMNSTORE INDEX
    ncci_OrderDetails
    ON [Order Details] (OrderID, ProductID, UnitPrice, Quantity, Discount);

-- Clustered: replaces rowstore entirely (best compression)
CREATE CLUSTERED COLUMNSTORE INDEX cci_SalesFact
    ON SalesFact;   -- no column list — stores ALL columns

-- Query that benefits from columnstore batch mode:
SELECT YEAR(OrderDate), SUM(Quantity * UnitPrice)
FROM   [Order Details] od
JOIN   Orders o ON od.OrderID = o.OrderID
GROUP BY YEAR(OrderDate);
-- SQL Server switches to batch mode automatically

What is Table Partitioning?
Divides a large table into smaller, independently managed chunks called partitions. Each partition is stored in its own filegroup. From the application's perspective it is one table — partitioning is transparent to queries and DML.
Partition Elimination
The query optimizer inspects the WHERE clause. If the filter matches the partition key (e.g. WHERE OrderYear = 2024), only the relevant partition(s) are scanned. A query on 1 of 12 monthly partitions reads ~1/12 the data → proportional speed-up.
Partition Switching
Bulk data moves between a staging table and a partition are an instantaneous metadata operation — no data copy. Pattern: load new month into staging → ALTER TABLE … SWITCH PARTITION n TO staging → archive → switch new data in. Minimal locking, near-zero logging.
Parallel aggregation
With multiple partitions on separate filegroups, SQL Server can scan partitions in parallel using multiple threads. Combined with a columnstore index, large aggregations (SUM over 1 B rows) decompose: sum each partition independently, then combine results.
Table Partitioning — Setup & Pattern 
-- Step 1: Partition Function — defines boundary points
CREATE PARTITION FUNCTION pf_OrderYear (int)
    AS RANGE RIGHT FOR VALUES (2022, 2023, 2024);
-- Partitions: [<2022] | [2022] | [2023] | [2024] | [≥2025]

-- Step 2: Partition Scheme — maps partitions to filegroups
CREATE PARTITION SCHEME ps_OrderYear
    AS PARTITION pf_OrderYear
    ALL TO ([PRIMARY]);

-- Step 3: Create partitioned table
CREATE TABLE OrdersFact (
    OrderID   int,  OrderYear int,
    Revenue   decimal(12,2)
) ON ps_OrderYear(OrderYear);  -- partition key column

-- Optimizer auto-eliminates partitions:
SELECT SUM(Revenue) FROM OrdersFact
WHERE  OrderYear = 2024;  -- reads only 2024 partition

Composite strategy: partition by year + clustered columnstore index on the fact table. Partition elimination reduces scanned segments; batch mode compresses and processes each segment efficiently. Best combination for large-scale OLAP workloads.
