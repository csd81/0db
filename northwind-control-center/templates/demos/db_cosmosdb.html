Azure Cosmos DB

Multi-API · 5 consistency levels · Global active-active · <10ms P99

Cosmos DB (2017, SIGMOD 2020 paper) is Microsoft's globally distributed, multi-model database. Its unique differentiator: one engine, multiple API surfaces — SQL (NoSQL), MongoDB, Cassandra, Gremlin (graph), and Azure Table. Applications using any of these wire protocols can benefit from global distribution, 5 tunable consistency levels, and guaranteed single-digit millisecond reads. Xbox Live uses Cosmos DB for 3.5 billion users' gaming state. Rolls-Royce uses it for jet engine telemetry.

One Engine, Multiple Wire Protocols

Cosmos DB's internal format — Atom-Record-Sequence (ARS) — is a binary encoding that can represent documents, key-value pairs, wide-column rows, and graph edges. The API you choose is just a wire protocol: MongoDB driver, Cassandra CQL, Gremlin, or Cosmos SQL all read/write to the same underlying ARS storage.

The billing unit is Request Units (RU/s) — a normalized measure of CPU + memory + IOPS consumed per operation. A 1KB document read costs 1 RU; complex queries cost more. You provision RU/s (or use Serverless for pay-per-operation). Every operation returns its consumed RUs in the response header.

5 Consistency Levels
Strong
Read returns most recent write. One region only (or single-write). Highest consistency, highest latency.
Bounded Staleness
Reads lag behind writes by K versions or T seconds. Predictable staleness window.
Session
Read-your-writes within a session. Default and most popular. Balances consistency with performance.
Consistent Prefix
Never see out-of-order writes. If A then B was written, reads see A, or A+B, never B alone.
Eventual
Lowest latency. No ordering guarantee. Best for read-heavy globally distributed apps.

Python SDK — CRUD Operations 
from azure.cosmos import CosmosClient, PartitionKey

client = CosmosClient(url=ENDPOINT, credential=KEY)
db = client.create_database_if_not_exists("northwind")
container = db.create_container_if_not_exists(
    id="orders",
    partition_key=PartitionKey(path="/customerId"),
    offer_throughput=400  # RU/s (autoscale: use default_ttl + max_throughput)
)

# CREATE — upsert a document
container.upsert_item({
    "id": "order-001",
    "customerId": "cust-42",
    "total": 199.99,
    "status": "pending"
})

# READ — by id + partition key (1 RU, fastest path)
item = container.read_item(item="order-001", partition_key="cust-42")

# QUERY — cross-partition query (costs more RUs)
results = list(container.query_items(
    query="SELECT * FROM c WHERE c.status = @status",
    parameters=[{"name": "@status", "value": "pending"}],
    enable_cross_partition_query=True
))

# PATCH — partial update (no read-modify-write needed)
container.patch_item(
    item="order-001", partition_key="cust-42",
    patch_operations=[
        {"op": "replace", "path": "/status", "value": "shipped"},
        {"op": "add", "path": "/shippedAt", "value": "2024-01-16"}
    ]
)

# BATCH — transactional batch within a partition
batch = [
    ("create", ({"id": "order-002", "customerId": "cust-42"},), {}),
    ("replace", ("order-001", {"id": "order-001", "customerId": "cust-42", "status": "complete"}), {}),
]
container.execute_item_batch(batch, partition_key="cust-42")
Cosmos SQL (NoSQL API) Query Examples 
-- ── 1. Basic SELECT with WHERE
SELECT c.id, c.customerId, c.total, c.status
FROM c
WHERE c.customerId = "cust-42"
  AND c.status IN ("pending", "processing")
ORDER BY c.orderDate DESC
OFFSET 0 LIMIT 10

-- ── 2. Nested document access
SELECT c.id, c.shipping.city, ARRAY_LENGTH(c.items) AS item_count
FROM c
WHERE c.shipping.city = "Seattle"

-- ── 3. JOIN (self-join for arrays — not cross-container!)
SELECT c.id, item.productId, item.qty
FROM c
JOIN item IN c.items          -- unnests the items array
WHERE item.qty > 1

-- ── 4. Aggregate with GROUP BY
SELECT c.status, COUNT(1) AS count, SUM(c.total) AS revenue
FROM c
GROUP BY c.status

-- ── 5. Array functions
SELECT c.id,
       ARRAY_CONTAINS(c.tags, "premium") AS is_premium,
       ARRAY_LENGTH(c.items)             AS num_items
FROM c
WHERE ARRAY_CONTAINS(c.tags, "urgent")

-- ── 6. String functions
SELECT c.id,
       UPPER(c.customerId)    AS upperId,
       SUBSTRING(c.id, 0, 8) AS shortId,
       CONTAINS(c.notes, "refund") AS has_refund_note
FROM c

-- ── 7. Type-checking (Cosmos stores schema-free JSON)
SELECT c.id, IS_DEFINED(c.discountCode) AS has_discount
FROM c
WHERE IS_NUMBER(c.total) AND c.total > 500

-- ── 8. Spatial query (Point, Polygon, LineString)
SELECT c.id, c.name
FROM c
WHERE ST_DISTANCE(c.location, {"type":"Point","coordinates":[-122.33,47.61]}) < 5000

Architecture Deep-Dive 
Cosmos DB internal architecture:

Physical partition (max 50 GB, ~10K RU/s)
  └── Replica set (4 replicas: 1 primary, 3 secondary)
      └── ARS (Atom-Record-Sequence) pages
          └── Backed by SSDs in Azure datacenter

Global distribution:
  Write region ──replication──▶ Region 2 (async, <2ms typical)
               ──replication──▶ Region 3
               ──replication──▶ Region 4

Consistency implementation:
  Strong: quorum write + quorum read across replicas
  Session: session token (LSN - Log Sequence Number)
  Eventual: read from any replica, no wait

Partition routing:
  Client ──▶ Gateway (resolves partition key to physical partition)
         ──▶ Replica set primary (write) or any replica (read)

Auto-partition split: when partition exceeds 50 GB or 10K RU/s
→ Splits into 2 physical partitions (transparent to application)

# ── Cosmos DB for PostgreSQL (Citus distributed tables)
-- Horizontally scale PostgreSQL with Cosmos as the managed service
CREATE TABLE events (
    id          BIGSERIAL,
    user_id     BIGINT,
    event_type  TEXT,
    occurred_at TIMESTAMPTZ DEFAULT NOW()
);
SELECT create_distributed_table('events', 'user_id');

-- Distributed query automatically sharded:
SELECT user_id, COUNT(*) FROM events
WHERE occurred_at > NOW() - INTERVAL '24 hours'
GROUP BY user_id
HAVING COUNT(*) > 100;
