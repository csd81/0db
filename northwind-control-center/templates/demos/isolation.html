Chapter 1 · The Webshop Race Condition — two sessions, one horse, two isolation levels

SQL Server's five transaction isolation levels — Read Uncommitted, Read Committed, Repeatable Read, Snapshot, and Serializable — each offer a different trade-off between data consistency and concurrent throughput. At Read Uncommitted a transaction reads uncommitted changes from other sessions (dirty read); at Serializable all reads are protected against phantom rows by range locks that block competing inserts for the duration of the transaction. The demo places two sessions in a horse-racing bet scenario, showing exactly which anomalies appear at each isolation level: dirty reads, non-repeatable reads, phantom rows, and the deadlocks that emerge at higher levels. Choosing the wrong isolation level is the most common source of subtle data corruption bugs that only manifest under concurrent load in production.

READ COMMITTED (default)
Shared locks are released immediately after each SELECT. Between the read and the update in Session 1, Session 2 can freely read the same row — it also sees 'for sale' and proceeds. Both transactions commit successfully. The horse is sold twice. Only one buyer receives it.
id=2: sold='sold', buyer='Your name'
Session 1 thinks it bought the horse — it did not.
REPEATABLE READ
Shared locks are held for the entire duration of the transaction. When Session 2 tries to update the row that Session 1 is reading, it blocks. When Session 1 then tries to update the same row, the server detects a deadlock cycle and kills one session as the victim. Only one horse is sold. Logical integrity preserved — but one transaction is aborted.
id=2: sold='sold', buyer='Your name'
Session 1: "Transaction was deadlocked … chosen as victim."
Correct Fix — Optimistic Locking with UPDLOCK
The right solution is to take an exclusive update lock at read time using WITH (UPDLOCK). This prevents a second session from even reading the row while the first is deciding whether to buy it — eliminating both the race condition and the deadlock.
-- The correct pattern (used in sp_new_order) BEGIN TRAN SELECT @sold = sold FROM test_product WITH (UPDLOCK, ROWLOCK) WHERE id = 2 -- Session 2 BLOCKS here — cannot read until S1 commits IF @sold = 'for sale' UPDATE test_product SET sold='sold', buyer='My name' WHERE id=2 COMMIT TRAN -- Session 2 unblocks → sees sold='sold' → aborts cleanly
