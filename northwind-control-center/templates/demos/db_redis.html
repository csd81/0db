Redis

In-Memory Data Store · 8 data types · Pub/Sub · Streams · Sub-millisecond latency

Redis (Remote Dictionary Server) was created by Salvatore Sanfilippo in 2009 as a key-value store that keeps its entire dataset in RAM. What sets Redis apart is not just speed — it is the richness of its data structures. Strings, lists, hashes, sets, sorted sets, streams, HyperLogLogs, and geospatial indexes each enable entirely different use cases from a single system, making Redis the first choice for session caches, gaming leaderboards, rate limiters, and real-time pub/sub messaging.

List, Hash, Set 
-- LIST (deque, O(1) push/pop at both ends)
RPUSH queue "job:1001" "job:1002"
LPOP queue          -- consume from left (FIFO)
LRANGE queue 0 -1   -- view all items

-- HASH (flat map, like a row)
HSET user:42 name "Alice" email "a@x.com" credits 500
HGET user:42 name       -- "Alice"
HMGET user:42 name credits
HINCRBY user:42 credits 100

-- SET (unique strings)
SADD online:users 42 99 17
SISMEMBER online:users 42   -- 1 (true)
SCARD online:users          -- 3 (count)
SINTER online:users premium:users  -- intersection
Sorted Set & Stream 
-- SORTED SET: leaderboard
ZADD leaderboard 9850 "alice" 7200 "bob" 12100 "carol"
ZREVRANK leaderboard "alice"   -- rank from top (0-based)
ZREVRANGE leaderboard 0 9 WITHSCORES  -- top 10
ZINCRBY leaderboard 500 "alice"        -- atomic score bump

-- STREAM: Kafka-lite event log
XADD events * action "purchase" user_id 42 amount 99.5
XADD events * action "view" user_id 17 product "SKU-9"

-- Consumer group (competing consumers)
XGROUP CREATE events mygroup $ MKSTREAM
XREADGROUP GROUP mygroup worker1 COUNT 10 STREAMS events >
XACK events mygroup
Core Patterns 
-- String + TTL (session token)
SET session:tok:abc123 '{"user":42,"role":"admin"}' EX 3600
GET session:tok:abc123
TTL session:tok:abc123       -- seconds remaining
PERSIST session:tok:abc123   -- remove expiry

-- Atomic counter / rate limiter
INCR rate:user:42            -- atomic increment
EXPIRE rate:user:42 60       -- reset window

-- Distributed lock (SET NX PX)
SET lock:checkout:42 "worker-1" NX PX 5000
-- NX = only set if Not eXists
-- PX 5000 = expire after 5000ms

-- Pipeline (batch commands, 1 round-trip)
MULTI
  INCR page_views
  ZADD top_pages 100 "homepage"
  HSET stats day "2024-01-15" views 100
EXEC
Pub/Sub & Lua 
-- Pub/Sub (fire-and-forget, not durable)
SUBSCRIBE notifications:user:42
PSUBSCRIBE notifications:*      -- pattern subscribe
PUBLISH notifications:user:42 '{"type":"order","id":99}'

-- Lua scripting (atomic, server-side)
EVAL "
  local current = redis.call('GET', KEYS[1])
  if current and tonumber(current) >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1
  end
  return 0
" 1 inventory:SKU-9 5

Killer Use Cases
Session cache: Twitter stored 300M user sessions in Redis. GET by session token <1ms. TTL auto-expires stale sessions.
Leaderboards: Sorted Sets provide O(log N) ZADD and ZREVRANGE. Gaming companies run real-time global leaderboards for millions of players.
Rate limiting: INCR + EXPIRE per user:endpoint key. Atomic, no race condition, auto-resets after window. Used by Stripe, Shopify APIs.
Task queues: Celery (Python) defaults to Redis as broker. RPUSH to enqueue, BLPOP to dequeue (blocking pop). Reliable with BRPOPLPUSH pattern.
Streams as Kafka-lite: For moderate throughput (<100k msg/s), Redis Streams with consumer groups replaces Kafka without the operational overhead.

Single-Thread + Persistence

Single-threaded event loop: All commands execute sequentially with no lock contention. This is why Redis commands have O(1)/O(log N) guarantees and why KEYS * or SMEMBERS on large sets blocks the server.

RDB (snapshot): Fork + copy-on-write snapshot at configured intervals (save 900 1 = after 900s if 1 key changed). Fast restore, but you may lose minutes of data on crash.

AOF (Append-Only File): Logs every write command. appendfsync everysec = at most 1s data loss. appendfsync always = no data loss but slower. AOF rewrite compacts history periodically.

Eviction Policies 
# redis.conf — key settings

maxmemory 8gb
# What to do when memory is full:
maxmemory-policy allkeys-lru
# Policies:
#   noeviction     -- return error (default)
#   allkeys-lru    -- evict least-recently-used
#   volatile-lru   -- LRU among keys with TTL
#   allkeys-lfu    -- least-frequently-used
#   allkeys-random -- random eviction

# Persistence
appendonly yes
appendfsync everysec   # balance of safety/speed

# Cluster
cluster-enabled yes
cluster-node-timeout 15000   # ms before failover
