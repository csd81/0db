Latency Scenarios
Speed of Light
Physical minimum
Light in fiber travels at ~200,000 km/s (≈ 2/3 of c in vacuum).

New York → London = 5,570 km
RTT minimum = 5,570 / 200,000 × 2 = 55.7 ms
Real RTT ≈ 80–90 ms (routing, queuing)

NY → Tokyo = 10,800 km
RTT minimum = 108 ms
Real RTT ≈ 150–180 ms

CDN principle
Move static content to edge nodes close to users. A London user hits a London PoP instead of a US data-center.

What CDN cannot help
Dynamic, personalised, or real-time data must still come from origin. Only cache-able (idempotent GET) responses benefit.

CDN cache-hit rate of 90%+ means 9 in 10 requests served from edge (< 10 ms) rather than from origin (50–200 ms).
Zone — single failure domain
A zone is an independent data-centre within a region. VMs and Persistent Disks live in a zone. Intra-zone RTT is < 1 ms. One power/network failure can take down a zone — always spread critical workloads across ≥ 2 zones.
Region — group of zones
A region (e.g. us-central1) contains 3–5 zones (-a / -b / -c / -f). Regional resources — like static external IPs and Cloud SQL replicas — are available to all zones in the region. Choose a region close to your users to minimise RTT.
Global — across all regions
Global resources (disk images, snapshots, VPC networks, Cloud CDN) are replicated or accessible worldwide. Cloud CDN is a global resource — edge PoPs serve cached content from the nearest node regardless of where your origin VM lives.
Why it matters for latency
Placing an origin in the wrong region adds a fixed RTT penalty to every dynamic request. CDN offloads static content globally, but dynamic API calls, WebSockets, and gRPC streams still pay the speed-of-light cost to the nearest zonal instance.

Fault tolerance patterns
Single zone — one failure takes down the app. Never for production.
Multi-zone — VMs in -a and -b. Survives zone outage. Standard HA.
Multi-region — VMs in us-east1 + europe-west1. Survives regional disaster + cuts latency for global users.
A global HTTP(S) load balancer routes each user to the nearest healthy region automatically — combining fault tolerance with geo-latency reduction in one resource.
