The Polling Problem 
-- Run every minute: "what changed since I last checked?"
SELECT * FROM Orders WHERE updated_at > @last_check_time
ProblemConsequence
Requires an updated_at columnMost legacy schemas do not have one
Misses hard-deletesA deleted row leaves no trace
Polling interval = latency floor1-minute poll = up to 1 minute of lag
Hammers the DBFrequent large scans under production load
Not transactionalA row updated twice between polls shows only final state
What CDC Does Instead — Transaction Log Tap 
Application → SQL Server engine
                    ↓
          Transaction Log (WAL / binlog)
                    ↓
           CDC connector reads log
                    ↓
          Downstream consumers get events
✓ Captures
INSERTs, UPDATEs, DELETEs — even hard deletes
✓ Sub-second latency
Log updated at commit time, no polling interval
✓ Zero polling overhead
Follows log tail — no table scans
✓ Transactional ordering
Events arrive in commit order, preserving causality

Live Demo — Enable CDC on Northwind

SQL Server has built-in CDC. Enable it at the database level, then per table, and query the resulting change table.

sp_cdc_enable_db + sp_cdc_enable_table
TOP 10 change rows
-- Enable CDC at DB level
EXEC sys.sp_cdc_enable_db;

-- Enable on Orders table
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name   = N'Orders',
    @role_name     = NULL;

-- Verify
SELECT name, is_cdc_enabled FROM sys.databases
WHERE name = 'Northwind';

-- Query changes
SELECT TOP 10 * FROM cdc.dbo_Orders_CT
ORDER BY __$start_lsn DESC;

Apache Kafka — Core Concepts
ConceptDescription
ProducerPublishes messages to a topic
TopicNamed, ordered, durable stream of messages
PartitionTopic split for parallelism; order guaranteed within partition
ConsumerReads messages, tracks position with an offset
Consumer groupMultiple consumers; each partition assigned to one consumer
OffsetInteger index of a message within a partition
BrokerA Kafka server node that stores partitions
Why Kafka Over Direct DB-to-DB Sync
✗ Tightly coupled:
SQL Server → [sync script] → Elasticsearch
If ES is down, script must buffer or fail. Only one consumer. No replay. No backpressure.
✓ Kafka decouples:
SQL Server → [Debezium] → Kafka topic → [consumer A] → Elasticsearch
                                      → [consumer B] → MongoDB
                                      → [consumer C] → BigQuery
Each consumer reads independently. 7-day retention. Reset to offset 0 for full replay.

Flyway Migration Naming Convention
V{version}__{description}.sql
Rules
  • V prefix (uppercase)
  • Double underscore __
  • Description with _ for spaces
  • Applied once, ascending version order
Examples
V1__create_orders_table.sql
V2__add_priority_column.sql
V3__create_index_orders.sql
V4__insert_seed_data.sql
Example Migrations for Northwind
V1__create_orders_table.sql
CREATE TABLE IF NOT EXISTS orders (
    OrderID      INT          PRIMARY KEY,
    CustomerID   CHAR(5)      NOT NULL,
    EmployeeID   INT,
    OrderDate    DATE,
    ShipCountry  VARCHAR(15),
    Freight      DECIMAL(10,2) DEFAULT 0
);
V2__add_priority_column_to_orders.sql
ALTER TABLE orders
  ADD COLUMN priority INT NOT NULL DEFAULT 0;

COMMENT ON COLUMN orders.priority IS
  '0=normal, 1=high, 2=urgent — SLA tracking';
V4__add_audit_columns.sql
ALTER TABLE orders ADD COLUMN created_at TIMESTAMP DEFAULT NOW();
ALTER TABLE orders ADD COLUMN updated_at TIMESTAMP DEFAULT NOW();

CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN NEW.updated_at = NOW(); RETURN NEW; END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER orders_set_updated_at
BEFORE UPDATE ON orders
FOR EACH ROW EXECUTE FUNCTION set_updated_at();

The Isolation Level Ladder
LevelDirty ReadNon-RepeatablePhantom
READ UNCOMMITTED Possible Possible Possible
READ COMMITTED Prevented Possible Possible
REPEATABLE READ Prevented Prevented Possible
SERIALIZABLE Prevented Prevented Prevented
Most production systems use READ COMMITTED (PostgreSQL default) or REPEATABLE READ (MySQL InnoDB default) as a balance of correctness vs throughput.
What acid_test.py Does
Creates 10 bank accounts (balance=1,000 each → total=10,000), then spawns concurrent transfer threads. Each thread:
  1. Picks two random account IDs
  2. Begins a database transaction
  3. Reads the source account balance
  4. Sleeps 0–50 ms (amplifies race conditions)
  5. Transfers 1–100 units if source has sufficient funds
  6. Commits the transaction
ACID Invariant: total balance must always equal 10,000 regardless of concurrency.

READ UNCOMMITTED — Dirty Read Race Condition
With READ UNCOMMITTED, threads can read balances modified by uncommitted transactions:
Thread A reads balance=500, decides to transfer 200
Thread B (not committed) subtracted 300 from same account
Thread A sees uncommitted balance → makes decision on phantom data
Thread B rolls back → balance restored, but Thread A acted on ghost value
→ Final total may NOT equal 10,000
