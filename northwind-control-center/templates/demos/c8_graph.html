Chapter 8 — Special Data Types

SQL Server Graph Tables · Neo4j / Cypher · BLOB storage

Node Tables

Created with AS NODE. SQL Server adds a hidden $node_id column (JSON: {"type":"node","schema":"dbo","table":"person","id":0}). Queried like normal tables — INSERT / SELECT work as usual.

CREATE TABLE person (id INT PK, name VARCHAR(100)) AS NODE
Edge Tables

Created with AS EDGE. Hidden $from_id and $to_id columns store node references as JSON. Edges are always directed — undirected relationships need two opposite edges. Duplicate edges are allowed (no auto-constraint).

CREATE TABLE friendof AS EDGE
ALTER TABLE friendof ADD CONSTRAINT ec_1 CONNECTION (person TO person)
MATCH Syntax

Graph queries use MATCH in the WHERE clause. Pattern: (n1-(edge)->n2) for directed traversal. Multiple constraints with AND. SHORTEST_PATH requires FOR PATH aliases and STRING_AGG … WITHIN GROUP (GRAPH PATH).

WHERE MATCH(person-(likes)->restaurant
AND person-(livesin)->city)

Duplicate Edges & $edge_id

        

          SQL Server graph allows duplicate edges (no uniqueness constraint). To detect or delete duplicates,
          read the hidden $edge_id JSON column. Internal IDs start at 0, not 1.
        

        

          

            

              
Detect duplicates

              
SELECT * FROM friendof
WHERE json_value($from_id,'$.id') = 2
  AND json_value($to_id,'$.id') = 5;

            

          

          

            

              
Delete by $edge_id

              
DELETE friendof
WHERE json_value($edge_id,'$.id') = 8;

RELATIONAL QUERIES (JOIN-BASED)

      

        
        
        
        
        
      

    


    

      

        
      

      

        

        

      

    


    
    

      

        Query Complexity: "Restaurants liked by John's friends" (Q3)
      

      

        

          

            
GRAPH — 1 MATCH clause

            
SELECT restaurant.name
FROM person person1, person person2,
     likes, friendof, restaurant
WHERE MATCH(
  person1-(friendof)->person2
           -(likes)->restaurant)
AND person1.name = 'john'

          

          

            
RELATIONAL — 4 JOINs

            
SELECT r.name
FROM r_person p1
  JOIN r_friendof f  ON p1.id = f.person1_id
  JOIN r_person p2   ON p2.id = f.person2_id
  JOIN r_likes l     ON l.person_id = p2.id
  JOIN r_restaurant r ON r.id = l.restaurant_id
WHERE p1.name = 'john'

          

        

      

    


    
    

      

        
SHORTEST_PATH Performance

        

          SQL Server's SHORTEST_PATH implementation uses temp tables + nested loops — it does not use
          native graph algorithms. Performance degrades rapidly on dense graphs (millions of nodes). For production
          social-graph workloads, Neo4j's native pointer-based storage traverses relationships at
          O(1) per hop, regardless of graph size.

CYPHER EQUIVALENT QUERIES

    


      

        

          

            Schema Setup (MERGE)
          

          

            
// Create persons (MERGE = CREATE if not exists)
MERGE (john:Person {name:'John'})
MERGE (mary:Person {name:'Mary'})
MERGE (alice:Person {name:'Alice'})

// Create relationship
MERGE (john)-[:FRIENDS_WITH]->(mary)
MERGE (mary)-[:FRIENDS_WITH]->(alice)

// Create restaurants
MERGE (t:Restaurant {name:'Taco Dell'})
MERGE (john)-[:LIKES {rating:9}]->(t)

// Import from CSV
LOAD CSV WITH HEADERS
FROM 'file:///persons.csv' AS line
MERGE (:Person {name: line.name})

          

        

      


      

        

          

            Query Examples (MATCH)
          

          

            
// John's friends
MATCH (john:Person {name:'John'})
      -[:FRIENDS_WITH]->(friend)
RETURN friend.name

// Friends-of-friends liked restaurants
MATCH (john:Person {name:'John'})
      -[:FRIENDS_WITH]->(friend)
      -[:LIKES]->(r:Restaurant)
RETURN r.name

// Shortest path
MATCH (a:Person {name:'John'}),
      (b:Person {name:'Jacob'}),
      p = shortestPath((a)-[:FRIENDS_WITH*]->(b))
RETURN [n IN nodes(p) | n.name] AS path,
       length(p) AS hops

// Graph diameter (longest shortest path)
MATCH (a:Person), (b:Person) WHERE a <> b
WITH shortestPath((a)-[:FRIENDS_WITH*]->(b)) AS p
RETURN max(length(p)) AS diameter

          

        

      


      

        

          

            DML: RETURN / DELETE / SET / REMOVE
          

          

            
// RETURN selected properties
MATCH (n:Person)
RETURN n.name, n.age ORDER BY n.name

// DELETE node and all its relationships
MATCH (n:Person {name:'Tom'})
DETACH DELETE n

// SET property
MATCH (n:Person {name:'Alice'})
SET n.age = 30
RETURN n

// REMOVE property
MATCH (n:Person {name:'Alice'})
REMOVE n.age

// UPDATE via SET
MATCH (n:Person {name:'John'})
SET n.city = 'Bellevue'

          

        

      


      

        

          

            Aggregation (no GROUP BY — implicit)
          

          

            
// Count friends per person
MATCH (p:Person)-[:FRIENDS_WITH]->(friend)
RETURN p.name, count(friend) AS friend_count
ORDER BY friend_count DESC

// Average rating per restaurant
MATCH (p:Person)-[l:LIKES]->(r:Restaurant)
RETURN r.name, avg(l.rating) AS avg_rating

// Note: no GROUP BY — Cypher aggregates
// automatically on non-aggregated cols

// Collect into list
MATCH (p:Person)-[:LIKES]->(r:Restaurant)
RETURN p.name, collect(r.name) AS restaurants

          

        

      


    


    
    

      

        
GraphQL — Not the Same as Graph DB

        

          

            

              GraphQL is a query interface specification (Facebook, 2015)
              for APIs — it defines how clients request exactly the fields they need. It runs on top of
              any backend (SQL, REST, NoSQL). It is not a database, not a graph, and not related to Neo4j.
            

          

          

            

              Graph DB (Neo4j, Cypher) is a storage model where
              data is fundamentally nodes + relationships. Traversal follows pointers, not table scans.
              Best for "5 degrees of separation" style queries that would require dozens of JOINs in SQL.
            

          

        

      

    


  


  
  


    

      
      Three thin views over the live Northwind database — vi_products, vi_orders,
      vi_orders_products — create a lightweight graph-ready projection (nodes + edge-like join table).
    


    

      

        

          

            vi_products
          

          

            
CREATE VIEW vi_products AS
SELECT productid,
       productname
FROM Products;

            
Product "nodes" — minimal projection, productid as node key.

          

        

      

      

        

          

            vi_orders
          

          

            
CREATE VIEW vi_orders AS
SELECT orderid,
       CAST(orderdate AS DATE) odate
FROM Orders;

            
Order "nodes" — strips time from datetime, exposes date only.

          

        

      

      

        

          

            vi_orders_products
          

          

            
CREATE VIEW vi_orders_products AS
SELECT o.orderid, p.productid,
    od.Quantity quantity,
    od.Quantity * od.UnitPrice
    * (1 - od.Discount) price
FROM Orders o
JOIN [Order Details] od ...
JOIN Products p ...

            
Edge table — connects order nodes to product nodes with quantity + price.

            VARBINARY(MAX) — In-Row BLOB
          

          

            

              Store binary data (images, PDFs, small files) directly in a SQL Server column.
              Loaded from disk using OPENROWSET(BULK …, SINGLE_BLOB).
              Suitable for files up to a few MB — larger files should use FileTables or Azure Blob Storage.
            

            
-- DDL
CREATE TABLE my_table (
    id   INT IDENTITY PRIMARY KEY,
    name VARCHAR(200),
    data VARBINARY(MAX)
);

-- Insert image from filesystem
INSERT my_table (name, data)
SELECT 'photo.png', BulkColumn
FROM OPENROWSET(
    BULK 'C:\path\my_photo.png',
    SINGLE_BLOB
) AS blob_data;

-- Read back (size check)
SELECT id, name,
    DATALENGTH(data) / 1024.0 AS size_kb
FROM my_table;

          

        

      

      

        

          

            FileTables — Large BLOB / File System
          

          

            

              FileTables extend FILESTREAM to expose a full Windows file-system share backed by SQL Server.
              Files appear as directories. Applications can read/write via UNC paths and via T-SQL simultaneously.
              Best for large video, audio, medical imaging — anything impractical to store in rows.
            

            
-- Enable FILESTREAM at instance level (SQL Server Configuration Manager)
-- then:

CREATE DATABASE docs
ON PRIMARY (...),
FILEGROUP fg_fs CONTAINS FILESTREAM
    (NAME='docs_fs', FILENAME='C:\data\docs_fs')
...
WITH FILESTREAM (NON_TRANSACTED_ACCESS = FULL,
                 DIRECTORY_NAME = 'docs');

-- Create FileTable
CREATE TABLE docs_ft AS FileTable
WITH (FileTable_Directory = 'documents',
      FileTable_Collate_Filename = database_default);

-- Files accessible via:
-- \\server\instance\docs\documents\

OPENROWSET Requires SQL Agent Perms

            

              OPENROWSET(BULK …) requires the service account to have read access to the
              file path. On Linux SQL Server, map the path into the container or use a bind mount.
              The file must be accessible to the SQL Server process, not the client.
            

          

        

      

      

        

          

            
Modern Pattern: Pointer + Object Store

            

              Modern apps rarely store BLOBs in SQL. Instead: store the URL in SQL, upload the actual bytes
              to S3 / Azure Blob / GCS. SQL handles metadata + ACID; the object store handles throughput + CDN.
              SELECT photo_url FROM products WHERE id = 5 → CDN delivers the image.
