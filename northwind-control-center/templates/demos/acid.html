⚡ ACID Bank Transfer Demo

This demo transfers money between two Northwind accounts inside a SQL Server transaction, then deliberately crashes mid-transfer to prove that ROLLBACK restores both balances atomically. Run the broken version first — no transaction, just two bare UPDATE statements — and watch the database end up in a corrupted half-updated state. The fixed version wraps the same logic in BEGIN TRAN / TRY / CATCH / ROLLBACK, so failure leaves every account exactly as it was. ACID guarantees are what separate a transactional database from a flat file.

Batch (köteg)
A T-SQL script is split into batches by GO on its own line. GO is not an SQL keyword — only client tools (SSMS, sqlcmd) recognise it. Variables declared inside a batch are scoped to that batch and invisible to others.
Parse → Compile → Execute
Each batch goes through three phases in order. Parse checks syntax; Compile builds an execution plan; Execute runs it. A syntax error aborts the entire batch — nothing executes. A runtime error aborts only from that statement onward in the current batch; the next batch continues normally.
Syntax error → whole batch skipped
selec * from pelda (typo) — parse fails before any statement in the same batch runs. Even an UPDATE before the typo is not executed. Split into separate batches to isolate the failure.
Special runtime errors — continue within batch
Division by zero (1/0) and constraint violations (FK / CHECK) are statement-level: the failing statement is skipped, but the next statement in the same batch still runs. Most other runtime errors (invalid object name) abort the batch at that point.

Dump vs Backup: A dump is a text script (DDL + DML) that recreates the database — portable across versions but larger. A backup is a binary snapshot — faster to restore but version-locked.
Why server-side logic?
Stored procedures, functions, and triggers let the business logic live inside the database. Benefits: simplicity (thin client connects directly, no extra tier); technology neutrality (any language calls the same stored proc via RPC); security (client needs only EXECUTE right, not read/write on every table); manageability (deploy logic changes without touching clients); efficiency (bulk data processed where it lives — less network traffic); readability (SQL is a data language — complex joins are cleaner in T-SQL than in OOP code).
Trade-offs
Low-level: general-purpose tasks (string processing, HTTP calls) are awkward in SQL.
Tooling: debugging, version control, and team workflows are harder in DB-side code.
Scalability: scaling the DB tier is far more expensive than adding app-server nodes.
Vendor lock-in and specialised expertise are shared drawbacks of both approaches.
Rule of thumb: data-intensive, well-structured operations → DB side; general logic → app server.

SELECT @var = col FROM … gotchas:
• No rows → variable keeps its previous value (still NULL if never set). Exception: COUNT(*) always assigns 0.
• Multiple rows → variable gets the last row's value; no error raised. Always filter by key or check COUNT first.
• Cannot mix variable assignment with data-retrieval in the same SELECT.

Key lesson — partial execution without transactions: Even a clean TRY/CATCH cannot undo work already committed before the error. In the phone-order example the customer's balance is decremented before the INSERT [Order Details] fails on a NOT NULL constraint — leaving the DB inconsistent. The fix: wrap the entire multi-step operation in BEGIN TRAN … COMMIT / ROLLBACK (see ACID demo above). @@ERROR is a simpler alternative for checking the immediately preceding statement only.
Client-side cursor
The server sends the entire result set to the client; the client processes it in its own memory. This is the default for most frameworks. ADO.NET: DataSet filled via DataAdapter. Good for: small result sets, disconnected scenarios. Causes a burst of network traffic at query time.
Server-side cursor
The result set stays on the server; rows arrive over the network only as fast as the client fetches them. Like video streaming — continuous but low-bandwidth. ADO.NET: DataReader. T-SQL: DECLARE … CURSOR FOR. Good for: very large result sets where loading everything into RAM would be too costly.
When to use a cursor
• The problem is naturally row-by-row and hard to express as set-based SQL
• Result set is small (performance not critical)
Typical cases: running totals, sequential pattern detection (e.g. consecutive small orders), dynamic SQL generation, conditional per-row branching (one row → insert order, next row → send email).
When NOT to use a cursor
• The task can be expressed cleanly with set-based DML (UPDATE, GROUP BY, window functions)
• Expected row count is in the millions
Cursors process one row at a time — orders of magnitude slower than a single set-based statement on large tables. ROW_NUMBER() OVER (…) often replaces what naively needs a cursor.
Cursor Lifecycle (T-SQL) 
-- 1. Declare variables to hold each row
DECLARE @id int, @name nvarchar(50), @addr nvarchar(60)

-- 2. Declare the cursor (FAST_FORWARD = forward-only, read-only)
DECLARE cursor_emp CURSOR FAST_FORWARD FOR
    SELECT EmployeeID, LastName, Address
    FROM   Employees ORDER BY LastName

-- 3. Open (allocates server-side result set)
OPEN cursor_emp

-- 4. Prime read
FETCH NEXT FROM cursor_emp INTO @id, @name, @addr

-- 5. Iterate (@@FETCH_STATUS = 0 while rows remain)
WHILE @@FETCH_STATUS = 0
BEGIN
    PRINT CONCAT('ID:', @id, ' ', @name, ' — ', @addr)
    FETCH NEXT FROM cursor_emp INTO @id, @name, @addr
END

-- 6. Close (releases row locks / position)
CLOSE cursor_emp
-- 7. Deallocate (frees the server-side result set memory)
DEALLOCATE cursor_emp
@@FETCH_STATUS = 0 → success; non-zero → past end or row deleted. Variable order and types must exactly match the cursor's SELECT list. DEALLOCATE (not CLOSE) is what frees server memory. Cursors can be nested — declare a second cursor inside the WHILE body.

FAST_FORWARD = FORWARD_ONLY + READ_ONLY + server-optimised fetch plan. Best choice for simple foreach patterns. Cannot update the current row through the cursor. Omitting the type usually gives GLOBAL DYNAMIC SCROLL — the most expensive option; always specify explicitly.
Atomicity
All steps of a transaction succeed, or none do. Guaranteed by the transaction log (WAL). Example failure: in the phone-order script the customer's balance was deducted before the INSERT [Order Details] failed — atomicity violated because there was no explicit transaction.
Consistency
The database must remain in a valid state at all times — foreign keys, CHECK constraints, and other integrity rules must hold after every transaction commits. Consistency is enforced jointly by constraints and ROLLBACK.
Isolation
Concurrent transactions interfere with each other as little as possible. Implemented via locking (shared, exclusive, update locks on rows / pages / tables). The degree of isolation is configurable — see isolation levels below.
Durability
Committed transactions survive hardware failure. Guaranteed by the Write-Ahead Log (WAL): every change is written to the log before it reaches the data pages. On crash recovery the log is replayed (roll forward) or reversed (roll back) as needed.
Explicit Transactions & XACT_ABORT
Every DML statement is an implicit transaction. Multi-step business logic needs an explicit one.
BEGIN TRAN
BEGIN TRY
    -- business-logic checks (stock, balance…)
    UPDATE Customers  SET balance = balance - @total WHERE …
    INSERT INTO Orders …
    INSERT INTO [Order Details] …

    IF @status = 0  COMMIT TRAN
    ELSE            ROLLBACK TRAN  -- business rule failed
END TRY
BEGIN CATCH
    ROLLBACK TRAN               -- unexpected error
    PRINT 'ERROR: ' + ERROR_MESSAGE()
END CATCH
XACT_ABORT controls what happens when a statement inside a transaction errors
Setting On error inside BEGIN TRAN Atomicity
SET XACT_ABORT OFF (default) Failing statement skipped; rest of batch continues; COMMIT commits partial work Can be violated
SET XACT_ABORT ON Entire transaction auto-rolled back immediately on any error Enforced automatically
Example: with OFF, inserting 3 rows where row 2 has a bad FK commits rows 1 and 3, skipping row 2 — atomicity violated. With ON the entire transaction is rolled back and t2 stays empty.
WAL + Nested Transactions
Write-Ahead Log (WAL)
Every data-modifying statement appends a sequentially numbered log record before touching the data page. Each record contains enough information to either
▸ roll forward — re-apply the change (used after a crash to recover committed work)
▸ roll back — undo the change (used by ROLLBACK or on partial failure)
WAL is also the foundation of log shipping and replication. Dump (Generate Scripts) contains the schema + data; Backup is the binary data file — WAL applies to both.
Nested transactions — @@TRANCOUNT tracks depth
BEGIN TRAN   -- @@TRANCOUNT = 1
    BEGIN TRAN -- @@TRANCOUNT = 2
    COMMIT TRAN -- @@TRANCOUNT = 1  (inner commit just decrements)
COMMIT TRAN   -- @@TRANCOUNT = 0  (outer commit actually writes)

-- But ROLLBACK always goes all the way to 0:
BEGIN TRAN   -- @@TRANCOUNT = 1
    BEGIN TRAN -- @@TRANCOUNT = 2
    ROLLBACK TRAN -- @@TRANCOUNT = 0 (ALL transactions rolled back)
Key rule: an inner COMMIT only decrements @@TRANCOUNT — data is not actually committed until the outermost COMMIT (count → 0). A ROLLBACK anywhere immediately resets @@TRANCOUNT to 0 and undoes all open transactions, not just the innermost one. Always check @@TRANCOUNT before issuing a ROLLBACK to avoid "no transaction active" errors.
Master tables (törzsállomány)
Tables that hold no foreign key to other tables — they are the roots of the FK graph. Typically small, static, and rarely modified. The further a table is from the transactional center in the FK chain, the more master-like it behaves.
Northwind examples: Categories, Suppliers, Shippers, Employees, Region
Transactional center (tranzakcionális középpont)
A table no other table references via FK, but which itself references many (or all) others — directly or indirectly. Large, frequently modified, often with a composite key (junction table). This is the table the main business transaction continuously writes to.
Northwind examples: [Order Details] (order entry), EmployeeTerritories (territory assignment)
How to read the structure
A well-designed OLTP schema reveals its most important transactions through its FK graph. Follow the FK arrows from the transactional center outward to the master tables — each hop toward a root is a step toward more static, less critical data. The center is the hottest table: most locks, most log writes, most contention. Index and partition strategy should prioritise it.

Memory-Optimized Tables (MOT)
Tables declared WITH (MEMORY_OPTIMIZED=ON) live entirely in RAM. No buffer pool, no page latches, no lock manager — data structures are designed for concurrent in-memory access. A durable copy is written to disk asynchronously for crash recovery.
Natively Compiled SPs
Stored procedures declared WITH (NATIVE_COMPILATION) are compiled to machine code at CREATE time — not interpreted at runtime like regular T-SQL. The resulting DLL is loaded into SQL Server's address space and eliminates interpreter overhead for high-frequency short transactions.
Optimistic Concurrency (MVCC)
MOTs use multi-version concurrency control instead of locks. Each row version is timestamped; readers never block writers. Conflicts are detected at commit time — transaction aborts and retries if another writer committed a conflicting update first. No deadlocks possible.
Use cases
• Extreme-throughput OLTP: session state, shopping carts, order queues
• High-frequency tick data or telemetry ingestion
• Latency-sensitive operations that saturate disk-based locking
⚠ Not suitable for large tables (must fit in RAM) or ad-hoc analytics
In-Memory OLTP — Syntax & Performance 
-- 1. Enable memory-optimized filegroup first
ALTER DATABASE Northwind
    ADD FILEGROUP fg_mem
    CONTAINS MEMORY_OPTIMIZED_DATA;

-- 2. Create memory-optimized table
CREATE TABLE ShoppingCart (
    SessionID  nvarchar(36) NOT NULL,
    ProductID  int          NOT NULL,
    Qty        int          NOT NULL,
    PRIMARY KEY NONCLUSTERED HASH (SessionID, ProductID)
        WITH (BUCKET_COUNT = 1048576)
) WITH (MEMORY_OPTIMIZED = ON, DURABILITY = SCHEMA_AND_DATA);

-- 3. Natively compiled stored procedure
CREATE PROCEDURE usp_AddToCart
    @sid nvarchar(36), @pid int, @qty int
WITH NATIVE_COMPILATION, SCHEMABINDING
AS BEGIN ATOMIC WITH
    (TRANSACTION ISOLATION LEVEL = SNAPSHOT, LANGUAGE = N'English')
    IF EXISTS (SELECT 1 FROM dbo.ShoppingCart
               WHERE SessionID=@sid AND ProductID=@pid)
        UPDATE dbo.ShoppingCart SET Qty += @qty
        WHERE  SessionID=@sid AND ProductID=@pid;
    ELSE
        INSERT dbo.ShoppingCart VALUES (@sid, @pid, @qty);
END;
