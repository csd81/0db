WebAssembly & the Post-Docker Edge

Docker's 500ms cold start was acceptable for centralized cloud deployments. At the edge — a 5G tower in Tokyo, a CDN node in Cairo — it was catastrophic. WebAssembly eliminated the OS entirely, cutting cold starts from milliseconds to microseconds and making truly global edge computing economically viable.

⚠ The Breaking Point: Docker Is Too Heavy for the Edge

Docker's isolation model boots a complete Linux userspace for each container. At 2020's scale of millions of edge functions globally, the overhead became the bottleneck itself. When a user in Bangkok clicks a button and triggers a serverless function, the function needs to respond in <50ms total — but the container might take 500ms just to cold-start.

✦ WebAssembly on the Server: Just Execute, Don't Boot

Wasm is a compact, portable binary instruction format. It was designed to be the assembly language of the web — safe, sandboxed, near-native speed. The key insight for server-side use: Wasm has no OS concept. No kernel to boot, no network stack to initialize, no filesystem to mount. The runtime just executes the binary. Any language (Rust, Go, C++, Python, JS) can compile to Wasm.

// Fermyon Spin: Wasm microservice (Rust → .wasm → edge execution)
// No Docker. No Kubernetes. No OS. Just the function.

// Rust source → compiled to .wasm by cargo build --target wasm32-wasi
use spin_sdk::{http::{IntoResponse, Request, Response}, http_component};

// This runs on a 5G tower edge node — binary is ~800KB
// Cold start: module is pre-instantiated in the Spin runtime pool
// First request: <500μs — not 500ms
#[http_component]
fn handle_request(req: Request) -> anyhow::Result<impl IntoResponse> {
    let body = req.body().clone();
    let result = process_order(body)?;   // business logic in pure Wasm
    Ok(Response::builder()
        .status(200)
        .body(result)
        .build())
}
// Same .wasm binary runs on: Cloudflare PoP Tokyo / AWS Wavelength Seoul
// / Fastly edge Chicago / your laptop (wasmtime) — zero recompilation
// Architecture-neutral: x86, ARM64, RISC-V — Wasm runtime handles translation

✦ Stateful Edge: Cloudflare Durable Objects

Stateless Wasm functions can't hold shared state — each invocation is independent. Cloudflare's Durable Objects solve this: each DO is a single globally-unique JavaScript/Wasm class instance, guaranteed to run at exactly one Cloudflare PoP worldwide. The DO holds in-memory state + durable KV storage. This makes real-time collaborative apps possible without any central AWS server.

// Durable Object: real-time collaborative document (Figma/Notion-style)
// One DO instance per document, worldwide — guaranteed single authoritative state

export class CollaborativeDoc {
  private clients = new Map<string, WebSocket>();
  private state:    DurableObjectState;           // persists across restarts
  private doc:      Record<string, any> = {};

  async fetch(request: Request) {
    const [client, server] = new WebSocketPair();
    server.accept();
    const clientId = crypto.randomUUID();
    this.clients.set(clientId, server);

    server.addEventListener('message', async e => {
      const op = JSON.parse(e.data as string);

      // Apply CRDT operation to document state
      this.doc = applyOp(this.doc, op);

      // Persist to durable storage (survives PoP restarts)
      await this.state.storage.put('doc', this.doc);

      // Broadcast to all connected clients — all hitting THIS PoP
      // Cloudflare routes any user opening this doc to the same PoP globally
      this.clients.forEach(ws => ws.send(JSON.stringify({op, doc: this.doc})));
    });
    return new Response(null, { status: 101, webSocket: client });
  }
}
// Result: real-time multiplayer doc — no AWS, no Redis, no central server
// All state at the Cloudflare edge, 30ms from 95% of internet users
WASI: The POSIX Interface for Server-Side Wasm WebAssembly System Interface (WASI) gives Wasm modules controlled access to filesystem, network, clocks, and environment — the minimum POSIX syscalls needed for server workloads, without a full OS kernel. A Rust program compiled with --target wasm32-wasip1 runs identically on any WASI-compatible runtime (wasmtime, Wasmer, Spin) on any hardware architecture. Docker's key value prop — "write once, run anywhere" — is now a property of the Wasm binary format itself, without the OS overhead.

The Verdict

WebAssembly on the server represents Docker's most credible long-term successor for edge workloads. It is not Docker's replacement for complex stateful backend services — Kubernetes will run those for decades. But for the billions of lightweight request-handling functions at the global edge, Wasm's sub-millisecond cold starts, tiny binary size, and architecture neutrality make Docker's overhead unjustifiable. The post-Docker era has already begun at the edge; it is spreading inward toward the data center.
