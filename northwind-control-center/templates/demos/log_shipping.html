What is Log Shipping?
A disaster-recovery technique that keeps a warm standby database by periodically backing up the primary's transaction log, copying the file to a secondary server, and restoring it. No dedicated replication infrastructure — uses standard SQL Server BACKUP/RESTORE.
Three-step pipeline
1. Backup Job — SQL Agent job on primary: BACKUP LOG … TO DISK every N minutes.
2. Copy Job — copies the .bak file to the secondary server share.
3. Restore Job — SQL Agent on secondary: RESTORE LOG … WITH NORECOVERY (keeps DB in RESTORING state so more logs can be applied later).
RPO & RTO
RPO (Recovery Point Objective) = backup interval — up to N minutes of data can be lost.
RTO (Recovery Time Objective) = restore remaining logs + RESTORE … WITH RECOVERY + redirect connections.
Typically minutes, not seconds — hence "warm," not "hot."
Cold / Warm / Hot standby
Cold — backup archive only; restore from scratch (hours).
Warm — log shipping; DB in RESTORING state, bring online in minutes.
Hot — Always On AG; synchronous mirror, automatic failover in seconds.
Log shipping sits in the middle: simpler and cheaper than Always On, faster than cold.

Log Shipping T-SQL steps:
-- Primary: backup every 15 min (SQL Agent job)
BACKUP LOG Northwind
    TO DISK = N'\\share\logs\NW_202601061500.bak'
    WITH COMPRESSION, STATS = 5;

-- Secondary: apply with NORECOVERY (stay in restoring)
RESTORE LOG Northwind
    FROM DISK = N'\\share\logs\NW_202601061500.bak'
    WITH NORECOVERY;

-- Failover: bring secondary fully online
RESTORE DATABASE Northwind WITH RECOVERY;
