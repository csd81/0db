MariaDB

Relational OLTP · MySQL-compatible SQL · Galera multi-master · Default in most Linux distros

Created in 2009 by Michael "Monty" Widenius — the original author of MySQL — as a community-driven fork after Oracle acquired Sun Microsystems (and MySQL). MariaDB maintains full MySQL wire-protocol compatibility while adding features that MySQL adopted years later: Recursive CTEs, JSON_TABLE, window functions, and the Galera Cluster extension for synchronous multi-master replication. It has replaced MySQL as the default in Debian, Ubuntu, Red Hat, and Arch Linux. Wikipedia, Deutsche Telekom, and cPanel all run MariaDB.

Storage Engines

InnoDB — default OLTP engine: MVCC, B+tree clustered index, foreign keys, row-level locking. Compatible with MySQL InnoDB.

Aria — MariaDB's improved MyISAM replacement: crash-safe, faster for read-heavy workloads without needing MVCC.

ColumnStore — columnar analytics engine (MPP): attach to a MariaDB server for HTAP — same SQL, different storage backend for analytics queries.

Spider — sharding engine: partition a table across multiple MariaDB servers, accessed as a single table via Spider engine metadata.

CONNECT — query external data sources (CSV, XML, ODBC, MongoDB, HDFS) as virtual tables.

MariaDB-Specific Features 
-- SEQUENCE objects (no AUTO_INCREMENT workarounds)
CREATE SEQUENCE order_seq START WITH 1000 INCREMENT BY 1;
SELECT NEXTVAL(order_seq);

-- JSON_TABLE (available before MySQL)
SELECT jt.*
FROM events,
JSON_TABLE(payload, '$.items[*]' COLUMNS (
    item_id INT PATH '$.id',
    qty     INT PATH '$.qty',
    price   DECIMAL(10,2) PATH '$.price'
)) AS jt;

-- Flashback (InnoDB via binlog)
-- (MariaDB-specific, not in MySQL)
FLASHBACK TABLE orders TO TIMESTAMP '2024-01-15 10:00:00';

-- Recursive CTE (added in MariaDB 10.2,
--   years before MySQL 8.0)
WITH RECURSIVE path(id, parent, depth) AS (
    SELECT id, parent_id, 0 FROM categories
    WHERE parent_id IS NULL
    UNION ALL
    SELECT c.id, c.parent_id, p.depth+1
    FROM categories c JOIN path p ON c.parent_id = p.id
)
SELECT * FROM path ORDER BY depth, id;
SQL Primer (MySQL-compatible + extras) 
-- 1. Window functions
SELECT customer_id,
       SUM(total) AS order_total,
       RANK() OVER (ORDER BY SUM(total) DESC) AS revenue_rank,
       NTILE(4) OVER (ORDER BY SUM(total)) AS quartile
FROM orders
GROUP BY customer_id;

-- 2. Dynamic columns (MariaDB-specific)
--    Store flexible attributes as a binary blob
CREATE TABLE products (
    id   INT PRIMARY KEY,
    name VARCHAR(100),
    attrs BLOB -- dynamic column store
);
INSERT INTO products VALUES (1, 'Shirt',
    COLUMN_CREATE('color','blue','size','M','material','cotton'));

SELECT COLUMN_GET(attrs, 'color' AS CHAR),
       COLUMN_GET(attrs, 'size' AS CHAR)
FROM products WHERE id = 1;

-- wsrep_local_state_comment: Synced

Killer Use Cases

LAMP Stack / Shared Hosting — MariaDB is the M in the modern LAMP/LEMP stack, pre-installed on virtually every Linux server and cPanel hosting environment.

Wikipedia — the Wikimedia Foundation runs MariaDB for one of the largest non-commercial database workloads: 60M+ articles, 500M+ monthly users.

MySQL Drop-in Replacement — the most common use case: replace MySQL 5.x with MariaDB 10.x with zero application changes. Get Galera clustering for free.

HTAP with ColumnStore — combine OLTP (InnoDB tables) and analytics (ColumnStore tables) in one MariaDB cluster using the same SQL interface.

vs MySQL

MariaDB wins: Galera multi-master, Aria engine, Spider sharding, dynamic columns, flashback, earlier feature parity (CTEs, JSON_TABLE arrived years before MySQL)

MySQL wins: InnoDB Group Replication, MySQL Router, Oracle commercial support, stricter type safety by default

Galera Replication (wsrep)

wsrep API — Write Set Replication protocol. Every transaction is packaged as a write set and broadcast to all nodes before commit.

Certification — each node independently certifies the write set against its own transaction history. Conflict detection via key set comparison.

Flow Control — if a node falls behind, Galera applies flow control to slow writes on fast nodes (pause signal). Monitor via wsrep_flow_control_paused.

SST / IST — new node joins via State Snapshot Transfer (full copy) or Incremental State Transfer (just the delta). Xtrabackup or mariabackup for SST avoids blocking the donor.

MaxScale Proxy

MaxScale is MariaDB's intelligent database proxy:

  • Read/Write Splitting — transparently routes writes to primary, reads to replicas
  • Connection Pooling — reduces connection overhead for thousands of app threads
  • Query Routing — route analytics queries to ColumnStore nodes
  • Automatic Failover — detect primary failure, elect new primary, update routing
