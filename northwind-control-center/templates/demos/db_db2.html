IBM Db2

Relational OLTP · SQL/PL · BLU Acceleration · Powers global banking & 95% of ATM transactions

Launched in 1983, IBM Db2 is the enterprise workhorse that silently runs a staggering portion of global finance. An estimated 95% of all ATM transactions touch a Db2 database. It dominates banking (Bank of America, HSBC), insurance claims processing, and US government agencies. Available on z/OS mainframes (where it processes truly extraordinary transaction volumes), Linux/UNIX/Windows, and IBM Cloud. Db2's BLU Acceleration feature delivers in-memory columnar analytics without changing a line of SQL — the same tables serve both OLTP and analytics workloads.

Core Features

Table Spaces — logical containers (SMS, DMS, Automatic). DMS gives full control of extents; Automatic Storage (default) manages space automatically.

pureXML — native XML storage with XQuery and SQL/XML. Stores XML as a parsed tree (not as CLOB), enabling efficient XPath/XQuery evaluation.

BLU Acceleration — column-organized tables stored in CPU cache-friendly format with SIMD compression. Up to 25x analytics speedup vs row tables, same SQL.

Temporal Tables — system-period and application-period temporal tables (bitemporal). Query data "as it was" at any historical moment.

Multi-Dimensional Clustering (MDC) — cluster rows along multiple dimension columns simultaneously, reducing I/O for multi-predicate scans.

Schema Example 
-- Bitemporal table (system + application time)
CREATE TABLE policy (
    policy_id    INTEGER NOT NULL,
    holder_name  VARCHAR(100),
    premium      DECIMAL(10,2),
    -- System time: when Db2 stored the row
    sys_start    TIMESTAMP(12) NOT NULL
                 GENERATED ALWAYS AS ROW BEGIN,
    sys_end      TIMESTAMP(12) NOT NULL
                 GENERATED ALWAYS AS ROW END,
    -- Application time: business validity
    bus_start    DATE NOT NULL,
    bus_end      DATE NOT NULL,
    PERIOD SYSTEM_TIME(sys_start, sys_end),
    PERIOD BUSINESS_TIME(bus_start, bus_end)
) WITH (DATA CAPTURE CHANGES);

-- Column-organized table for BLU Acceleration
CREATE TABLE sales_facts (
    date_id     INTEGER,
    product_id  INTEGER,
    region_id   SMALLINT,
    revenue     DECIMAL(15,2),
    units       INTEGER
)
ORGANIZE BY COLUMN;  -- BLU columnar storage
SQL + SQL/PL Primer 
-- 1. Temporal query — as-of historical data
SELECT policy_id, premium
FROM policy
FOR SYSTEM_TIME AS OF TIMESTAMP '2023-06-01 00:00:00'
WHERE holder_name = 'Smith';

-- 2. OLAP window functions
SELECT product_id, region_id, revenue,
       SUM(revenue) OVER (
           PARTITION BY region_id
           ORDER BY date_id
           RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_total,
       RATIO_TO_REPORT(revenue)
           OVER (PARTITION BY region_id) AS pct_of_region
FROM sales_facts
WHERE date_id BETWEEN 20240101 AND 20241231;

-- 3. SQL/PL stored procedure
CREATE OR REPLACE PROCEDURE process_payment(
    IN  p_acct    INTEGER,
    IN  p_amount  DECIMAL(10,2),
    OUT p_status  VARCHAR(20)
)
LANGUAGE SQL
BEGIN
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        ROLLBACK;
        SET p_status = 'ERROR';
    END;

    UPDATE accounts SET balance = balance - p_amount
    WHERE account_id = p_acct AND balance >= p_amount;

    IF (SQL%ROWCOUNT = 0) THEN
        SET p_status = 'INSUFFICIENT_FUNDS';
        ROLLBACK;
    ELSE
        COMMIT;
        SET p_status = 'OK';
    END IF;
END;

SELECT p.product_name, SUM(s.revenue)
FROM sales_facts s JOIN products p ON s.product_id = p.id
GROUP BY p.product_name
ORDER BY 2 DESC FETCH FIRST 10 ROWS ONLY;

Killer Use Cases

Banking Core Systems (z/OS) — Db2 for z/OS processes millions of transactions per minute with five-nines availability. The IBM mainframe + Db2 combination is the backbone of global interbank clearing.

Insurance Claims Processing — complex eligibility rules, historical policy data, and regulatory audit trails. Db2's bitemporal tables are uniquely suited here.

Airline Reservation (SABRE) — American Airlines' SABRE reservation system, one of the earliest real-time transaction processing systems, ran on Db2 predecessors and continues on Db2.

HTAP with BLU — add ORGANIZE BY COLUMN to analytical tables and query them alongside row-organized OLTP tables in the same SQL workload.

Performance

z/OS: Millions of TPS with Sysplex coupling

BLU analytics: 25x over row tables

HADR: RPO near-zero with sync replication

Adaptive Compression: 70-80% storage reduction

Storage & Buffer Pools

Buffer Pools — separate pools for different page sizes (4K/8K/16K/32K). Create dedicated buffer pools for hot tables. Key performance knob: BUFFERPOOL size.

Logging — write-ahead log to primary and mirror log paths. LOGARCHMETH1/2 controls archiving (local disk, TSM, S3). Log full = halt = emergency.

Adaptive Compression — row-level dictionary compression applied per page. Typically 70-80% size reduction, with decompression in CPU L3 cache.

HADR & Sysplex

HADR — High Availability Disaster Recovery: log shipping to standby server, automatic client reroute (ACR) on failover. SYNC/NEARSYNC/ASYNC modes.

z/OS Sysplex — IBM's proprietary data sharing architecture: multiple Db2 subsystems share one disk pool via a coupling facility (CF). Scales to hundreds of TPS millions.

pureScale — Linux/UNIX equivalent of Sysplex: clustered Db2 instances sharing storage via high-speed interconnect, continuous availability on commodity hardware.
