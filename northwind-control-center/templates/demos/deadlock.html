🔒 Deadlock Detection Demo

A deadlock occurs when Transaction A holds Lock 1 and waits for Lock 2 while Transaction B holds Lock 2 and waits for Lock 1 — a circular dependency that neither can resolve without external help. SQL Server's deadlock monitor scans the wait-for graph every 5 seconds and, upon detecting a cycle, selects the cheapest transaction as a victim and terminates it with error 1205. The demo places two sessions racing to update the same horse-racing bet in opposite column order, visualizing the lock graph forming and the moment the monitor fires. Applying WITH (UPDLOCK) at read time prevents the deadlock entirely by acquiring an update lock before write contention arises, enforcing a consistent lock acquisition order.

Result
Shared lock (S)
Acquired by SELECT. Multiple transactions can hold S locks on the same row simultaneously. Prevents X locks from being granted (no dirty reads). Duration depends on isolation level — released immediately after the read at Read Committed, held until end of transaction at Repeatable Read.
Exclusive lock (X)
Acquired by INSERT / UPDATE / DELETE. Only one transaction can hold an X lock; no other S or X locks are allowed on the same object. Always held until the transaction ends (COMMIT or ROLLBACK) — this prevents the lost update anomaly.
Update lock (U)
Temporary lock acquired while evaluating the WHERE filter before an update. Compatible with S (reads can proceed during the scan), but not with another U or X. Converted to X if the row matches the filter. Prevents the update starvation problem that would occur if two S+X upgrades raced each other.
Lock compatibility
A lock is granted if the requested mode is compatible with all existing locks on the object; otherwise the transaction enters wait (blocked) state until blocking locks are released.
S + S → both granted  |  S + U → both granted
S + X → X waits  |  X + X → second waits
U + U → second waits  |  U + X → X waits

Higher isolation → fewer anomalies → more blocking → lower throughput.
Even Read Uncommitted is valid when reads happen outside business hours with no concurrent writers (e.g. nightly ETL export from an OLTP DB).
SQL Server also provides SNAPSHOT and READ_COMMITTED_SNAPSHOT (row versioning — avoids read/write blocking by keeping old row versions in tempdb).
Isolation in Action — Horse Auction
Two concurrent buyers. Both read for sale, wait 10 s to charge their card, then update.
READ COMMITTED — logical error
1. My name reads → S lock acquired then released
2. Your name reads → S lock acquired then released
3. Both wait 10 s (no locks on the row)
4. My name UPDATE → X lock granted → COMMIT
5. Your name UPDATE → X lock granted → COMMIT
Result: both cards charged, horse sold twice (last writer wins)
REPEATABLE READ — deadlock → correct
1. My name reads → S lock held
2. Your name reads → S lock held (two S locks on row)
3. Both wait 10 s
4. My name tries X → blocked by Your name's S
5. Your name tries X → blocked by My name's S
6. Engine detects cycle → rolls back victim
Result: one buyer succeeds, one rolled back — only one card charged

Inspect live locks for a session:
SELECT resource_type, request_mode, request_status,
       resource_associated_entity_id, request_session_id
FROM   sys.dm_tran_locks
WHERE  request_session_id = @@SPID
Performance trade-off: Higher isolation → fewer anomalies, more blocking, lower throughput (transactions/sec). The right level depends on the workload — a nightly read-only ETL can safely use Read Uncommitted because no concurrent writers are active.
