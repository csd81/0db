IoT Communication Protocols
MQTT
Pub/Sub over TCP. QoS 0/1/2.
Best for: constrained devices, high-latency networks, millions of connections. Broker: EMQX, Mosquitto, AWS IoT Core. Header overhead: 2 bytes. Default port: 1883 (plain), 8883 (TLS).
CoAP
REST over UDP. Request/Response.
Best for: very constrained devices (Arduino, sensors with <100KB RAM). Binary encoding. Optional ACK. Maps to HTTP methods. Used in smart meters and embedded systems.
AMQP
Message queue protocol over TCP.
Best for: industrial IoT with reliable delivery requirements. RabbitMQ implements AMQP. Heavier than MQTT but richer routing (exchange types, binding keys). Used in industrial control systems.
WebSocket
Full-duplex over HTTP/1.1.
Best for: browser-based dashboards receiving real-time sensor updates. Works through corporate firewalls (port 80/443). Higher overhead than MQTT for constrained devices.
gRPC + Protobuf
Binary RPC over HTTP/2.
Best for: edge-to-cloud with structured schemas. Protobuf: 3–10× smaller than JSON. HTTP/2 multiplexing: many streams on one connection. Used by Tesla's Autopilot telemetry pipeline.
LwM2M
OMA standard over CoAP/UDP.
Device management: firmware OTA updates, object model for device resources. Standard for cellular IoT (LTE-M, NB-IoT). Used by telecom operators for SIM-based IoT devices globally.
🗄️ Storage & Data Layer
InfluxDB
Purpose-built time-series DB. TSM (Time-Structured Merge Tree) engine optimized for time-ordered writes. Automatic data downsampling (Flux tasks): raw data at 1s intervals → 1m averages → 1h averages for long-term storage. 10M writes/second on commodity hardware. Used by Tesla, SpaceX for telemetry.
TimescaleDB
PostgreSQL extension — time-series as hypertables. Automatic partitioning by time (chunk intervals). Continuous aggregates: materialized views that update automatically. Familiar SQL + time-series functions (time_bucket(), first(), last()). Compression: 95% space reduction on old chunks.
QuestDB
Fastest open-source time-series: 4M rows/second ingestion, 1B row queries in <1 second. SIMD-accelerated column scans. SQL with time-series extensions. Used by financial data and IoT companies that need extreme write throughput. Grafana-compatible.
AWS S3 / Azure Data Lake (Cold)
Raw sensor data tiered to object storage after hot window (30 days). Parquet format: column-oriented, compressed, readable by Spark/Athena for historical analysis. Sensor data from 2017 still queryable for ML training. Cost: ~$23/TB-month on S3 Standard vs $0.004/TB-month on Glacier.
⚙️ Compute & Processing
AWS IoT Core / Azure IoT Hub
Managed MQTT brokers that scale to billions of devices. Device Registry: unique device IDs, X.509 certificates, device twins (desired/reported state JSON). Rules Engine: SQL-like rules route messages to Lambda, Kinesis, DynamoDB based on topic/payload content. TLS mutual auth with device certificates.
Apache Kafka (Buffer Layer)
IoT broker → Kafka → time-series DB. Kafka acts as durable buffer: if InfluxDB is slow, messages queue in Kafka rather than backpressuring devices. Partitioned by device_id for ordered per-device processing. Kafka Connect sink connector writes batches to InfluxDB efficiently (10K messages/batch).
Apache Flink / Spark Streaming
Stateful stream processing. Tumbling windows: "average temperature per device per 5 minutes". Sliding windows: "max vibration in last 60 seconds, updated every 10 seconds". Complex event processing: "3 out-of-range readings within 1 minute → trigger maintenance alert". State stored in RocksDB.
Edge Compute (AWS Greengrass/Azure IoT Edge)
Run Lambda/containers on the device itself. Pre-filters noise before cloud upload. Anomaly detection at the edge: if sensor reading is within normal range → discard, don't transmit. Saves 80% of bandwidth costs. Critical for latency-sensitive control loops (sub-10ms reaction time).
🌐 Connectivity & Device Mgmt
Cellular IoT: LTE-M / NB-IoT
LPWAN (Low Power Wide Area Network) technologies for battery-powered devices. NB-IoT: 250 kbps, 10+ year battery life on AA cells, works underground. LTE-M: 1 Mbps, supports voice/mobility (asset tracking). Used in smart meters, parking sensors, agricultural soil monitors.
LoRaWAN
License-free sub-GHz radio. Range: 5 km urban, 15 km rural, 700 km via satellite (space LoRa). Ultra-low bandwidth (250 bps – 50 kbps). Perfect for: smart agriculture (soil moisture once/hour), city infrastructure (waste bin fill levels), flood sensors in remote areas. Battery: 10 years.
Device Twins / Shadow State
Cloud maintains JSON representation of each device's current state ("reported") and desired state ("desired"). Device syncs delta on reconnect. Example: desired={"firmware": "v2.1.0"} → device downloads and installs OTA update. Decouples device availability from configuration updates.
X.509 Certificate Auth
Each device has a unique private key and certificate provisioned at manufacture. Mutual TLS: device proves identity to broker + broker proves identity to device. Certificate Revocation: compromised device certificates revoked in device registry → blocked immediately. No shared secrets.
Workflow Deep Dive
1
Edge Filtering: Reduce Upload by 80%
Tesla FSD computer: 2,300 frames/second from 8 cameras + radar + ultrasonic. Raw bandwidth: ~40 Gbps. Onboard neural network processes locally — only anomalous events, disengagements, and training-relevant scenarios uploaded. Simple threshold filter: engine temp sampled every 100ms, only uploaded if delta > 2°C or absolute > 150°C. Reduces cellular data 95%.
2
MQTT QoS Levels: Delivery Guarantees
QoS 0 (At most once): fire-and-forget UDP-like. For non-critical metrics (ambient temperature). QoS 1 (At least once): PUBACK confirmation. May duplicate — receiver must deduplicate by message_id. For alerts and threshold events. QoS 2 (Exactly once): 4-message handshake (PUBLISH/PUBREC/PUBREL/PUBCOMP). For financial/billing-relevant telemetry (electricity meters).
3
Continuous Queries: Real-time Alerting
InfluxDB continuous query: SELECT mean(temp) INTO alert_check FROM sensors WHERE time > now()-5m GROUP BY device_id — runs every 30s. If mean > 180°C: emit alert event to Kafka → PagerDuty webhook → maintenance ticket created in ServiceNow. For time-critical systems: Flink stateful operator evaluates threshold on every incoming message with <10ms latency — vs 30s polling.
4
Data Retention & Downsampling
InfluxDB retention policy: hot data (raw, 1s resolution) retained 30 days. Continuous aggregate: every night, compute 1-minute averages for the past day → store in "warm" measurement (3 years). Compute 1-hour averages for past month → "cold" measurement (forever). Raw data deleted after 30 days. Storage cost: 1M devices × 10 metrics × 1/s = 864B data points/day. Compressed: ~50 GB/day.

🔋 Why MQTT, Not HTTP?
HTTP: 800+ byte headers per request. MQTT: 2-byte fixed header + topic. For a sensor sending 10 readings/second over cellular (NB-IoT at 10 kbps): HTTP would spend 64% of bandwidth on headers. MQTT: 5%. At 1M devices, MQTT saves $2M/month in cellular data costs. Also: MQTT broker maintains persistent sessions — device reconnects and gets missed messages without re-registering.
🎯 CAP Position
AP system for telemetry — a few lost sensor readings are acceptable. InfluxDB and TimescaleDB prioritize write availability: if a node fails, writes continue to other nodes (replication factor 3). The alert system, however, needs consistency — you can't miss a critical alert. Critical alerts use Kafka (durable, replicated) + Flink for guaranteed processing.
