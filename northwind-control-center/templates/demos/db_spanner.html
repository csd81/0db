Google Cloud Spanner

NewSQL · Standard SQL · TrueTime · Global strong consistency at planetary scale

Introduced internally at Google in 2012 and made public in 2017, Cloud Spanner is the first database to provide full SQL relational semantics with external consistency (effectively serializable + linearizable) across multiple geographically distributed datacenters — simultaneously. Its secret weapon is TrueTime: a globally synchronized clock using GPS receivers and atomic clocks in every Google datacenter, enabling Spanner to assign commit timestamps with bounded uncertainty and guarantee that a transaction committed at T1 is always visible to transactions starting after T1. Google's F1 advertising database (processing billions of dollars in ad revenue) runs on Spanner.

Unique Features

Interleaved Tables — co-locate child rows physically next to parent rows. An Orders table interleaved in Customers stores each customer's orders adjacent to the customer row, eliminating cross-node joins for parent-child queries.

Non-blocking DDL — schema changes (ADD COLUMN, CREATE INDEX) run online without locking the table. Spanner validates and applies schema changes gradually using its version system.

Stale Reads — read data as of a past timestamp (e.g., 10 seconds ago) without acquiring locks. Great for analytics queries that don't need the latest data.

STRUCT & ARRAY types — columns can be ARRAYs of any type, or STRUCTs (records). Enables denormalized storage patterns without sacrificing SQL.

Schema with Interleaving 
-- Parent table
CREATE TABLE Customers (
    CustomerID  STRING(36) NOT NULL,
    Name        STRING(MAX),
    Country     STRING(2),
    CreatedAt   TIMESTAMP  NOT NULL OPTIONS (allow_commit_timestamp=true)
) PRIMARY KEY (CustomerID);

-- Interleaved child — stored co-located with parent
CREATE TABLE Orders (
    CustomerID  STRING(36) NOT NULL,
    OrderID     STRING(36) NOT NULL,
    Total       NUMERIC,
    Status      STRING(20),
    OrderDate   DATE
) PRIMARY KEY (CustomerID, OrderID),
  INTERLEAVE IN PARENT Customers ON DELETE CASCADE;

-- Global index on non-key column
CREATE INDEX idx_orders_status
ON Orders(Status, OrderDate DESC)
STORING (Total);  -- avoid index join for Total
SQL Primer 
-- 1. Standard SQL with ARRAY and STRUCT
SELECT c.Name, c.Country,
       ARRAY(SELECT o.OrderID FROM Orders o
             WHERE o.CustomerID = c.CustomerID
             ORDER BY o.OrderDate DESC LIMIT 5) AS recent_orders,
       (SELECT COUNT(*) FROM Orders o
        WHERE o.CustomerID = c.CustomerID
          AND o.Status = 'SHIPPED') AS shipped_count
FROM Customers c WHERE c.Country = 'US';

-- 2. Stale read (bounded staleness, 10 seconds)
-- In client SDK:
-- spanner.single(TimestampBound.ofExactStaleness(10, TimeUnit.SECONDS))

-- Via SQL hint:
SELECT * FROM Orders@{FORCE_INDEX=idx_orders_status}
WHERE Status = 'PENDING'
ORDER BY OrderDate;

-- 3. Partitioned DML (large updates without locking)
-- Updates millions of rows efficiently
UPDATE Orders SET Status = 'ARCHIVED'
WHERE OrderDate < DATE_SUB(CURRENT_DATE(), INTERVAL 2 YEAR)
  AND Status = 'COMPLETED';

-- 4. Window functions (full SQL 2011)
SELECT CustomerID, OrderDate, Total,
       SUM(Total) OVER (
           PARTITION BY CustomerID
           ORDER BY OrderDate
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_total
FROM Orders ORDER BY CustomerID, OrderDate;

-- 5. JSON column operations
SELECT OrderID,
       JSON_VALUE(metadata, '$.source') AS source,
       JSON_QUERY(metadata, '$.items') AS items
FROM Orders WHERE metadata IS NOT NULL;

-- 6. EXPLAIN QUERY PLAN
EXPLAIN
SELECT c.Name, COUNT(o.OrderID)
FROM Customers c LEFT JOIN Orders o USING (CustomerID)
GROUP BY c.Name;
Killer Use Cases

Google F1 (AdWords) — Google's ad system migrated from sharded MySQL to Spanner. F1 serves hundreds of thousands of queries/second with external consistency across datacenters. This proved the concept at real scale.

Financial Services — multi-region financial ledgers that need serializable ACID transactions with zero data loss on regional failure. Spanner's RPO = 0 is unique for a distributed SQL database.

Global SaaS Applications — write-anywhere, read-locally from any region. No shard management. Automatic load balancing. Ideal for global B2B SaaS needing 99.999% uptime SLA.

Gaming (Snap) — Snapchat uses Spanner for its core data, benefiting from global scale and strong consistency for user profiles and snaps.

TrueTime API

The Problem — distributed systems can't rely on clocks being in sync (network delays, clock drift). Traditional approaches use vector clocks or logical timestamps.

Google's Solution — GPS receivers + atomic clocks in every Google datacenter. TrueTime exposes TT.now() which returns a time interval [earliest, latest] with bounded uncertainty (typically ±7ms).

Commit Wait — Spanner waits out the TrueTime uncertainty interval before returning from COMMIT. This guarantees that any transaction started after COMMIT sees the committed data.

Result: external consistency — if commit T1 completes before T2 starts, T2 always sees T1's writes. This is stronger than serializability alone.

Paxos & Splits

Splits — a Spanner instance is divided into "splits" (like shards). Each split covers a contiguous key range of one or more tables. Splits are automatically moved between nodes to balance load.

Paxos per split — each split is replicated across 3-5 replicas using Paxos consensus. A write requires a quorum (2 of 3, or 3 of 5). The leader replica processes writes; followers can serve stale reads.

Distributed transactions — transactions that span multiple splits use a 2-phase commit coordinated by a single transaction coordinator, with Paxos ensuring each split's portion commits atomically.
