Apache Cassandra

Wide-Column Store · CQL · Ring topology · Tunable consistency · Linear scaling

Cassandra was born at Facebook in 2008 to handle the inbox search problem — billions of messages written at any time from anywhere. Released as open-source in 2010, it became the database of choice when you need linear horizontal scaling with no single point of failure. Apple runs 75,000+ Cassandra nodes. Netflix uses it for watchlist and streaming metadata. Discord stored trillions of messages in Cassandra before migrating parts to ScyllaDB.

The Golden Rule

Design queries first, then schema. Cassandra cannot do joins, GROUP BY, or ad-hoc WHERE on non-key columns. Every query must be served by exactly one table designed for it. Denormalization is not a smell — it is the pattern.

If you need to query messages by user AND by time AND by conversation, you create three separate tables. Disk is cheap; unplanned full-cluster scans are not.

Ring Topology 
         ┌──── Token Ring ────┐
         │                    │
    Node A ──────────── Node B
    (0..63)             (64..127)
        │                   │
    Node D ──────────── Node C
   (192..255)          (128..191)

Each node owns a token range.
Partition key is hashed (Murmur3)
to determine which node is "home."

With Replication Factor = 3:
  Each partition is stored on 3
  consecutive nodes in the ring.

  Write to Partition X:
    Coordinator receives write
    → forwards to replica nodes 1,2,3
    → waits for QUORUM (2/3) acks
    → returns success to client

  No Primary! All nodes equal.
  Any node can be coordinator.
Keyspace > Table > Partition 
-- Keyspace (like a database/schema)
CREATE KEYSPACE messenger
  WITH replication = {
    'class': 'NetworkTopologyStrategy',
    'dc1': 3,   -- 3 replicas in dc1
    'dc2': 2    -- 2 replicas in dc2
  };

USE messenger;

-- Table: messages by conversation
-- Partition key: conversation_id
-- Clustering key: created_at (DESC = newest first)
CREATE TABLE messages_by_conversation (
  conversation_id UUID,
  created_at      TIMEUUID,   -- time + UUID, sortable
  sender_id       UUID,
  content         TEXT,
  attachment_url  TEXT,
  PRIMARY KEY (conversation_id, created_at)
) WITH CLUSTERING ORDER BY (created_at DESC)
  AND compaction = {
    'class': 'TimeWindowCompactionStrategy',
    'compaction_window_size': '1',
    'compaction_window_unit': 'DAYS'
  };
Primary Key Anatomy 
PRIMARY KEY ((partition_key), clustering_col1, ...)
              ─────┬─────    ──────────┬────────
                   │                  │
              Determines         Controls sort
              which node         order within
              owns the data      the partition

-- Examples:
PRIMARY KEY (user_id)
  → one partition per user (can get large)

PRIMARY KEY ((user_id, month), created_at)
  → composite partition (bounds partition size)
  → clustering on created_at within bucket

-- Counter table (distributed counters)
CREATE TABLE page_views (
  page_url TEXT PRIMARY KEY,
  views    COUNTER
);
UPDATE page_views SET views = views + 1
WHERE page_url = '/home';
Basic CQL 
-- Write (upsert by default — last-write-wins)
INSERT INTO messages_by_conversation
  (conversation_id, created_at, sender_id, content)
VALUES (
  a1b2c3d4-...,
  now(),
  f9e8d7c6-...,
  'Hello!'
) USING TTL 2592000;   -- auto-delete after 30 days

-- Read a partition (efficient)
SELECT * FROM messages_by_conversation
WHERE conversation_id = a1b2c3d4-...
LIMIT 50;

-- Slice within partition (by clustering key)
SELECT * FROM messages_by_conversation
WHERE conversation_id = a1b2c3d4-...
  AND created_at > minTimeuuid('2024-01-01 00:00')
  AND created_at < maxTimeuuid('2024-02-01 00:00');
Consistency Tuning 
-- Consistency levels per statement
SELECT * FROM orders WHERE id = ?
USING CONSISTENCY QUORUM;
-- QUORUM = ceil(RF/2) + 1 acks required
-- With RF=3: 2 nodes must respond

-- Levels (weakest → strongest):
--  ONE      – fastest, risk stale read
--  LOCAL_ONE – one replica in local DC
--  QUORUM   – majority cluster-wide
--  LOCAL_QUORUM – majority in local DC (recommended)
--  ALL      – all replicas (slowest, max consistency)

-- Strong consistency trick:
-- Write QUORUM + Read QUORUM → linearizable
-- (W + R > RF ensures overlap)

-- Lightweight transactions (compare-and-set)
INSERT INTO users (id, email)
VALUES (?, ?) IF NOT EXISTS;
-- Uses Paxos — 4 round trips, use sparingly
Real-World Deployments
Apple: Operates over 75,000 Cassandra nodes — one of the largest known deployments. Used for iCloud storage metadata, Siri backend, and app analytics.
Netflix: Stores watchlist, viewing history, and streaming metadata. Chose Cassandra for multi-region active-active — content must be available even when a region goes down.
Discord: Stored trillions of messages. Hit Cassandra's hot partition limits (large partitions, GC pressure) at scale; migrated to ScyllaDB (C++ Cassandra-compatible) for lower tail latency.

Storage Engine (LSM)

LSM Tree (Log-Structured Merge): Writes go to an in-memory structure (Memtable) + commit log. When full, Memtable flushes to an immutable SSTable on disk. SSTables are periodically merged (compacted) to remove tombstones and reduce read amplification.

Compaction strategies: STCS (Size-Tiered, default, great for write-heavy), LCS (Leveled, better for reads, write amplification), TWCS (Time-Window, ideal for time-series with TTLs).

Gossip protocol: Nodes share state by gossiping with 3 random peers every second. Failure detection uses a phi accrual detector. The entire cluster topology is known by every node within seconds.

Key Config 
# cassandra.yaml
cluster_name: 'prod-cluster'
num_tokens: 256           # vnodes (virtual nodes)

# Snitch: determines DC/rack topology
endpoint_snitch: GossipingPropertyFileSnitch

# Heap (tune for GC pressure)
# Set MAX_HEAP_SIZE in cassandra-env.sh
# Typically 8-16GB; G1GC recommended

# Read/write path
memtable_allocation_type: heap_buffers
commitlog_sync: periodic
commitlog_sync_period_in_ms: 10000

# Partitioner (consistent hashing)
partitioner: org.apache.cassandra.dht.Murmur3Partitioner

# nodetool commands:
nodetool status          # cluster state
nodetool tpstats         # thread pool stats
nodetool compactionstats # compaction progress
nodetool repair          # anti-entropy repair
