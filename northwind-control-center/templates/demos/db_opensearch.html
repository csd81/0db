OpenSearch

Search & Analytics · Lucene · Apache 2.0 fork of Elasticsearch · Security free, ML built-in

OpenSearch was born in 2021 when Amazon Web Services forked Elasticsearch 7.10 (the last Apache 2.0 version) after Elastic changed its license to SSPL. Today it is an independent Apache 2.0 project under the Linux Foundation umbrella, with contributions from AWS, SAP, Aryn, Eightfold.ai, and others. Compared to Elasticsearch, OpenSearch ships enterprise security (RBAC, field-level security, audit logging), k-NN vector search, and ML inference nodes as completely free, no-X-Pack-paywall features. AWS OpenSearch Service is the dominant managed search service on AWS, used by tens of thousands of customers for log analytics, application search, and security analytics.

Index & Mapping

Index — logical collection of documents (like a table). Sharded across data nodes. Each index has a mapping defining field types.

Document — JSON object stored in an index. Has a unique _id. Fields are analyzed/indexed based on the mapping.

Mapping — defines field types: text (analyzed, full-text), keyword (exact match), date, integer, float, knn_vector, nested.

Shard — a Lucene index. Primary shards = parallelism for writes. Replica shards = redundancy + read parallelism.

Index Templates — define settings/mappings for new indexes matching a pattern. Essential for log pipelines (daily indexes: logs-2024.01.01).

Query DSL Primer 
// 1. Full-text search with boosting
POST /products/_search
{
  "query": {
    "bool": {
      "must": {
        "multi_match": {
          "query": "wireless headphones",
          "fields": ["name^3", "description"],
          "type": "best_fields"
        }
      },
      "filter": [
        { "term":  { "category": "electronics" } },
        { "range": { "price": { "gte": 50, "lte": 300 } } },
        { "term":  { "in_stock": true } }
      ]
    }
  },
  "sort": [{ "_score": "desc" }, { "price": "asc" }],
  "from": 0, "size": 10
}

// 2. Aggregation — faceted search counts
POST /products/_search
{
  "query": { "match": { "description": "laptop" } },
  "aggs": {
    "by_category": {
      "terms": { "field": "category", "size": 10 },
      "aggs": {
        "avg_price": { "avg": { "field": "price" } },
        "price_ranges": {
          "range": {
            "field": "price",
            "ranges": [
              { "to": 500 },
              { "from": 500, "to": 1000 },
              { "from": 1000 }
            ]
          }
        }
      }
    }
  },
  "size": 0
}

// 3. k-NN vector search (semantic search)
POST /products/_search
{
  "query": {
    "knn": {
      "embedding": {
        "vector": [0.12, -0.45, 0.89, ...],  // 1536 dims
        "k": 10
      }
    }
  }
}

// 4. Hybrid search (keyword + vector)
POST /products/_search
{
  "query": {
    "hybrid": {
      "queries": [
        { "match": { "name": "wireless headphones" } },
        { "knn": { "embedding": { "vector": [...], "k": 10 } } }
      ]
    }
  },
  "search_pipeline": "normalization-processor"
}

// 5. PPL (Piped Processing Language) — alternative query syntax

Killer Use Cases

Log Analytics (AWS OpenSearch Service) — ingest terabytes of application/infrastructure logs from Kinesis/Firehose, search and visualize in OpenSearch Dashboards. The #1 use case on AWS.

Security Analytics (Wazuh) — Wazuh SIEM uses OpenSearch as its backend for storing and querying security events. Free security plugin eliminates the X-Pack paywall that Elasticsearch imposes.

E-commerce Product Search — full-text search with faceted filters (category, price range, brand), spell correction, and now hybrid search combining keyword + semantic (k-NN) results.

RAG / Semantic Search — store LLM-generated embeddings alongside documents, perform k-NN search to retrieve relevant context for Retrieval-Augmented Generation pipelines.

vs Elasticsearch

OpenSearch wins: Apache 2.0 license, security plugin free, k-NN and ML free, PPL query language, AWS managed service

Elasticsearch wins: EQL (Event Query Language), more mature vector search (ES 8.x), better fleet management, larger community historically

Lucene & Segments

Lucene Segments — each shard is a Lucene index composed of immutable segments. Writes create new segments; background merge combines small segments into larger ones (reducing search overhead).

Inverted Index — for text fields: term → list of document IDs (posting list). Fast full-text search via posting list intersection. Compression via FOR (Frame of Reference) encoding.

BKD Trees — for numeric/geo fields: block KD-tree for fast range queries. Enables efficient range filters on price, timestamp, coordinates.

Translog — write-ahead log for durability. Writes go to translog first (fsync configurable), then indexed into Lucene in-memory buffers, then flushed to disk segments.

Key Settings
number_of_shardsImmutable; set based on expected data size (10GB/shard rule of thumb)
number_of_replicasMutable; 1 for HA, 0 for bulk ingestion speed
refresh_interval-1 during bulk load; 1s for near-real-time search
