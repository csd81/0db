What is Merge Replication?
Bi-directional synchronisation: both the Publisher and each Subscriber can independently make changes while disconnected. When they reconnect, the Merge Agent reconciles all changes. A rowguid column tracks each row's identity across all participants.
Conflict Detection
A conflict occurs when the same row is modified on both sides between syncs. SQL Server detects conflicts by comparing the generation counter and timestamps. Conflicts are logged in the MSmerge_conflicts_info table for later inspection.
Conflict Resolution Strategies
Must be defined before deployment:
• Publisher wins (default) — central authority
• Subscriber wins — field override
• Latest timestamp wins — requires accurate clocks (NTP)
• Custom resolver — COM object or stored procedure
Use cases
• Mobile users / field workers who go offline
• Remote branch offices with unreliable WAN
• Disconnected sites that need full local autonomy
• Multi-master topologies (requires carefully designed conflict rules)

Northwind Lab Scenario — Merge: Traveling Sales Employees
Scenario
Sales employees travel and place/modify orders while offline. The Orders table in northwind on PRIM is published as nw_merge. Two subscribers (nw_merge_1 on THIRD and nw_merge_2 on PRIM) hold pull subscriptions so each employee schedules their own sync when connectivity allows.
ROWGUIDCOL auto-added — legacy app risk: If the Orders table has no UNIQUEIDENTIFIER ROWGUIDCOL column, merge replication adds one automatically (e.g. rowguid). This changes the table's column list and can crash legacy application code that uses SELECT * or positional column access. The column is removed when the publication is deleted.

"First to Publisher wins" — the real rule
The default resolver is often called Publisher wins, but the precise rule is:
Conflict typeWinner
Publisher vs. SubscriberPublisher change wins
Subscriber A vs. Subscriber B (pull)First to sync with Publisher wins
The Replication Conflict Viewer (pop-up menu on the publication) shows all conflicts and lets you override the resolution by clicking Submit Winner or Submit Loser. Conflicts are stored in MSmerge_conflicts_info.
CAP Theorem & Eventual Consistency
Merge replication sacrifices strong consistency (C) in favour of availability (A) and partition tolerance (P). While disconnected, each node is available but may hold stale data. After sync, all nodes converge — this is eventual consistency. According to CAP, no distributed system can fully satisfy all three properties simultaneously; merge replication makes A+P the priority, accepting temporary C violations.
