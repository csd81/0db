Azure SQL Database

Managed SQL Server · Hyperscale · Intelligent Query Processing · Always On

Azure SQL Database (2010) is Microsoft's fully managed relational database built on SQL Server (Socrates SIGMOD 2019 paper). Unlike RDS for SQL Server (which runs SQL Server on VMs), Azure SQL Database is a cloud-native service engineered from the ground up for Azure infrastructure. It offers three tiers: General Purpose (remote storage), Business Critical (local SSD with Always On AG under the hood), and Hyperscale (up to 100 TB with snapshot-based instant backups). Intelligent Query Processing (IQP) automatically optimizes queries without index hints or manual tuning.

Three Service Tiers
General Purpose
  • Remote Azure Premium Storage
  • 2–80 vCores, up to 4TB
  • Best for: typical business apps
  • RPO: 1–6 hours (geo-backup)
Business Critical
  • Local SSD (NVMe), In-Memory OLTP
  • 4 replicas (Always On AG internal)
  • Read replica included free
  • 99.99% SLA, sub-5ms failover
Hyperscale
  • Up to 100 TB, auto-grow
  • Snapshot-based instant backup
  • Named replicas (up to 30)
  • Log service architecture

Intelligent Query Processing (IQP) 
-- IQP: automatic query optimization (compatibility level 150+)
ALTER DATABASE NorthwindDB SET COMPATIBILITY_LEVEL = 160;

-- ── Adaptive Joins: switches between hash/loop/merge at runtime
-- No query hints needed — optimizer picks at first row
SELECT o.OrderID, c.CompanyName
FROM Orders o JOIN Customers c ON o.CustomerID = c.CustomerID
WHERE o.OrderDate > '2024-01-01';
-- Optimizer may start with nested-loop, switch to hash if rows exceed threshold

-- ── Batch Mode on Rowstore: columnar processing for row heaps
-- Previously only columnstore indexes got batch mode
-- Now regular B-tree tables get batch mode for analytics

-- ── Approximate COUNT DISTINCT (Hyperloglog, 2% error, 97% faster)
SELECT APPROX_COUNT_DISTINCT(CustomerID) FROM Orders;
-- vs COUNT(DISTINCT CustomerID) — exact but 97x slower

-- ── Parameter Sensitive Plan Optimization (SQL 2022)
-- Automatically generates multiple plans for skewed parameters
-- Previously: one cached plan → bad for all but the "average" parameter

-- ── Columnstore indexes for HTAP on same table
CREATE NONCLUSTERED COLUMNSTORE INDEX NCCI_Orders_Analytics
ON Orders (OrderDate, CustomerID, Freight, ShipCountry);
-- OLTP: B-tree for point lookups
-- Analytics: columnstore for full-table scans
-- Same table, both indexes active simultaneously
Elastic Pools & Serverless 
-- ── Elastic Pool: share DTU/vCore budget across many DBs
-- SaaS multi-tenant: 1000 tenant DBs sharing 200 vCores
-- Peak usage is rarely concurrent → pooling saves 60-80%

-- ── Serverless compute tier (auto-pause + auto-resume)
CREATE DATABASE NorthwindDev
( SERVICE_OBJECTIVE = 'GP_S_Gen5_2',  -- Serverless, 2 vCores max
  AUTO_PAUSE_DELAY = 60,              -- pause after 60 min idle
  MIN_CAPACITY = 0.5 );               -- 0.5 vCore minimum when active

-- Billed per second of compute used (not provisioned)
-- Database auto-pauses when idle, auto-resumes on connection

-- ── Ledger: tamper-proof auditable tables
-- Blockchain-like verification using SQL Server 2022 ledger
CREATE TABLE account_transactions (
    transaction_id INT PRIMARY KEY,
    account_id INT,
    amount DECIMAL(10,2),
    balance DECIMAL(10,2)
)
WITH (LEDGER = ON);  -- all rows cryptographically chained
-- Azure Confidential Ledger stores root digests externally

-- Verify integrity:
EXECUTE sp_verify_database_ledger_from_digest_storage;
T-SQL — Azure SQL Specific Features 
-- ── 1. Temporal tables — system-versioned history
CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name NVARCHAR(100),
    Salary DECIMAL(10,2),
    Department NVARCHAR(50),
    ValidFrom DATETIME2 GENERATED ALWAYS AS ROW START,
    ValidTo   DATETIME2 GENERATED ALWAYS AS ROW END,
    PERIOD FOR SYSTEM_TIME (ValidFrom, ValidTo)
)
WITH (SYSTEM_VERSIONING = ON (HISTORY_TABLE = dbo.EmployeesHistory));

-- Query past state (as of a point in time)
SELECT * FROM Employees
FOR SYSTEM_TIME AS OF '2024-01-01T00:00:00';

-- ── 2. JSON support (full round-trip)
SELECT e.EmployeeID,
       JSON_VALUE(e.ContactInfo, '$.email') AS Email,
       JSON_QUERY(e.ContactInfo, '$.phones') AS Phones
FROM Employees e
WHERE JSON_VALUE(e.ContactInfo, '$.city') = 'Seattle';

-- ── 3. Graph tables (SQL Server / Azure SQL)
CREATE TABLE Employees AS NODE;
CREATE TABLE Reports AS EDGE;

INSERT INTO Reports ($from_id, $to_id)
SELECT e1.$node_id, e2.$node_id
FROM Employees e1, Employees e2
WHERE e1.EmployeeID = 5 AND e2.EmployeeID = 2;

-- MATCH clause for graph traversal
SELECT manager.Name AS Manager, emp.Name AS DirectReport
FROM Employees manager, Reports, Employees emp
WHERE MATCH(manager-(Reports)->emp);

-- ── 4. Query Store — automatically captures plans and regressions
ALTER DATABASE NorthwindDB SET QUERY_STORE = ON;
-- Force a previous good plan (plan pinning):
EXEC sp_query_store_force_plan @query_id = 1, @plan_id = 3;

-- ── 5. External REST endpoint (SQL 2022 / Azure SQL)
-- Call REST APIs directly from T-SQL
EXEC sp_invoke_external_rest_endpoint
    @url = 'https://api.openai.com/v1/embeddings',
    @method = 'POST',
    @headers = '{"Authorization":"Bearer ...","Content-Type":"application/json"}',
    @payload = '{"model":"text-embedding-ada-002","input":"Hello world"}',
    @response = @result OUTPUT;
Ideal Use Cases
.NET / Azure Enterprise Apps

The natural choice for .NET applications on Azure. Entity Framework, Dapper, ADO.NET all have first-class support. Azure AD integrated auth for zero-credential database access.

SaaS Multi-Tenant (Elastic Pools)

Law firms, medical SaaS, financial SaaS — each tenant gets their own database (data isolation) but shares compute via Elastic Pool. Scales economically to thousands of tenants.

Compliance / Auditing (Ledger)

Financial transactions requiring tamper-proof audit trails. SQL Ledger tables provide cryptographic chain-of-custody without external blockchain infrastructure.
