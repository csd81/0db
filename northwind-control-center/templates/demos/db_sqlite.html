SQLite

Embedded Relational · SQL · Serverless · Most deployed database engine on Earth

Created by D. Richard Hipp in 2000, SQLite is the world's most widely deployed database — estimated to run on over a trillion instances. Every iPhone, Android device, Chrome browser, Firefox, macOS, Python installation, and Electron app ships with SQLite. Its killer feature is simplicity: no server process, no configuration, no network protocol — just a single C library linked into your application that reads and writes a single portable file. It's the right choice for local data, mobile apps, caching, and testing, but not for high-concurrency server workloads.

Type System

SQLite uses dynamic typing with type affinity — columns declare a preferred type but any value can be stored in any column. This is the biggest gotcha for SQL Server/Oracle developers.

5 storage classes: NULL · INTEGER (1-8 bytes) · REAL (float64) · TEXT (UTF-8/16) · BLOB

Type affinity determines coercion: declare INT and SQLite stores integers efficiently; declare TEXT and numbers become strings.

Strict Tables (3.37+) — add STRICT to CREATE TABLE to enforce proper type checking like traditional RDBMS.

WITHOUT ROWID tables — use the PRIMARY KEY as the B-tree key, eliminating the implicit 64-bit rowid and saving space for narrow-key tables.

Schema Example 
-- Strict table (3.37+) — type-safe
CREATE TABLE orders (
    id       INTEGER PRIMARY KEY, -- alias for rowid
    cust_id  INTEGER NOT NULL,
    total    REAL    NOT NULL,
    status   TEXT    DEFAULT 'pending',
    created  TEXT    DEFAULT (datetime('now'))
) STRICT;

-- WITHOUT ROWID — for key-value lookups
CREATE TABLE sessions (
    token    TEXT PRIMARY KEY,
    user_id  INTEGER,
    expires  INTEGER -- Unix timestamp
) WITHOUT ROWID;

-- Virtual table (FTS5 full-text search)
CREATE VIRTUAL TABLE docs USING fts5(
    title, body,
    content='articles',
    content_rowid='id'
);

-- Partial index
CREATE INDEX idx_active_orders
ON orders(cust_id)
WHERE status = 'pending';
SQL Primer 
-- 1. Window functions (3.25+)
SELECT name, salary,
       RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS dept_rank,
       AVG(salary) OVER (PARTITION BY dept) AS dept_avg
FROM employees;

-- 2. Recursive CTE
WITH RECURSIVE fib(a, b) AS (
    VALUES (0, 1)
    UNION ALL
    SELECT b, a+b FROM fib WHERE b < 1000
)
SELECT a FROM fib;

-- 3. JSON functions (built-in extension)
SELECT json_extract(payload, '$.user.name') AS username,
       json_each.value AS item
FROM events, json_each(events.payload, '$.items')
WHERE json_extract(payload, '$.type') = 'order';

-- 4. FTS5 full-text search
INSERT INTO docs(title, body) VALUES
    ('SQLite Guide', 'SQLite is an embedded database...');

SELECT title, snippet(docs, 1, '','','...',10)
FROM docs WHERE docs MATCH 'embedded database'
ORDER BY rank;

-- 5. UPSERT (INSERT OR REPLACE / ON CONFLICT)
INSERT INTO sessions(token, user_id, expires)
VALUES ('abc123', 42, strftime('%s','now')+3600)
ON CONFLICT(token) DO UPDATE SET
    expires = excluded.expires,
    user_id = excluded.user_id;

Killer Use Cases

Mobile Apps (iOS/Android) — CoreData on iOS wraps SQLite. Android Room is SQLite. WhatsApp stores message history in SQLite.

Browser Local Storage — Chrome uses SQLite for cookies, history, bookmarks, and web storage. Firefox uses it for profiles.

Unit Testing — spin up an in-memory SQLite database (:memory:) for each test, no cleanup needed. Fast and isolated.

Local-First Applications — offline-capable apps store data locally, sync to server. Used in VS Code, Obsidian, and many Electron apps.

When NOT to Use SQLite

High concurrency writes — SQLite allows only one writer at a time (even in WAL mode). For multiple concurrent writers, use PostgreSQL or MySQL.

Network access — SQLite has no server; it can't be shared across machines without a file system mount (which is unreliable over NFS).

Very large datasets with complex queries — no query parallelism, no partitioning, single-threaded B-tree. PostgreSQL handles this better.

Performance

In-memory (:memory:): <1μs reads

WAL mode reads: concurrent, no lock

Bulk insert: wrap in a single transaction for 100x speedup

PRAGMA cache_size: default 2000 pages; increase for read-heavy workloads

PRAGMA synchronous=NORMAL: safe but faster than FULL

Storage Engine

B-tree Pages — all tables and indexes are stored as B-trees in fixed-size pages (default 4096 bytes). The pager module manages the page cache.

Journal Mode — default: rollback journal (writes original page before modifying). WAL mode: writes to a separate WAL file, enabling concurrent readers.

WAL Mode — enables read/write concurrency (readers don't block writer, writer doesn't block readers). Use PRAGMA journal_mode=WAL;.

VDBE — Virtual DataBase Engine: a register-based virtual machine that executes compiled SQL bytecode. EXPLAIN shows the bytecode.

Key PRAGMAs
