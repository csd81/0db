Snapshot Replication — How It Works
  1. Configure PRIM as its own distributor (or separate server)
  2. Create publication on Northwind DB, choose Snapshot type
  3. Add filter: WHERE ShipCountry = 'USA' on Orders article
  4. Set snapshot agent schedule (e.g. every 2 min for demo)
  5. Create push subscription → nw_repl destination DB
  6. Snapshot agent runs bcp → places .bcp/.sch/.pre/.idx files in snapshot folder
  7. Distribution agent applies snapshot to subscriber
Note: bcp locks the whole table during copy. Schedule snapshot agent in off-peak hours in production.

Transactional Replication

Copies only changed data — near real-time. The Log Reader Agent reads the publisher's WAL and copies matching transactions to the distribution database. The Distribution Agent runs those transactions on the subscriber.

Flow
Publisher WAL → Log Reader Agent → Distribution DB → Distribution Agent → Subscriber
Scenario: Beverages-Only Branch Sync
-- Filter for Order Details:

Peer-to-Peer Transactional Replication

Allows writes to any node — all changes propagate to all nodes automatically, exactly once. Designed for read scale-out load balancing.

Concurrent writes to the same record on different nodes = write conflict → critical error → manual resolution required.
PropertyP2P Transactional
Write nodesAny peer
Write conflictsNot supported — manual fix
Read scalingYes — direct any read to any peer
Use caseHigh-read, low-write, global read perf
Merge Replication

Scenario: traveling sales employees make new orders offline. When they reconnect, changes merge with the central DB — even if two people modified the same record.

How It Works
  • Triggers detect changes: MSmerge_ins_*, MSmerge_upd_*, MSmerge_del_*
  • Changes logged in MSmerge_* system tables at both publisher and subscriber
  • Merge Agent syncs changes bi-directionally
  • Each article needs a UNIQUEIDENTIFIER ROWGUIDCOL column for global record identity
  • Distribution DB stores only history and error info
Adding GUID column can crash legacy apps already using the table — check before enabling.
Conflict Resolution

When two nodes modify the same record, the Merge Agent applies a resolver. Default strategy: "first to Publisher wins".

Conflict TypeWinner
Publisher vs SubscriberPublisher change wins
Two pull SubscribersFirst to sync with Publisher wins
Server subscriptionExplicit numeric priority wins

CAP Theorem & Eventual Consistency

Merge replication demonstrates eventual consistency: the system may be inconsistent at any moment, but if no further updates occur, all nodes will converge to the same state after some time.

According to the CAP theorem, a distributed system cannot simultaneously guarantee all three of:

{% for letter,name,color,desc in [ ('C','Consistency','#f59e0b','All reads return the most recent write or an error'), ('A','Availability','#60a5fa','All reads contain data (may not be most recent)'), ('P','Partition Tolerance','#34d399','System tolerates network failures'), ] %}

Merge replication trades off:
Strong Consistency (may be stale)
Availability (always readable)
Partition Tolerance (offline ok)
Since network partitions always happen, C ↔ A is the real trade-off in practice.
Log Shipping — Warm Standby

Log shipping creates an exact copy of a database on a secondary server for disaster recovery. Unlike replication, it operates on the whole database, not articles. It involves three jobs:

Failover Clusters (Always On)

Failover Clustering (FC) requires Windows Server + multiple physical machines. Cannot be demonstrated in a single-machine lab.

Always On Availability Groups (SQL Server 2012+)
  • Define availability databases that fail over as a group
  • Supports automatic read load balancing for secondaries
  • Sub-second RTO (Recovery Time Objective) vs minutes for log shipping
  • Synchronous or asynchronous commit modes
  • Primary for writes, secondaries for reads (off-loads reporting)
