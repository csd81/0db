Pinecone

Vector Database · HNSW ANN · Semantic Search · RAG Infrastructure

Pinecone launched in 2021 as the first managed vector database, arriving just as large language models began producing high-dimensional embeddings that needed to be stored and searched efficiently. A vector database answers questions traditional databases cannot: "find the 10 most semantically similar documents to this query." It powers Retrieval-Augmented Generation (RAG), semantic search, recommendation systems, duplicate detection, and anomaly detection — making it the core infrastructure of the AI application layer.

What is a Vector?

An embedding model (like OpenAI text-embedding-3-small) converts any text, image, or audio into a dense float array — e.g., [0.021, -0.143, 0.892, ...] with 1536 dimensions.

Semantically similar inputs have vectors that are close in this high-dimensional space. "The cat sat on the mat" and "A feline rested on a rug" will have nearly identical vectors even though they share no keywords.

RAG Architecture 
─── INDEXING PHASE (one-time) ───────────────

  Documents
    │
    ▼
  [Chunking] → split into 500-token chunks
    │
    ▼
  [Embedding Model] → OpenAI / Cohere / local
    │ 1536-dim float vector per chunk
    ▼
  [Pinecone upsert] → vector + metadata stored

─── QUERY PHASE (runtime) ───────────────────

  User question: "How do I reset my password?"
    │
    ▼
  [Embedding Model] → question vector
    │
    ▼
  [Pinecone query(vector, top_k=5)]
    │ Returns 5 most similar document chunks
    ▼
  [LLM prompt: "Using this context: {chunks}
    Answer: {question}"]
    │
    ▼
  Grounded, accurate answer
Index / Namespace / Vector 
-- Pinecone data model hierarchy:

Index (one per application or use case)
  └─ Namespace (logical partition, e.g. by tenant)
       └─ Vector {
            id: "doc-chunk-00042",    // unique string
            values: [0.021, -0.143, ...],  // 1536 floats
            metadata: {               // JSON for filtering
              "source": "docs/setup.md",
              "page": 3,
              "language": "en",
              "category": "troubleshooting"
            }
          }

-- Index configurations:
-- dimension: must match embedding model output
-- metric: cosine | dotproduct | euclidean
-- pod_type: p1 (balanced) | p2 (low-latency) | s1 (storage)
-- replicas: copies for read throughput

-- Serverless (recommended, 2024+):
-- No pod management, pay per query
Hybrid Filtering 
-- Metadata filter: combine ANN + exact filter
-- Find top 10 similar English troubleshooting docs

results = index.query(
    vector=query_embedding,   # 1536-dim float list
    top_k=10,
    namespace="docs-v2",
    filter={
        "language": {"$eq": "en"},
        "category": {"$in": ["troubleshooting", "faq"]},
        "page": {"$lt": 50}
    },
    include_metadata=True
)

-- Sparse-dense hybrid (keyword + semantic)
results = index.query(
    vector=dense_embedding,
    sparse_vector={             # BM25 keyword weights
        "indices": [42, 91, 201],
        "values": [0.8, 0.6, 0.4]
    },
    top_k=10
)
Python SDK: Index + Upsert 
from pinecone import Pinecone
from openai import OpenAI

pc = Pinecone(api_key="YOUR_API_KEY")
oai = OpenAI()

# Create serverless index
pc.create_index(
    name="docs",
    dimension=1536,
    metric="cosine",
    spec=ServerlessSpec(cloud="aws", region="us-east-1")
)
index = pc.Index("docs")

# Embed + upsert chunks
chunks = load_and_chunk_documents("docs/")
embeddings = oai.embeddings.create(
    model="text-embedding-3-small",
    input=[c.text for c in chunks]
).data

vectors = [
    {"id": f"chunk-{i}",
     "values": e.embedding,
     "metadata": {"text": chunks[i].text,
                  "source": chunks[i].source}}
    for i, e in enumerate(embeddings)
]
index.upsert(vectors=vectors, namespace="v1")
RAG Query + LLM 
def answer(question: str) -> str:
    # 1. Embed the question
    q_vec = oai.embeddings.create(
        model="text-embedding-3-small",
        input=question
    ).data[0].embedding

    # 2. Find semantically similar chunks
    results = index.query(
        vector=q_vec,
        top_k=5,
        namespace="v1",
        include_metadata=True
    )

    # 3. Build context from retrieved chunks
    context = "\n\n".join([
        m["metadata"]["text"]
        for m in results["matches"]
    ])

    # 4. LLM generates grounded answer
    resp = oai.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system",
             "content": f"Context:\n{context}"},
            {"role": "user", "content": question}
        ]
    )
    return resp.choices[0].message.content
Use Cases
RAG (Q&A over documents): Store embedded company docs, code, policies. Query returns relevant chunks as context for the LLM. Eliminates hallucination on domain-specific knowledge.
Semantic search: Replace keyword search (BM25) with meaning-based retrieval. "Show me comfortable running shoes for wide feet" matches "athletic footwear broad toe box" without keyword overlap.
Recommendations: User's embedding (from past behavior) queries item embeddings. Returns items similar to what they've interacted with. Used by Spotify, Netflix, Etsy.
Duplicate detection: Embed all records, query each against the index. High cosine similarity (>0.98) indicates duplicates. Scales to millions of records with ANN.

HNSW: How ANN Works

Approximate Nearest Neighbor (ANN): Exact nearest-neighbor search in 1536 dimensions requires scanning every vector — O(N×D). HNSW builds a multi-layer navigable small-world graph where each node connects to M neighbors across layers.

HNSW query: Start at a random entry point in the top layer, greedily navigate toward the query vector, descend layers until reaching the bottom. Returns ~top-k results in O(log N) time. Recall typically 95–99%.

Pinecone's twist: Shards (pods) each hold a partition of the index. Queries fan out to all shards, each returns local top-k, global top-k is merged. Replicas provide read throughput scaling.

pgvector Alternative 
-- pgvector: vector search inside PostgreSQL
-- No separate service, ACID guarantees

CREATE EXTENSION vector;

CREATE TABLE documents (
  id    BIGSERIAL PRIMARY KEY,
  chunk TEXT,
  vec   VECTOR(1536),
  source TEXT,
  category TEXT
);

-- HNSW index (pgvector 0.5+)
CREATE INDEX ON documents
  USING hnsw (vec vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);

-- Semantic search with SQL join
SELECT d.chunk, d.source,
  1 - (d.vec <=> $1::vector) AS similarity
FROM documents d
WHERE d.category = 'troubleshooting'
ORDER BY d.vec <=> $1::vector
LIMIT 5;
-- $1 = query embedding from app layer

-- pgvector vs Pinecone:
-- pgvector: <10M vectors, SQL joins, no extra cost
-- Pinecone: 100M+ vectors, managed, faster ANN
