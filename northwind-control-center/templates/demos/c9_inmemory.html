Chapter 9 — In-Memory OLTP

Memory-Optimized Tables · Hash Indexes · Native Compilation · Benchmark

Click Check Status to see if the benchmark database is ready.

Memory-Optimized Tables (MOT) — the Formula-1 of OLTP — store all data in RAM using lock-free data structures. For every MOT table or natively compiled procedure, SQL Server generates a dedicated DLL (compiled C code), bypassing the interpreted T-SQL engine entirely. Full ACID is maintained via an append-only log on disk.

🔒
Lock-Free Concurrency

Row versioning instead of locks. When a row is updated, a new version is created. Deletes are logical — old versions stay until all readers are done. Result: no lock contention at peak concurrency.

⚙️
Native Compilation

Stored procedures compiled to machine code (DLL) at CREATE time. Skip the T-SQL interpreter, the query plan cache, and lock management. Only MOT tables allowed inside; requires BEGIN ATOMIC block.

💾
Full ACID

All changes written to the non-volatile transaction log on disk. Crash recovery replays the log into RAM. MOT tables co-exist with disk-based tables in the same database.

🗑️
Non-Durable Mode

DURABILITY = SCHEMA_ONLY: data lost on restart, but schema survives. Zero disk I/O → fastest possible cache or ETL staging table. Alternative to tempdb temp tables (structure preserved, no re-CREATE needed).

Row Versioning — Automatic Snapshot Isolation

No reader ever blocks a writer. No writer ever blocks a reader. The garbage collector (GC thread) reclaims old versions only after all transactions that could have read them have committed.

2. Durable MOT table (SCHEMA_AND_DATA) 
CREATE TABLE dbo.t_mot (
    id  INT   NOT NULL,
    val FLOAT NOT NULL,
    -- Hash index: O(1) point lookup,
    -- BUCKET_COUNT ≈ 2× expected unique keys
    CONSTRAINT pk_t_mot
        PRIMARY KEY NONCLUSTERED HASH (id)
        WITH (BUCKET_COUNT = 1024)
)
WITH (
    MEMORY_OPTIMIZED = ON,
    DURABILITY = SCHEMA_AND_DATA  -- survive restart
);
3. Non-durable cache table (SCHEMA_ONLY) 
-- Zero disk I/O — data lost on restart
-- Use as: cache, ETL staging, session store
CREATE TABLE dbo.t_cache (
    key_col VARCHAR(200)
        COLLATE Latin1_General_100_BIN2
        NOT NULL,
    val_col NVARCHAR(MAX) NULL,
    -- BTree index: supports range queries
    PRIMARY KEY NONCLUSTERED (key_col)
)
WITH (
    MEMORY_OPTIMIZED = ON,
    DURABILITY = SCHEMA_ONLY  -- no log writes
);
4. Natively compiled stored procedure 
CREATE PROCEDURE dbo.p_insert_mot
    @id  INT,
    @val FLOAT
WITH
    NATIVE_COMPILATION,  -- compile to DLL
    SCHEMABINDING        -- schema lock
AS
-- BEGIN ATOMIC required (explicit isolation)
BEGIN ATOMIC WITH (
    TRANSACTION ISOLATION LEVEL = SNAPSHOT,
    LANGUAGE = N'english'
)
    INSERT INTO dbo.t_mot VALUES (@id, @val);
END;
GO

-- Call normally:
EXEC dbo.p_insert_mot 1, 3.14;
5. Memory-optimized table type (TVP) 
-- Table types can also be memory-optimized
-- → enables high-perf table-valued parameters
CREATE TYPE dbo.OrderLinesType
AS TABLE (
    product_id INT     NOT NULL,
    qty        SMALLINT NOT NULL,
    price      MONEY   NOT NULL,
    INDEX ix_prod NONCLUSTERED (product_id)
)
WITH (MEMORY_OPTIMIZED = ON);
GO

-- Used as TVP in a natively compiled proc:
CREATE PROCEDURE dbo.bulk_insert_lines
    @lines dbo.OrderLinesType READONLY
WITH NATIVE_COMPILATION, SCHEMABINDING
AS BEGIN ATOMIC WITH (...)
    INSERT INTO dbo.order_lines
    SELECT * FROM @lines;
END;
6. Monitoring memory consumers 
-- Memory used by MOT tables + indexes
SELECT object_name(object_id) tbl,
       type_desc,
       pages_allocated_count,
       pages_used_count
FROM sys.dm_db_xtp_memory_consumers
WHERE object_id > 0
ORDER BY pages_used_count DESC;

-- Overall XTP memory usage
SELECT memory_consumer_type_desc,
       SUM(allocated_bytes)/1024/1024 mb_alloc,
       SUM(used_bytes)/1024/1024      mb_used
FROM sys.dm_xtp_system_memory_consumers
GROUP BY memory_consumer_type_desc;
Why MEMORY_OPTIMIZED_ELEVATE_TO_SNAPSHOT = ON?

MOT tables are accessed using optimistic concurrency control. They cannot participate in READ COMMITTED isolation (the SQL Server default). When interpreted T-SQL touches a MOT table, SQL Server must upgrade the isolation level to at least SNAPSHOT — otherwise the statement fails with error 41368. 

-- Without this setting:
--   MSG 41368: Accessing memory optimized tables
--   using the READ COMMITTED isolation level is
--   not supported with the current settings.

ALTER DATABASE CURRENT
SET MEMORY_OPTIMIZED_ELEVATE_TO_SNAPSHOT = ON;

-- Now interpreted T-SQL can INSERT/SELECT
-- from MOT tables without explicit WITH(SNAPSHOT)

MOT migration requires dropping all foreign keys involving the candidate tables. They must be manually re-created as regular constraints after migration (MOT tables cannot have FKs to disk-based tables).
4c Run Table Memory Optimization Advisor on each candidate table
SSMS: Table → Table Memory Optimization Advisor → plan & execute. The Advisor renames the old disk table and creates the MOT replacement. Select "include data" to copy existing rows.

6 Re-run load test with the existing (interpreted) sp_new_order()
Same 10 000 × 3 sessions. Compare with disk-based baseline — expect ~6–7× improvement from lock elimination alone.

RECOMMENDED USE CASES
Ideal for MOT
  • Stock exchange order entry — thousands of concurrent low-latency transactions
  • IoT sensor ingestion — real-time data acquisition from many sources; rotate to disk columnstore via SQL Agent job
  • ETL staging — SCHEMA_ONLY non-durable tables as staging area; zero disk I/O, no tempdb contention
  • Session cache / key-value store — replace Redis for small-to-medium caches accessible via T-SQL
  • Temporal tables — history stored on disk, hot rows in MOT; handles high-update temporal data
Poor fit for MOT
  • Tables with foreign keys to/from disk-based tables (must be dropped)
  • Analytics / reporting queries — no columnstore index support; use OLAP DB instead
  • Very large tables (>available RAM) — entire dataset must fit in memory
  • Tables with DDL triggers, computed columns, page/row compression
  • Cross-database queries or distributed transactions
  • Any table where the CPU cost dominates reads (not writes) — read-heavy workloads benefit less
