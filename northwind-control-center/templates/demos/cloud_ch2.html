Cloud Book — Chapter 2

Technological Developments: The Road to Cloud Computing

Tracing the conceptual path from 1960s mainframe batch scheduling to virtualised utility computing — Cloud is the result of decades of systematic evolution, not a sudden invention.

The Core Tension

Every era of computing faces the same triangle: minimise cost (high utilisation), maximise performance (exclusive access), and maximise autonomy (no restrictions). These conflict. The history of computing is a sequence of partial resolutions to this tension.

Cost penalty of idle hardware:
System: $10,000  ·  Lifetime: 100,000 h
100% utilisation → $0.10 / CPU-hour
10% utilisation → $1.00 / CPU-hour
10× cost increase from 90% idle time.
1960s
Mainframes — Centralised Batch

Large, expensive machines at computing centres. Administrators scheduled jobs to keep utilisation high. Users reserved time slots. Time-sharing via terminals reduced per-user performance proportionally to connected users.

1980s
Personal Computers — Decentralised, Isolated

Miniaturisation made computers affordable. No sharing needed — but printers/storage duplicated expensively per desk. Data on local HDDs, shared via floppy disks. Peripheral utilisation very low.

1980s–90s
Client-Server & LANs — Shared Services

Ethernet + TCP/IP + Novell NetWare → print servers, shared directories. Specialised servers (FTP, email, DB, web). Global Internet by late 1990s. Still tightly coupled — one server failure blocks all callers.

Always
Supercomputers — Elite Centralised HPC

Continuation of mainframe model for extreme computation. Batch queuing, approved users, centralised centres. Performance: sequential → vector/pipeline → massively parallel (100,000+ cores). Languages: Fortran/C/C++ + OpenMP · MPI · CUDA.

The Problem — Raw Sockets

Without abstraction, every distributed call requires: create socket → configure address → connect → send message bytes → receive bytes → handle errors. This boilerplate repeats throughout the codebase; each team invents its own conventions.

C — Raw Socket Client
clientSocket = socket(PF_INET, SOCK_STREAM, 0);
serverAddr.sin_family      = AF_INET;
serverAddr.sin_port        = htons(7891);
serverAddr.sin_addr.s_addr = inet_addr("127.0.0.1");
connect(clientSocket, (struct sockaddr*)&serverAddr,
        sizeof serverAddr);
recv(clientSocket, buffer, 1024, 0);  /* raw bytes */
RPC — Remote Procedure Call (Sun, 1984)

Key philosophy: use an abstraction programmers already know — the function call. An RPC compiler generates stub functions that hide connection/marshalling. Middleware handles transport. Programmer sees only the call.

RPC mechanism (hidden from programmer):
Client calls print_list_1(l, cl)
 → client stub marshals args into message bytes
  → network transport to server
   → server stub unmarshals, calls real function
    → result marshalled back to client
Programmer sees only: result = print_list_1(l, cl);
C — RPC Client (Sun RPC)
CLIENT *cl;
cl = clnt_create(argv[1], PRINTER, PRINTER_V1, "tcp");
if (cl == NULL) {
    printf("error: could not connect.\n");
    return 1;
}
/* Looks identical to a local function call: */
result = print_list_1(l, cl);
if (result == NULL) {
    printf("error: RPC failed!\n");
}
RMI — Java Remote Method Invocation

Java's platform independence + object serialisation → ship classes between JVMs. RMI uses interfaces (not IP:port) to describe remote services; clients look up by name via RMIRegistry. The stub object looks like a local Java object.

Java — BankInterface (remote contract)
public interface BankInterface extends Remote {
    int deposit(int amount)  throws RemoteException;
    int withdraw(int amount) throws RemoteException;
    int getBalance()         throws RemoteException;
    // every method MUST declare RemoteException
}
Java — BankServer (registers in RMIRegistry)
public class BankServer extends UnicastRemoteObject
        implements BankInterface {
    static int balance = 0;
    public int getBalance() throws RemoteException {
        return balance;
    }
    public static void main(String[] args) {
        BankServer server = new BankServer();
        Naming.rebind("BankServer", server); // publish
    }
}
Java — ATMClient (stub looks like local object)
String name = "//" + args[0] + "/BankServer";
BankInterface server =
    (BankInterface) Naming.lookup(name);
// Remote call — indistinguishable from local:
balance = server.getBalance();
The Loose Coupling Problem — Why RPC/RMI is Fragile
Client C → Server A → Server B → Server D
↑ Error in D propagates as RemoteException chain: D→B→A→C
Strong coupling: a deadlock in D blocks the entire call chain indefinitely. The abstraction "pretends" remote calls are local — but they fail in entirely non-local ways (network partition, server crash, timeout). Industry response: loose coupling via message queues and async execution.
Service-Oriented Architecture (SOA)

Goal: decouple what a server does from where it runs. A service has a published contract; clients find services by capability, not by IP:port.

Four SOA Principles
1
Interface description
Service described by a contract (interface), not implementation details
2
Directory mediation
Services register themselves; clients look up by capability
3
Transient interaction
Clients and services interact only for the duration of service execution
4
Open standards
Platform-neutral technologies — any language, any OS can participate
XML Web Services (WSDL + UDDI + SOAP)
WSDL
Web Service Description Language — XML contract: location, operations, parameters
UDDI
Universal Description Discovery & Integration — the public XML directory
SOAP
Simple Object Access Protocol — XML message envelope over HTTP
Flow: query UDDI → retrieve WSDL → generate stub → send SOAP over HTTP → receive SOAP response.
The WS-* Disaster — 70+ Specifications
Each additional need (state, security, transactions, orchestration) spawned a new spec. Each spec depends on others. The result: essentially impossible to understand completely.

REST — Representational State Transfer (Fielding, 2000)

Roy Fielding's PhD reaction to WS-* complexity. Instead of computation-centric ("execute this procedure"), REST is resource-centric: everything has a URL; use HTTP verbs to query and modify state. No stubs, no WSDL, no code generation required.

SOAP Request (verbose XML)
<?xml version="1.0"?>
<soap:Envelope
  xmlns:soap="http://schemas.xmlsoap.org/
              soap/envelope/">
  <soap:Body>
    <m:GetBalance
      xmlns:m="http://bank.example/ws">
      <m:AccountId>12345</m:AccountId>
    </m:GetBalance>
  </soap:Body>
</soap:Envelope>
REST Request (simple HTTP)
GET /accounts/12345/balance HTTP/1.1
Host: api.bank.example
Accept: application/json

# Response:
{
  "account": "12345",
  "balance": 4250.00,
  "currency": "USD"
}

Grid Computing — Distributed Supercomputing

Two simultaneous 1990s movements: (1) US supercomputer centres had uneven loads — one overbooked while another sat idle. (2) Ethernet clusters of commodity PCs could be linked via WAN. Both sought to increase utilisation via distributed resource management.

The Grid = "Computing Power Grid"
Analogy: US electrical power grid — plug in anywhere, get power. Goal: plug a computer into "the Grid" and access compute/storage resources regardless of geography. Automatic load balancing across supercomputer centres.
Globus Toolkit — Key Components
GRAM
Globus Resource Allocation Manager — collects resource info per site, accepts/denies RSL job requests, monitors execution
Information Service
Real-time resource status aggregation across all grid nodes — used by brokers for scheduling decisions
RSL
Resource Specification Language — express compute, memory, storage, scheduling requirements declaratively
Resource Broker
Translates user requests into site-specific allocations; co-allocates across multiple sites
OGSA
Open Grid Service Architecture — Globus evolved into Web Services; governed by academic/industry consortium
DataGrid / CERN (2005)
European DataGrid proved international multi-site grid works at scale — purpose-built for processing Large Hadron Collider data. By 2005, most developed countries had national grid systems.
Voluntary Computing — SETI@home & BOINC

Alternative to push-style Grid: computer owners donate idle CPU time. Programs are deployed pull-style — the client downloads work units from a central task database and returns results.

SETI@home (1999, Berkeley)
Screensaver that downloads radio telescope data segments and analyses them for signs of extraterrestrial life. Millions of users donated idle CPU cycles.
Inversion: "server" = the researcher;
"clients" = users donating compute.

Fault tolerance: disconnected client → task reissued automatically.
BOINC Infrastructure
Berkeley Open Infrastructure for Network Computing — generalised runtime for volunteer computing.

Hosting & Utility Computing

By early 2000s, businesses needed professional IT but couldn't afford full-time staff. Outsourced web hosting → shared server farms → general managed hosting → "utility computing": pay a flat fee for compute, just like electricity.

SUN Grid — First Utility Computing (2005)
SUN Microsystems ("The Network Is The Computer") deployed portable containerised server systems. In 2005 opened them to the public at flat pay-per-use rates — a direct precursor to EC2.

Why Virtualisation Enables Cloud

Legacy problem: software compiled for one hardware/OS couldn't run anywhere else. Vendor lock-in. Obsolete hardware meant losing the software.

Solution: A Virtual Machine Monitor (VMM) interposes between guest OS and physical hardware, emulating the expected instruction set. Multiple VMs share one machine. Each VM is isolated, snapshotable, migratable.

Advantages & Disadvantages
Resource efficiency
Multiple VMs on one machine; maximise CPU vs one-app-per-server idle waste
Portability & live migration
VM image moved to new hardware transparently; snapshot before risky operations
Backup & DR
VM state snapshots are trivial to archive and restore
Performance overhead
Extra translation layers → slightly slower; mitigated by hardware-assisted virtualisation
Licensing & training
Some software prohibits virtualised execution; IT staff must learn new tooling
Three Types of Hardware Virtualisation
Full Virtualisation
Complete hardware simulation. Guest OS runs unmodified. VMM translates every instruction in software.
