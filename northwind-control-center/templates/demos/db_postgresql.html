PostgreSQL

Relational OLTP · SQL + JSONB · MVCC · Most feature-rich open-source RDBMS

PostgreSQL emerged from UC Berkeley's POSTGRES project in 1986 and has evolved into the world's most advanced open-source relational database. It combines strict ACID guarantees with an extraordinary extension ecosystem — JSONB for semi-structured data, PostGIS for geospatial queries, and pgvector for AI embeddings — making it the database of choice for Instagram, Shopify, Apple, and countless SaaS companies.

Relational + Extended Types 
-- Standard relational schema
CREATE TABLE orders (
  order_id   SERIAL PRIMARY KEY,
  customer   VARCHAR(100) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  total      NUMERIC(12,2),
  tags       TEXT[],          -- native array
  metadata   JSONB            -- semi-structured
);

-- JSONB: fast, indexed, binary
CREATE INDEX idx_meta ON orders USING GIN (metadata);

-- PostGIS geometry
CREATE TABLE stores (
  id       SERIAL PRIMARY KEY,
  name     TEXT,
  location GEOMETRY(Point, 4326)  -- lat/lon WGS84
);

-- pgvector for AI embeddings (1536-dim for OpenAI)
CREATE EXTENSION vector;
CREATE TABLE embeddings (
  id    SERIAL PRIMARY KEY,
  chunk TEXT,
  vec   VECTOR(1536)
);
CREATE INDEX ON embeddings USING ivfflat (vec vector_cosine_ops);

Window Functions & CTEs 
-- Running total with window function
SELECT order_id, customer, total,
  SUM(total) OVER (
    PARTITION BY customer
    ORDER BY created_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total,
  RANK() OVER (PARTITION BY customer ORDER BY total DESC) AS rank
FROM orders;

-- Recursive CTE: org chart traversal
WITH RECURSIVE org AS (
  SELECT id, name, manager_id, 1 AS level
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  SELECT e.id, e.name, e.manager_id, o.level + 1
  FROM employees e JOIN org o ON e.manager_id = o.id
)
SELECT level, name FROM org ORDER BY level, name;
JSONB Operators 
-- Extract nested value
SELECT metadata->'shipping'->>'city' AS city
FROM orders WHERE metadata @> '{"priority": "express"}';

-- JSON path query (PostgreSQL 12+)
SELECT * FROM orders
WHERE metadata @? '$.items[*].sku ? (@ == "SKU-999")';

-- Update JSONB in-place
UPDATE orders
SET metadata = jsonb_set(metadata, '{status}', '"shipped"')
WHERE order_id = 42;

-- Aggregate to JSON
SELECT customer,
  jsonb_agg(jsonb_build_object('id', order_id, 'total', total))
FROM orders GROUP BY customer;

-- Analyze query plan with actual timings
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT c.name, COUNT(o.order_id), SUM(o.total)
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE c.region = 'Europe'
GROUP BY c.name
ORDER BY SUM(o.total) DESC
LIMIT 20;
-- Output: Hash Aggregate → Hash Join → Bitmap Heap Scan
-- Buffers: shared hit=1284, read=48

PostGIS & pgvector 
-- Nearest stores within 5km
SELECT name, ST_Distance(location::geography,
  ST_MakePoint(-73.9857, 40.7484)::geography) AS metres
FROM stores
WHERE ST_DWithin(location::geography,
  ST_MakePoint(-73.9857, 40.7484)::geography, 5000)
ORDER BY metres LIMIT 5;

-- Semantic similarity search (pgvector)
SELECT chunk, 1 - (vec <=> $1::vector) AS similarity
FROM embeddings
ORDER BY vec <=> $1::vector
LIMIT 10;

MVCC & Storage

MVCC (Multi-Version Concurrency Control): Every row update creates a new tuple version. Old versions remain visible to concurrent transactions. Readers never block writers; writers never block readers.

Heap Storage: Tables stored as 8KB pages in base/<dboid>/<relfilenode>. Each page has a header, item pointers (line pointers), and row data (tuples) with xmin/xmax visibility markers.

autovacuum: Reclaims dead tuples left by MVCC. Without it, table bloat accumulates. Key parameters: autovacuum_vacuum_cost_delay, autovacuum_vacuum_scale_factor.

WAL (Write-Ahead Log): All changes written to WAL before data pages. Enables crash recovery, streaming replication, and point-in-time recovery (PITR). WAL segment = 16MB.

Key Config Knobs 
# postgresql.conf — production tuning

# Memory
shared_buffers = 8GB          # 25% of RAM
effective_cache_size = 24GB   # hint for planner
work_mem = 64MB               # per sort/hash op
maintenance_work_mem = 2GB    # VACUUM, CREATE INDEX

# WAL & replication
wal_level = logical           # enables replication
max_wal_senders = 10
synchronous_commit = on       # durability guarantee

# Connections (use PgBouncer for pooling!)
max_connections = 200

# Checkpointing
checkpoint_completion_target = 0.9
max_wal_size = 4GB
