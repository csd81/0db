B-Tree / Row Store
FactInternetSales_BTREE
Traditional row-oriented heap. Each 8 KB page stores complete rows. Ideal for OLTP point lookups; inefficient for wide analytical scans — every column is read even if only 2 are needed.
Space used
— KB
Page Compression
FactInternetSales_PageComp
Row store + LZ-based page compression. SQL Server compresses repeated byte patterns across rows on the same page. Effective for low-cardinality columns. No engine change — just fewer pages to read.
Space used
— KB
Clustered Columnstore
FactInternetSales_CCI
Columns stored as compressed segments (~1M row groups). SIMD batch mode processes 900 rows per CPU instruction. Aggregation queries skip entire row groups via segment metadata. 10–100× faster for analytics.
Space used
— KB

I/O Statistics — SET STATISTICS IO ON
Logical Reads
8 KB page reads served from the data buffer (memory). Zero I/O wait. Minimizing logical reads = smaller table = compressed formats win.
Physical Reads
Pages fetched from storage (SSD/HDD). These block query execution. Flush with DBCC DROPCLEANBUFFERS before benchmarking.
Read-Ahead Reads
Asynchronous prefetch requests sent to the OS for pages likely needed soon. Do not block query. SQL Server's read-ahead thread issues these speculatively.
Workfile / Worktable = internal temp tables for hash join partitions and hash aggregates (always appear with row-mode hash joins).

Partition Function
Maps each row's key value to a partition number. RANGE LEFT: boundary value belongs to the left partition. Value 10 → rows with PcInternetSalesYear ≤ 10 go to partition 1.
Partition Scheme
Maps partition numbers to filegroups. Here all go to [PRIMARY], but in production each partition can be on a different physical drive for parallel I/O.
Aligned Index
CCI uses the same partition scheme as the table. ALTER TABLE … SWITCH requires aligned indexes — no data moves, only metadata pointers change. Instant.
SWITCH (Step 7)
Metadata-only — reassigns the staging table as partition 4 of InternetSales. No rows copied. The staging table becomes empty after the switch. Enabling minimally-logged bulk loads.
Partition Function & Scheme 
-- Part. function: 4 boundaries → 5 partitions
CREATE PARTITION FUNCTION PfInternetSalesYear (TINYINT)
  AS RANGE LEFT FOR VALUES (10, 11, 12, 13)
-- Value 10 → ≤2010, 11 → 2011, 12 → 2012, 13 → 2013, else → 5

CREATE PARTITION SCHEME PsInternetSalesYear
  AS PARTITION PfInternetSalesYear
  ALL TO ([PRIMARY])
Step 7 — SWITCH (no data copy!) 
-- Load 2013 into staging table (empty → minimally logged)
INSERT INTO dbo.InternetSalesNew ...
WHERE PcInternetSalesYear = 13

-- Build aligned CCI on staging
CREATE COLUMNSTORE INDEX CSI_ISN ON dbo.InternetSalesNew ...

-- Atomic metadata-only partition assignment
ALTER TABLE dbo.InternetSalesNew
  SWITCH TO dbo.InternetSales PARTITION 4

-- After: InternetSalesNew is empty (0 rows)
SELECT COUNT(*) FROM InternetSalesNew --0
PostgreSQL supports two partitioning mechanisms. Declarative (PG 10+, recommended) uses native syntax. Table inheritance (legacy) requires RULE or TRIGGER to redirect inserts. Both demo against the Northwind products table (77 rows), partitioning by first letter of product name.
Declarative Partitioning (PG 10+) 
-- Parent table is virtual (stores no data)
CREATE TABLE products_p
  (productid int, productname varchar(40),
   part_key char(1), unitprice money)
PARTITION BY RANGE(part_key)

-- Child partitions (ranges are half-open: [from, to))
CREATE TABLE products_p_a_m
  PARTITION OF products_p
  FOR VALUES FROM ('a') TO ('m')

CREATE TABLE products_p_m_z
  PARTITION OF products_p
  FOR VALUES FROM ('m') TO ('z')

-- Insert via parent → auto-routed to correct partition
INSERT INTO products_p ...
  SELECT ..., LOWER(SUBSTRING(productname,1,1)) FROM products

-- Query partition directly or via parent
SELECT * FROM products_p WHERE productname < 'd'
SELECT * FROM products_p_a_m  -- direct

-- Lifecycle management
ALTER TABLE products_p DETACH PARTITION products_p_a_m
ALTER TABLE products_p ATTACH PARTITION products_p_a_m
  FOR VALUES FROM ('a') TO ('m')
Table Inheritance (Legacy, pre-PG 10) 
-- Parent table (virtual)
CREATE TABLE products_i
  (productid int, productname varchar(40), unitprice money)

-- Child tables inherit schema from parent
CREATE TABLE products_i_a_m () INHERITS (products_i)
ALTER TABLE products_i_a_m ADD CONSTRAINT df_a_m
  CHECK (lower(left(productname,1)) >= 'a'
     AND lower(left(productname,1)) <= 'm')

CREATE TABLE products_i_n_z () INHERITS (products_i)
-- + check constraint for n..z

-- RULE redirects inserts to correct child
CREATE RULE products_insert_a_m AS
  ON INSERT TO products_i
  WHERE lower(left(productname,1)) <= 'm'
  DO INSTEAD INSERT INTO products_i_a_m VALUES (new.*)

-- SELECT via parent sees ALL children
SELECT * FROM products_i          --77 (all)
SELECT * FROM ONLY products_i     --0  (parent only)
SELECT * FROM products_i_a_m      --41
SELECT * FROM products_i_n_z      --36
