Couchbase

NoSQL Document · N1QL (SQL for JSON) · Built-in cache · Mobile sync via Couchbase Lite

Born in 2011 from the merger of CouchDB and Membase, Couchbase uniquely combines a managed document database with a built-in in-memory caching layer — giving you sub-millisecond latency on hot data without a separate Redis tier. Its N1QL query language brings full SQL semantics (including JOINs, GROUP BY, and subqueries) to JSON documents, making it more approachable than MongoDB's aggregation pipeline. Couchbase Lite enables offline-first mobile applications with automatic sync to the server. Used by LinkedIn for profile data, Amadeus for airline reservations, and major telecom operators for subscriber management.

Data Hierarchy

Bucket — top-level namespace with dedicated memory quota. Think "database" in MongoDB terms. Can be Couchbase (persistent) or Ephemeral (memory-only).

Scope — logical namespace within a bucket. Like a schema in relational DBs. Default scope is _default.

Collection — groups of documents within a scope. Like a table/collection. Documents in a collection share indexes and access patterns.

Document — JSON object with a unique key (string). Max 20MB. Supports nested objects, arrays, and all JSON types.

Sub-Document API — atomically modify nested fields within a document without fetching the whole document. Key performance optimization.

Document Design 
// Document key: "user::42"
{
  "type": "user",
  "id": 42,
  "name": "Alice Chen",
  "email": "alice@example.com",
  "preferences": {
    "currency": "USD",
    "timezone": "America/New_York",
    "notifications": true
  },
  "loyalty": {
    "tier": "gold",
    "points": 12500,
    "since": "2021-03-15"
  },
  "tags": ["premium", "newsletter"]
}

// Sub-Document operation: add loyalty points
// without loading the full document
bucket.mutate_in("user::42", [
  SD.increment("loyalty.points", 500)
])
N1QL — SQL for JSON 
-- 1. Basic SELECT (like SQL on JSON)
SELECT name, email, loyalty.tier, loyalty.points
FROM users
WHERE loyalty.tier = 'gold' AND loyalty.points > 10000
ORDER BY loyalty.points DESC
LIMIT 10;

-- 2. JOIN across collections
SELECT o.order_id, o.total, u.name AS customer
FROM orders o
JOIN users u ON KEYS o.user_key
WHERE o.status = 'pending'
  AND o.created > '2024-01-01';

-- 3. UNNEST — flatten array elements to rows
SELECT u.name, tag
FROM users u
UNNEST u.tags AS tag
WHERE tag IN ['premium', 'vip'];

-- 4. Aggregation with GROUP BY
SELECT loyalty.tier,
       COUNT(*) AS user_count,
       AVG(loyalty.points) AS avg_points,
       SUM(loyalty.points) AS total_points
FROM users
GROUP BY loyalty.tier
ORDER BY avg_points DESC;

-- 5. Sub-document query
SELECT name, preferences.timezone
FROM users
WHERE preferences.currency = 'USD'
  AND preferences.notifications = true;

-- 6. ARRAY functions
SELECT name,
       ARRAY_LENGTH(tags) AS tag_count,
       ARRAY_CONTAINS(tags, 'premium') AS is_premium
FROM users
WHERE ARRAY_CONTAINS(tags, 'newsletter');

-- 7. Full-Text Search integration
SELECT META().id, name, email,
       SEARCH_SCORE() AS score
FROM users
WHERE SEARCH(users, {
    "query": {"match": "chen", "field": "name"},
    "index": "users_fts_index"
});

-- 8. CREATE INDEX
CREATE INDEX idx_loyalty_tier
ON users(loyalty.tier, loyalty.points DESC)
WHERE type = 'user';
Killer Use Cases

Telecom Subscriber Management — subscriber profile documents with complex JSON structures, queried via N1QL, with sub-millisecond KV access for real-time enrichment. Vodafone uses Couchbase here.

Airline Booking (Amadeus) — Amadeus uses Couchbase for session state and booking flow data. The built-in cache eliminates a separate Redis tier.

Offline-First Mobile Apps — Couchbase Lite (embedded) + Sync Gateway enables truly offline-capable apps that sync changes when connectivity resumes. Airlines use this for check-in apps.

User Profile & Session Store — LinkedIn used Couchbase for member profile serving. The combination of KV access (sub-ms) and N1QL (complex queries) covers both read and analytics.

Performance

KV get/set: <1ms (from memory tier)

N1QL: GSI (Global Secondary Index) enables sub-10ms indexed lookups

DCP replication: near-real-time cross-datacenter

Magma storage: write-optimized LSM for high write throughput

Service Architecture

Data Service — the KV engine. Stores documents in managed memory cache + Magma persistent storage. vBuckets (1024 virtual buckets) distribute documents via consistent hashing.

Query Service — N1QL query engine. Parses, optimizes, and executes N1QL queries. Can run on separate nodes from data nodes for independent scaling.

Index Service — Global Secondary Indexes (GSI) stored on separate nodes. Indexes are "global" — not tied to a specific data node's vBuckets.

Search Service — Bleve-based full-text search. CRUD operations automatically reflected via DCP stream from Data Service.

Eventing Service — JavaScript functions triggered by document mutations. Like database triggers, but in JS.

DCP & XDCR

DCP (Data Change Protocol) — internal streaming protocol that propagates document mutations to replicas, indexes, search, analytics, and Sync Gateway in near-real-time.

XDCR (Cross Datacenter Replication) — async replication between Couchbase clusters in different datacenters. Bidirectional with conflict resolution. Used for geo-distributed active-active deployments.

Magma Storage Engine — LSM-tree based engine designed for very large data sets (TB+) with write amplification optimizations. Replaces ForestDB as the default storage engine.
