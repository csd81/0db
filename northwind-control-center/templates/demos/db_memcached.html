Memcached

Key-Value Cache · In-Memory · Sub-millisecond · Simple distributed object caching

Created in 2003 by Brad Fitzpatrick at LiveJournal to dramatically reduce database load, Memcached pioneered the idea of a shared distributed in-memory cache as a separate tier. Its design philosophy is radical simplicity: no persistence, no data types beyond opaque blobs, no clustering logic (that's the client's job), no pub/sub, no scripting — just the fastest possible get/set over a binary or ASCII protocol. Facebook served over 1 billion requests per day from Memcached at its peak. While Redis has largely superseded it for new projects (offering persistence and richer data structures), Memcached remains deployed at massive scale at Facebook, YouTube, and Wikipedia precisely because its simplicity makes it fast and predictable.

The Data Model: Radical Simplicity

One data type: key → value

Every item stored in Memcached is just: key (string, max 250 bytes) → value (opaque byte blob, max 1MB) + expiry (TTL in seconds).

There are no lists, sets, hashes, sorted sets, or streams. If you need those, use Redis.

Slab Allocator

Memory is pre-allocated in slab classes (groups of same-size chunks). Items stored in the smallest slab class that fits. This eliminates memory fragmentation at the cost of some waste.

LRU Eviction

When memory is full, the Least Recently Used item is evicted. Applications must handle cache misses gracefully — Memcached is a best-effort cache, not a store.

No Replication

Standard Memcached has no built-in replication. If a server goes down, its data is gone (but your database still has the canonical copy). Facebook built their own replication layer (mcrouter) on top.

Memcached Commands 
# ASCII protocol (telnet to memcached:11211)

# SET — store unconditionally
set user:42 0 3600 18
{"name":"Alice","age":30}
STORED

# GET — retrieve value
get user:42
VALUE user:42 0 18
{"name":"Alice","age":30}
END

# ADD — only if key doesn't exist
add lock:resource1 0 30 1
STORED   # (or NOT_STORED if key exists)

# REPLACE — only if key exists
replace user:42 0 3600 15
{"name":"Alice*"}

# DELETE
delete user:42
DELETED

# INCR / DECR — atomic counter on integer values
set page_views:home 0 0 1
0
incr page_views:home 1
1
incr page_views:home 1
2

# GETS + CAS (Check-And-Set) — optimistic locking
gets user:42
VALUE user:42 0 18 12345678   # CAS token = 12345678
{"name":"Alice","age":30}
END

cas user:42 0 3600 15 12345678  # update only if token matches
{"name":"Alice*"}
STORED   # or EXISTS (CAS token changed)

Killer Use Cases

Database Query Cache — cache expensive SQL query results for seconds/minutes. Dramatically reduces database load for read-heavy apps. The classic use case that birthed Memcached.

Session Store — store web session data (stateless app servers can retrieve any user's session from Memcached). Simple, fast, TTL-based cleanup is perfect here.

HTML Fragment Cache — cache rendered HTML snippets (product listings, navigation menus) to avoid re-rendering on every request.

API Response Cache — cache responses from expensive upstream APIs with short TTLs. Rate-limit protection + latency reduction.

Performance

Read latency: <1ms (typically 50-200μs)

Throughput: 200K+ ops/sec per core (multi-threaded)

Multi-threaded: better than Redis for many-core servers with simple get/set workloads

Scale: horizontal via client-side hashing — add nodes linearly

Slab Allocator

Slab Classes — memory divided into slab classes of increasing chunk sizes (e.g., 96B, 120B, 152B, ..., 1MB). Each class pre-allocates 1MB slabs divided into equal chunks.

Item Storage — an item is stored in the smallest slab class whose chunk size ≥ item size. Some space is wasted per item (internal fragmentation), but malloc() fragmentation is eliminated.

LRU per Slab — each slab class maintains its own LRU list. Eviction happens within the slab that's full, not globally. Use stats slabs and stats items to diagnose slab imbalance.

Client-Side Consistent Hashing

Memcached has no cluster awareness. The client library (libmemcached, pylibmc, spymemcached) implements consistent hashing to distribute keys across servers.

When a server is added/removed, only ~1/N of keys remap (where N = number of servers), minimizing cache invalidation. The hash ring is computed entirely client-side — servers don't talk to each other.

Facebook's mcrouter adds a routing proxy layer, enabling replication, failover, and cold cluster warm-up while keeping the simple Memcached protocol.
