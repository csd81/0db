Word count is the canonical MapReduce example because its structure perfectly illustrates the entire execution model with trivial business logic that does not obscure the framework mechanics. The Map phase emits one (word, 1) pair for every token in each document shard; the Shuffle phase groups all pairs with the same key across all mappers onto a single reducer; the Reduce phase sums the counts for each key. The demo runs the full four-phase pipeline against Northwind product descriptions, printing real intermediate key-value pairs from the Map phase and a Chart.js bar chart of the final word-frequency distribution. Every distributed computing framework — Spark, Flink, Beam, Dask — teaches its API with this same example because it exercises every part of the execution model without requiring any domain knowledge.

Why WordCount?
The "Hello World" of MapReduce — simple enough to read in 10 lines of Java, yet it exercises every stage: split, map, shuffle/sort, reduce. Every production job (log aggregation, click counting, inverted index) is a variation of this pattern.
Type contract
Mapper output key/value types must match Reducer input types. The framework enforces this at compile time via Java generics — a mismatch is a compile error, not a runtime crash.
Shuffle is free
You write zero shuffle code. The framework sorts all (word,1) pairs by key across all mapper outputs and delivers a sorted iterator to each reducer. This is the core value of the MapReduce abstraction.
Hadoop on GCP
Compile to a .jar, upload to Cloud Storage, submit via Dataproc. HDFS is backed by GCS. YARN schedules the containers. You pay only per-minute cluster use.
Mapper Class (Java)
// K_in=LongWritable (line offset), V_in=Text (line) // K_out=Text (word), V_out=IntWritable (1) class Map extends Mapper<LongWritable,Text,Text,IntWritable> { void map(LongWritable key, Text value, Context context) { String line = value.toString(); StringTokenizer tok = new StringTokenizer(line); while (tok.hasMoreTokens()) { // emit (word, 1) for every token context.write(tok.nextToken(), new IntWritable(1)); } } }
The line offset key (LongWritable) is ignored — it only exists because HDFS delivers records as (byte_offset, line_text) pairs. Each token emits an independent (word, 1) — no state shared between tokens or documents.
Reducer Class (Java)
// K_in=Text (word), V_in=IntWritable (counts) // K_out=Text (word), V_out=IntWritable (sum) class Reduce extends Reducer<Text,IntWritable,Text,IntWritable> { void reduce(Text key, Iterable<IntWritable> values, Context context) { int sum = 0; for (IntWritable x : values) sum += x.get(); context.write(key, new IntWritable(sum)); } }
The framework guarantees the reducer sees all values for one key before moving to the next. Iterable<IntWritable> is a lazy stream — Hadoop may spill values to disk if the list is huge.
Data Flow — Concrete Example
── Input (K1=offset, V1=line) ────────── 0 "Deer Bear River" 1 "Car Car River" 2 "Deer Car Bear" ── Map → (K2=word, V2=1) ─────────────── Deer,1 Bear,1 River,1 Car,1 Car,1 River,1 Deer,1 Car,1 Bear,1 ── Shuffle (K2, list(V2)) ────────────── Bear → [1, 1] Car → [1, 1, 1] Deer → [1, 1] River → [1, 1] ── Reduce → (K3=word, V3=sum) ────────── Bear 2 Car 3 Deer 2 River 2
This is the exact diagram from the Dean & Ghemawat paper. Every production MapReduce job maps to this same flow — only the key/value types and the user functions change.
