InfluxDB

Time-Series · TSM Engine · Line Protocol · Flux · Telegraf-InfluxDB-Grafana stack

InfluxDB was created in 2013 by InfluxData specifically for time-indexed data — metrics, events, and sensor readings where every record has a timestamp as its primary dimension. Its measurement → tags → fields model is purpose-built for the way monitoring data is produced and queried. InfluxDB is the "I" in the TIG stack (Telegraf collects, InfluxDB stores, Grafana visualizes) and is the default choice for infrastructure monitoring, IoT sensor ingestion, and application performance telemetry.

Why Time-Series Databases Exist

The append-only nature: Time-series data is almost always written once in timestamp order and never updated. This allows optimizations impossible in general-purpose databases: no MVCC overhead, no row-level locking, no B-tree rebalancing on writes.

Compression wins: Consecutive timestamps differ by a fixed interval (1s, 10s). Delta-of-delta encoding compresses a day of 1-second timestamps into a few bytes. Gorilla encoding for floats achieves similar ratios. InfluxDB's TSM achieves 10-40x compression ratios on typical metrics.

Retention is built-in: Infrastructure metrics from 6 months ago are worthless. Retention policies auto-delete data after a configured duration without expensive DELETE queries. Continuous queries downsample: keep 1-year of hourly averages, 30 days of per-minute, 7 days of per-second.

Measurement / Tags / Fields 
-- Conceptual model:
-- Measurement = table name (like "cpu", "disk")
-- Tags = indexed string metadata (host, region)
-- Fields = actual values (float/int/bool/string)
-- Time = timestamp (nanosecond precision)

-- Line Protocol (write format):
measurement,tag1=val1,tag2=val2 field1=x,field2=y timestamp

-- Examples:
cpu,host=web01,region=us-east cpu_percent=45.2,load_1m=1.8 1706054400000000000
cpu,host=web02,region=us-east cpu_percent=62.1,load_1m=2.3 1706054400000000000
disk,host=web01,device=sda1 used_bytes=50000000000i,free_bytes=30000000000i

-- Tags vs Fields:
-- Tags: indexed, great for filtering/grouping
--       keep cardinality LOW (<100k distinct values)
-- Fields: not indexed, numeric data, measured values
-- NEVER put a timestamp or UUID in a tag!
Retention Policies 
-- InfluxQL (v1) retention policy
CREATE RETENTION POLICY "7d" ON "monitoring"
  DURATION 7d
  REPLICATION 1
  DEFAULT;

CREATE RETENTION POLICY "1y" ON "monitoring"
  DURATION 52w
  REPLICATION 1;

-- Continuous query: downsample per-second to hourly
CREATE CONTINUOUS QUERY cq_hourly
ON monitoring BEGIN
  SELECT mean(cpu_percent) AS cpu_mean,
         max(cpu_percent)  AS cpu_max
  INTO "1y"."cpu_hourly"
  FROM "7d"."cpu"
  GROUP BY time(1h), host, region
END;

-- InfluxDB v3 (IOx): bucket + retention
influx bucket create --name metrics --retention 7d
Flux Language (Functional Pipeline) 
// Flux: pipe-forward operator |>
// Average CPU per host, last 24h
from(bucket: "monitoring")
  |> range(start: -24h)
  |> filter(fn: (r) => r._measurement == "cpu")
  |> filter(fn: (r) => r._field == "cpu_percent")
  |> aggregateWindow(every: 1h, fn: mean)
  |> group(columns: ["host"])
  |> mean()

// Moving average (rolling window)
from(bucket: "monitoring")
  |> range(start: -7d)
  |> filter(fn: (r) => r._measurement == "http_requests")
  |> timedMovingAverage(every: 1h, period: 6h)

// Join two measurements
cpuData = from(bucket: "monitoring")
  |> range(start: -1h)
  |> filter(fn: (r) => r._measurement == "cpu")
memData = from(bucket: "monitoring")
  |> range(start: -1h)
  |> filter(fn: (r) => r._measurement == "memory")
join(tables: {cpu: cpuData, mem: memData},
     on: ["_time", "host"])
InfluxQL (SQL-like, v1) 
-- InfluxQL: familiar SQL-like syntax
SELECT mean("cpu_percent"), max("cpu_percent")
FROM "cpu"
WHERE time > now() - 24h
  AND "host" = 'web01'
GROUP BY time(1h), "host"
ORDER BY time DESC
LIMIT 24;

-- fill(previous): forward-fill gaps
SELECT mean("temperature")
FROM "sensors"
WHERE time > now() - 7d
GROUP BY time(5m), "device_id"
fill(previous);

-- Derivative (rate of change)
SELECT derivative(mean("bytes_sent"), 1s) AS bps
FROM "network"
WHERE time > now() - 1h
GROUP BY time(10s), "interface";
Killer Use Cases
Infrastructure monitoring: The TIG stack (Telegraf + InfluxDB + Grafana) is the most popular open-source monitoring stack. Telegraf's 500+ plugins collect from everything — Linux system stats, Docker containers, Kubernetes pods, PostgreSQL, Kafka.
IoT sensor data: A factory floor with 10,000 sensors writing temperature every second generates 864M readings/day. InfluxDB handles this with write batching and automatic retention expiry.
Financial tick data: Stock price ticks, order book snapshots, and trade records are inherently time-indexed. InfluxDB's nanosecond timestamp precision and fast range queries make it suitable for backtesting pipelines.

TSM Storage Engine

TSM (Time-Structured Merge Tree): Inspired by LSM trees but optimized for time-series. Writes go to an in-memory Write-Ahead Log (WAL) and Cache. The Cache flushes to immutable TSM files periodically. Background compaction merges TSM files.

Series cardinality: Each unique combination of measurement + tag set is a "series." High cardinality (millions of unique series, e.g., one series per user_id) bloats the in-memory index and causes OOM. Keep tag values to bounded enumerations.

Shard groups: Data is physically partitioned into time-based shard groups (default: 7 days). Each shard group is an independent TSM storage directory. Retention policies delete entire expired shard groups atomically.
