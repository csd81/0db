What is Transactional Replication?
Changes (INSERT/UPDATE/DELETE) are captured from the Publisher's transaction log and streamed to one or more Subscribers in near-real-time. Each committed transaction is replayed at the Subscriber in the same order, preserving transactional consistency.
Log Reader Agent
Runs continuously on the Distributor. Reads the Publisher's transaction log, extracts committed DML for replicated articles, and writes them as commands to the distribution database (a persistent queue). Latency from Publisher commit to Distributor: typically <1 second on a healthy network.
Distribution Agent
Runs per Subscriber. Reads queued commands from the distribution database and applies them at the Subscriber using stored procedure calls (sp_MSins_*, sp_MSupd_*, sp_MSdel_*). Supports both push (runs at Distributor) and pull (runs at Subscriber) subscriptions.
Use cases
• High-availability read replicas (offload reporting queries)
• Site autonomy — remote offices need a local copy
• One-way sync OLTP → data warehouse staging
• Near-real-time warm standby (before Always On was available)

The Distributor's distribution database is a durable queue. If a Subscriber goes offline, commands accumulate until it reconnects. Queue retention is configurable (default 72 h).
