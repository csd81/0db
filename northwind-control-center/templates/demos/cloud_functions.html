▶ Cloud Functions FaaS Cold/Warm Start · HTTP + Background Triggers

Google Cloud Functions runs each invocation in an isolated container that is created on first request (cold start: 300–2000 ms) and kept alive for subsequent requests (warm start: sub-10 ms) until idle for several minutes. The platform manages all infrastructure, scaling from zero to thousands of parallel instances in seconds, and bills only for CPU time consumed during execution. The demo tracks the cold/warm/hot state of function instances across a burst of requests, showing how the warm pool fills, how idle instances expire, and how a concurrency spike forces new cold starts. This execution model makes FaaS ideal for event-driven workloads with unpredictable traffic but costly for high-throughput services that would maintain permanently warm instances anyway.
