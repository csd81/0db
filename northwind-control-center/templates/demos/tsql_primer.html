Setup — no transaction
Autocommit is ON by default. Every statement commits the instant it succeeds — there is no way to undo them.
-- ⚠ No BEGIN TRAN — autocommit mode SET NOCOUNT ON; DECLARE @prod_name varchar(20) = 'boston', @quantity int = 10, @cust_id nchar(5) = 'AROUT';
1
Product lookup
Find the product by partial name. Fail if 0 or >1 matches.
SELECT @res_no = COUNT(*) FROM Products WHERE ProductName LIKE '%' + @prod_name + '%' -- Boston Crab Meat → 1 match
2
Stock check
Retrieve stock level and unit price. Abort if stock < quantity.
SELECT @prod_id = ProductID, @stock = UnitsInStock, @unitprice = UnitPrice FROM Products WHERE ProductName LIKE '%boston%' -- UnitsInStock=900 ≥ 10 → OK
3
Balance check
Verify the customer has enough credit. If balance IS NULL, reject.
SELECT @cust_balance = balance FROM Customers WHERE CustomerID = @cust_id -- balance=1000, cost=184 → OK
4
Deduct balance ← AUTO-COMMITTED
The balance is deducted immediately. There is no pending transaction — this is permanent the moment it succeeds.
UPDATE Customers SET balance = balance - (@quantity * @unitprice) WHERE CustomerID = @cust_id -- ✓ COMMITTED immediately — cannot be undone
5
INSERT Orders header ← AUTO-COMMITTED
Order header row is inserted and committed immediately.
INSERT INTO Orders (CustomerID, OrderDate) VALUES (@cust_id, GETDATE()) SET @order_id = @@IDENTITY -- ✓ COMMITTED immediately — cannot be undone
6
INSERT [Order Details] ← FAILS
The Discount column is NOT NULL but is not supplied. This raises an error.
INSERT [Order Details] (OrderID, ProductID, Quantity, UnitPrice) -- ⚠ Missing: Discount (NOT NULL column!) VALUES(@order_id, @prod_id, @quantity, @unitprice) -- ✗ ERROR: Cannot insert NULL into 'Discount'
7
CATCH — but no ROLLBACK possible
Steps 4 and 5 are already committed. The database is in an inconsistent state: balance reduced, Orders row orphaned.
BEGIN CATCH -- Cannot rollback — no transaction was open PRINT 'ERROR: ' + ERROR_MESSAGE() END CATCH -- ✗ INCONSISTENT STATE: balance ↓, order missing
0
BEGIN TRAN — one atomic unit
All subsequent statements are pending. Nothing is written to disk until COMMIT. Any error triggers ROLLBACK — all or nothing.
SET XACT_ABORT ON; BEGIN TRAN; -- All statements below are PENDING (not committed)
1
Product lookup (same as before)
SELECT @res_no = COUNT(*) FROM Products WHERE ProductName LIKE '%boston%'
2
Stock + price check (same)
SELECT @prod_id = ProductID, @stock = UnitsInStock, @unitprice = UnitPrice FROM Products WHERE ProductName LIKE '%boston%'
3
Balance check (same)
SELECT @cust_balance = balance FROM Customers WHERE CustomerID = @cust_id
4
Deduct balance ⏳ PENDING
Balance is updated in memory — pending inside the open transaction. Not yet on disk.
UPDATE Customers SET balance = balance - (@quantity * @unitprice) WHERE CustomerID = @cust_id -- ⏳ PENDING — will be committed or rolled back together
5
INSERT Orders header ⏳ PENDING
INSERT INTO Orders (CustomerID, OrderDate) VALUES (@cust_id, GETDATE()) SET @order_id = @@IDENTITY -- ⏳ PENDING — part of the same transaction
6
INSERT [Order Details] ← FAILS (same bug)
Same missing Discount. Error is caught — but now we have a transaction to roll back.
INSERT [Order Details] (OrderID, ProductID, Quantity, UnitPrice) -- ⚠ Missing Discount VALUES(@order_id, @prod_id, @quantity, @unitprice) -- ✗ FAILS — same NOT NULL error
7
CATCH → ROLLBACK TRAN ✓
All pending changes (steps 4, 5) are undone atomically. The database is back to its original state.
BEGIN CATCH PRINT 'ERROR: ' + ERROR_MESSAGE() PRINT 'Rolling back transaction' ROLLBACK TRAN END CATCH -- ✓ CONSISTENT: balance unchanged, no orphaned rows
