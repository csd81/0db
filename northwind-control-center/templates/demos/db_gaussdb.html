Huawei GaussDB

AI-native · PostgreSQL-compatible · HTAP · China financial-grade

GaussDB is Huawei Cloud's enterprise-grade distributed database, commercially launched in 2019. The open-source variant openGauss is Apache 2.0. Built on PostgreSQL-compatible internals, it comes in two primary forms: GaussDB (OLTP, shared-nothing distributed relational) and GaussDB (DWS) for analytics. Its defining feature is AI4DB (AI for Database) — an AI layer that automatically recommends indexes, diagnoses slow queries, tunes knobs, and predicts failures. GaussDB is Huawei's answer to Oracle for China's financial sector, which is legally required to eliminate foreign databases by 2027 under government directives.

China's Oracle Replacement at Banking Scale

China's "Xinchuang" (信创) policy mandates replacement of Western IT infrastructure with domestic alternatives in government, banking, and critical infrastructure. GaussDB is Huawei's offering for this transition, specifically targeting Oracle and IBM Db2 installations in the financial sector.

The AI4DB component includes: self-diagnosis (automatic slow query root cause analysis), index advisor (recommends indexes based on actual workload), knob tuning (adjusts memory and parallelism parameters automatically), and performance predictor (forecasts degradation before it happens). This is particularly valuable for DBAs migrating from Oracle who aren't familiar with PostgreSQL tuning.

Distributed Relational Model 
-- GaussDB OLTP: standard PostgreSQL-compatible DDL

CREATE TABLE transactions (
    txn_id       BIGSERIAL PRIMARY KEY,
    account_from BIGINT NOT NULL,
    account_to   BIGINT NOT NULL,
    amount       DECIMAL(18,4) NOT NULL,
    currency     CHAR(3) DEFAULT 'CNY',
    txn_type     VARCHAR(20),
    status       VARCHAR(10) DEFAULT 'pending',
    created_at   TIMESTAMPTZ DEFAULT NOW(),
    CONSTRAINT chk_amount CHECK (amount > 0)
)
DISTRIBUTE BY HASH (account_from)  -- GaussDB-specific: distribute by column
PARTITION BY RANGE (created_at) (
    PARTITION p2024q1 VALUES LESS THAN ('2024-04-01'),
    PARTITION p2024q2 VALUES LESS THAN ('2024-07-01'),
    PARTITION p2024q3 VALUES LESS THAN ('2024-10-01'),
    PARTITION p2024q4 VALUES LESS THAN ('2025-01-01')
);

-- Column-store table (for analytics on same system)
CREATE TABLE txn_summary (
    day         DATE,
    txn_type    VARCHAR(20),
    total_amount DECIMAL(18,4),
    count       BIGINT
) WITH (ORIENTATION = COLUMN)  -- columnar storage in GaussDB DWS
  DISTRIBUTE BY HASH (txn_type);

-- Compressed table (OLTP hot data → compress old partitions)
ALTER TABLE transactions
PARTITION p2024q1
SET (COMPRESSION = LOW);

PostgreSQL-Compatible SQL + GaussDB Extensions 
-- ── 1. Standard PostgreSQL SQL (fully compatible)
SELECT t.txn_id,
       a1.account_name AS sender,
       a2.account_name AS receiver,
       t.amount,
       t.created_at
FROM transactions t
JOIN accounts a1 ON t.account_from = a1.id
JOIN accounts a2 ON t.account_to   = a2.id
WHERE t.created_at > NOW() - INTERVAL '1 day'
  AND t.amount > 10000
ORDER BY t.created_at DESC;

-- ── 2. Window functions (same as PostgreSQL)
SELECT account_from,
       amount,
       SUM(amount) OVER (
           PARTITION BY account_from
           ORDER BY created_at
           ROWS UNBOUNDED PRECEDING
       ) AS cumulative_outflow
FROM transactions
WHERE status = 'completed';

-- ── 3. CTE with recursive (detect circular transfers — fraud)
WITH RECURSIVE transfer_chain AS (
    SELECT txn_id, account_from, account_to, amount, ARRAY[account_from] AS path
    FROM transactions
    WHERE account_from = 123456

    UNION ALL

    SELECT t.txn_id, t.account_from, t.account_to, t.amount, path || t.account_from
    FROM transactions t
    JOIN transfer_chain tc ON t.account_from = tc.account_to
    WHERE NOT t.account_from = ANY(path)  -- avoid infinite loop
      AND array_length(path, 1) < 10
)
SELECT * FROM transfer_chain WHERE account_to = 123456;  -- circular transfer

-- ── 4. MERGE (upsert — PostgreSQL 15 compatible)
MERGE INTO account_balances b
USING (SELECT account_id, SUM(amount) AS net FROM transactions GROUP BY account_id) t
ON b.account_id = t.account_id
WHEN MATCHED THEN
    UPDATE SET balance = balance + t.net, updated_at = NOW()
WHEN NOT MATCHED THEN
    INSERT (account_id, balance, updated_at)
    VALUES (t.account_id, t.net, NOW());

-- ── 5. Stored procedure (PL/pgSQL compatible)
CREATE OR REPLACE PROCEDURE transfer_funds(
    p_from BIGINT, p_to BIGINT, p_amount DECIMAL, OUT p_txn_id BIGINT
) LANGUAGE plpgsql AS $$
BEGIN
    -- Check balance
    IF (SELECT balance FROM accounts WHERE id = p_from FOR UPDATE) < p_amount THEN
        RAISE EXCEPTION 'Insufficient funds';
    END IF;
    UPDATE accounts SET balance = balance - p_amount WHERE id = p_from;
    UPDATE accounts SET balance = balance + p_amount WHERE id = p_to;
    INSERT INTO transactions (account_from, account_to, amount, status)
    VALUES (p_from, p_to, p_amount, 'completed')
    RETURNING txn_id INTO p_txn_id;
END;
$$;
