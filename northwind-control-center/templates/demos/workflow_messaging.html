Delivery Scenarios: Online vs Offline
✅ Recipient is Online
  1. Router looks up recipient's gateway server ID from Redis: GET ws:{user_b_id} → "gateway-server-7"
  2. Internal gRPC call to gateway-server-7: DeliverMessage(user_b, encrypted_payload)
  3. Gateway pushes to client WebSocket immediately
  4. Client sends ACK → Delivery receipt (single tick → double tick)
  5. When message is read: READ event → Read receipt (blue ticks)
⚡ End-to-end latency: ~100ms
📵 Recipient is Offline
  1. Redis lookup returns null → recipient offline
  2. Message stored in ScyllaDB/Cassandra: partition(user_b_id) → sorted(timestamp, msg_id)
  3. Push notification sent via APNs (iOS) or FCM (Android)
  4. Push payload is minimal (no message content for E2E): {"new_message": true}
  5. When user reconnects, client pulls pending messages from DB since last seen
📦 Message stored up to 30 days (WhatsApp)
🗄️ Storage & Data Layer
ScyllaDB (Discord)
Discord migrated from Cassandra to ScyllaDB in 2022. ScyllaDB is Cassandra-compatible but written in C++ — eliminates JVM garbage collection pauses that caused latency spikes. Discord serves 4 billion messages per day. Wide-row per channel: partition(channel_id, bucket_month) → sorted(message_id DESC).
Apache Cassandra (WhatsApp/Slack)
Leaderless, multi-master wide-column store. WhatsApp's message store: never deletes messages older than delivery window — moves them to off-server storage after delivery. Slack uses Cassandra for message history with 10-year retention on paid tiers.
Redis (Presence / Routing)
SET ws:{user_id} {gateway_server_id} EX 30 — expires if heartbeat stops. Used to route messages to the correct gateway server. 30-second TTL means presence state is eventually consistent (30s max lag for offline detection). Redis Cluster with 6 nodes for HA.
Erlang Mnesia (WhatsApp)
WhatsApp runs on Erlang/OTP. Mnesia is Erlang's built-in distributed DB — used for router state and session tables. Erlang was chosen specifically because a single BEAM VM can maintain 2 million concurrent connections with lightweight process-per-connection model.
⚙️ Compute & Concurrency
Erlang/OTP (WhatsApp)
Actor model: each user connection is a lightweight Erlang process (~2KB heap). Supervisor tree restarts crashed processes automatically. Message passing between actors — no shared memory, no locks. 2 million concurrent connections per server is a well-documented benchmark. Pattern-matching for protocol handling.
Elixir / Phoenix (Slack alternatives)
Elixir runs on BEAM (same VM as Erlang). Phoenix Channels provide WebSocket abstraction. Phoenix LiveView for real-time UI. Used by platforms that want Erlang's concurrency with Ruby-like syntax. Discord originally used Elixir before migrating hot paths to Rust/Go.
Go (Discord routing)
Discord's message routing and gateway services rewritten in Go for better goroutine memory efficiency vs Elixir at their scale. Go goroutines: ~4KB stack. 10 million concurrent connections → 40 GB for goroutines alone. GC pauses <1ms with Go's tricolor mark-sweep.
Rust (Discord read states)
Discord's "read states" service (tracking which messages you've seen) handles 1 billion state updates/day. Rewritten from Go to Rust: eliminated GC pauses entirely. Rust's ownership model prevents data races at compile time. Discord published this case study in 2020.
🌐 Protocols & Encryption
WebSocket (RFC 6455)
Persistent TCP connection. Upgrade from HTTP/1.1 via Upgrade: websocket header. Full-duplex — both sides can send at any time. Heartbeat (PING/PONG frames) every 30 seconds. If pong not received → connection closed → client reconnects. Binary frames for efficiency (not text/JSON).
MTProto (Telegram)
Telegram's custom binary protocol. Designed for mobile networks: works over TCP, UDP (for speed), and HTTP (as fallback). Authorization keys generated via Diffie-Hellman. 256-bit AES-IGE encryption. MTProto 2.0 adds perfect forward secrecy. Controversial: designed in-house, not Signal Protocol.
Signal Protocol (WhatsApp/Signal)
Double Ratchet algorithm: combines Diffie-Hellman key exchange (session keys) with KDF ratchet (per-message keys). Forward secrecy: compromise of today's key doesn't expose past messages. Pre-keys allow offline key exchange. Open-source, widely audited. WhatsApp uses Signal Protocol for E2EE.
WebRTC (Voice/Video)
STUN/TURN servers for NAT traversal. SRTP (Secure RTP) for encrypted media. ICE (Interactive Connectivity Establishment) finds best path. WhatsApp calls use Opus codec (variable bitrate, 6–510 kbps). Discord: Opus audio + VP8/VP9 video. DTLS for key exchange.
Workflow Deep Dive
1
WebSocket Handshake + TLS
Client sends HTTP GET with Upgrade: websocket and Sec-WebSocket-Key: [base64 random]. Server responds 101 Switching Protocols. TLS 1.3 handshake: 1 RTT (vs 2 RTT for TLS 1.2) — critical for mobile networks. Connection kept alive with PING/PONG every 30s. On 4G, a round-trip from London to New York: ~80ms RTT. Total handshake: ~200ms.
2
Presence Registration in Redis
After auth, gateway server writes: SET ws:{user_id} {this_server_id} EX 30. Every 25 seconds: EXPIRE ws:{user_id} 30 (heartbeat refresh). If server crashes or client disconnects without clean close, key expires → user marked offline within 30 seconds. Redis Pub/Sub for cross-server presence events (user-joined-channel, user-left).
3
End-to-End Encryption (Signal Protocol)
Session key exchange uses X3DH (Extended Triple Diffie-Hellman): combines identity keys, signed pre-keys, and one-time pre-keys. Result: shared session key, established without server seeing it. Messages encrypted with AES-256-GCM. Server receives and forwards ciphertext — cannot read content. Group chats: per-sender key distributed to each member (Sender Key Protocol — O(1) encryptions vs O(N)).
4
Message Storage in ScyllaDB
Discord schema: CREATE TABLE messages (channel_id bigint, message_id bigint, author_id bigint, content text, PRIMARY KEY ((channel_id, bucket), message_id)) WITH CLUSTERING ORDER BY (message_id DESC). Bucket = floor(message_id / 1M). This prevents unbounded partition growth. Snowflake message IDs (Discord) encode timestamp, worker, sequence — sortable without DB clock. ScyllaDB shards data across nodes, replication factor 3.
🐘 Why Not PostgreSQL?
PostgreSQL can't maintain 2M concurrent connections — each connection is a process (~5 MB). Even with PgBouncer pooling, the OLTP model doesn't fit messaging. Cassandra/ScyllaDB's wide-row model maps naturally to "messages in a channel" — single partition scan, no JOINs needed. Write throughput: 1M msg/s with Cassandra vs ~50K/s with Postgres.
⚡ Erlang: 2M Connections
WhatsApp: 50 engineers, 900 million users. Each Erlang process: ~2 KB heap (vs ~2 MB for OS thread). 2M processes on a 64 GB server = entirely feasible. Message passing between processes is lock-free. If a process crashes, the supervisor restarts it — isolated failure, no cascading crash. This is how 1 server handles 2M simultaneous chat connections.
🎯 CAP Position
AP system for delivery, CP for auth. Message delivery: if a Cassandra node is partitioned, messages still write (eventual consistency — minor reordering acceptable). Authentication and session keys: must be CP (you can't have two valid sessions for the same user delivering to different keys — that's a security breach).
