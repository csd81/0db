Connection Pooling

A naive app opens one DB connection per HTTP request. At 500 concurrent users → 500 connections, each consuming ~5 MB RAM on the DB server.

PgBouncer (GCP/AWS)
Lightweight pooler between app and PostgreSQL. Transaction-mode pooling allows pool much smaller than thread count.
RDS Proxy (AWS)
Fully managed pooler native to RDS/Aurora. Integrates with IAM for credential rotation without connection drops — ideal for Lambda.
Maintenance Windows

Managed services apply patches during a configurable 30-minute slot during off-peak hours.

  1. Set window to lowest-traffic period (e.g., Sunday 03:00–03:30)
  2. Enable notifications so on-call engineers are aware
  3. Test next minor version on staging replica before window
Parameter changes: static params require reboot · dynamic params apply immediately

Spanner TrueTime

Cloud Spanner achieves external consistency without a global lock via TrueTime — GPS receivers and atomic clocks in every Google data center.

TrueTime API
Returns interval [earliest, latest] — guaranteed to contain true wall-clock time. Uncertainty: typically 1–7 ms.
Commit Wait
Spanner waits out the TrueTime uncertainty interval before declaring commit complete. Timestamp is provably in the past for all subsequent readers.
Clock Drift Effect
If GPS drifts, TrueTime widens the interval → higher commit-wait latency. System remains correct at the cost of throughput.
Aurora Global Database

One primary region (read/write) + up to 5 secondary regions (read-only). Dedicated log-shipping layer with typical lag under 1 second.

Disaster Recovery
Secondary region promoted to primary in under 1 minute.
Limitation
Does not offer true multi-region writes — primary region is still the single write authority.

Bigtable Row Key Design

The row key is the only indexed field. Its design determines whether load is spread evenly or concentrated on a hotspot.

✗ Anti-pattern — Monotonic Timestamp
# BAD: all writes go to same tablet (hotspot)
row_key = f"{timestamp_ms}#sensor_{sensor_id}"
# Example: "1714694400000#sensor_42"
# New writes always land on the LAST tablet
✓ Pattern 1 — Reverse Timestamp
MAX_TIMESTAMP = 9999999999999
row_key = f"{MAX_TIMESTAMP - timestamp_ms}#sensor_{sensor_id}"
# Newest data sorts first; avoids leading-edge hotspot
✓ Pattern 2 — Salted Prefix
import hashlib

def make_row_key(sensor_id, timestamp_ms, buckets=100):
    salt = int(hashlib.md5(sensor_id.encode()
             ).hexdigest(), 16) % buckets
    return f"{salt:03d}#{sensor_id}#{timestamp_ms}"
# "042#sensor_42#1714694400000"
# Distributes sensors across 100 evenly-spaced prefixes
Tradeoff: to read all data for one sensor, must query all 100 buckets and merge.

BigQuery ML — Train & Score in SQL 
-- Train a linear regression model on Northwind orders
CREATE OR REPLACE MODEL northwind.revenue_model
OPTIONS (
    model_type = 'linear_reg',
    input_label_cols = ['total_revenue']
) AS
SELECT
    c.country,
    p.category_id,
    EXTRACT(MONTH FROM o.order_date) AS order_month,
    SUM(od.unit_price * od.quantity * (1 - od.discount))
        AS total_revenue
FROM northwind.orders o
JOIN northwind.order_details od ON o.order_id = od.order_id
JOIN northwind.customers c ON o.customer_id = c.customer_id
JOIN northwind.products p ON od.product_id = p.product_id
GROUP BY 1, 2, 3;

-- Score new data with the trained model
SELECT * FROM ML.PREDICT(
    MODEL northwind.revenue_model,
    (SELECT 'Germany' AS country, 1 AS category_id, 12 AS order_month)
);
BigQuery Cost Optimization — Partition Filters
$2.50
per query without partition filter
500 GB × $5/TB
$0.025
per query with partition filter
5 GB × $5/TB — 100× cheaper
-- Expensive: scans entire table (500 GB)
SELECT customer_id, SUM(freight)
FROM northwind.orders GROUP BY customer_id;

-- Cheap: partition filter limits scan to ~5 GB
SELECT customer_id, SUM(freight)
FROM northwind.orders
WHERE order_date >= DATE_TRUNC(CURRENT_DATE(), MONTH)
GROUP BY customer_id;
Aurora Serverless — Cold Start Retry 
def connect_with_retry(dsn, max_attempts=5):
    delay = 1.0
    for attempt in range(max_attempts):
        try:
            return psycopg2.connect(dsn)
        except psycopg2.OperationalError as e:
            if attempt == max_attempts - 1:
                raise
            time.sleep(delay)
            delay = min(delay * 2, 30.0)
# Aurora Serverless pause → 25–35s resume latency

SQL Server Columnstore Index (Northwind) 
-- Create NCCI alongside the clustered B-tree index
CREATE NONCLUSTERED COLUMNSTORE INDEX ncci_OrderDetails
ON dbo.[Order Details] (OrderID, ProductID, UnitPrice, Quantity, Discount);

-- This aggregation is routed to the columnstore:
SELECT cat.category_name,
       SUM(od.UnitPrice * od.Quantity * (1 - od.Discount)) AS total_revenue,
       COUNT(DISTINCT o.OrderID) AS order_count
FROM dbo.Orders o
JOIN dbo.[Order Details] od ON o.OrderID = od.OrderID
JOIN dbo.Products p ON od.ProductID = p.ProductID
JOIN dbo.Categories cat ON p.CategoryID = cat.CategoryID
GROUP BY cat.category_name
ORDER BY total_revenue DESC;

SQL Server 2022 Ledger Tables 
-- Create a ledger-enabled transactions table
CREATE TABLE dbo.AccountTransactions (
    TransactionId   INT IDENTITY PRIMARY KEY,
    AccountId       INT NOT NULL,
    Amount          DECIMAL(18, 2) NOT NULL,
    Description     NVARCHAR(255),
    TransactionDate DATETIME2 DEFAULT GETDATE()
) WITH (SYSTEM_VERSIONING = ON, LEDGER = ON);
-- SQL Server auto-creates: dbo.AccountTransactions_Ledger

-- Query full immutable history
SELECT lt.ledger_transaction_id, lt.commit_time,
       lt.principal_name, t.AccountId, t.Amount,
       lt.ledger_operation_type_desc  -- INSERT/UPDATE/DELETE
FROM dbo.AccountTransactions_Ledger lt
WHERE lt.AccountId = 1001
ORDER BY lt.commit_time;

-- Generate cryptographic digest for audit submission
EXECUTE sp_generate_database_ledger_digest;
-- Returns Merkle root — auditors store this independently
-- Later: sp_verify_database_ledger to detect tampering
Service Selection Decision Tree 
Need SQL?
  → Global multi-region writes required?
      YES → Spanner / Aurora Global / Cosmos DB SQL API (multi-master)
      NO  → Single-region OLTP?
              YES → Cloud SQL / RDS / Azure SQL Database
              NO  → Analytical queries on petabyte scale?
                      YES → BigQuery / Redshift / Synapse Analytics

Need NoSQL?
  → Real-time mobile/web push sync?
      YES → Firestore
      NO  → High-throughput key-value / document?
              YES → DynamoDB (AWS-native) / Cosmos DB NoSQL API
              NO  → Graph traversal?
                      YES → Neptune / Cosmos DB Gremlin / Neo4j AuraDB
                      NO  → Time-series / IoT sensor streams?
                              YES → Bigtable / Keyspaces / InfluxDB Cloud

Need HTAP (OLTP + OLAP, no ETL lag)?
  → TiDB / AlloyDB / SingleStore / SQL Server columnstore

Need immutable audit trail?
  → AWS QLDB / SQL Server Ledger Tables

Portability Strategies
  1. Use standard engines — RDS PostgreSQL is dramatically more portable than DynamoDB. PostgreSQL wire protocol supported everywhere.
  2. Abstract behind an interface — A thin data access layer hides provider-specific query patterns behind a repository interface.
  3. Export data regularly — Daily export to cloud storage (Parquet/CSV) ensures data is never hostage to the vendor.
