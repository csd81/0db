OceanBase

NewSQL · MySQL + Oracle compatible · Broke TPC-C world record · Powers Alipay at 92,000 TPS

Developed inside Alibaba starting in 2010 and open-sourced in 2021, OceanBase was purpose-built to handle Alipay's astronomical transaction volumes — the payment system processing Alibaba's Singles' Day (11.11) peak loads. In 2021, OceanBase set the TPC-C benchmark record at 707 million tpmC (transactions per minute), surpassing Oracle, IBM, and AWS Aurora. It features a multi-tenant architecture, MySQL and Oracle SQL compatibility modes, and a Paxos-based distributed transaction engine with zero-data-loss guarantees. Used by the Agricultural Bank of China (the world's largest bank by deposits), WeBank, and over 1,000 financial institutions in China.

Multi-Tenant Architecture

Tenants — OceanBase is natively multi-tenant. Each tenant has isolated CPU, memory, and storage quotas. The "sys" tenant administers all others. This is how Alipay runs multiple isolated business units on one cluster.

MySQL Mode vs Oracle Mode — choose per tenant. MySQL mode: MySQL syntax, MySQL system tables. Oracle mode: Oracle PL/SQL, Oracle functions, sequence objects. The same cluster can host both.

Table Groups — group related tables so their partitions are co-located on the same OBServer nodes, eliminating cross-node joins for common access patterns.

Partition types — HASH, RANGE, LIST, and COMPOSITE (sub-)partitioning. Partition pruning eliminates unnecessary node access.

DDL Examples 
-- MySQL mode: partitioned table
CREATE TABLE transactions (
    txn_id     BIGINT NOT NULL AUTO_INCREMENT,
    account_id BIGINT NOT NULL,
    amount     DECIMAL(15,2),
    txn_time   DATETIME,
    status     TINYINT DEFAULT 1,
    PRIMARY KEY (txn_id, txn_time)  -- partition key must be in PK
)
PARTITION BY RANGE COLUMNS(txn_time) (
    PARTITION p202401 VALUES LESS THAN ('2024-02-01'),
    PARTITION p202402 VALUES LESS THAN ('2024-03-01'),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- Create table group for co-location
CREATE TABLEGROUP tg_banking;

ALTER TABLE accounts  TABLEGROUP = tg_banking;
ALTER TABLE transactions TABLEGROUP = tg_banking;

-- Oracle mode: sequences and PL/SQL
CREATE SEQUENCE order_seq START WITH 1000000
    INCREMENT BY 1 NOCACHE NOCYCLE;

DECLARE
    v_balance NUMBER;
BEGIN
    SELECT balance INTO v_balance
    FROM accounts WHERE acct_id = 42;
    IF v_balance >= 1000 THEN
        UPDATE accounts SET balance = balance - 1000
        WHERE acct_id = 42;
        COMMIT;
    END IF;
END;
SQL Primer (MySQL Mode) 
-- 1. Distributed query with partition pruning
SELECT account_id, SUM(amount) AS net_flow
FROM transactions
WHERE txn_time >= '2024-01-01'
  AND txn_time  < '2024-02-01'
  AND status = 1
GROUP BY account_id
ORDER BY net_flow DESC
LIMIT 100;
-- Only p202401 partition accessed (partition pruning)

-- 2. EXPLAIN EXTENDED — shows distributed plan
EXPLAIN EXTENDED
SELECT t.txn_id, t.amount, a.name
FROM transactions t
JOIN accounts a ON t.account_id = a.id
WHERE t.status = 1 AND t.amount > 10000;
-- Shows: Exchange → TableScan on TiKV-style storage

-- 3. Cross-partition transaction
BEGIN;
UPDATE accounts SET balance = balance - 5000
WHERE acct_id = 1001;
UPDATE accounts SET balance = balance + 5000
WHERE acct_id = 2002;
INSERT INTO transactions(account_id, amount, txn_time, status)
VALUES (1001, -5000, NOW(), 1);
COMMIT;

-- 4. Parallel DML (large batch operations)
INSERT /*+ PARALLEL(4) */ INTO archive_transactions
SELECT * FROM transactions
WHERE txn_time < '2023-01-01';

-- 5. Check cluster status
SELECT zone, status, count(*) AS servers
FROM oceanbase.dba_ob_servers
GROUP BY zone, status;

-- 6. Monitor slow queries
SELECT sql_text, elapsed_time, plan_type
FROM oceanbase.gv$sql_audit
WHERE elapsed_time > 1000000  -- > 1 second (microseconds)
ORDER BY elapsed_time DESC LIMIT 20;
Killer Use Cases

Alipay (Singles' Day) — 92,000 TPS sustained during 11.11 peak (2021), replacing an Oracle + MySQL cluster combination. Zero downtime. The benchmark that proved OceanBase at financial scale.

Storage Engine

LSM-Tree with Major Compaction — OceanBase uses an LSM-tree variant optimized for financial workloads. Writes go to a MemTable (in-memory), spilled to SSTables on disk. Major compaction merges all levels once daily (default 2 AM) for read performance recovery.

Hybrid row-column storage — row storage for OLTP; optional columnar encoding for analytics workloads within the same engine (without TiFlash-style separate replicas).

MVCC — multi-version concurrency control with Snapshot Isolation as default and serializable as option. Old versions retained for configurable time (undo window).

Paxos & HA

3-zone deployment — standard deployment has one OBServer leader in each of 3 zones (IDCs). Paxos requires 2/3 votes to commit, tolerating one zone failure.

5-zone for geo — for multi-region deployments, 5 zones (3 in primary region, 2 in secondary) enable tolerating a full primary region failure (3/5 quorum maintained).

OBProxy — intelligent routing proxy: reads from the Leader replica of each partition, routes writes to the appropriate OBServer, handles failover transparently.

RPO=0 — since every commit requires Paxos quorum acknowledgment (including at least one remote replica), committed data is never lost on single-zone failure.
