MySQL

Relational OLTP · InnoDB engine · GTIDs · Most deployed open-source database

MySQL launched in 1995 and became the backbone of the LAMP stack, powering the first generation of web applications. Now owned by Oracle, it remains the world's most deployed open-source RDBMS by sheer installation count. InnoDB's clustered B+tree primary key, MVCC, and binary log replication make it a reliable workhorse for WordPress, Shopify, GitHub, and Twitter's original read-heavy workloads.

InnoDB Strengths
  • Clustered primary key — PK lookups read fewer pages
  • Row-level locking (not table-level like MyISAM)
  • Foreign key constraints enforced at engine level
  • MVCC with undo log space management
  • Doublewrite buffer prevents partial page writes on crash
  • Buffer pool (equivalent to PostgreSQL's shared_buffers)
  • Binary log enables point-in-time recovery and replication

Schema Definition 
-- InnoDB table: clustered on PRIMARY KEY
CREATE TABLE orders (
  order_id  INT AUTO_INCREMENT PRIMARY KEY,
  user_id   INT NOT NULL,
  status    ENUM('pending','paid','shipped','done'),
  total     DECIMAL(12,2) NOT NULL,
  payload   JSON,           -- MySQL 5.7.8+ JSON column
  created   DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_user (user_id),
  INDEX idx_status_created (status, created),
  FOREIGN KEY (user_id) REFERENCES users(id)
    ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Range partitioning by year
ALTER TABLE orders PARTITION BY RANGE (YEAR(created)) (

  PARTITION p2023 VALUES LESS THAN (2024),

  PARTITION future VALUES LESS THAN MAXVALUE

-- Extract with JSON_EXTRACT / -> shorthand
SELECT order_id,
  payload->'$.shipping' AS ship_type,
  JSON_EXTRACT(payload, '$.items[0].sku') AS first_sku
FROM orders WHERE user_id = 42;

-- Functional index on JSON path (MySQL 8.0+)
ALTER TABLE orders ADD COLUMN ship_type VARCHAR(50)
  GENERATED ALWAYS AS (payload->>'$.shipping') VIRTUAL;
CREATE INDEX idx_ship ON orders (ship_type);
Window Functions (8.0+) 
-- Rank users by spend per region
SELECT user_id, region, total_spend,
  RANK() OVER (PARTITION BY region
               ORDER BY total_spend DESC) AS rk,
  NTILE(4) OVER (ORDER BY total_spend) AS quartile
FROM user_summary;

-- Running total
SELECT created, total,
  SUM(total) OVER (ORDER BY created
    ROWS UNBOUNDED PRECEDING) AS cumulative
FROM orders WHERE user_id = 42;

-- LAG / LEAD for period comparisons
SELECT month, revenue,
  LAG(revenue) OVER (ORDER BY month) AS prev_month,
  revenue - LAG(revenue) OVER (ORDER BY month) AS delta
FROM monthly_revenue;

Stored Procedure 
DELIMITER //
CREATE PROCEDURE ProcessOrder(IN p_order_id INT)
BEGIN
  DECLARE v_total DECIMAL(12,2);
  START TRANSACTION;
    SELECT total INTO v_total FROM orders
    WHERE order_id = p_order_id FOR UPDATE;

    UPDATE inventory SET qty = qty - 1
    WHERE sku IN (SELECT sku FROM order_items
                  WHERE order_id = p_order_id);

    UPDATE orders SET status='paid'
    WHERE order_id = p_order_id;
  COMMIT;
END//
DELIMITER ;

InnoDB Internals

Clustered Index: The primary key IS the table. Rows are stored sorted by PK on disk. Secondary indexes store the PK value as the row locator. Implications: PK inserts should be monotonically increasing (AUTO_INCREMENT) to avoid B+tree page splits.

Buffer Pool: MySQL's page cache. Default 128MB — should be 70–80% of RAM in production. The LRU list has a "young" sublist (recently accessed) and "old" sublist (cold pages). Full table scans won't evict hot working set.

Doublewrite Buffer: Before writing a 16KB page to disk, InnoDB first writes it to the doublewrite buffer (sequential I/O). On crash, InnoDB can recover a torn page from the doublewrite copy.

Binary Log: Statement-level or row-based record of all DDL + DML. Primary → Replica replication reads binlog. GTIDs (Global Transaction IDs) make failover safe by giving every transaction a globally unique ID.

Key Config Knobs 
# my.cnf — production InnoDB tuning

[mysqld]
# InnoDB buffer pool (most important!)
innodb_buffer_pool_size = 12G
innodb_buffer_pool_instances = 8

# I/O
innodb_io_capacity = 2000
innodb_io_capacity_max = 4000
innodb_flush_method = O_DIRECT

# Logging
innodb_log_file_size = 2G      # larger = fewer flushes
sync_binlog = 1                # durable binlog
innodb_flush_log_at_trx_commit = 1  # full ACID

# Replication
server_id = 1
log_bin = mysql-bin
binlog_format = ROW
gtid_mode = ON
enforce_gtid_consistency = ON
