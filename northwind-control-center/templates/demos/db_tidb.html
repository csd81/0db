TiDB

NewSQL · MySQL-compatible · HTAP (TiFlash) · Scale-out MySQL without sharding

Developed by PingCAP and open-sourced in 2015, TiDB is a MySQL-compatible distributed SQL database that scales horizontally like a NoSQL system while maintaining full ACID transactions. Its headline feature: applications written for MySQL can migrate to TiDB with zero code changes — the same SQL queries, same wire protocol, same JDBC/Python drivers. Internally, TiDB separates the SQL layer (TiDB servers) from storage (TiKV, a distributed key-value store with Raft consensus). TiFlash adds a columnar replica for analytics (HTAP), enabling mixed OLTP and OLAP workloads on the same data. Used heavily at Zhihu (China's Quora), JD.com, and Meituan.

Regions & Sharding

Transparent Sharding — TiDB automatically shards data into "Regions" (default 96MB). Applications use standard MySQL SQL; sharding is invisible.

Region = Raft Group — each Region is replicated 3x via Raft consensus. A write succeeds when 2 of 3 replicas acknowledge (quorum).

PD (Placement Driver) — stores Region metadata (which nodes hold which key ranges), schedules Region splits/merges, and rebalances load across TiKV nodes.

TiFlash Replicas — opt-in columnar replicas synchronized from TiKV via Raft. Analytics queries are automatically routed to TiFlash by the optimizer if a TiFlash replica exists.

DDL Examples 
-- Standard MySQL DDL — works as-is
CREATE TABLE orders (
    id         BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    total      DECIMAL(10,2),
    status     VARCHAR(20) DEFAULT 'pending',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user (user_id),
    INDEX idx_status_date (status, created_at)
) ENGINE=InnoDB;

-- Add TiFlash replica for analytics
ALTER TABLE orders
SET TIFLASH REPLICA 1;

-- Shard the table across regions:
-- No action needed — TiDB does this automatically!
-- Monitor regions:
SHOW TABLE orders REGIONS;

-- Partition hint (force read from TiFlash)
SELECT /*+ READ_FROM_STORAGE(tiflash[orders]) */
    status, COUNT(*), SUM(total)
FROM orders
GROUP BY status;
MySQL-Compatible SQL 
-- 1. Same queries as MySQL — no changes needed
SELECT u.name, COUNT(o.id) AS order_count, SUM(o.total) AS ltv
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.country = 'US' AND o.status = 'completed'
GROUP BY u.id, u.name
HAVING ltv > 1000
ORDER BY ltv DESC
LIMIT 100;

-- 2. Window functions (MySQL 8.0 compatible)
SELECT id, user_id, total,
       RANK() OVER (PARTITION BY user_id
                    ORDER BY total DESC) AS user_rank,
       SUM(total) OVER (PARTITION BY user_id) AS user_total
FROM orders WHERE status = 'completed';

-- 3. Pessimistic transactions (like MySQL InnoDB)
BEGIN PESSIMISTIC;
SELECT balance FROM accounts WHERE id = 42 FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 42;
UPDATE accounts SET balance = balance + 100 WHERE id = 99;
COMMIT;

-- 4. Optimistic transactions (TiDB default for older versions)
BEGIN OPTIMISTIC;
UPDATE inventory SET qty = qty - 1 WHERE product_id = 7;
COMMIT;  -- may fail if write conflict detected at commit

-- 5. EXPLAIN ANALYZE (shows TiKV vs TiFlash routing)
EXPLAIN ANALYZE
SELECT /*+ READ_FROM_STORAGE(tiflash[orders]) */
    DATE(created_at) AS dt, SUM(total)
FROM orders WHERE created_at > '2024-01-01'
GROUP BY dt ORDER BY dt;

-- 6. Check TiFlash sync status
SELECT * FROM information_schema.tiflash_replica
WHERE TABLE_NAME = 'orders';
Killer Use Cases

MySQL Scale-Out — the #1 use case: a MySQL table has grown to 10TB+ and queries are slow. Migrate to TiDB (zero code changes), scale horizontally, performance restored.

HTAP Workloads — e-commerce platforms (JD.com, Meituan) run both OLTP (order processing) and analytics (daily sales reports) on the same TiDB cluster — OLTP on TiKV, analytics on TiFlash.

Zhihu (Chinese Quora) — 100+ TiDB nodes, hundreds of TB of data, serving Q&A content for hundreds of millions of users. Replaced multiple MySQL clusters.

Financial Services — serializable transactions, automatic HA, cross-datacenter replication. Used by WeBank and other Chinese fintech companies.

Performance

OLTP: Similar to MySQL per node, scales horizontally

TiFlash analytics: Columnar MPP, much faster than TiKV for scans

Scale: 100+ nodes in production deployments

Raft: Cross-AZ replication ~20-50ms

TiKV Storage

RocksDB — TiKV uses RocksDB as its local storage engine (LSM-tree). Each TiKV node stores multiple Regions as RocksDB column families.

Raft Log — all writes are first appended to the Raft log, replicated to the quorum, then applied to RocksDB. This guarantees durability and consistency.

MVCC — TiKV implements multi-version concurrency control at the KV layer. Every key has multiple versions (timestamp → value). Old versions are garbage collected by TiDB's GC worker.

2-Phase Commit — distributed transactions use Percolator-style 2PC (Google's approach) with TiKV locks stored in the KV store itself.

TiFlash Columnar Engine

Async Replication — TiFlash receives data changes from TiKV via Raft learner protocol (not part of the Raft quorum — no write latency impact on OLTP).

Columnar Storage — data is physically reorganized from row format (TiKV) to columnar format (TiFlash). SIMD-accelerated vectorized execution for analytics.

Automatic Routing — the TiDB optimizer automatically routes queries to TiFlash when: (1) a TiFlash replica exists, (2) a full table scan is needed, (3) it estimates TiFlash will be faster.

TiSpark — Apache Spark connector reads from TiKV/TiFlash directly, enabling large-scale ETL and ML workloads on TiDB data without exporting.
