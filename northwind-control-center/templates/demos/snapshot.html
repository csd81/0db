Snapshot Replication Demo

Idle

SQL Server snapshot replication copies a complete, consistent image of a publisher's data to subscribers rather than streaming incremental changes, making it the simplest replication topology to configure and maintain. A Snapshot Agent photographs the entire publication at a scheduled interval; a Distribution Agent delivers that snapshot to each subscriber, overwriting their existing data with the fresh baseline. This full-refresh approach is ideal for read-only reference data that changes infrequently — product catalogues, price lists, lookup tables — because subscribers never need to apply incremental change streams. Unlike transactional or merge replication, snapshot replication imposes zero ongoing overhead on the publisher's transaction log between snapshot intervals.

Replication Topology
What is Snapshot Replication?
The Publisher takes a complete copy (snapshot) of all replicated tables and ships it to Subscribers. The Distributor stores the snapshot files and coordinates delivery. On each scheduled interval the Subscriber's data is completely replaced — no incremental tracking.
Publisher → Distributor → Subscriber
Three roles in every SQL Server replication topology:
Publisher — owns source data; defines articles (tables/views to replicate).
Distributor — staging server; holds the distribution database (queue).
Subscriber — receives and applies the replicated data.
When to use
• Data that changes infrequently (price lists, lookup tables)
• Subscribers that need a fresh full copy on a schedule (daily reports)
• Simple setup — no need to track individual changes
• Off-site read-only reporting / data warehouse seeding
Limitations
• High network bandwidth — full dataset transferred every interval
• Subscriber data is stale between snapshots (not near-real-time)
• Large tables cause long apply windows → Subscriber briefly locked
• Not suitable for high-change transactional data
Replication Types — Comparison
Three SQL Server replication types side by side:
TypeHow it worksLatencyBest for
Snapshot Full copy of all articles at each intervalMinutes–hoursStatic lookups, reports
Transactional Row-level changes streamed via log in near-real-timeSub-secondOLTP HA, site autonomy
Merge Bi-directional sync; conflict resolution at merge timeOn syncMobile/disconnected
All three use the same Publisher → Distributor → Subscriber topology. Transactional replication also starts with an initial snapshot before switching to log-based streaming.
Snapshot Agent workflow:
-- 1. Snapshot Agent locks tables, generates BCP files
EXEC sp_startpublication_snapshot
    @publication = N'NorthwindPub',
    @publisher    = N'SQLSERVER01';

-- 2. Distribution Agent applies to Subscriber:
--    TRUNCATE + BULK INSERT per article
--    Runs on schedule (e.g. every 24 h)

-- 3. Monitor replication status:
SELECT publisher, publication, subscriber,
       last_sync_status, last_sync_summary
FROM   distribution..MSdistribution_status;
The Distribution Agent truncates the Subscriber's tables and bulk-inserts the new snapshot. During this window the Subscriber data is temporarily stale or unavailable.
Northwind Lab Scenario — Snapshot: USA Orders
Scenario
Replicate only American customer orders from the northwind database on PRIM into a reporting database nw_repl (also on PRIM) — refreshed overnight via snapshot replication. Three named SQL Server instances are used: PRIM (publisher + distributor), SECOND, THIRD.
Setup fix — database owner
After restoring Northwind you may hit: "Cannot execute as the database principal because the principal 'dbo' does not exist". Fix it with: 
EXEC sp_changedbowner 'sa';
ALTER AUTHORIZATION ON DATABASE::northwind TO sa;
Table-lock warning: The Snapshot Agent runs bcp which places a shared lock on the entire table for the duration of the copy. In production, schedule snapshots out of business hours to avoid blocking OLTP transactions. In the lab demo the agent is set to run every 2 minutes — only for test purposes.
