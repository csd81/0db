Migrated Stored Function — new_order() 
-- PostgreSQL stored FUNCTION (SQL Server had stored PROCEDURE)
CREATE OR REPLACE FUNCTION new_order(
    var_custid CHAR(5), var_prodid INT, var_qty SMALLINT)
RETURNS VOID LANGUAGE plpgsql AS $$
DECLARE
    var_orderid INT; var_price NUMERIC(10,4);
    var_value   NUMERIC(10,4); var_balance NUMERIC(10,2);
    var_stock   SMALLINT;
BEGIN
    SELECT unitprice, unitsinstock
    INTO var_price, var_stock
    FROM products WHERE productid = var_prodid;

    var_value := var_price * var_qty;

    SELECT balance INTO var_balance
    FROM customers WHERE customerid = var_custid;

    IF var_balance < var_value THEN
        RAISE EXCEPTION 'Insufficient balance';
    END IF;
    IF var_stock < var_qty THEN
        RAISE EXCEPTION 'Insufficient stock';
    END IF;

    SELECT COALESCE(MAX(orderid),0)+1
    INTO var_orderid FROM orders;

    UPDATE customers
      SET balance = balance - var_value
      WHERE customerid = var_custid;    -- ① balance decremented

    UPDATE products
      SET unitsinstock = unitsinstock - var_qty
      WHERE productid = var_prodid;     -- ② stock decremented

    INSERT INTO orders (orderid, customerid)
      VALUES (var_orderid, var_custid);  -- ③ FK check here!

    INSERT INTO orderdetails
      (orderid,productid,unitprice,quantity)
      VALUES (var_orderid,var_prodid,var_price,var_qty);
END; $$

    Cloud Firestore — Document Model
    Firestore = Collections of documents. Each document is a JSON record. No fixed schema — documents in the same collection may have different fields (schema-on-read). Document IDs must be unique strings (no numbers), and can be auto-generated.
    Collections used for Northwind migration:
    • products — flat, one document per product
    • customers — flat, one document per customer (includes balance)
    • orders → orderitems — hierarchical sub-collection
    Orders are NOT nested inside Customers — querying sub-collections requires iterating the parent.
    Many-to-Many Migration Problem
    Relational → NoSQL: hierarchical one-to-many maps naturally. Many-to-many (Orders ↔ Products via OrderDetails) has no automatic solution — must be designed manually.
    Relational Firestore Options
    Orders ↔ OrderDetails (FK join) ① Embed items as array inside Order document
    ② Sub-collection Orders/items
    ③ Reference fields pointing to other documents
    We chose sub-collection (Orders → Orderitems) to support time-ordered listing of orders via a single collection query.
    Data Integrity Enforcement — PostgreSQL vs Firestore
    Feature PostgreSQL Cloud Firestore
    Schema Enforced on every DML — schema-on-write Documents can have different fields — schema-on-read
    Data types Rich: NUMERIC, MONEY, DATE, ENUM, JSONB, ARRAY… Limited: no MONEY, no DECIMAL — use strings or integers for currency
    Primary keys Any data type, composite, auto-increment SERIAL String only, cannot be a number, auto-generated UUIDs
    Foreign keys Enforced by the engine at insert/update time Must be stored as reference fields; enforced only by Security Rules (bypassed by Admin SDK!)
    Constraints UNIQUE, CHECK, NOT NULL — enforced server-side Security Rules only (client-side bypass possible)
    Transactions Full ACID — serializability, rollback, savepoints Read-then-write transactions with auto-retry; batched writes for atomicity. Cannot combine both!
    Concurrency Demo — Without Transactions
    1. Set stock = 3, balance = 10000 for all 3 customers
    2. Open 3 browser tabs simultaneously
    3. Each orders 2 units of Chai (value = 2×18 = 36)
    4. Result: ALL 3 succeed — all charged 36, final stock = −3 (impossible!)
    Without transactions, the read-then-write race condition allows all three concurrent sessions to see stock=3 and proceed, resulting in overselling.
    Firestore Transaction — With Auto-Retry 
    @firestore.transactional
def update_in_transaction(transaction, cust_ref):
    snapshot = cust_ref.get(transaction=transaction)
    new_balance = snapshot.get('balance') + 1
    if new_balance <= 10:
        transaction.update(cust_ref, {'balance': new_balance})
    # If another client modified the doc between
    # this read and the write, Firestore ROLLS BACK
    # and retries this function (up to 5 times).
    Firestore transactions retry on conflict — eventually only one succeeds. However: transactions cannot be combined with batched writes. Choose: Atomicity (batch) XOR Isolation (transaction).
    Batched Writes — Atomicity Without Isolation
    What it provides:
    Atomic commit of multiple set/update/delete operations — all succeed or all fail. Works offline too.
    What it does NOT provide:
    No protection against external updates on documents already read by the client. Cannot prevent test-and-set logical errors (race conditions).
    Key constraint:
    Cannot nest batched writes inside a Firestore transaction. Maximum 500 documents per batch or transaction. Read operations must precede write operations in a transaction.
    Conclusion: NoSQL technologies delegate low-level data processing, transaction control, and integrity enforcement to the client application. They are best suited for simple business domains (e.g., blogs, user sessions) rather than complex transactional systems like order management.
