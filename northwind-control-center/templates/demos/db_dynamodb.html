Amazon DynamoDB

Managed NoSQL · Partition Key + Sort Key · GSI/LSI · Single-Table Design

DynamoDB is the spiritual successor to Amazon's 2007 Dynamo paper — the database that kept the Amazon shopping cart available during Black Friday outages when relational databases couldn't keep up. Launched as a managed service in 2012, it delivers single-digit millisecond latency at any scale with no servers to manage, no schema migrations, and no capacity planning when using on-demand mode. It powers billions of calls per day across Amazon, Duolingo, Lyft, and Snapchat.

The Access-Pattern Rule

Like Cassandra, DynamoDB requires you to know your access patterns before designing your table. There are NO joins, NO GROUP BY, NO ad-hoc queries.

Single-table design: Store all entity types in one table using composite keys and item type prefixes (PK: USER#42, SK: ORDER#20240115). Use GSIs to support alternate access patterns.

Primary Key Design 
-- Single-table design (e-commerce example)
-- PK (partition key) + SK (sort key)

-- User profile item:
{
  "PK": "USER#alice",
  "SK": "PROFILE",
  "name": "Alice Smith",
  "email": "alice@example.com",
  "created": "2024-01-15T10:30:00Z"
}

-- Order by user (PK = user, SK = order):
{
  "PK": "USER#alice",
  "SK": "ORDER#20240215-001",
  "total": 99.95,
  "status": "shipped",
  "GSI1PK": "STATUS#shipped",   -- for GSI
  "GSI1SK": "2024-02-15"
}

-- Query: all orders for alice
-- PK = "USER#alice", SK begins_with "ORDER#"
GSI / LSI for Access Patterns 
-- GSI: Global Secondary Index
-- Different PK+SK, replicated asynchronously
-- Use for: "all orders with status=shipped"

GSI1: GSI1PK = "STATUS#shipped"
      GSI1SK = order_date
→ Query: GSI1PK = "STATUS#shipped"
         AND GSI1SK BETWEEN "2024-01" AND "2024-02"

-- LSI: Local Secondary Index
-- Same PK, different SK
-- Must be defined at table creation time
-- Use for: "user orders sorted by total"
LSI1: PK = "USER#alice", SK = total (numeric)
→ Query orders by value within a user partition

-- Table creation (AWS SDK / CloudFormation)
AttributeDefinitions:
  - { AttributeName: PK, AttributeType: S }
  - { AttributeName: SK, AttributeType: S }
  - { AttributeName: GSI1PK, AttributeType: S }
KeySchema:
  - { AttributeName: PK, KeyType: HASH }
  - { AttributeName: SK, KeyType: RANGE }
Core API (Python boto3) 
import boto3
from boto3.dynamodb.conditions import Key, Attr

table = boto3.resource('dynamodb').Table('ecommerce')

# GetItem (single item by exact key, ~1ms)
item = table.get_item(
    Key={'PK': 'USER#alice', 'SK': 'PROFILE'}
)['Item']

# PutItem (upsert)
table.put_item(Item={
    'PK': 'USER#alice',
    'SK': 'ORDER#20240215-001',
    'total': Decimal('99.95'),
    'status': 'pending'
})

# Query (PK required, SK optional range)
resp = table.query(
    KeyConditionExpression=
        Key('PK').eq('USER#alice') &
        Key('SK').begins_with('ORDER#'),
    FilterExpression=Attr('status').eq('shipped'),
    Limit=20
)
Transactions & Conditional Writes 
# Conditional write: only if item doesn't exist
table.put_item(
    Item={'PK': 'USER#alice', 'SK': 'PROFILE', ...},
    ConditionExpression='attribute_not_exists(PK)'
)

# Atomic counter
table.update_item(
    Key={'PK': 'COUNTER', 'SK': 'page_views'},
    UpdateExpression='ADD #v :one',
    ExpressionAttributeNames={'#v': 'views'},
    ExpressionAttributeValues={':one': 1}
)

# Transaction (ACID across items/tables)
client.transact_write(TransactItems=[
    {'Update': {
        'TableName': 'ecommerce',
        'Key': {'PK': 'INV#SKU-9', 'SK': 'STOCK'},
        'UpdateExpression': 'SET qty = qty - :n',
        'ConditionExpression': 'qty >= :n',
        'ExpressionAttributeValues': {':n': 1}
    }},
    {'Put': {'TableName': 'ecommerce',
             'Item': {new_order_item}}}
])
Real-World Use Cases
Amazon shopping cart: The original Dynamo use case. Every item in every cart for every customer must be writable at any time from any region — even during network partitions. AP model accepts temporary divergence.
Duolingo: Stores 500M+ learner records. DynamoDB's auto-scaling handles traffic spikes from "streak at midnight" surges without pre-provisioning.
Gaming leaderboards: GSI on score column provides sorted leaderboard queries. DynamoDB Streams → Lambda for real-time push notifications on rank changes.

Partitioning & Adaptive Capacity

Consistent hashing: Partition key is hashed to determine storage node. Each partition holds up to 10GB and handles up to 3,000 RCU or 1,000 WCU. Hot partitions get split automatically.

Adaptive capacity: If one partition key is "hot" (traffic spike), DynamoDB automatically shifts capacity to that partition without you doing anything. In on-demand mode, this happens instantaneously.

DynamoDB Streams: Every write produces an ordered change record (INSERT/MODIFY/REMOVE) with old and new images. Lambda reads the stream for real-time reactions. Guaranteed exactly-once delivery within a shard.

resource "aws_dynamodb_table" "ecommerce" {
  name         = "ecommerce"
  billing_mode = "PAY_PER_REQUEST"  # on-demand
  # or: PROVISIONED with read/write_capacity

  hash_key  = "PK"
  range_key = "SK"

  attribute { name = "PK"; type = "S" }
  attribute { name = "SK"; type = "S" }
  attribute { name = "GSI1PK"; type = "S" }

  global_secondary_index {
    name            = "GSI1"
    hash_key        = "GSI1PK"
    range_key       = "SK"
    projection_type = "ALL"
  }

  stream_enabled   = true
  stream_view_type = "NEW_AND_OLD_IMAGES"

  point_in_time_recovery { enabled = true }
  ttl { enabled = true; attribute_name = "expires" }
}
