The Bandwidth Budget Problem
Fortnite: 100 players per match. Each player needs position updates from 99 others. At 30 Hz: 99 × 30 = 2,970 state updates/second per client. Each player position packet: ~20 bytes. Total: ~60 Kbps upstream. Multiply by 250K concurrent matches → 3.75 Tbps total game server uplink bandwidth.
Area of Interest (AoI)
MMOs send updates only for entities within a player's visible range. Players in a different continent of the map → no updates. Reduces per-player bandwidth by 90%. Epic Fortnite: circle shrinks → AoI naturally reduces as match progresses.
🗄️ Storage & Data Layer
DynamoDB (Player Profiles)
Single-table design: partition key = player_id, sort key = record_type. One table stores profile, friends list, match history, achievements. DynamoDB DAX (in-memory cache): sub-microsecond profile reads during matchmaking. Used by Riot for League of Legends player stats (200M+ player records).
MongoDB (Inventory / Items)
Game items have wildly varying attributes: swords have attack/speed, armor has defense/weight, potions have effect/duration. MongoDB flexible document model — no schema migration needed when new item types added. Aggregation pipeline for inventory queries. Used by multiple AAA studios for loot systems.
Redis Sorted Sets (Leaderboards)
ZADD leaderboard {score} {player_id} — O(log N) insert. ZREVRANK leaderboard {player_id} → global rank in O(log N). ZREVRANGE leaderboard 0 99 → top 100 in O(log N + K). Redis Sorted Set is the canonical leaderboard solution — handles 100M players with sub-millisecond rank queries. Used by every major mobile and PC game.
Cassandra (Match History)
Write-heavy, append-only match results. Wide-row per player: partition(player_id) → sorted(match_timestamp DESC, match_id). Riot's League of Legends processes 40M matches/day — Cassandra never rejects a write even under peak load (AP system). Each row: ~2 KB of match stats.
⚙️ Game Server Infrastructure
Agones (K8s Game Server Scaling)
Google-developed Kubernetes extension for managing game servers. Defines GameServer and Fleet CRDs. Lifecycle: Ready → Allocated → Shutdown. Agones auto-scales fleets based on match queue depth. When a match is found, Agones allocates a Ready server and returns its IP:port to matchmaker. Server shuts down after match ends, releasing pod back to pool.
Dedicated Game Servers (DGS)
Authoritative server: the "truth" of the game state. Clients send inputs (button presses, mouse positions), server validates physics, checks anti-cheat, and broadcasts resulting state. No client can hack their position — the server re-simulates everything. Runs at tick rate (e.g., 64 Hz = 64 physics steps/second). C++ for performance; Unreal Engine server mode for many AAA games.
Bare Metal at IXPs
Riot operates dedicated servers at major Internet Exchange Points (AMS-IX, DE-CIX, LINX). Co-location inside IXPs: direct peering with ISPs, bypasses public internet routing. Reduces average latency from 60ms → 30ms for European players. Latin America: servers in São Paulo, Miami, Buenos Aires for regional low-latency coverage.
Anti-Cheat (Vanguard/VAC)
Kernel-level driver (Riot Vanguard): runs at ring-0, detects injected DLLs and memory modification. Server-side: statistical anomaly detection (headshot rate, movement speed, reaction time distributions). VAC (Valve Anti-Cheat): retroactive bans — log suspicious data, ban in waves to prevent reverse engineering of detection methods.
🌐 Networking & Protocols
UDP Netcode
TCP is unusable for real-time games: head-of-line blocking (one lost packet stalls all subsequent data). UDP: no ordering, no delivery guarantee → game handles this at application layer. Position packets: if newer update received, discard older. Critical events (damage, death): application-layer ACK + retry. Custom reliability layer over UDP = best of both.
Client-Side Prediction
Client predicts outcome of local input immediately (movement applied instantly without waiting for server). Server processes input, sends authoritative state. Client reconciles: if server disagrees → correct position (snap or smooth). Without prediction: 60ms RTT feels like 120ms (action + response). With prediction: feels instant. Used by every competitive FPS.
Lag Compensation (Rollback)
When player A shoots player B: A's crosshair was on B at 100ms ago (due to A's latency). Server rewinds game state 100ms, checks if the shot was valid at that historical position. If yes → hit registered. Players with 200ms latency still get fair hit registration. GGPO: fighting game library implementing rollback netcode for frame-perfect fighting game inputs.
STUN/TURN (P2P fallback)
For peer-to-peer games (fighting games, old-school): STUN discovers public IP/port behind NAT. TURN relays traffic when NAT traversal fails (symmetric NAT). ICE tries direct → STUN → TURN in order. P2P: ~2× lower server cost but susceptible to IP exposure and host-advantage cheating. Riot moved League away from P2P to dedicated servers in 2018 specifically to eliminate host advantage.
Workflow Deep Dive
1
Matchmaking: MMR + Ping
Player enters queue. Matchmaker reads MMR (elo-like rating) from DynamoDB. Searches for 9 others (5v5) within ±100 MMR within 30 seconds. If no match in 30s: widen to ±200 MMR. If no match in 60s: ±400 MMR. Ping budget: measure latency to 3 nearest server regions → only match players sharing a server where all have <80ms ping. Riot's matchmaker: 40M matches/day = ~462 matches/second.
2
Server Allocation: Agones
Matchmaker calls Agones allocation API: POST /v1/gameserverallocations with region preference and label selectors. Agones picks a Ready GameServer pod in the target region, transitions it to Allocated, returns {"address": "34.100.200.50", "ports": [{"port": 7777}]}. Matchmaker distributes server address to all 10 clients. Fleet autoscaling: if queue depth > 100, scale fleet by 20% (warm servers pre-warmed to avoid startup latency).
3
Game Loop: 64 Hz Authority
Server runs tight loop: READ (collect input packets from all 10 clients) → SIMULATE (advance physics 15.6ms, apply all inputs, check collisions, resolve abilities) → BROADCAST (send authoritative game state snapshot to all clients). Input buffer: server holds 2-frame input buffer per client — if client's input arrives late, extrapolate previous input rather than stalling simulation. "Dropped input" rather than "stalled server".
🌐 Why Dedicated Servers?
P2P: one player hosts → host advantage (no latency for host). Cheating: host controls simulation → easy to hack. Dedicated: authoritative simulation no player can influence. Cost: ~$0.03 per player-hour on AWS. For a 40-minute match with 10 players → $0.02/match → $800K/day for League's 40M daily matches. Cheaper than the customer service cost of host-advantage complaints.
⚡ Redis: The Leaderboard
ZADD leaderboard 4521.7 "player123" inserts in O(log N). ZREVRANK leaderboard "player123" returns rank in O(log N). For 100M players: log₂(100M) ≈ 27 operations — rank query in microseconds. No SQL SELECT COUNT(*) WHERE score > ? (full table scan). Redis Sorted Set is a skip list + hash table — impossible to replicate efficiently in relational DB.
🎯 CAP Position
AP for game state, CP for inventory. Game state (positions, health): eventual consistency — a 1-frame desync is resolved by next tick broadcast. Inventory (item purchases, microtransactions): strictly CP — cannot charge a player twice for the same item or allow duplicate item creation. Two separate subsystems, two different CAP choices.
