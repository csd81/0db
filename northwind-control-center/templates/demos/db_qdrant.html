Qdrant

Vector Database · Rust-native · Filterable HNSW · Fastest benchmarks · On-premise friendly

Founded in Berlin in 2021 and written entirely in Rust, Qdrant consistently tops vector database benchmarks for query speed, precision, and memory efficiency. Its key architectural innovation is the filterable HNSW — unlike other engines that build the HNSW graph over all vectors and post-filter, Qdrant traverses the HNSW graph while respecting payload filters, avoiding the re-ranking problem entirely. This makes filtered searches dramatically more accurate. Qdrant supports named vectors (multiple vector spaces per point), sparse vectors (for BM25 hybrid search), and binary quantization (97% memory reduction). Popular in European AI startups and enterprises that prefer on-premise or self-hosted deployment.

Filterable HNSW 
Standard HNSW + post-filter approach:
Search ALL vectors → get top-K → filter
Problem: if filter removes 99%, you only
get 1% of requested results (low recall)

Qdrant's Filterable HNSW:
Payload conditions pre-indexed →
during graph traversal, skip nodes
that don't match the filter
→ Full recall even with selective filters

query(vector=[...], filter={
  "must": [
    {"key": "category", "match": {"value": "electronics"}},
    {"key": "price", "range": {"lt": 1000}}
  ]
})
→ Traverses HNSW, skips non-matching
   nodes at graph level
Collections & Points

Collection — top-level entity. Defines vector dimensions, distance metric, quantization, and on-disk storage config.

Point — a single record: ID (uint64 or UUID) + vector(s) + payload (JSON metadata).

Named Vectors — a point can have multiple named vector spaces (e.g., "title_embedding", "image_embedding", "user_embedding"). Search each independently or combine.

Sparse Vectors — token→weight maps for BM25/TF-IDF. Combined with dense vectors for true hybrid search (no pipeline needed).

Payload — arbitrary JSON metadata per point. Fields can be indexed for fast pre-filtering during ANN search. Indexed types: keyword, integer, float, bool, datetime, text (full-text).

Collection Setup 
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, SparseVectorParams,
    ScalarQuantizationConfig, ScalarType,
    PayloadSchemaType
)

client = QdrantClient(url="http://localhost:6333")

# Create collection with named vectors + quantization
client.create_collection(
    collection_name="products",
    vectors_config={
        "dense": VectorParams(
            size=1536,
            distance=Distance.COSINE,
            quantization_config=ScalarQuantizationConfig(
                type=ScalarType.INT8,  # 4x memory reduction
                quantile=0.99,
                always_ram=True
            )
        )
    },
    sparse_vectors_config={
        "sparse": SparseVectorParams()
    }
)

# Create payload indexes for fast filtering
client.create_payload_index(
    collection_name="products",
    field_name="category",
    field_schema=PayloadSchemaType.KEYWORD
)
client.create_payload_index(
    collection_name="products",
    field_name="price",
    field_schema=PayloadSchemaType.FLOAT
)
Python SDK Primer 
from qdrant_client import QdrantClient
from qdrant_client.models import (
    PointStruct, Filter, FieldCondition, MatchValue,
    Range, SearchRequest, SparseVector, NamedSparseVector,
    FusionQuery, Prefetch, Query
)
import numpy as np

client = QdrantClient(url="http://localhost:6333")

# 1. Upsert points
points = [
    PointStruct(
        id=1,
        vector={"dense": np.random.rand(1536).tolist()},
        payload={
            "product_id": 101,
            "title": "iPhone 15 Pro",
            "category": "electronics",
            "price": 999.0,
            "in_stock": True
        }
    ),
]
client.upsert(collection_name="products", points=points)

# 2. Filtered ANN search
results = client.search(
    collection_name="products",
    query_vector=("dense", np.random.rand(1536).tolist()),
    query_filter=Filter(
        must=[
            FieldCondition(key="category", match=MatchValue(value="electronics")),
            FieldCondition(key="price", range=Range(lt=1500)),
            FieldCondition(key="in_stock", match=MatchValue(value=True))
        ]
    ),
    limit=10,
    with_payload=True
)
for r in results:
    print(f"id={r.id}, score={r.score:.4f}, title={r.payload['title']}")

# 3. Hybrid search (dense + sparse with RRF fusion)
results = client.query_points(
    collection_name="products",
    prefetch=[
        Prefetch(
            query=np.random.rand(1536).tolist(),
            using="dense",
            limit=50
        ),
        Prefetch(
            query=NamedSparseVector(
                name="sparse",
                vector=SparseVector(
                    indices=[12, 45, 89],
                    values=[0.8, 1.2, 0.5]
                )
            ),
            limit=50
        )
    ],
    query=FusionQuery(fusion="rrf"),  # Reciprocal Rank Fusion
    limit=10
)

# 4. Recommendations API
similar = client.recommend(
    collection_name="products",
    positive=[1, 5, 12],   # point IDs to use as positive examples
    negative=[99],          # point IDs to use as negative examples
    limit=10,
    using="dense"
)

# 5. Scroll through all points
offset = None
while True:
    records, offset = client.scroll(
        collection_name="products",
        limit=100,
        offset=offset,
        with_payload=True
    )
    if not records:
        break
Killer Use Cases

Tripadvisor Recommendations — hotel/restaurant embeddings matched to user preference vectors with mandatory filters (city, price range, amenities). Qdrant's filterable HNSW maintains recall even with narrow filters.

RAG for Enterprise (Dust.tt) — company knowledge base embeddings. Employees search via natural language; results filtered by document access permissions. Qdrant handles permission filtering without accuracy loss.

On-Premise AI for European Enterprises — GDPR and data sovereignty requirements push many EU companies to self-host. Qdrant's single binary deployment (no JVM, no Python) makes it operationally simple.

Multi-Modal Search — named vectors enable searching across modalities: image embedding + text embedding per product, searching with either image or text query.

Storage Engine

Segment-based — data is stored in immutable segments. New data goes into a growing mutable segment; when full it's converted to an immutable segment with an HNSW index built on it.

Memmap — large segments are memory-mapped from disk. OS manages which pages stay in RAM (hot data) vs disk (cold data). This enables working sets larger than RAM without manual management.

On-disk HNSW — optionally store the HNSW graph on disk (SSD) rather than RAM. Slightly slower but dramatically reduces memory requirements for large collections.

Write-Ahead Log — all mutations go to a WAL before application. Enables crash recovery and is the source for replication in distributed mode.

Distributed Mode

Sharding — collections are split into shards distributed across nodes. Shards are balanced automatically. Custom sharding key enables co-location of related points.

Raft Consensus — shard replicas use Raft for leader election and log replication. No external coordination service needed (unlike Milvus which needs etcd).

Resharding (2.x+) — live resharding: increase shard count without downtime, migrating data in the background while serving queries.

REST + gRPC — primary API is REST (human-readable, easy testing). gRPC available for high-throughput ingestion with ~30% lower overhead than REST.
