Amazon Redshift

MPP columnar warehouse · Redshift Spectrum · RA3 compute-storage separation

Amazon Redshift launched in 2013 as the first cloud data warehouse at scale (SIGMOD 2022 paper), bringing petabyte-scale analytics to companies that couldn't afford Oracle Exadata. Its MPP (Massively Parallel Processing) architecture distributes data across compute nodes (slices), each processing its partition in parallel. RA3 nodes (2019) decoupled compute from storage via managed S3-backed storage, enabling independent scaling. Redshift Spectrum queries S3 data directly without loading it. AQUA (Advanced Query Accelerator) pushes compute to the storage layer.

MPP Columnar at Cloud Scale

Redshift derives from PostgreSQL but shares almost no code at the storage/execution level. Data is stored column-by-column with per-column compression (up to 80% reduction). Queries are compiled to machine code at planning time (not interpreted), then executed in parallel across all compute node slices.

Distribution keys control which rows go to which node — choosing the right distribution key eliminates cross-node data movement ("redistribution") for JOIN operations. Sort keys enable zone map pruning: Redshift tracks min/max per 1MB block and skips blocks that can't contain rows matching the WHERE predicate.

Distribution Styles & Sort Keys 
-- Distribution styles control data placement:

-- KEY: rows with same key go to same slice → JOINs are local
CREATE TABLE orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  DATE,
    total       DECIMAL(10,2)
)
DISTSTYLE KEY
DISTKEY (customer_id)         -- hash to node by customer_id
SORTKEY (order_date);         -- rows sorted on disk by date (zone maps)

-- EVEN: round-robin (uniform distribution, bad for JOINs)
CREATE TABLE staging_raw (...)
DISTSTYLE EVEN;

-- ALL: copy full table to every node (small dimension tables)
CREATE TABLE countries (country_code CHAR(2), name TEXT)
DISTSTYLE ALL;

-- AUTO: Redshift decides based on table size
CREATE TABLE products (...) DISTSTYLE AUTO;

-- Compound sort key (range scan on multiple columns)
CREATE TABLE events (
    event_time  TIMESTAMP,
    event_type  VARCHAR(50),
    user_id     BIGINT
)
COMPOUND SORTKEY (event_time, event_type);

-- Interleaved sort key (equal weight to each column)
CREATE TABLE fact_sales (...)
INTERLEAVED SORTKEY (sale_date, product_id, region);
Columnar Compression Encodings 
-- Column-level compression encodings
CREATE TABLE page_views (
    view_id    BIGINT     ENCODE ZSTD,
    url        VARCHAR(2048) ENCODE LZO,     -- text → LZO
    status     SMALLINT   ENCODE BYTEDICT,   -- few distinct values → dict
    user_id    BIGINT     ENCODE DELTA32K,   -- sorted sequential IDs
    view_time  TIMESTAMP  ENCODE AZ64,       -- numeric → AZ64 (best for numbers)
    referer    VARCHAR(512) ENCODE ZSTD      -- general → ZSTD
);

-- ANALYZE COMPRESSION: let Redshift recommend encodings
ANALYZE COMPRESSION page_views;
-- Result: Redshift suggests optimal encoding per column

-- Materialized views for pre-aggregated analytics
CREATE MATERIALIZED VIEW daily_revenue AS
SELECT
    DATE_TRUNC('day', order_date) AS day,
    product_category,
    SUM(total)      AS revenue,
    COUNT(*)        AS order_count
FROM orders
JOIN products USING (product_id)
GROUP BY 1, 2;

-- Auto-refresh when base tables change
ALTER MATERIALIZED VIEW daily_revenue
AUTO REFRESH YES;
Redshift SQL — Data Warehouse Patterns 
-- ── 1. COPY from S3 (bulk load — orders of magnitude faster than INSERT)
COPY orders
FROM 's3://my-bucket/orders/2024/'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftS3'
FORMAT PARQUET;  -- or CSV, JSON, ORC, AVRO

-- ── 2. UNLOAD to S3 (export results)
UNLOAD ('SELECT * FROM orders WHERE order_date > ''2024-01-01''')
TO 's3://my-bucket/exports/orders/'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftS3'
FORMAT PARQUET PARALLEL ON;

-- ── 3. Redshift Spectrum: query S3 without loading
CREATE EXTERNAL SCHEMA spectrum_schema
FROM DATA CATALOG
DATABASE 'mydb'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftSpectrum'
CREATE EXTERNAL DATABASE IF NOT EXISTS;

CREATE EXTERNAL TABLE spectrum_schema.historical_orders (
    order_id BIGINT, customer_id BIGINT, total FLOAT
)
STORED AS PARQUET
LOCATION 's3://my-bucket/historical/';

-- JOIN Redshift table with S3 data seamlessly
SELECT r.customer_id, COUNT(*) AS recent_orders, SUM(h.total) AS historical_spend
FROM orders r
JOIN spectrum_schema.historical_orders h ON r.customer_id = h.customer_id
WHERE r.order_date > CURRENT_DATE - 30
GROUP BY r.customer_id;

-- ── 4. Window functions for cohort analysis
SELECT
    DATE_TRUNC('month', first_purchase_date)  AS cohort_month,
    DATE_TRUNC('month', order_date)           AS order_month,
    COUNT(DISTINCT customer_id)               AS active_customers,
    ROUND(COUNT(DISTINCT customer_id) * 100.0
        / FIRST_VALUE(COUNT(DISTINCT customer_id)) OVER (
              PARTITION BY DATE_TRUNC('month', first_purchase_date)
              ORDER BY DATE_TRUNC('month', order_date)
          ), 2)                               AS retention_rate
FROM customer_orders
GROUP BY 1, 2
ORDER BY 1, 2;

-- ── 5. EXPLAIN to see distribution and broadcast operations
EXPLAIN
SELECT c.name, SUM(o.total)
FROM customers c JOIN orders o ON c.id = o.customer_id
GROUP BY c.name;

Killer Use Cases
Enterprise BI & Reporting

Nasdaq runs market data analytics on Redshift. McDonald's enterprise BI. Heavily used as the destination for AWS Glue / Kinesis Firehose ETL pipelines feeding into Tableau, QuickSight, or Looker.

Data Lake Analytics

Redshift Spectrum enables query-in-place on S3 data lake. Hot data in Redshift cluster; cold historical data in S3. Single SQL query spans both. Lyft uses this pattern for event analytics.

Real-Time Streaming Analytics

Redshift Streaming Ingestion: directly ingest from Kinesis Data Streams or MSK (Kafka) into materialized views. Sub-second latency from stream to queryable table.

RA3 Architecture: Compute-Storage Separation 
┌────────────────────────────────────────────────────────────────────┐
│  Leader Node (single)                                              │
│  ├── Query parsing and planning                                    │
│  ├── Query compilation to C++ (LLVM)                               │
│  ├── WLM (Workload Management) — assign queries to queues          │
│  └── Aggregates partial results from compute nodes                 │
├────────────────────────────────────────────────────────────────────┤
│  Compute Nodes (2–128 RA3 nodes, 2–32 slices each)                 │
│                                                                    │
│  RA3.xlplus  — 4 vCPU,  32 GB RAM,  64 TB managed storage          │
│  RA3.4xlarge — 12 vCPU, 96 GB RAM,  128 TB managed storage         │
│  RA3.16xlarge— 48 vCPU, 384 GB RAM, 128 TB managed storage         │
│                                                                    │
│  Each node: NVMe SSD cache (hot data) + S3-backed managed storage  │
│  AQUA: custom hardware (FPGAs) in storage layer run predicates     │
│  before data hits compute nodes                                    │
├────────────────────────────────────────────────────────────────────┤
│  Managed Storage (S3-backed, per-node namespace)                   │
│  ├── Columnar blocks (1 MB each)                                   │
│  ├── Zone maps (min/max per block for pruning)                     │
│  ├── Per-column compression                                        │
│  └── Automatic tiering: SSD cache → S3 (LRU)                       │
└────────────────────────────────────────────────────────────────────┘

WLM Queues example:
  Queue 1: "ETL" — 2 concurrent queries, 60% memory
  Queue 2: "BI Reports" — 10 concurrent, 30% memory
  Queue 3: "Default" — 5 concurrent, 10% memory
  Concurrency scaling: overflow queries → auto-add compute nodes
