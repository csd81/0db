CREATE PROCEDURE
The script is compiled into an execution plan and stored as a database object. The code does not run at creation time — only when called via EXECUTE proc_name [args]. Can contain any T-SQL: variables, loops, cursors, TRY/CATCH, transactions, calls to other procedures.
OUTPUT parameters
Declared with the OUTPUT keyword; passed by reference — the caller reads the value after execution. EXEC sp_name @in_param, @out_var OUTPUT. Useful for returning a status code, a generated ID, or a computed result alongside the main operation.
Return value (RETURN n)
Every procedure implicitly returns an int. RETURN 0 means success by convention; negative values are set automatically on error. Captured with EXEC @result = proc_name. Distinct from OUTPUT params — use for a single success/failure code, not for data.
Result sets from SELECT
Any SELECT whose results are not stored into variables is sent as a result set to the caller — like a table returned from the procedure. A single procedure can return multiple result sets. SQL Server specific: other DBMS require a cursor OUTPUT parameter for this.
Stored Procedure Anatomy 
CREATE PROCEDURE sp_inc_salary
    @name    nvarchar(20),          -- input parameter
    @result  varchar(50) OUTPUT    -- output parameter
AS
BEGIN TRY
    SET NOCOUNT ON
    DECLARE @cnt int, @emp_id int

    SELECT @cnt = COUNT(*) FROM Employees
    WHERE  LastName LIKE @name

    IF @cnt = 0      SET @result = 'NOT_FOUND'
    ELSE IF @cnt > 1 SET @result = 'AMBIGUOUS'
    ELSE BEGIN
        SELECT @emp_id = EmployeeID FROM Employees
        WHERE  LastName LIKE @name
        UPDATE Employees SET salary = 1.1*salary
        WHERE  EmployeeID = @emp_id
        SET @result = 'OK'
    END
END TRY
BEGIN CATCH
    SET @result = 'ERROR:' + ERROR_MESSAGE()
END CATCH
GO

-- Call with OUTPUT parameter:
DECLARE @r varchar(50)
EXEC sp_inc_salary 'Fuller%', @r OUTPUT
PRINT @r   -- 'OK' / 'NOT_FOUND' / 'AMBIGUOUS'
SET NOCOUNT ON suppresses the "n rows affected" message — important for procedures that return result sets, so the client doesn't confuse the count with a result set. Verify the new procedure under Programmability → Stored Procedures in SSMS.
Nested Procedure Pattern — Order Entry
The order-entry transaction is split into two cooperating stored procedures. The outer procedure owns the transaction; the inner handles stock update and signals errors via OUTPUT.
uj_rendeles (outer)
BEGIN TRAN / BEGIN TRY
① check product count → ambiguous?
② check customer balance → insufficient?
③ UPDATE Customers SET balance -= total
④ INSERT Orders → @@IDENTITY → @order_id
⑤ INSERT [Order Details]
⑥ EXEC sp_raktárkészlet_aktualizalo
@order_id, @result OUTPUT
⑦ IF @result='OK' → COMMIT ELSE ROLLBACK
END TRY / BEGIN CATCH → ROLLBACK
▶
sp_raktárkészlet_aktualizalo (inner)
@orderid IN, @result OUT
BEGIN TRY
UPDATE Products
SET UnitsInStock -= od.Quantity
FROM [Order Details] od
WHERE od.OrderID = @orderid
SET @result = 'OK'
END TRY / BEGIN CATCH
SET @result = 'HIBA'
END CATCH
Why split? The stock-update logic is reusable (e.g. for cancellations, adjustments) without coupling it to the order-entry transaction. The outer procedure owns the transaction boundary — it decides whether to COMMIT or ROLLBACK based on the inner result. If the inner proc raises an error (CHECK constraint on UnitsInStock ≥ 0), the outer CATCH rolls back everything atomically.
Return value codes:
EXEC @r = proc_name — captures the integer return code.
0 = success (default), negative = system error, positive = user-defined error code.
Example: EXEC @r = p where p does SELECT 1/0 → @r = −6.

Scalar User-Defined Function
Restrictions vs stored procedures: No UPDATE / INSERT / DELETE; no non-deterministic functions (RAND, GETDATE); no calls to user stored procedures; no TRY/CATCH or RAISERROR.
Must prefix with schema on call: dbo.fn_name(…) 
CREATE FUNCTION fn_ev_honap (@datum datetime)
RETURNS varchar(50)
AS BEGIN
    DECLARE @r varchar(50)
    SET @r = CAST(YEAR(@datum) AS varchar)
            + CASE WHEN MONTH(@datum) < 10
                  THEN '_0' + CAST(MONTH(@datum) AS varchar)
                  ELSE '_'  + CAST(MONTH(@datum) AS varchar)
             END
    RETURN @r
END
GO
-- Used directly in GROUP BY / SELECT:
SELECT lastname,
       dbo.fn_ev_honap(OrderDate) AS honap,
       COUNT(OrderID) AS rend_szam
FROM Employees LEFT JOIN Orders USING(EmployeeID)
GROUP BY lastname, dbo.fn_ev_honap(OrderDate)
ORDER BY lastname, honap
A scalar UDF can appear anywhere a scalar expression is valid: SELECT list, WHERE, GROUP BY, ORDER BY, default column value.
Table-Valued Function (TVF)
Return type is TABLE. In the inline form (single SELECT) the table structure is inferred — no need to declare columns explicitly. Use the TVF in FROM like a table or view. 
-- Inline TVF: returns products and order count for a category
CREATE FUNCTION kateg_termekei (@kat_id int)
RETURNS TABLE AS RETURN
(
    SELECT p.ProductName  AS termeknev,
           COUNT(*)        AS rendelesek_szama
    FROM   Products p
    JOIN   [Order Details] od ON p.ProductID = od.ProductID
    WHERE  p.CategoryID = @kat_id
    GROUP BY p.ProductID, p.ProductName
)
GO
-- Used in FROM — can filter on returned columns:
SELECT * FROM dbo.kateg_termekei(1)
WHERE  rendelesek_szama > 30

What is a trigger?
A special stored procedure attached to a table, fired automatically by the engine when a specified DML event occurs (INSERT, UPDATE, DELETE). No input parameters and no return value. Errors inside the trigger roll back the triggering statement. Makes a passive database active — the DB reacts to changes without client involvement.
inserted / deleted virtual tables
SQL Server exposes two read-only pseudo-tables inside every DML trigger:
inserted — new row versions (for INSERT and UPDATE)
deleted — old row versions (for DELETE and UPDATE)
A single UPDATE touching 1 000 rows puts all 1 000 new rows in inserted and all 1 000 old rows in deleted — always write set-based logic, not row-by-row.
AFTER vs INSTEAD OF
AFTER INSERT|UPDATE|DELETE — trigger fires after the statement succeeds (default).
INSTEAD OF — trigger fires in place of the statement; used on views to make non-updatable views writable.
Multiple triggers of the same type can exist on one table; execution order is determined by system rules. Triggers can cascade to other triggers or recurse into themselves.
Typical uses
• Cross-table integrity — rules that span multiple columns or tables (beyond a single CHECK constraint)
• Business automation — e.g. update salary on every new order insert
• Audit logging — copy old/new row to a log table on any modification
• Derived data — maintain a running total or denormalised cache column
AFTER INSERT — Salary Increment
Every new order row inserted into Orders raises the employee's salary 2 %. Works correctly even when a single INSERT adds multiple rows at once. 
CREATE TRIGGER tr_uj_rend
ON Orders AFTER INSERT
AS
BEGIN
    DECLARE @i int
    SELECT @i = COUNT(*) FROM inserted
    PRINT 'Rows inserted: ' + CAST(@i AS varchar)

    UPDATE Employees
    SET    Salary = Salary * 1.02
    WHERE  EmployeeID IN (
        SELECT EmployeeID FROM inserted
    )
END
GO
-- Test:
-- SELECT Salary FROM Employees WHERE EmployeeID=3  → 100
-- INSERT Orders (EmployeeID) VALUES (3)
-- SELECT Salary FROM Employees WHERE EmployeeID=3  → 102
PRINT inside a trigger is useful for debugging but is invisible to most application clients. Never write row-by-row cursor logic in a trigger — it breaks on multi-row DML statements.
AFTER UPDATE — Stock Adjustment (inserted + deleted)
When an order detail quantity changes, adjust UnitsInStock by the delta (new qty − old qty). Uses both inserted and deleted in a three-way join. 
CREATE TRIGGER tr_stock_update
ON [Order Details] AFTER UPDATE
AS
BEGIN TRY
    PRINT 'Updating ' + CAST(COUNT(*) FROM inserted) + ' rows'

    UPDATE Products
    SET    UnitsInStock = UnitsInStock - (i.Quantity - d.Quantity)
    FROM   Products p
    JOIN   inserted i ON p.ProductID = i.ProductID
    JOIN   deleted  d ON i.ProductID = d.ProductID
END TRY
BEGIN CATCH
    PRINT 'Error: quantity too large.'
    -- triggering UPDATE auto-rolls back
END CATCH
GO

CHECK constraint fires → CATCH → UPDATE rolled back
When an error occurs inside a trigger, the triggering statement is automatically rolled back — you do not need an explicit ROLLBACK inside the trigger's CATCH block (and issuing one would roll back the entire surrounding transaction).
