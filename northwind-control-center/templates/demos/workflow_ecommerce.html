Saga Pattern — Distributed Checkout Transaction Each step has a compensating rollback action
Step 1 → Reserve InventoryUPDATE stock SET reserved=reserved+qty WHERE product_id=? AND (stock-reserved) >= qty
↩ Compensate: UPDATE stock SET reserved=reserved-qty
Step 2 → Create Order RecordINSERT INTO orders (user_id, total, status='PENDING') → returns order_id
↩ Compensate: UPDATE orders SET status='CANCELLED'
Step 3 → Charge PaymentStripe API: create_charge(amount, card_token, idempotency_key=order_id)
↩ Compensate: Stripe API: create_refund(charge_id)
Step 4 → Confirm OrderUPDATE orders SET status='CONFIRMED'; UPDATE stock SET stock=stock-qty, reserved=reserved-qty
✓ Success — emit OrderConfirmed event to Kafka
Step 5 → Notify FulfillmentKafka message consumed by Warehouse, Shipping, Email services independently
✓ Each service handles its own failure/retry independently
🗄️ Storage & Data Layer
Amazon Aurora / CockroachDB
ACID relational DB for orders, inventory, customers. Aurora: MySQL-compatible, 6-way replication across 3 AZs. CockroachDB: distributed SQL, geo-partitioned tables — Paris orders stay in EU region for GDPR compliance. SELECT FOR UPDATE prevents overselling.
Amazon DynamoDB / Redis
Shopping cart state. DynamoDB: TTL on items (cart expires after 24h). Redis: sub-millisecond SET/GET for cart data with session TTL. Eventual consistency acceptable — the cart is non-critical until checkout.
Elasticsearch / Apache Solr
Product catalog search. 300M+ product listings at Amazon. Full-text search, faceted filtering (brand, price_range, rating). Autocomplete via edge n-gram tokenizer. Fuzzy matching ("running shose" finds "running shoes"). Separate from relational DB — denormalized.
MongoDB (Product Catalog)
Flexible schema for product attributes. Electronics: {"voltage": "110V", "wattage": 1200}. Clothing: {"sizes": ["S","M","L"], "material": "cotton"}. No fixed schema means new product categories don't require DB migrations.
⚙️ Compute & Processing
AWS SQS / RabbitMQ
Order fulfillment queue. FIFO queues for strict ordering. Dead letter queue (DLQ) captures failed messages for manual review. Message visibility timeout: 30s (prevents duplicate processing). Decouples checkout from fulfillment — payment service can succeed even if warehouse is down.
Apache Airflow
Batch orchestration. Daily: Recalculate seller rankings, sync inventory counts from warehouses, process refund batches, generate financial reconciliation reports. DAG-based workflow with retry/alerting. Shopify uses Airflow for merchant payout calculations.
AWS Lambda (Event-driven)
Serverless microservices for stateless operations: send order confirmation email, resize product image, calculate estimated delivery date. Pay per invocation — no idle servers. Cold start penalty (~100ms) mitigated by provisioned concurrency for critical paths.
Recommendation Engine (A9/BM25)
Amazon's A9 search algorithm combines BM25 relevance with purchase signal (conversion rate, revenue per click). "Also bought" recommendations from collaborative filtering. Personalized ranking re-orders results per user based on purchase history.
🌐 Infrastructure & Resilience
Microservice Architecture (SOA)
Amazon: 1000s of microservices. Each team owns their service end-to-end ("you build it, you run it"). Services communicate via REST or gRPC. Service mesh (Envoy/Istio) handles mTLS, circuit breaking, retries. Internal APIs: "Amazon Marketplace" was built to sell Amazon's own infrastructure as AWS.
Circuit Breaker Pattern
Resilience4j / Hystrix. If Product Recommendation service fails: open circuit, return cached recommendations. Prevents cascading failure from one slow microservice bringing down checkout. States: Closed (normal) → Open (fail fast) → Half-Open (probe recovery).
Idempotency Keys
Every payment request includes an idempotency key (UUID). Stripe deduplicates: same key within 24h returns the same result without charging twice. Critical for retry safety — network timeout doesn't mean the charge failed.
Shopify Sagas / 2PC alternatives
Shopify uses the Saga pattern (choreography-based) — no distributed 2PC which would require all services to be available simultaneously. Each step publishes events, compensating transactions handle failures. ~99.99% checkout success rate even with partial service degradation.
⚡ Why No 2-Phase Commit?
2PC requires all participants to be available simultaneously. In a microservices world with 1000s of services, one unavailable service would block every transaction. The Saga pattern (compensating transactions) sacrifices some atomicity for availability — acceptable for business logic, not for money movement.
🛡️ Inventory Overselling
Amazon's "1 left in stock!" — maintained by atomic SQL: UPDATE inventory SET stock=stock-1 WHERE id=? AND stock>0. Affected rows = 0 → out of stock. For high-demand items (Black Friday), optimistic locking with version columns prevents race conditions across concurrent checkouts.
🎯 CAP Position
CP for orders, AP for catalog. Order/payment must be CP — you can't have two concurrent transactions both think they got the last item. Product catalog and recommendations can be AP — a slightly stale stock count is acceptable. Two different databases, two different CAP choices in the same system.
