⚖️ CAP Theorem

Brewer's CAP Theorem proves that any distributed data store can guarantee at most two of three properties simultaneously: Consistency (every read returns the most recent write), Availability (every non-failing node always responds), and Partition Tolerance (the system operates correctly during network splits). Since network partitions cannot be prevented in practice, the real engineering choice is always CP versus AP — sacrifice availability to preserve consistency, or serve potentially stale data to remain available. The demo maps real databases — Cassandra (AP), HBase (CP), Zookeeper (CP), CouchDB (AP) — onto the triangle and simulates what each design choice means for a shopping cart during a datacenter split. PACELC extends CAP to also model the latency-consistency trade-off that exists even when there is no active partition.

Brewer's CAP Theorem (2000): A distributed data store can guarantee at most two of the following three properties simultaneously. Since network partitions cannot be avoided in practice, the real design choice is CP (Consistency + Partition Tolerance) vs AP (Availability + Partition Tolerance).
🔒
C — Consistency
Every read receives the most recent write or an error. All nodes in the distributed system see the same data at the same time. No stale reads are ever served — if a write hasn't propagated yet, the system either waits or rejects the read.

SQL analogy: SERIALIZABLE isolation applied globally across all nodes.
✅
A — Availability
Every request receives a response — not an error — even if the data might be stale. The system remains operational and responsive at all times, prioritising uptime over data freshness.

Example: TikTok feed always loads even if "Like" count is 30 seconds behind.
🌐
P — Partition Tolerance
The system continues to operate even when network messages between nodes are delayed or dropped (a network partition). In any real multi-node deployment partitions will happen — hardware fails, cables are cut, datacentres become isolated.

P is non-negotiable for distributed systems. The choice is really C vs A.
The CAP Triangle — Pick Two
Why CA is not a real option: CA means sacrificing Partition Tolerance. In a single-server database (no network) this is fine — Postgres or SQL Server on one machine is perfectly CA. But once you have two or more machines communicating over a network, you will eventually have a partition. Therefore, all distributed databases must choose CP or AP.
CP systems — when partition occurs:
→ Refuse writes (return error)
→ Refuse stale reads
→ Wait for partition to heal
Sacrifices: Availability
Examples: HBase, ZooKeeper, etcd, Google Spanner, SQL Server Always On (sync)
AP systems — when partition occurs:
→ Serve stale data
→ Accept writes on both sides
→ Reconcile later (eventual consistency)
Sacrifices: Consistency
Examples: Cassandra, DynamoDB, CouchDB, Riak, Facebook TAO

Bank account — why CP is mandatory:
-- AP system: network partition between NY and London
NY node:  account.balance = $100
  User A withdraws $100 → NY says OK  ✓
London:   also reads $100 (stale!)
  User B withdraws $100 → London says OK  ✓
→ Bank lost $100 — double spend problem

-- CP system: quorum required before committing
NY node:  User A withdraws $100
  → must confirm with London before OK
London partition: unreachable
  → NY returns: "Service temporarily unavailable"
→ No data loss — consistency preserved

Eventual Consistency
AP systems guarantee that if no new writes occur, all replicas will eventually converge to the same value. There is no guarantee on how long this takes. Common in social media (feeds, like counts), DNS propagation, and CDN cache invalidation. Strategies: Last-Write-Wins (timestamp), CRDT (conflict-free replicated data types), read-repair, anti-entropy gossip.
PACELC Extension (2012)
CAP only addresses the partition case. PACELC extends it: even when the system is running normally (no partition, Else), there is a trade-off between Latency and Consistency. A strongly consistent system must wait for replication acknowledgment (higher latency). An eventually consistent system can return immediately (lower latency). Formula: PA/EL (e.g. Cassandra) vs PC/EC (e.g. Spanner).
Two Generals Problem
Formal proof that it is impossible to achieve 100% guaranteed message delivery across an unreliable network. This is why "Exactly-Once" delivery in distributed systems (Kafka, message queues) requires idempotency keys and write-ahead logging — you can get arbitrarily close to exactly-once, but never provably guarantee it without additional assumptions about the network.
CAP in SQL Server
SQL Server Always On Availability Groups:
• Synchronous commit = CP (waits for secondary ACK → zero data loss, higher latency)
• Asynchronous commit = AP-like (returns immediately → possible data loss on failover, lower latency)
Merge Replication = AP (bi-directional; conflicts resolved with user-defined rules after the fact).
RTT & Quorum — The Physics of Consistency
Why strong consistency costs latency:
-- CP write requires quorum ACK:
Client → Primary (London)
Primary → Replica A (NY)    RTT ≈ 80ms
Primary → Replica B (Tokyo) RTT ≈ 230ms
Client waits for majority ACK before commit
→ Write latency ≥ 80ms (fastest quorum member)

-- AP write returns immediately:
Client → Primary (London)   RTT ≈ 1ms
Primary: "OK" immediately
Async replication to NY, Tokyo later
→ Write latency ≈ 1ms (local ack only)

-- Speed of light: London → NY via fibre
-- ~5,570 km / (200,000 km/s) ≈ 28ms one-way
-- RTT ≈ 70–80ms (routing overhead included)
Quorum voting — how CP systems decide:
A Raft/Paxos cluster of 5 nodes requires 3 (majority) to agree before committing a write. If a network partition splits the cluster into a group of 3 and a group of 2:
Majority partition (3 nodes)
Can still reach quorum → continues to accept writes → stays Consistent
Minority partition (2 nodes)
Cannot reach quorum → becomes read-only → sacrifices Availability
This is why ZooKeeper, etcd, and Raft-based systems always have an odd number of nodes (3, 5, 7). An even split (2/2, 3/3) would leave no majority — both sides become unavailable.
