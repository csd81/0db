# 14. fejezet — Elosztott konszenzus: PoW vs Raft

## Tanulási célok

A fejezet végére képes leszel:

- Megmagyarázni, miért van szükség konszenzusra az elosztott adatbázisokban, és mi mehet félre nélküle
- Ismertetni, hogyan ér el megállapodást a Proof-of-Work (PoW), és miért alkalmatlan adatbázisokhoz
- Leírni a Raft algoritmust: vezető-választás, naplóreplikáció, commit
- Helyi 3 csomópontos rqlite clustert elindítani és SQL lekérdezéseket kiadni rajta
- Elvégezni a **Chaos Monkey** kísérletet: megölni a vezetőt és megfigyelni az új választást
- Összehasonlítani a PoW-t és a Raft-et egy táblázatban

---

## 14.1 Miért fontos a konszenzus?

Egyetlen csomópontos adatbázis egyszerű: egy író, egy igazság. Amint hozzáadsz egy második csomópontot a hibatűrés érdekében, szembesülsz az **elosztott konszenzus problémájával**: minden csomópontnak azonos sorrendű írásokkal kell egyetértenie, még akkor is, ha üzenetek elvesznek, késnek, vagy csomópontok összeomlanak.

Konszenzus nélkül:

| A csomópont ír | B csomópont olvas | Eredmény |
|---|---|---|
| INSERT order 999 | SELECT * FROM orders | 0 vagy 1 sort adhat vissza az időzítéstől függően |
| Két csomópont ütköző frissítéseket fogad el | — | *Split brain* — két különböző „igazság" |

Minden elosztott adatbázis (PostgreSQL streaming replikáció, MySQL Group Replication, CockroachDB, MongoDB replica set-ek, rqlite) más módon oldja meg a konszenzust.

---

## 14.2 Proof-of-Work konszenzus (visszatekintés a 4. fejezetből)

A PoW-t **engedély nélküli** hálózatokhoz találták ki, ahol a résztvevők anonimak és potenciálisan rosszindulatúak (Bitcoin). A mechanizmus:

1. Az adatot hozzáadni kívánó csomópontnak meg kell oldania egy **kriptográfiai rejtvényt** (olyan `nonce`-t kell találni, amelyre `SHA256(payload + nonce)` N nullával kezdődik).
2. Az első megoldó közvetíti a megoldást.
3. A többi csomópont ellenőrzi (olcsó) és elfogadja a blokkot.

**Miért alkalmatlan a PoW adatbázisokhoz:**

| Tulajdonság | PoW | Adatbázis-követelmény |
|---|---|---|
| Áteresztőképesség | ~7 TPS (Bitcoin) | Ezres nagyságrendű TPS |
| Írási késleltetés | Percek | Ezredmásodpercek |
| Energiafogyasztás | Óriási | Elhanyagolható |
| Hibatűrési modell | Bizánci (rosszindulatú csomópontok is OK) | Összeomlás-hibatűrő (csomópontok meghalnak, nem hazudnak) |
| Tagság | Nyílt / engedély nélküli | Zárt / ismert résztvevők |

A 4. fejezetbeli szimuláció vizuálisan mutatta a bányászati fázist — minden „VM" versenyez a nonce megtalálásáért. Ez a késleltetés a Bitcoin biztonságának *jellemzője*; az OLTP-hez viszont katasztrofális.

## 14.3 A Raft algoritmus

A Raft (Ongaro & Ousterhout, 2014) célja az érthetőség volt. Ismert résztvevők zárt clusterein belül oldja meg a konszenzust, összeomlás-hibatűrő modellben.

### 14.3.1 Szerepek

Minden pillanatban minden csomópont az alábbiak egyike:

- **Vezető (Leader)** — fogadja az összes kliens írást, replikálja a követőkhöz
- **Követő (Follower)** — passzív; a vezető naplóbejegyzéseit alkalmazza
- **Jelölt (Candidate)** — átmeneti állapot választás során

Egyszerre pontosan egy vezető létezik (vagy nulla, választás alatt).

### 14.3.2 Vezető-választás

Minden követőnek véletlenszerű **választási időkorlátja** van (150–300 ms). Ha ennyi ideig nem hall a vezető felől, **Jelölt** lesz, növeli a **cikluszát (term)**, és `RequestVote` RPC-ket küld.

Egy csomópont **igen**-nel szavaz, ha:
- ebben a ciklusban még nem szavazott, ÉS
- a jelölt naplója legalább olyan naprakész, mint a sajátja

Többségi szavazatot kapó jelölt lesz az új vezető erre a ciklusra.

**Miért véletlenszerű az időkorlát?** Véletlenszerűség nélkül minden követő egyszerre indítana választást, örökké osztva a szavazatokat. A véletlenszerűség biztosítja, hogy általában egy csomópont indul el elsőként.

### 14.3.3 Naplóreplikáció

```
Kliens         Vezető          Követő 2        Követő 3
  |  INSERT ... |               |               |
  |------------>|               |               |
  |             |--AppendEntry->|               |
  |             |--AppendEntry----------------->|
  |             |<---ACK--------|               |
  |             |<---ACK--------------------------
  |             | (többség)                     |
  |             | COMMIT helyben                |
  |<--OK--------|               |               |
  |             |--Értesít----->|               |
  |             |--Értesít---------------------->
```

Egy írás **commitált**, amint a csomópontok többsége a naplójába írta — még akkor is, ha néhány követő lassú vagy átmenetileg lecsatlakozott.

### 14.3.4 Biztonsági garancia

A Raft garantálja: ha egy naplóbejegyzés commitált, *minden jövőbeli vezető is rendelkezni fog azzal a bejegyzéssel*. Ez megakadályozza az „elfogadott írás visszavonása" típusú hibát.

## 14.4 rqlite — Raft-konszenzusos SQLite

Az **rqlite** egy nyílt forráskódú elosztott relációs adatbázis, amelyet Philip O'Toole épített. SQLite-ot burkol Raft konszenzusréteggel, egyszerű HTTP+JSON API-t nyújtva.

Architektúra:

```
┌─────────────────────────────────────────┐
│              HTTP API                   │  ← Flask alkalmazás (requests könyvtár)
│   GET /db/query    POST /db/execute     │
├─────────────────────────────────────────┤
│              SQLite motor               │  ← csomópontonkénti beágyazott adatbázis
├─────────────────────────────────────────┤
│              Raft napló                 │  ← replikált előre-írási napló
└─────────────────────────────────────────┘
         1. csomópont    2. csomópont    3. csomópont
         :4001           :4003           :4005
```

Főbb tulajdonságok:

- **Nincs szükség driverre** — tiszta HTTP; `curl`-ből, Python `requests`-ből, bármely nyelvből működik
- **Erős konzisztencia** alapértelmezés szerint — olvasások a vezető felé mennek
- **Gyenge olvasás** opció — `?level=weak` bármely csomópontnak megengedi a válaszadást (esetleg kicsit elavult adat)
- **SQLite SQL** — teljes SELECT, INSERT, UPDATE, DELETE, PRAGMA támogatás
- **Automatikus átvétel** — ha a vezető meghal, a követők ~1 mp alatt választanak újat

## 14.5 A helyi cluster elindítása

```bash
# A northwind-control-center könyvtárból:
bash start_rqlite_cluster.sh
```

A szkript:
1. Leállítja az esetleg futó `rqlited` folyamatokat
2. Elindítja az **1. csomópontot** a 4001/4002 portokon (HTTP/Raft)
3. Megvárja, amíg az 1. csomópont készen áll, majd elindítja a **2.** (4003/4004) és **3.** (4005/4006) csomópontot `-join http://127.0.0.1:4001` kapcsolóval
4. Northwind mintaadatokkal tölti fel az `orders` és `products` táblákat

A cluster ellenőrzése:

```bash
# Vezető lekérdezése
curl -s http://127.0.0.1:4001/status | python3 -m json.tool | grep -A3 '"leader"'

# Lekérdezés bármely csomóponton keresztül
curl 'http://127.0.0.1:4001/db/query?q=SELECT+*+FROM+orders'
curl 'http://127.0.0.1:4003/db/query?q=SELECT+COUNT(*)+FROM+orders'
```

## 14.6 HTTP API gyors referencia

### Olvasás (query)

```bash
curl 'http://localhost:4001/db/query?q=SELECT+*+FROM+orders+LIMIT+5'
```

Válasz:

```json
{
  "results": [{
    "columns": ["order_id", "customer_id", "total"],
    "types":   ["integer", "text", "real"],
    "values":  [[10248, "VINET", 440.0]]
  }],
  "time": 0.000123
}
```

### Írás (execute)

```bash
curl -XPOST http://localhost:4001/db/execute \
  -H 'Content-Type: application/json' \
  -d '["INSERT INTO orders VALUES (10999,\"ALFKI\",1,\"2025-06-01\",\"Germany\",999.99)"]'
```

## 14.7 Flask integráció

A Northwind Control Center az rqlite-ot a `services/rqlite_service.py`-on keresztül éri el — driver nélkül, csak `requests`:

```python
import requests

class RqliteClient:
    def query(self, sql):
        r = requests.get(f'{self.url}/db/query', params={'q': sql})
        r.raise_for_status()
        res = r.json()['results'][0]
        return res.get('columns', []), res.get('values', []) or []
```

A Query Studio **Target Engine** legördülőjében kiválasztható bármely regisztrált rqlite kapcsolat. Kijelölés után a lekérdezés a `rqlite_service.run_query()`-n keresztül fut, nem az SQL adapteren.

## 14.8 Chaos Monkey labor — öld meg a vezetőt

### 1. lépés — Alaplekérdezés

A Query Studioban válaszd ki az `rqlite-node1`-et és futtasd:

```sql
SELECT COUNT(*) FROM orders
```

Jegyezd fel a válaszidőt (~1–5 ms).

### 2. lépés — Öld meg a vezetőt

Egy terminálban:

```bash
pkill -f "node-id node1"
```

Az 1. csomópont halott. A 2. és 3. csomópont észleli a heartbeat-hiányt (~300 ms) és választást indít. Az egyik kb. 1 másodperc alatt új vezető lesz.

### 3. lépés — Lekérdezés a 2. csomóponton

Regisztrálj egy második kapcsolatot az alkalmazásban: `rqlite-node2`, URL `http://127.0.0.1:4003`. Válaszd ki a Target Engine legördülőben és futtasd ugyanazt a lekérdezést.

**Elvárt eredmény:** a lekérdezés sikerül. A Raft már választott új vezetőt — semmit sem kellett újraindítani, semmilyen konfigurációt nem kellett módosítani. Ez az **automatikus átvétel (automatic failover)**.

### 4. lépés — Nézd meg a naplókat

```bash
grep -E "election|leader|term" /tmp/rqlite/logs/node2.log | tail -20
```

Ilyen bejegyzéseket fogsz látni:
```
raft: entering candidate state node=node2 term=2
raft: election won tally=2
raft: entering leader state node=node2
```

## 14.9 PoW vs Raft — összehasonlítási táblázat

| Dimenzió | Proof-of-Work (4. fejezet) | Raft (ez a fejezet) |
|---|---|---|
| **Megoldott probléma** | Bizánci hibatűrés nyílt hálózatokban | Összeomlás-hibatűrés zárt clusterekben |
| **Írási késleltetés** | Másodpercektől percekig (bányászat) | 1–5 ms |
| **Áteresztőképesség** | ~7 TPS (Bitcoin) | Ezres TPS |
| **Energia** | Hatalmas | Elhanyagolható |
| **Résztvevői modell** | Anonim, nem megbízható | Ismert, megbízható partnerek |
| **Vezető** | Nincs — a leghosszabb lánc alapján | Explicit választás ciklusonként |
| **Hibatűrés** | Az összes hash-erő 51%-a legyen őszinte | A csomópontok bármely kisebbsége meghalhat |
| **Felhasználási eset** | Kriptovaluta, nyilvános blokklánc | Elosztott adatbázisok, kulcs-érték tárolók |
| **Példák** | Bitcoin, Ethereum PoW | rqlite, etcd, CockroachDB, Consul |

## 14.10 Feladatok

1. **Olvasási konzisztencia szintek** — Futtasd ugyanazt a SELECT-et az 1. csomóponton `?level=strong` és `?level=weak` paraméterrel. Mérd a késleltetést. Magyarázd el, miért lassabb az erős olvasás.

2. **Párhuzamos írások** — Két terminálban egyidejűleg küldj 50-50 INSERT-et. Ellenőrizd, hogy a végső `COUNT(*)` pontosan 100 (nincs elveszett írás, nincs duplikátum).

3. **Hálózati partíció szimulálása** — `iptables` segítségével blokkold a forgalmat az 1. csomópont és a 2-3. csomópontok között. Mi történik az 1. csomópontnak küldött írásokkal? Mi a 2. csomópont olvasásaival? Állítsd vissza a kapcsolatot és figyeld meg az egyeztetést.

4. **Tervezési kérdés** — Egy startup felhasználói munkameneteket akar tárolni (10 000 olvasás/mp, 100 írás/mp) egy elosztott tárolóban magas rendelkezésre állás érdekében. Hasonlítsd össze az rqlite-ot, a Redis Sentinel-t és a MongoDB replica set-et. Melyik illik legjobban és miért?

## Összefoglalás

- Az elosztott konszenzus szükséges, valahányszor több csomópontnak azonos írássorrendben kell megegyeznie
- A PoW helyes, de túl lassú és energiaigényes adatbázis-munkaterhelésekhez; ellenséges, nyílt hálózatokhoz tervezték
- A Raft zárt clustereken old meg konszenzust: egy vezető, többségi visszaigazolás írásként, automatikus vezető-választás meghibásodás esetén
- Az rqlite Raft-tal burkolja az SQLite-ot és egyszerű HTTP API-t nyújt — nincs szükség driverre
- A Chaos Monkey kísérlet a Raft ~1 mp-es automatikus átvételét demonstrálja a gyakorlatban
- A Northwind CC Query Studiója közvetlenül megcélozhat bármely regisztrált rqlite csomópontot
