## 14.1 Why Consensus Matters

A single-node database is simple: one writer, one truth. The moment you add a second node for fault tolerance, you face the **distributed consensus problem**: every node must agree on the *same* sequence of writes, even when messages are lost, delayed, or nodes crash.

Without consensus:

| Node A writes | Node B reads | Result |
|---|---|---|
| INSERT order 999 | SELECT * FROM orders | May return 0 or 1 rows depending on timing |
| Two nodes accept conflicting updates | — | *Split brain* — two different "truths" |

Every distributed database (PostgreSQL streaming replication, MySQL Group Replication, CockroachDB, MongoDB replica sets, rqlite) solves consensus differently.

---

## 14.2 Proof-of-Work Consensus (Revisited from Chapter 4)

PoW was invented for *permissionless* networks where participants are anonymous and potentially malicious (Bitcoin). The mechanism:

1. A node that wants to add data must solve a **cryptographic puzzle** (find a `nonce` such that `SHA256(payload + nonce)` starts with N zeros).
2. The first node to solve it broadcasts the solution.
3. All other nodes verify it (cheap) and accept the block.

**What makes PoW unsuitable for databases:**

| Property | PoW | Database requirement |
|---|---|---|
| Throughput | ~7 TPS (Bitcoin) | Thousands of TPS |
| Latency per write | Minutes | Milliseconds |
| Energy | Enormous | Negligible |
| Fault model | Byzantine (malicious nodes OK) | Crash-fault (nodes fail, don't lie) |
| Membership | Open / permissionless | Closed / known peers |

The simulation in Chapter 4 showed the mining phase visually — each "VM" racing to find the nonce. That latency is a *feature* for Bitcoin security; it is catastrophic for OLTP.

## 14.3 The Raft Algorithm

Raft (Ongaro & Ousterhout, 2014) was designed to be understandable. It solves consensus for *crash-fault-tolerant* clusters of known peers.

### 14.3.1 Roles

At any moment each node is one of:

- **Leader** — accepts all client writes, replicates to followers
- **Follower** — passive; applies log entries from the leader
- **Candidate** — temporary state during election

There is exactly one leader at a time (or zero during election).

### 14.3.2 Leader Election

Each follower has a randomized **election timeout** (150–300 ms). If it does not hear from the leader within that window, it transitions to *Candidate*, increments its **term**, and sends `RequestVote` RPCs.

A node votes *yes* if:
- it has not voted in this term yet, AND
- the candidate's log is at least as up-to-date as its own

A candidate with a majority of votes becomes the new leader for this term.

**Why randomized timeouts?** Without randomization, all followers would call elections simultaneously, splitting votes forever. Randomization ensures one node usually fires first.

### 14.3.3 Log Replication

```
Client          Leader          Follower 2      Follower 3
  |  INSERT ...  |               |               |
  |------------->|               |               |
  |              |--AppendEntry->|               |
  |              |--AppendEntry----------------->|
  |              |<---ACK--------|               |
  |              |<---ACK--------------------------
  |              |  (majority)                   |
  |              | COMMIT locally                |
  |<--OK---------|               |               |
  |              |--Notify------>|               |
  |              |--Notify---------------------> |
```

A write is **committed** once a majority of nodes have written it to their log — even if some followers are slow or temporarily disconnected.

### 14.3.4 Safety Guarantee

Raft guarantees: if a log entry is committed, *every future leader will also have that entry*. This prevents the "revert accepted write" failure mode.

## 14.4 rqlite — Raft-Consensus SQLite

**rqlite** is an open-source distributed relational database built by Philip O'Toole. It wraps SQLite with a Raft consensus layer, exposing a simple HTTP+JSON API.

Architecture:

```
┌─────────────────────────────────────────┐
│              HTTP API                   │  ← your Flask app (requests library)
│   GET /db/query    POST /db/execute     │
├─────────────────────────────────────────┤
│              SQLite engine              │  ← per-node embedded database
├─────────────────────────────────────────┤
│              Raft log                   │  ← replicated write-ahead log
└─────────────────────────────────────────┘
         Node 1          Node 2          Node 3
         :4001           :4003           :4005
```

Key properties:

- **No driver needed** — pure HTTP; works from `curl`, Python `requests`, any language
- **Strong consistency** by default — reads go through the leader
- **Eventual read** option — `?level=weak` lets any node answer reads (may be slightly stale)
- **SQLite SQL** — full SELECT, INSERT, UPDATE, DELETE, PRAGMA support
- **Automatic failover** — if the leader dies, followers elect a new one in ~1 s

## 14.5 Running the Local Cluster

```bash
# From the northwind-control-center directory:
bash start_rqlite_cluster.sh
```

The script:
1. Kills any existing `rqlited` processes
2. Starts **Node 1** on ports 4001/4002 (HTTP/Raft)
3. Waits for Node 1 to be ready, then starts **Node 2** (4003/4004) and **Node 3** (4005/4006) with `-join http://127.0.0.1:4001`
4. Seeds `orders` and `products` tables with Northwind sample data

Verify the cluster:

```bash
# Check leader
curl -s http://127.0.0.1:4001/status | python3 -m json.tool | grep -A3 '"leader"'

# Query through any node
curl 'http://127.0.0.1:4001/db/query?q=SELECT+*+FROM+orders'
curl 'http://127.0.0.1:4003/db/query?q=SELECT+COUNT(*)+FROM+orders'
curl 'http://127.0.0.1:4005/db/query?q=SELECT+*+FROM+products'
```

## 14.6 HTTP API Quick Reference

### Read (query)

```bash
# Any SQL SELECT
curl 'http://localhost:4001/db/query?q=SELECT+*+FROM+orders+LIMIT+5'

# Weak consistency (any node, possibly stale)
curl 'http://localhost:4003/db/query?q=SELECT+COUNT(*)+FROM+orders&level=weak'
```

Response:

```json
{
  "results": [{
    "columns": ["order_id", "customer_id", "total"],
    "types":   ["integer", "text", "real"],
    "values":  [[10248, "VINET", 440.0], [10249, "TOMSP", 1863.4]]
  }],
  "time": 0.000123
}
```

### Write (execute)

```bash
curl -XPOST http://localhost:4001/db/execute \
  -H 'Content-Type: application/json' \
  -d '["INSERT INTO orders VALUES (10999,\"ALFKI\",1,\"2025-06-01\",\"Germany\",999.99)"]'
```

Response:

```json
{
  "results": [{"last_insert_id": 10999, "rows_affected": 1}],
  "time": 0.00089
}
```

### Cluster status

```bash
curl http://localhost:4001/status   # full node info
curl http://localhost:4001/nodes    # peer list with leader flag
```

## 14.7 Flask Integration

The Northwind Control Center talks to rqlite through `services/rqlite_service.py`:

```python
import requests

class RqliteClient:
    def __init__(self, url):
        self.url = url.rstrip('/')

    def query(self, sql):
        r = requests.get(f'{self.url}/db/query', params={'q': sql, 'level': 'weak'})
        r.raise_for_status()
        result = r.json()['results'][0]
        return result.get('columns', []), result.get('values', []) or []

    def execute(self, sql):
        r = requests.post(f'{self.url}/db/execute', json=[sql])
        r.raise_for_status()
        return r.json()['results'][0]
```

**No ORM, no driver, no connection pool** — just HTTP.

To add rqlite in the app:
1. Go to **Control Plane → Connections → New Connection**
2. Type: `rqlite`, URL: `http://127.0.0.1:4001`, Name: `rqlite-node1`
3. Test Connection → should show green
4. Go to **Control Plane → rqlite (Raft)** — cluster status, node list, SQL console

Query Studio integration: the **Target Engine** dropdown shows all registered rqlite connections. Selecting one routes the query through `rqlite_service.run_query()` instead of the SQL adapter.

## 14.8 Chaos Monkey Lab — Kill the Leader

This experiment demonstrates Raft's automatic failover. Do it in three steps:

### Step 1 — Baseline query

In the app, select `rqlite-node1` (Node 1) as target and run:

```sql
SELECT COUNT(*) FROM orders
```

Note the response time (~1–5 ms).

### Step 2 — Kill the leader

In a terminal:

```bash
pkill -f "node-id node1"
```

Node 1 is dead. Nodes 2 and 3 detect the heartbeat timeout (~300 ms) and start an election. One of them becomes the new leader in about 1 second.

### Step 3 — Query through Node 2

Register a second connection in the app: `rqlite-node2`, URL `http://127.0.0.1:4003`. Select it in the Target Engine dropdown and run the same query.

**Expected result:** the query succeeds. Raft has already elected a new leader — you did not need to restart anything or change any configuration. This is **automatic failover**.

Compare to what would happen with a single SQL Server node: the app would show a connection error until you manually restored the server.

### Step 4 — Observe the election in logs

```bash
grep -E "election|leader|term" /tmp/rqlite/logs/node2.log | tail -20
```

You will see entries like:
```
raft: entering candidate state node=node2 term=2
raft: election won tally=2
raft: entering leader state node=node2
```

### Step 5 — Restart Node 1

```bash
rqlited -node-id node1 -http-addr 127.0.0.1:4001 -raft-addr 127.0.0.1:4002 \
  -join http://127.0.0.1:4003 /tmp/rqlite/node1 &
```

Node 1 rejoins as a follower, catches up the log, and the cluster is healthy again — all with zero downtime for the application.

## 14.9 PoW vs Raft — Comparison Table

| Dimension | Proof-of-Work (Chapter 4) | Raft (this chapter) |
|---|---|---|
| **Problem solved** | Byzantine fault tolerance in open networks | Crash fault tolerance in closed clusters |
| **Write latency** | Seconds to minutes (mining) | 1–5 ms |
| **Throughput** | ~7 TPS (Bitcoin) | Thousands of TPS |
| **Energy** | Proportional to hash rate | Negligible |
| **Participant model** | Anonymous, untrusted | Known, trusted peers |
| **Leader** | None — emergent via longest chain | Elected explicitly per term |
| **Failure tolerance** | 51 % of hash power can be honest | Any minority of nodes can fail |
| **Use case** | Cryptocurrency, public blockchain | Distributed databases, key-value stores |
| **Examples** | Bitcoin, Ethereum PoW | rqlite, etcd, CockroachDB, Consul |

## Summary

- Distributed consensus is necessary any time multiple nodes must agree on a write order
- PoW is correct but too slow and energy-intensive for database workloads; it is designed for adversarial, open networks
- Raft solves consensus for closed clusters: one leader, majority acknowledgment per write, automatic leader election on failure
- rqlite wraps SQLite with Raft and exposes a simple HTTP API — no driver needed
- The Chaos Monkey experiment shows Raft's ~1 s automatic failover in practice
- Query Studio in Northwind CC can target any registered rqlite node directly
