## 15.1 Why SQL LIKE Falls Short

The SQL `LIKE` operator is simple and universally supported, but it has fundamental limitations for full-text workloads:

```sql
-- Find products containing "chai"
SELECT ProductName FROM Products WHERE ProductName LIKE '%chai%';
```

| Limitation | Explanation |
|---|---|
| **No relevance ranking** | All matching rows are equally "good"; there is no score |
| **Leading wildcard kills indexes** | `LIKE '%chai%'` cannot use a B-tree index — it scans every row |
| **No tokenization** | `'chai tea'` does not match `'tea chai'` unless you add extra wildcards |
| **No fuzziness** | `'Chaai'` (typo) returns nothing |
| **Language-unaware** | Plural forms, stemming, synonyms require complex manual workarounds |

A search engine like Elasticsearch solves all of these by processing text at index time rather than query time.

## 15.2 How Elasticsearch Indexes Text

When a document is indexed, Elasticsearch runs the field value through an **analyzer pipeline**:

```
"Grandma's Boysenberry Spread"
          │
          ▼  tokenizer (standard)
["Grandma's", "Boysenberry", "Spread"]
          │
          ▼  token filters (lowercase, stop words, stemmer)
["grandma", "boysenberri", "spread"]
          │
          ▼  inverted index
grandma      → [doc 7]
boysenberri  → [doc 7]
spread       → [doc 7, doc 12, doc 31]
```

At query time, the same analyzer runs on the query text. The engine then looks up matching terms in the inverted index in O(1) — no table scan required.

### 15.2.1 Elasticsearch Data Model

| Concept | SQL equivalent | Description |
|---|---|---|
| **Index** | Table | A collection of documents with a common schema |
| **Document** | Row | A JSON object stored in an index |
| **Field** | Column | A named value inside a document |
| **Mapping** | DDL / column types | Defines field types (text, keyword, integer, float …) |
| **Shard** | Partition | A Lucene index; ES distributes shards across nodes |
| **Replica** | Standby node | A copy of a shard for high availability |

## 15.3 SQL vs Elasticsearch Query Comparison

| Goal | SQL | Elasticsearch |
|---|---|---|
| Exact match | `WHERE name = 'Chai'` | `{ "term": { "product_name.keyword": "Chai" } }` |
| Substring | `WHERE name LIKE '%chai%'` | `{ "match": { "product_name": "chai" } }` |
| Multi-field | `WHERE name LIKE '%x%' OR desc LIKE '%x%'` | `{ "multi_match": { "query": "x", "fields": ["*"] } }` |
| Fuzzy (typo) | Not supported natively | `{ "match": { "product_name": { "query": "cahai", "fuzziness": "AUTO" } } }` |
| Relevance sort | `ORDER BY` a computed column | Automatic `_score` based on BM25 |
| Range bucket | `CASE WHEN price < 10 THEN …` in GROUP BY | `{ "range": { "field": "unit_price", "ranges": […] } }` |
| Average per group | `AVG(price) GROUP BY category` | `{ "terms": { "field": "category.keyword" }, "aggs": { "avg_price": { "avg": … } } }` |

## 15.4 The Northwind Product Search Lab

The Northwind Control Center provides a ready-made Elasticsearch integration. Follow these steps:

### Step 1 — Register the connection

Go to **Control Plane → Connections → New Connection**:

- Type: `elasticsearch`
- URL: `https://localhost:9200`
- Username: `elastic`
- Password: (from your `.env` or the elastic user's password)
- Name: `ES Local`

Click **Test Connection** — you should see a green success banner.

### Step 2 — Import Northwind data

Navigate to **Control Plane → Elasticsearch**. Select your `ES Local` connection. In the left panel, choose the Northwind SQL connection as the source and click **Import**.

The service executes:

```python
# Products
SELECT ProductID, ProductName, QuantityPerUnit,
       UnitPrice, UnitsInStock, Discontinued
FROM Products

# Categories
SELECT CategoryID, CategoryName, Description FROM Categories
```

Each row becomes a JSON document indexed under `northwind_products` and `northwind_categories` respectively.

### Step 3 — Search

With the `northwind_products` index selected, type **chai** in the search bar and click Search. The ES query sent is:

```json
{
  "query": {
    "multi_match": {
      "query":     "chai",
      "fields":    ["*"],
      "fuzziness": "AUTO",
      "operator":  "or"
    }
  },
  "size": 50
}
```

You will see "Chai" and "Ipoh Coffee" (both are beverages) returned with relevance scores computed by BM25.

### Step 4 — Fuzzy search

Try searching for **"cahai"** (deliberate typo) with fuzziness set to AUTO. Elasticsearch allows up to 2 edit-distance substitutions for longer terms, so it still finds "Chai". A SQL `LIKE '%cahai%'` would return nothing.

## 15.5 BM25 Relevance Scoring

Elasticsearch scores each hit using **BM25** (Best Match 25), the industry standard for full-text ranking since the 1990s.

The BM25 score for term `t` in document `d` is:

```
score(t, d) = IDF(t) × (tf(t,d) × (k1 + 1)) / (tf(t,d) + k1 × (1 − b + b × |d|/avgdl))
```

Where:
- **IDF(t)** — Inverse Document Frequency: rare terms score higher than common ones
- **tf(t,d)** — how many times `t` appears in `d`
- **k1** (default 1.2) — term-frequency saturation; diminishing returns after a few occurrences
- **b** (default 0.75) — length normalization; long documents are penalized slightly
- **|d| / avgdl** — document length relative to the average

**Practical implication**: a product whose name is *exactly* "Chai" ranks above a product whose description *mentions* "chai" five times buried in a long paragraph, because `IDF × tf saturation × length normalization` collectively prefer precise, short-field matches.

## 15.6 Aggregations: Bucket vs Metric

Elasticsearch aggregations run in parallel with search and do not require a second query.

### Bucket aggregations

Group documents into buckets. Analogous to `GROUP BY`:

```json
{
  "aggs": {
    "price_ranges": {
      "range": {
        "field": "unit_price",
        "ranges": [
          { "key": "$0–$10",  "from": 0,  "to": 10 },
          { "key": "$10–$25", "from": 10, "to": 25 },
          { "key": "$25+",    "from": 25 }
        ]
      }
    }
  }
}
```

### Metric aggregations

Compute a single value per bucket. Analogous to `AVG()`, `SUM()`, `MAX()`:

```json
{
  "aggs": {
    "categories": {
      "terms": { "field": "category_name.keyword", "size": 20 },
      "aggs": {
        "avg_price": { "avg": { "field": "unit_price" } }
      }
    }
  }
}
```

### Nested aggregations

Bucket and metric aggregations can be nested to arbitrary depth — the equivalent of `SELECT category, AVG(price), COUNT(*) GROUP BY category` requires only a single ES request.

The Northwind Control Center exposes two built-in aggregations under the **Aggregations** panel:

1. **Products by price range** — bucket aggregation on `unit_price`
2. **Average price by category** — terms bucket + avg metric on `northwind_categories`

## 15.7 When to Use Elasticsearch Alongside SQL

Elasticsearch is not a replacement for a relational database. The two systems are complementary:

| Requirement | Use SQL | Use Elasticsearch |
|---|---|---|
| ACID transactions | Yes | No |
| Referential integrity (FK) | Yes | No |
| Precise aggregations on normalized data | Yes | Not primary use case |
| Full-text search with relevance | No | Yes |
| Fuzzy / typo-tolerant search | No | Yes |
| Log and event analytics at scale | No (too slow) | Yes |
| Autocomplete / suggest | No | Yes |
| Horizontal read scaling | Limited | Yes (shards + replicas) |

**Typical architecture**: store authoritative data in SQL Server / PostgreSQL; keep an Elasticsearch index as a read-optimized projection for search. A trigger or message queue event syncs writes from SQL to ES. This is the pattern used by e-commerce platforms, content management systems, and SaaS applications at scale.

## Summary

- SQL `LIKE` is unsuitable for full-text search: it cannot rank results, it requires full table scans with leading wildcards, and it has no typo tolerance
- Elasticsearch builds an inverted index at write time; queries look up terms in O(1) regardless of dataset size
- The ES data model maps naturally to SQL: index = table, document = row, field = column, mapping = DDL
- BM25 scoring combines term frequency, inverse document frequency, and length normalization to rank results by relevance
- Aggregations (bucket + metric) provide GROUP BY-equivalent analytics in a single search request
- ES works best alongside a relational database, not instead of one: SQL owns the authoritative data, ES owns the search projection
