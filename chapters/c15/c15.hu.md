# 15. fejezet — Teljes szöveges keresés Elasticsearch-szal

## Tanulási célok

A fejezet végére képes leszel:

- Megmagyarázni, miért jobb az Elasticsearch az SQL `LIKE`-nál teljes szöveges keresésnél és relevanciás rangsorolásnál
- Ismertetni az Elasticsearch adatmodelljét: index, dokumentum, mező, leképezés
- Összehasonlítani az SQL `LIKE` lekérdezéseket az Elasticsearch `match` és `multi_match` lekérdezéseivel
- Northwind termékeket és kategóriákat importálni Elasticsearch-be a Control Centerből
- Fuzzy kereséseket futtatni konfigurálható fuzziness értékkel, és értelmezni a BM25 relevanciapont-számítást
- Bucket és metrika aggregációkat készíteni Elasticsearch-ben
- Megfogalmazni, mikor érdemes Elasticsearch-t relációs adatbázis mellé, nem helyette használni

---

## 15.1 Miért nem elégséges az SQL LIKE?

Az SQL `LIKE` operátor egyszerű és mindenütt támogatott, de alapvető korlátai vannak teljes szöveges munkaterhelések esetén:

```sql
-- Termékek keresése, amelyek "chai"-t tartalmaznak
SELECT ProductName FROM Products WHERE ProductName LIKE '%chai%';
```

| Korlát | Magyarázat |
|---|---|
| **Nincs relevanciás rangsorolás** | Minden egyező sor egyformán "jó"; nincs pontszám |
| **Vezető helyettesítő karakter letiltja az indexet** | A `LIKE '%chai%'` nem tud B-fa indexet használni — minden sort átvizsgál |
| **Nincs tokenizáció** | A `'chai tea'` nem egyezik a `'tea chai'`-ra, hacsak nem adsz hozzá extra helyettesítő karaktereket |
| **Nincs fuzziness** | A `'Chaai'` (elgépelés) nem ad vissza semmit |
| **Nem érti a nyelvet** | Többes szám, töredékek, szinonimák kezeléséhez bonyolult manuális megoldások kellnek |

Egy kereső motor, mint az Elasticsearch, ezeket mind megoldja azzal, hogy a szöveget indexeléskor dolgozza fel, nem lekérdezéskor.

## 15.2 Hogyan indexeli az Elasticsearch a szöveget?

Amikor egy dokumentumot indexelnek, az Elasticsearch az elemzői csővezetéken futtatja a mezőértéket:

```
"Grandma's Boysenberry Spread"
          │
          ▼  tokenizáló (standard)
["Grandma's", "Boysenberry", "Spread"]
          │
          ▼  token szűrők (kisbetűs, stop szavak, tövedékképző)
["grandma", "boysenberri", "spread"]
          │
          ▼  fordított index
grandma      → [7. dok]
boysenberri  → [7. dok]
spread       → [7. dok, 12. dok, 31. dok]
```

Lekérdezéskor ugyanaz az elemző fut a keresőkifejezésen. A motor ezután O(1) időben keresi meg az egyező kifejezéseket a fordított indexben — nincs szükség táblaszkennelésre.

### 15.2.1 Az Elasticsearch adatmodellje

| Fogalom | SQL megfelelője | Leírás |
|---|---|---|
| **Index** | Tábla | Közös sémával rendelkező dokumentumok gyűjteménye |
| **Dokumentum** | Sor | Egy JSON objektum az indexben |
| **Mező** | Oszlop | Egy elnevezett érték a dokumentumon belül |
| **Leképezés (Mapping)** | DDL / oszloptípusok | Meghatározza a mezőtípusokat (text, keyword, integer, float …) |
| **Shard** | Partíció | Egy Lucene index; az ES shardokat oszt szét a csomópontok között |
| **Replika** | Készenléti csomópont | Egy shard másolata a magas rendelkezésre állás érdekében |

## 15.3 SQL vs Elasticsearch lekérdezés összehasonlítás

| Cél | SQL | Elasticsearch |
|---|---|---|
| Pontos egyezés | `WHERE name = 'Chai'` | `{ "term": { "product_name.keyword": "Chai" } }` |
| Részstring | `WHERE name LIKE '%chai%'` | `{ "match": { "product_name": "chai" } }` |
| Több mező | `WHERE name LIKE '%x%' OR desc LIKE '%x%'` | `{ "multi_match": { "query": "x", "fields": ["*"] } }` |
| Fuzzy (elgépelés) | Nem támogatott natívan | `{ "match": { "product_name": { "query": "cahai", "fuzziness": "AUTO" } } }` |
| Relevancia szerinti rendezés | `ORDER BY` egy számított oszlop alapján | Automatikus `_score` BM25 alapján |
| Tartomány-buckets | `CASE WHEN price < 10 THEN …` a GROUP BY-ban | `{ "range": { "field": "unit_price", "ranges": […] } }` |
| Átlag csoportonként | `AVG(price) GROUP BY category` | `{ "terms": { "field": "category.keyword" }, "aggs": { "avg_price": { "avg": … } } }` |

## 15.4 A Northwind termékkeresés labor

A Northwind Control Center kész Elasticsearch integrációt nyújt. Kövesd az alábbi lépéseket:

### 1. lépés — A kapcsolat regisztrálása

Menj a **Control Plane → Connections → New Connection** oldalra:

- Típus: `elasticsearch`
- URL: `https://localhost:9200`
- Felhasználónév: `elastic`
- Jelszó: (a `.env` fájlból vagy az elastic felhasználó jelszavából)
- Név: `ES Local`

Kattints a **Test Connection** gombra — zöld sikerbannert kell látni.

### 2. lépés — Northwind adatok importálása

Navigálj a **Control Plane → Elasticsearch** oldalra. Válaszd ki az `ES Local` kapcsolatot. A bal panelen válaszd ki a Northwind SQL kapcsolatot forrásként, majd kattints az **Import** gombra.

A szolgáltatás a következőket hajtja végre:

```python
# Termékek
SELECT ProductID, ProductName, QuantityPerUnit,
       UnitPrice, UnitsInStock, Discontinued
FROM Products

# Kategóriák
SELECT CategoryID, CategoryName, Description FROM Categories
```

Minden sor egy JSON dokumentumként kerül indexelésre a `northwind_products` és `northwind_categories` indexekbe.

### 3. lépés — Keresés

Ha a `northwind_products` index ki van választva, gépelje be a **chai** szót a keresőmezőbe, és kattintson a Search gombra. Az elküldött ES lekérdezés:

```json
{
  "query": {
    "multi_match": {
      "query":     "chai",
      "fields":    ["*"],
      "fuzziness": "AUTO",
      "operator":  "or"
    }
  },
  "size": 50
}
```

A "Chai" és az "Ipoh Coffee" (mindkettő ital) BM25 által számított relevanciapont alapján jelenik meg.

### 4. lépés — Fuzzy keresés

Próbáld meg keresni a **"cahai"** szót (szándékos elgépelés) AUTO fuzziness beállítással. Az Elasticsearch legfeljebb 2 szerkesztési távolságú cserét enged meg hosszabb kifejezéseknél, így a "Chai"-t még megtalálja. Egy SQL `LIKE '%cahai%'` semmit nem adna vissza.

## 15.5 BM25 relevanciapont-számítás

Az Elasticsearch minden találatot **BM25** (Best Match 25) segítségével pontoz — ez az iparági standard a teljes szöveges rangsoroláshoz az 1990-es évek óta.

A BM25 pontszám a `t` kifejezésre a `d` dokumentumban:

```
score(t, d) = IDF(t) × (tf(t,d) × (k1 + 1)) / (tf(t,d) + k1 × (1 − b + b × |d|/avgdl))
```

Ahol:
- **IDF(t)** — Inverz dokumentumfrekvencia: a ritka kifejezések magasabb pontszámot kapnak, mint a közönségesek
- **tf(t,d)** — hányszor szerepel `t` a `d`-ben
- **k1** (alapértelmezés: 1,2) — kifejezésfrekvencia-telítés; néhány előfordulás után csökkenő hozam
- **b** (alapértelmezés: 0,75) — hossznormalizálás; a hosszú dokumentumokat kissé büntetik
- **|d| / avgdl** — dokumentumhossz az átlaghoz képest

**Gyakorlati következmény**: egy termék, amelynek neve pontosan "Chai", magasabb rangot kap, mint egy olyan, amelynek leírásában ötször szerepel a "chai" egy hosszú bekezdés közepén — mert az `IDF × tf telítés × hossznormalizálás` összességében az pontos, rövid mezőegyezéseket részesíti előnyben.

## 15.6 Aggregációk: Bucket vs Metrika

Az Elasticsearch aggregációk párhuzamosan futnak a kereséssel, és nem igényelnek második lekérdezést.

### Bucket aggregációk

Dokumentumokat csoportosítanak bucketekbe. Analóg a `GROUP BY`-jal:

```json
{
  "aggs": {
    "price_ranges": {
      "range": {
        "field": "unit_price",
        "ranges": [
          { "key": "$0–$10",  "from": 0,  "to": 10 },
          { "key": "$10–$25", "from": 10, "to": 25 },
          { "key": "$25+",    "from": 25 }
        ]
      }
    }
  }
}
```

### Metrika aggregációk

Bucketenként egyetlen értéket számítanak. Analóg az `AVG()`, `SUM()`, `MAX()` függvényekkel:

```json
{
  "aggs": {
    "categories": {
      "terms": { "field": "category_name.keyword", "size": 20 },
      "aggs": {
        "avg_price": { "avg": { "field": "unit_price" } }
      }
    }
  }
}
```

### Beágyazott aggregációk

A bucket és metrika aggregációk tetszőleges mélységig egymásba ágyazhatók — a `SELECT category, AVG(price), COUNT(*) GROUP BY category` egyenértékű egyetlen ES kéréssel valósítható meg.

A Northwind Control Center két beépített aggregációt kínál az **Aggregations** panelen:

1. **Termékek árkategória szerint** — bucket aggregáció az `unit_price` mezőn
2. **Átlagár kategóriánként** — terms bucket + avg metrika a `northwind_categories` indexen

## 15.7 Mikor érdemes az Elasticsearch-t SQL mellé használni?

Az Elasticsearch nem helyettesíti a relációs adatbázist. A két rendszer kiegészíti egymást:

| Követelmény | Használd az SQL-t | Használd az Elasticsearch-t |
|---|---|---|
| ACID tranzakciók | Igen | Nem |
| Hivatkozási integritás (FK) | Igen | Nem |
| Pontos aggregációk normalizált adatokon | Igen | Nem elsődleges felhasználási eset |
| Teljes szöveges keresés relevanciával | Nem | Igen |
| Fuzzy / elgépeléstűrő keresés | Nem | Igen |
| Napló- és eseményelemzés nagy léptékben | Nem (túl lassú) | Igen |
| Automatikus kiegészítés / javaslat | Nem | Igen |
| Vízszintes olvasási skálázás | Korlátozott | Igen (shardok + replikák) |

**Tipikus architektúra**: az hiteles adatokat SQL Server / PostgreSQL tárolja; az Elasticsearch indexet kereső-optimalizált vetítésként tartják fenn. Egy trigger vagy üzenetsor-esemény szinkronizálja az SQL-ből ES-be írt adatokat. Ezt a mintát használják e-kereskedelmi platformok, tartalomkezelő rendszerek és SaaS alkalmazások nagy léptékben.

## 15.8 Feladatok

1. **Fuzzy keresési kísérlet** — Keresd a "choclate" szót (elgépelés a "chocolate" helyett) 0, 1 és 2 fuzziness értékkel. Jegyezd fel, mely termékek jelennek meg az egyes szinteken, és magyarázd el a szerkesztési távolság különbségét.

2. **Aggregáció** — Futtasd a "Termékek árkategória szerint" aggregációt, és hasonlítsd össze a bucket-számokat az egyenértékű SQL-lel:
   ```sql
   SELECT
     CASE
       WHEN UnitPrice < 10  THEN '$0–$10'
       WHEN UnitPrice < 25  THEN '$10–$25'
       WHEN UnitPrice < 50  THEN '$25–$50'
       WHEN UnitPrice < 100 THEN '$50–$100'
       ELSE '$100+'
     END AS price_range,
     COUNT(*) AS product_count
   FROM Products
   GROUP BY
     CASE
       WHEN UnitPrice < 10  THEN '$0–$10'
       WHEN UnitPrice < 25  THEN '$10–$25'
       WHEN UnitPrice < 50  THEN '$25–$50'
       WHEN UnitPrice < 100 THEN '$50–$100'
       ELSE '$100+'
     END;
   ```
   Megegyeznek-e a darabszámok? Miért vagy miért nem?

3. **LIKE teljesítményösszehasonlítás** — Engedélyezd az SQL Server lekérdezési statisztikáit, és mérd meg a `LIKE '%sauce%'` eltelt idejét 1 000 000 szintetikus terméken (generáld azokat egy ciklussal). Majd mérd meg ugyanezt a keresést Elasticsearch-ben. Rögzítsd a különbséget, és magyarázd el a fordított index fogalmával.

4. **Egyéni leképezés** — Töröld a `northwind_products` indexet, és hozd létre újra egy explicit leképezéssel, amely a `product_name`-t `text`-ként jelöli angol elemzővel, az `unit_price`-t pedig `scaled_float`-ként `scaling_factor: 100`-zal. Importálj újra, és futtass keresést. A tő-képzés most egyezik-e a "beverages" szóra, ha "beverage" (egyes szám) alakban keresel?

5. **Tervezési kérdés** — Egy könyvesbolt 10 millió könyvcímet tárol SQL Serverben. A felhasználók panaszkodnak, hogy a keresőmező rossz eredményeket ad, amikor elgépelik a szerzők nevét. Tervezz Elasticsearch integrációt: mely mezőket indexeld, milyen elemzőt használj, hogyan tartsd szinkronban az SQL-t és az ES-t, és mit tegyél, ha az ES átmenetileg nem elérhető.

## Összefoglalás

- Az SQL `LIKE` nem alkalmas teljes szöveges keresésre: nem tud rangsorolni, vezető helyettesítő karakterekkel teljes táblaszkennelést igényel, és nincs elgépeléstűrése
- Az Elasticsearch íráskor épít fordított indexet; a lekérdezések O(1) időben keresnek ki kifejezéseket az adatkészlet méretétől függetlenül
- Az ES adatmodellje természetesen illeszkedik az SQL-hez: index = tábla, dokumentum = sor, mező = oszlop, mapping = DDL
- A BM25 pontszámítás a kifejezésfrekvenciát, az inverz dokumentumfrekvenciát és a hossznormalizálást kombinálja a találatok relevanciás rangsorolásához
- Az aggregációk (bucket + metrika) GROUP BY-egyenértékű elemzést nyújtanak egyetlen keresési kérésen belül
- Az ES relációs adatbázis mellett a legjobb, nem helyette: az SQL az hiteles adatoké, az ES a keresési vetítésé
