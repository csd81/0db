# Chapter 13 – Neo4j: Native Graph Databases

## 13.1 From Relational to Native Graph

In Chapter 8 we saw how SQL Server can model graph data using `AS NODE` and `AS EDGE` tables. This works well when graph data lives inside an existing relational schema, but it has limits: only `SHORTEST_PATH` is available as a graph algorithm, multi-label nodes are not supported, and every query still passes through the relational engine.

Neo4j is a **native graph database**. Its storage engine is built from the ground up to store and traverse connected data. Every node stores a pointer to its first relationship, and every relationship stores pointers to the next relationship of each endpoint — making arbitrary-depth traversals O(relationship count), not O(table size).

## 13.2 Data Model

Neo4j uses the **Property Graph** model:

- **Node**: A vertex. Has zero or more **labels** (`:Person`, `:Restaurant`) and a set of key-value **properties** (`{name: 'john', id: 1}`).
- **Relationship**: A directed edge with exactly one **type** (`:LIKES`, `:FRIEND_OF`) and optional properties (`{rating: 9}`).
- **Path**: An alternating sequence of nodes and relationships.

Unlike SQL Server graph tables — where the node type is determined by which table the row lives in — Neo4j nodes are schema-optional. You can mix properties freely and add multiple labels to a single node.

## 13.3 Cypher Query Language

Cypher uses ASCII-art patterns to describe graph shapes:

```cypher
MATCH (a:Person)-[:FRIEND_OF]->(b:Person)
RETURN a.name, b.name
```

- `(a:Person)` — node with label `Person`, bound to variable `a`
- `-[:FRIEND_OF]->` — directed relationship of type `FRIEND_OF`
- `(b:Person)` — another `Person` node

### Comparison with SQL Server MATCH

```sql
-- SQL Server
SELECT p1.name, p2.name
FROM person p1, person p2, friendof
WHERE MATCH(p1-(friendof)->p2)
```

```cypher
-- Cypher
MATCH (p1:Person)-[:FRIEND_OF]->(p2:Person)
RETURN p1.name, p2.name
```

Cypher is more concise because the relationship type is part of the pattern, not a separate table reference. See `cypher/05_sql_vs_cypher_comparison.md` for a full side-by-side comparison.

## 13.5 Basic Queries

All 7 queries from Chapter 8 have direct Cypher equivalents (see `cypher/03_basic_queries.cypher`).

### John's friends
```cypher
MATCH (:Person {name: 'john'})-[:FRIEND_OF]->(f:Person)
RETURN f.name AS friend
```

### Two-hop: friends' liked restaurants
```cypher
MATCH (:Person {name: 'john'})-[:FRIEND_OF]->(p2:Person)-[:LIKES]->(r:Restaurant)
RETURN p2.name AS friend, r.name AS restaurant
```

### Shortest path
```cypher
MATCH path = shortestPath(
    (:Person {name: 'john'})-[:FRIEND_OF*]->(dest:Person)
)
WHERE dest.name <> 'john'
RETURN dest.name, length(path) AS hops, [n IN nodes(path) | n.name] AS route
```

## 13.6 Advanced Patterns

### Variable-length traversal
```cypher
-- All persons within 2 hops of john
MATCH (:Person {name: 'john'})-[:FRIEND_OF*1..2]->(p:Person)
RETURN DISTINCT p.name
```

### Aggregation
```cypher
MATCH (p:Person)-[l:LIKES]->(r:Restaurant)
RETURN r.name, avg(l.rating) AS avg_rating, count(p) AS fans
ORDER BY fans DESC
```

## 13.7 Migration from SQL Server

`cypher/02_migrate_from_sql.py` reads the Chapter 8 graph tables from SQL Server using `pyodbc` and writes them to Neo4j using the official Python driver:

```python
from neo4j import GraphDatabase
driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))
with driver.session() as session:
    session.run("CREATE (:Person {id: $id, name: $name})", id=1, name='john')
```

## 13.9 When to Choose Neo4j

| Scenario | SQL Server Graph | Neo4j |
|---|---|---|
| Graph data in an existing relational DB | ✓ Ideal | Overkill |
| Deeply connected traversals (6+ hops) | Slow | Fast (pointer-chain storage) |
| Multiple graph algorithms needed | Limited | Full GDS library |
| Standalone graph workload | Works | Ideal |
| Full ACID transactions | ✓ | ✓ |
