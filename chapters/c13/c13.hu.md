# 13. fejezet – Neo4j: Natív gráfadatbázisok

## 13.1 A relációstól a natív gráfig

A 8. fejezetben láttuk, hogyan lehet SQL Serverben gráfadatokat modellezni `AS NODE` és `AS EDGE` táblákkal. Ez jól működik, ha a gráfadat egy meglévő relációs sémában él, de vannak korlátai: csak `SHORTEST_PATH` érhető el gráfalgoritmusként, a csomópontok nem kaphatnak több típust, és minden lekérdezés a relációs motoron megy keresztül.

A Neo4j egy **natív gráfadatbázis**. Tárolómotorja a kapcsolt adatok tárolására és bejárására lett felépítve. Minden csomópont mutatót tárol az első kapcsolatára, és minden kapcsolat mutatókat tárol mindkét végpont következő kapcsolatára — így a tetszőleges mélységű bejárás O(kapcsolatok száma), nem O(táblaméret).

## 13.2 Adatmodell

A Neo4j a **Tulajdonsággráf** (Property Graph) modellt használja:

- **Csomópont (Node)**: Csúcs. Nulla vagy több **label**-t (`:Person`, `:Restaurant`) és kulcs-érték **tulajdonságokat** tartalmaz (`{name: 'john', id: 1}`).
- **Kapcsolat (Relationship)**: Irányított él, pontosan egy **típussal** (`:LIKES`, `:FRIEND_OF`) és opcionális tulajdonságokkal (`{rating: 9}`).
- **Útvonal (Path)**: Csomópontok és kapcsolatok váltakozó sorozata.

Az SQL Server gráftáblákkal ellentétben — ahol a csomópont típusát a tábla határozza meg — a Neo4j csomópontjai sémaopcionálisak: egy csomóponthoz több label is rendelhető.

## 13.3 Cypher lekérdezőnyelv

A Cypher ASCII-art mintákkal írja le a gráf alakzatait:

```cypher
MATCH (a:Person)-[:FRIEND_OF]->(b:Person)
RETURN a.name, b.name
```

- `(a:Person)` — `Person` labelű csomópont, `a` változóhoz kötve
- `-[:FRIEND_OF]->` — `FRIEND_OF` típusú irányított kapcsolat
- `(b:Person)` — másik `Person` csomópont

### Összehasonlítás az SQL Server MATCH szintaxisával

```sql
-- SQL Server
SELECT p1.name, p2.name
FROM person p1, person p2, friendof
WHERE MATCH(p1-(friendof)->p2)
```

```cypher
-- Cypher
MATCH (p1:Person)-[:FRIEND_OF]->(p2:Person)
RETURN p1.name, p2.name
```

A Cypher tömörebb, mert a kapcsolat típusa a mintában szerepel. Teljes összehasonlítást lásd: `cypher/05_sql_vs_cypher_comparison.md`.

## 13.5 Alaplekérdezések

A 8. fejezet mind a 7 lekérdezésének van közvetlen Cypher megfelelője (lásd `cypher/03_basic_queries.cypher`).

### John barátai
```cypher
MATCH (:Person {name: 'john'})-[:FRIEND_OF]->(f:Person)
RETURN f.name AS barat
```

### Kétlépéses: barátok kedvelt éttermei
```cypher
MATCH (:Person {name: 'john'})-[:FRIEND_OF]->(p2:Person)-[:LIKES]->(r:Restaurant)
RETURN p2.name AS barat, r.name AS etterem
```

### Legrövidebb út
```cypher
MATCH path = shortestPath(
    (:Person {name: 'john'})-[:FRIEND_OF*]->(cel:Person)
)
WHERE cel.name <> 'john'
RETURN cel.name, length(path) AS lepesek, [n IN nodes(path) | n.name] AS utvonal
```

## 13.6 Haladó minták

### Változó hosszú bejárás
```cypher
-- Johntól max. 2 lépésen belül elérhető személyek
MATCH (:Person {name: 'john'})-[:FRIEND_OF*1..2]->(p:Person)
RETURN DISTINCT p.name
```

### Aggregáció
```cypher
MATCH (p:Person)-[l:LIKES]->(r:Restaurant)
RETURN r.name, avg(l.rating) AS atlag_ertekeles, count(p) AS rajongok
ORDER BY rajongok DESC
```

## 13.7 Migráció SQL Serverből

A `cypher/02_migrate_from_sql.py` a 8. fejezet gráftábláit olvassa SQL Serverből `pyodbc`-vel, és a hivatalos Python driver segítségével írja Neo4j-ba:

```python
from neo4j import GraphDatabase
driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "jelszo"))
with driver.session() as session:
    session.run("CREATE (:Person {id: $id, name: $name})", id=1, name='john')
```

2. A séma megtekintését (csomópont-labelek és kapcsolattípusok)
3. A 7 beépített szociálhálózati lekérdezés futtatását
4. Egyéni Cypher kód végrehajtását

## 13.9 Mikor válasszuk a Neo4j-t?

| Helyzet | SQL Server Graph | Neo4j |
|---|---|---|
| Gráfadat meglévő relációs DB-ben | ✓ Ideális | Túlzás |
| Mély bejárások (6+ lépés) | Lassú | Gyors (mutatóláncos tárolás) |
| Több gráfalgoritmus szükséges | Korlátozott | Teljes GDS könyvtár |
| Önálló gráf munkaterhelés | Működik | Ideális |
| Teljes ACID tranzakciók | ✓ | ✓ |
