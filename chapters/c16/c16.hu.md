## 16.1 Change Data Capture (CDC) Debeziummal

### 16.1.1 A polling problémája

A táblaváltozások naív detektálási módja az időszakos lekérdezés (polling):

```sql
-- Percenként futtatva: "mi változott az utolsó ellenőrzés óta?"
SELECT * FROM orders WHERE updated_at > @last_check_time
```

Ennek a megközelítésnek több súlyos hibája van:

| Probléma | Következmény |
|---|---|
| `updated_at` oszlopot igényel | A legtöbb örökölt séma nem rendelkezik ilyennel |
| Nem veszi észre a törléseket | A törölt sor nyom nélkül eltűnik |
| A polling intervallum = késési határ | 1 perces poll = akár 1 perc késés |
| Megterhelő az adatbázisnak | Sűrű, nagy szkennelések termelési terhelés alatt |
| Nem tranzakcionális | Egy közbülső kétszeres módosítás csak végállapotot mutat |

### 16.1.2 Mit csinál helyette a CDC?

Minden nagyobb adatbázismotor minden jóváhagyott változást egy tartós, rendezett **tranzakciós naplóba** ír, mielőtt alkalmazza azt. Ez a napló a hiteles rekord arról, hogy mi és milyen sorrendben történt. A CDC-eszközök közvetlenül ebből a naplóból olvasnak, ahelyett, hogy a táblát kérdeznék le.

```
Alkalmazás → SQL Server motor
                    ↓
       Tranzakciós napló (WAL / binlog)
                    ↓
         CDC-connector olvassa a naplót
                    ↓
         Downstream fogyasztók eseményeket kapnak
```

Előnyök:

- **Rögzíti az INSERT, UPDATE és DELETE műveleteket** — beleértve a kemény törléseket
- **Másodperc alatti késés** — a napló a commit pillanatában frissül
- **Nulla polling-terhelés** — a connector a napló végét követi, nincs táblaolvasás
- **Tranzakcionális sorrend** — az eseményeket commit sorrendben kapjuk

### 16.1.3 Debezium architektúra

A Debezium egy nyílt forráskódú CDC platform, amely az Apache Kafka Connect-re épül. Részei:

1. **Source connector** — adatbázis-specifikus plugin, amely olvassa a tranzakciós naplót
2. **Kafka topic** — tartós, rendezett változási eseményfolyam
3. **Sink connector** — a Kafka-ból eseményeket ír egy downstream rendszerbe (MongoDB, Elasticsearch, BigQuery stb.)

```
┌──────────────────┐    tranzakciós napló    ┌───────────────────────┐
│   SQL Server /   │ ───────────────────────► │  Debezium Source      │
│   PostgreSQL /   │                          │  Connector            │
│   MySQL          │                          └──────────┬────────────┘
└──────────────────┘                                     │ Kafka topic táblánként
                                                         ↓
                                             ┌───────────────────────┐
                                             │   Apache Kafka        │
                                             │   (üzenetközvetítő)   │
                                             └──────────┬────────────┘
                                                        │
                               ┌────────────────────────┼────────────────────────┐
                               ↓                        ↓                        ↓
                    ┌──────────────────┐   ┌─────────────────┐   ┌──────────────────┐
                    │  MongoDB Sink    │   │  Elastic Sink   │   │  BigQuery Sink   │
                    └──────────────────┘   └─────────────────┘   └──────────────────┘
```

### 16.1.4 CDC engedélyezése adatbázis-motoronként

**SQL Server**

A Microsoft SQL Server beépített CDC-támogatással rendelkezik. Engedélyezd adatbázis szinten, majd táblánként:

```sql
-- CDC engedélyezése a Northwind adatbázisban
USE Northwind;
EXEC sys.sp_cdc_enable_db;

-- CDC engedélyezése az Orders táblán
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name   = N'Orders',
    @role_name     = NULL;

-- Ellenőrzés
SELECT name, is_cdc_enabled FROM sys.databases WHERE name = 'Northwind';
```

Az SQL Server CDC a változási sorokat a `cdc` sémában lévő rendszertáblákba írja. A Debezium ezeket olvassa.

**PostgreSQL**

A PostgreSQL a write-ahead logot (WAL) logikai replikáción keresztül teszi elérhetővé. Állítsd be a `postgresql.conf`-ban:

```
wal_level = logical
max_replication_slots = 4
max_wal_senders = 4
```

A Debezium replikációs slotot hoz létre és abból fogyasztja a változásokat:

```sql
-- Ellenőrzés: logikai replikáció aktív-e?
SELECT slot_name, plugin, database FROM pg_replication_slots;
```

**MySQL**

A MySQL a bináris naplót (binlog) használja. Állítsd be a `my.cnf`-ben:

```ini
[mysqld]
server-id        = 1
log_bin          = mysql-bin
binlog_format    = ROW
binlog_row_image = FULL
expire_logs_days = 10
```

### 16.1.5 Debezium connector konfiguráció (SQL Server példa)

Regisztráld a Debezium SQL Server connectort a Kafka Connect REST API-n keresztül:

```json
{
  "name": "northwind-sqlserver-source",
  "config": {
    "connector.class": "io.debezium.connector.sqlserver.SqlServerConnector",
    "database.hostname": "sqlserver",
    "database.port": "1433",
    "database.user": "sa",
    "database.password": "YourStrong!Passw0rd",
    "database.dbname": "Northwind",
    "database.server.name": "northwind",
    "table.include.list": "dbo.Orders,dbo.Order_Details,dbo.Products",
    "database.history.kafka.bootstrap.servers": "kafka:9092",
    "database.history.kafka.topic": "schema-changes.northwind",
    "decimal.handling.mode": "double",
    "include.schema.changes": "true"
  }
}
```

Telepítés:

```bash
curl -X POST http://localhost:8083/connectors \
  -H "Content-Type: application/json" \
  -d @northwind-sqlserver-source.json
```

A Debezium mostantól minden `INSERT`, `UPDATE` és `DELETE` műveletet közzétesz az `Orders`, `Order_Details` és `Products` táblákon a Kafka `northwind.dbo.Orders` stb. topicokba.

### 16.1.6 CDC vs eseménynapló-megközelítés (2. fejezet visszatekintés)

A 2. fejezetben manuálisan kezelt `OrderLog` táblát és triggereket használtunk:

```sql
-- 2. fejezetes megközelítés: manuális eseménynapló trigger segítségével
CREATE TRIGGER trg_orders_after_insert
ON dbo.Orders AFTER INSERT
AS BEGIN
    INSERT INTO OrderLog (OrderID, Action, ChangedAt)
    SELECT i.OrderID, 'INSERT', GETDATE() FROM inserted i
END
```

| Szempont | 2. fejezetes trigger-napló | CDC (Debezium) |
|---|---|---|
| Beállítási nehézség | Alacsony (csak SQL) | Magasabb (Kafka infrastruktúra) |
| Törlések rögzítése | Csak ha a trigger lefedi | Igen, mindig |
| Sémaváltozások | Frissíteni kell a triggert | Automatikus |
| Teljesítményre hatás | Trigger a tranzakcióban fut | Aszinkron, naplót olvas |
| Régi események visszajátszása | Nem (a napló csak append-only tábla) | Igen (Kafka megőrzés) |
| Több fogyasztó | A naplótáblát kell pollozni | Natív fanout |
| Termelési alkalmasság | Egyszerű audithoz jó | Streaming pipeline-okhoz jobb |

Használj trigger-megközelítést egyszerű auditáláshoz, egyetlen alkalmazáson belül. Használj Debeziumot, ha több downstream rendszerbe kell megbízhatóan és nagy méretben streamelni a változásokat.

## 16.2 Apache Kafka mint üzenetközvetítő

### 16.2.1 Alapfogalmak

Az Apache Kafka egy elosztott, tartós, rendezett üzenetnapló. Legfontosabb elemei:

| Fogalom | Leírás |
|---|---|
| **Producer** | Üzeneteket tesz közzé egy topicba |
| **Topic** | Elnevezett, rendezett, tartós üzenetfolyam |
| **Partition** | A topic partíciókra van osztva a párhuzamosság érdekében; a sorrendet partíción belül garantálják |
| **Consumer** | Topicból olvas üzeneteket, **offset**-tel követi a pozícióját |
| **Consumer group** | Topicot megosztó több consumer; minden partíciót pontosan egy consumer kap a csoportban |
| **Offset** | Üzenet egész indexe egy partíción belül; a consumerek commitálják az offsetet az előrehaladás jelzésére |
| **Broker** | Kafka szervercsomópont, amely partíciókat tárol |

### 16.2.2 Miért Kafka a közvetlen adatbázis-szinkron helyett?

A közvetlen szinkronizálás (pl. egy szkript, amely SQL Serverből olvas és Elasticsearch-be ír) szorosan összekapcsolja a két rendszert:

```
SQL Server → [szinkron szkript] → Elasticsearch
```

Problémák:
- Ha az Elasticsearch nem elérhető, a szkriptnek pufferelnie vagy hibásodnia kell
- Csak egy fogyasztó: MongoDB hozzáadása külön, a forrást újra olvasó szkriptet igényel
- Nincs visszajátszás: ha az Elasticsearch-index megsérül, nem lehet újra előállítani a forrás DB újraolvasása nélkül
- A nyomásszabályozás nem terjed: a lassú Elasticsearch import túlterhelheti a szinkron szkriptet

Kafka mindezt megoldja:

```
SQL Server → [Debezium] → Kafka topic → [A fogyasztó] → Elasticsearch
                                       → [B fogyasztó] → MongoDB
                                       → [C fogyasztó] → BigQuery
```

Minden fogyasztó saját tempójában olvas függetlenül. A Kafka konfigurálható ideig (pl. 7 napig) megtartja az üzeneteket. Egy fogyasztó visszaállítható 0-s offsetre a teljes előzmény visszajátszásához.

### 16.2.3 Northwind pipeline: Orders MongoDB-be és Elasticsearch-be

**Felhasználási eset:**
- A Northwind OLTP adatbázis (SQL Server) a rekordrendszer
- Az analitikai csapat MongoDB kollekcióra van szüksége rugalmas dokumentumlekérdezésekhez
- A keresési csapat Elasticsearch indexet igényel a rendelések teljes szöveges kereséshez

**Architektúra:**

```
┌─────────────────────────────────────────────────────────────┐
│                    Northwind Pipeline                        │
│                                                             │
│  SQL Server ──[Debezium]──► Kafka: northwind.dbo.Orders     │
│                                        │                    │
│                         ┌─────────────┴──────────────┐     │
│                         ▼                            ▼     │
│              MongoDB Sink Connector       Elasticsearch     │
│              (northwind.orders           Sink Connector     │
│               kollekció)                (orders-idx)       │
└─────────────────────────────────────────────────────────────┘
```

**Topic megvizsgálása (Kafka CLI-vel):**

```bash
# Topicok listázása
kafka-topics.sh --bootstrap-server localhost:9092 --list

# Orders topic követése
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic northwind.dbo.Orders \
  --from-beginning \
  --max-messages 5 | jq .
```

Egy Debezium változási esemény INSERT esetén:

```json
{
  "op": "c",
  "ts_ms": 1714000000000,
  "before": null,
  "after": {
    "OrderID": 10248,
    "CustomerID": "VINET",
    "EmployeeID": 5,
    "OrderDate": "1996-07-04",
    "ShipCountry": "France"
  }
}
```

Az `op` mező értékei: `c` = létrehozás (INSERT), `u` = frissítés, `d` = törlés, `r` = olvasás (kezdeti pillanatkép).

### 16.2.4 Consumer group-ok és offset-kezelés

```bash
# Consumer group-ok listázása
kafka-consumer-groups.sh --bootstrap-server localhost:9092 --list

# Lemaradás (lag) megjelenítése
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group mongodb-sink-group
```

Ha a MongoDB sink lemarad (nagy lag), a Kafka természetes módon alkalmaz **nyomásszabályozást**: az üzenetek a topicban halmozódnak (a megőrzési határon belül), és a MongoDB sink saját tempójában zárkózik fel, anélkül, hogy befolyásolná az SQL Server forrást vagy az Elasticsearch sinket.

## 16.3 Adatbázis CI/CD Flywayval

### 16.3.1 A manuális SQL szkriptek problémája

Sémamigrációs eszköz nélkül az adatbázis-telepítések így néznek ki:

1. Egy fejlesztő megírja: `ALTER TABLE Orders ADD COLUMN priority INT`
2. Manuálisan futtatja termelésben — és elfelejti dokumentálni, melyik környezetben van
3. Egy kolléga ugyanezt a szkriptet futtatja staging-en — hiba: az oszlop már létezik
4. Visszavonásra van szükség — senki sem tudja, melyik környezet milyen állapotban van

Ez egyenértékű az alkalmazáskód verziókezelés nélküli telepítésével.

### 16.3.2 Flyway migrációs elnevezési konvenció

A Flyway szigorú elnevezési konvenciót követ a migrációs szkriptekhez:

```
V{verzió}__{leírás}.sql
```

Szabályok:
- `V` előtag (nagybetű) verzionált migrációkhoz
- Dupla aláhúzás `__` elválasztó
- A leírásban aláhúzás helyettesíti a szóközöket
- A fájlok **egyszer** kerülnek alkalmazásra, növekvő verziósorrendben
- Az alkalmazott migrációkat a `flyway_schema_history` tábla rögzíti

Példák:

```
V1__create_orders_table.sql
V2__add_priority_column_to_orders.sql
V3__create_index_orders_customer.sql
V4__insert_seed_data.sql
```

### 16.3.3 Flyway migrációk példái a Northwindhez

**V1__create_orders_table.sql**

```sql
CREATE TABLE IF NOT EXISTS orders (
    OrderID      INT          PRIMARY KEY,
    CustomerID   CHAR(5)      NOT NULL,
    EmployeeID   INT,
    OrderDate    DATE,
    ShipCountry  VARCHAR(15),
    Freight      DECIMAL(10,2) DEFAULT 0
);
```

**V2__add_priority_column_to_orders.sql**

```sql
ALTER TABLE orders ADD COLUMN priority INT NOT NULL DEFAULT 0;

COMMENT ON COLUMN orders.priority IS
    '0=normál, 1=magas, 2=sürgős — SLA-követéshez hozzáadva';
```

**V3__create_index_orders_customer.sql**

```sql
CREATE INDEX IF NOT EXISTS idx_orders_customer
    ON orders (CustomerID, OrderDate DESC);
```

**V4__add_audit_columns.sql**

```sql
ALTER TABLE orders ADD COLUMN created_at  TIMESTAMP DEFAULT NOW();
ALTER TABLE orders ADD COLUMN updated_at  TIMESTAMP DEFAULT NOW();

-- Trigger az updated_at automatikus frissítéséhez PostgreSQL-en
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER orders_set_updated_at
BEFORE UPDATE ON orders
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

### 16.3.5 Flyway vs ad-hoc ALTER TABLE

| Szempont | Ad-hoc ALTER TABLE | Flyway |
|---|---|---|
| Verziókezelés | Nincs | Minden migráció egy Git-fájl |
| Reprodukálhatóság | A környezetek eltérnek egymástól | Minden környezet azonos migrációkat kap |
| Visszavonás | Manuális, hibalehetőséges | Visszavonó migráció vagy kompenzáló V{n} |
| CI/CD integráció | Nem automatizálható biztonságosan | `flyway migrate` a pipeline-ban |
| Naplózás | Nincs | `flyway_schema_history` tábla |
| Ütközésdetektálás | Futtatod és reméled | Ellenőrzőösszeg-eltérés gyorsan meghibásodik |

## 16.4 Az ACID tesztlabor

### 16.4.1 Mit csinál a szkript?

Az `scripts/acid_test.py` szkript 10 bankszámlát hoz létre, mindegyiken 1 000 egység egyenleggel (összesen: 10 000). Ezután 20 párhuzamos szálat indít. Minden szál:

1. Két véletlenszerű számlaazonosítót választ
2. Elindít egy adatbázis-tranzakciót
3. Beolvassa a forrásszámla egyenlegét
4. 0–50 ms-t vár a feldolgozás szimulálásához és a versenyhelyzetek felerősítéséhez
5. Véletlenszerű összeget (1–100) utal át a forrásról a célra, ha van elég egyenleg
6. Commitálja a tranzakciót

Az alapvető **ACID-invariáns** az, hogy a teljes egyenlegnek mindig 10 000-nek kell lennie, párhuzamosságtól függetlenül.


### 16.4.3 Mit kell megfigyelni?

**Alapértelmezett izolációval (SERIALIZABLE / READ COMMITTED):**

A teljes egyenleg mindig 10 000. Az adatbázismotor biztosítja, hogy a párhuzamos olvasások és írások megfelelően sorba rendezve legyenek, akár zárolással (pesszimista), akár ütközésdetektálással és visszavonással (optimista, mint a PostgreSQL SERIALIZABLE esetén).

**READ UNCOMMITTED izolációval:**

Egy szál beolvashat olyan egyenleget, amelyet egy másik, még nem commitált tranzakció módosított — ez **piszkos olvasás**. Ez vezethet ahhoz, hogy:

- Az A szál beolvassa az egyenleget=500, dönt a 200 egységes átutalásról
- A B szál (még nem commitált) már levonta a 300-at ugyanarról a számláról
- Az A szál a nem commitált egyenleget látja és egy soha nem létező értéken alapuló döntést hoz
- A B szál visszavon — a számlaegyenleg visszaáll, de az A szál döntése egy fiktív értéken alapult

A gyakorlatban a READ UNCOMMITTED egy átutalási munkaterhelésen olyan végső összeget produkálhat, amely nem egyenlő 10 000-rel, demonstrálva az **izolációs szabálysértést**.

Futtasd az összehasonlítást:

```bash
# PASS: megfelelő izoláció
python3 scripts/acid_test.py --db sqlite

# READ_UNCOMMITTED megfigyelése (az SQLite figyelmen kívül hagyja ezt írásokra,
# de PostgreSQL/MySQL esetén piszkos olvasások jelennek meg)
python3 scripts/acid_test.py --db postgresql --isolation READ_UNCOMMITTED
```

### 16.4.4 Az izolációs szintek létrája

| Izolációs szint | Piszkos olvasás | Nem megismételhető olvasás | Fantomolvasás |
|---|---|---|---|
| READ UNCOMMITTED | Lehetséges | Lehetséges | Lehetséges |
| READ COMMITTED | Megakadályozott | Lehetséges | Lehetséges |
| REPEATABLE READ | Megakadályozott | Megakadályozott | Lehetséges |
| SERIALIZABLE | Megakadályozott | Megakadályozott | Megakadályozott |

Minél magasabb az izolációs szint, annál több koordinációt igényel az adatbázis a párhuzamos tranzakciók között — ami növeli a késést és a zárolási versenyt. A legtöbb termelési rendszer READ COMMITTED (PostgreSQL alapértelmezés) vagy REPEATABLE READ (MySQL InnoDB alapértelmezés) izolációs szintet használ egyensúlyként.

## 16.5 Feladatok

**1. feladat — CDC engedélyezése SQL Serveren**

Engedélyezd a CDC-t a helyi Northwind adatbázison, és ellenőrizd, hogy változási sorok jelennek-e meg a CDC táblákban egy rendelés beszúrása után:

```sql
EXEC sys.sp_cdc_enable_db;
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name   = N'Orders',
    @role_name     = NULL;

-- Engedélyezés után szúrj be egy teszt-rendelést, majd kérdezd le:
SELECT * FROM cdc.dbo_Orders_CT ORDER BY __$start_lsn DESC;
```

**2. feladat — Kafka topic megvizsgálása**

Ha fut egy helyi Kafka cluster Debeziummal, fogyassz 10 üzenetet az orders topicból, és azonosítsd az `op` mezőt minden eseménynél:

```bash
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic northwind.dbo.Orders \
  --from-beginning \
  --max-messages 10 | python3 -c "
import sys, json
for line in sys.stdin:
    try:
        msg = json.loads(line)
        print(f\"op={msg.get('op','?')}  OrderID={msg.get('after',{}).get('OrderID','?')}\")
    except:
        pass
"
```

**3. feladat — Flyway migráció**

Hozz létre egy `migrations/` könyvtárat, és írj három verzionált SQL fájlt. Alkalmazd őket Flywayval, és ellenőrizd a `flyway_schema_history` táblát:

```bash
mkdir -p migrations
# Hozd létre a V1__create_test_table.sql, V2__add_column.sql, V3__create_index.sql fájlokat
flyway -url=jdbc:postgresql://localhost:5432/northwind -user=postgres migrate
flyway -url=jdbc:postgresql://localhost:5432/northwind -user=postgres info
```

**4. feladat — ACID teszt: izolációs szintek**

Futtasd az `acid_test.py`-t háromszor PostgreSQL-lel különböző izolációs szintekkel, és rögzítsd az eredményeket:

```bash
for level in READ_COMMITTED REPEATABLE_READ SERIALIZABLE; do
    echo "=== $level ==="
    python3 scripts/acid_test.py \
        --db postgresql --db-name northwind \
        --user postgres --password secret \
        --threads 50 --isolation $level
done
```

Hasonlítsd össze: melyik szintek produkálnak mindig pontosan 10 000-es végösszeget? Melyek a leggyorsabbak? Ábrázold az eltelt időt az izolációs szint függvényében.

**5. feladat — Elveszett frissítések bemutatása**

Futtasd a párhuzamos insert versenyhelyzet szkriptet, és figyeld meg az elveszett frissítések hatását:

```bash
# Alapértelmezett: 100 szál, nincs mesterséges késleltetés
python3 scripts/concurrent_insert_race.py

# A versenyablak felerősítése 10 ms késleltetéssel a READ és WRITE között
python3 scripts/concurrent_insert_race.py --delay-ms 10 --runs 5
```

Magyarázd el saját szavaiddal: miért produkál az UNSAFE futtatás 100-nál (vagy 1000 szálas esetén 1000-nél) kisebb értéket? Rajzold fel az elveszett frissítést okozó átlapolást.
