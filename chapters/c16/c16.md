## 16.1 Change Data Capture (CDC) with Debezium

### 16.1.1 The Polling Problem

The naïve way to detect changes in a database table is to poll it periodically:

```sql
-- Run every minute: "what changed since I last checked?"
SELECT * FROM orders WHERE updated_at > @last_check_time
```

This approach has several fatal flaws:

| Problem | Consequence |
|---|---|
| Requires an `updated_at` column | Most legacy schemas do not have one |
| Misses hard-deletes | A deleted row leaves no trace |
| Polling interval = latency floor | 1-minute poll = up to 1 minute of lag |
| Hammers the DB | Frequent large scans under production load |
| Not transactional | A row updated twice between polls shows only the final state |

### 16.1.2 What CDC Does Instead

Every major database engine writes every committed change into a persistent, ordered **transaction log** before it is applied. The log is the authoritative record of what happened and in what order. CDC tools tap into this log directly, instead of polling the table.

```
Application → SQL Server engine
                    ↓
          Transaction Log (WAL / binlog)
                    ↓
           CDC connector reads log
                    ↓
          Downstream consumers get events
```

Benefits:

- **Captures inserts, updates, and deletes** — even hard deletes
- **Sub-second latency** — the log is updated at commit time
- **Zero polling overhead** — the connector follows the log tail, no table scans
- **Transactional ordering** — events arrive in commit order, preserving causality

### 16.1.3 Debezium Architecture

Debezium is an open-source CDC platform built on top of Apache Kafka Connect. It consists of:

1. **Source connector** — database-specific plugin that reads the transaction log
2. **Kafka topic** — durable, ordered stream of change events
3. **Sink connector** — writes events from Kafka to a downstream system (MongoDB, Elasticsearch, BigQuery, etc.)

```
┌──────────────────┐    transaction log    ┌───────────────────────┐
│   SQL Server /   │ ──────────────────→   │  Debezium Source      │
│   PostgreSQL /   │                       │  Connector            │
│   MySQL          │                       └──────────┬────────────┘
└──────────────────┘                                  │ Kafka topic per table
                                                      ↓
                                          ┌───────────────────────┐
                                          │   Apache Kafka        │
                                          │   (message broker)    │
                                          └──────────┬────────────┘
                                                     │
                              ┌──────────────────────┼─────────────────────┐
                              ↓                      ↓                     ↓
                   ┌──────────────────┐  ┌─────────────────┐  ┌──────────────────┐
                   │  MongoDB Sink    │  │ Elastic Sink    │  │  BigQuery Sink   │
                   └──────────────────┘  └─────────────────┘  └──────────────────┘
```

### 16.1.4 Enabling CDC Per Database Engine

**SQL Server**

Microsoft SQL Server has built-in CDC support. Enable it at the database level, then per table:

```sql
-- Enable CDC on the Northwind database
USE Northwind;
EXEC sys.sp_cdc_enable_db;

-- Enable CDC on the Orders table
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name   = N'Orders',
    @role_name     = NULL;

-- Verify
SELECT name, is_cdc_enabled FROM sys.databases WHERE name = 'Northwind';
```

SQL Server CDC writes change rows to system tables in the `cdc` schema. Debezium reads these instead of the transaction log directly (which requires special permissions).

**PostgreSQL**

PostgreSQL exposes its write-ahead log (WAL) via logical replication. Set `wal_level = logical` in `postgresql.conf`:

```
wal_level = logical
max_replication_slots = 4
max_wal_senders = 4
```

Debezium creates a replication slot and consumes changes from it:

```sql
-- Verify logical replication is active
SELECT slot_name, plugin, database FROM pg_replication_slots;
```

**MySQL**

MySQL uses the binary log (`binlog`). Set `binlog_format = ROW` in `my.cnf`:

```ini
[mysqld]
server-id        = 1
log_bin          = mysql-bin
binlog_format    = ROW
binlog_row_image = FULL
expire_logs_days = 10
```

### 16.1.5 Debezium Connector Configuration (SQL Server Example)

Register the Debezium SQL Server connector via the Kafka Connect REST API:

```json
{
  "name": "northwind-sqlserver-source",
  "config": {
    "connector.class": "io.debezium.connector.sqlserver.SqlServerConnector",
    "database.hostname": "sqlserver",
    "database.port": "1433",
    "database.user": "sa",
    "database.password": "YourStrong!Passw0rd",
    "database.dbname": "Northwind",
    "database.server.name": "northwind",
    "table.include.list": "dbo.Orders,dbo.Order_Details,dbo.Products",
    "database.history.kafka.bootstrap.servers": "kafka:9092",
    "database.history.kafka.topic": "schema-changes.northwind",
    "decimal.handling.mode": "double",
    "include.schema.changes": "true"
  }
}
```

Deploy it:

```bash
curl -X POST http://localhost:8083/connectors \
  -H "Content-Type: application/json" \
  -d @northwind-sqlserver-source.json
```

Debezium will now publish every `INSERT`, `UPDATE`, and `DELETE` on `Orders`, `Order_Details`, and `Products` to Kafka topics named `northwind.dbo.Orders`, etc.

### 16.1.6 CDC vs the Event Log Approach (Chapter 2 Revisited)

In Chapter 2 we built an `OrderLog` table and used triggers to write events manually:

```sql
-- Chapter 2 approach: manual event log via trigger
CREATE TRIGGER trg_orders_after_insert
ON dbo.Orders AFTER INSERT
AS BEGIN
    INSERT INTO OrderLog (OrderID, Action, ChangedAt)
    SELECT i.OrderID, 'INSERT', GETDATE() FROM inserted i
END
```

| Dimension | Chapter 2 Trigger Log | CDC (Debezium) |
|---|---|---|
| Setup effort | Low (just SQL) | Higher (Kafka infra) |
| Captures deletes | Only if trigger covers DELETE | Yes, always |
| Schema changes | Must update trigger | Automatic |
| Performance impact | Trigger runs in transaction | Async, reads log |
| Replay old events | No (log is append-only table) | Yes (Kafka retention) |
| Multiple consumers | Requires polling the log table | Native fan-out |
| Production readiness | Good for simple audit | Better for streaming pipelines |

Use the trigger approach for simple auditing inside a single application. Use Debezium when you need to stream changes to multiple downstream systems reliably at scale.

## 16.2 Apache Kafka as Message Broker

### 16.2.1 Core Concepts

Apache Kafka is a distributed, durable, ordered message log. Its key primitives:

| Concept | Description |
|---|---|
| **Producer** | Publishes messages to a topic |
| **Topic** | Named, ordered, durable stream of messages |
| **Partition** | A topic is split into partitions for parallelism; ordering is guaranteed within a partition |
| **Consumer** | Reads messages from a topic, tracking its position with an **offset** |
| **Consumer group** | Multiple consumers sharing a topic; each partition is assigned to exactly one consumer in the group |
| **Offset** | Integer index of a message within a partition; consumers commit offsets to mark progress |
| **Broker** | A Kafka server node that stores partitions |

### 16.2.2 Why Kafka Over Direct DB-to-DB Sync

Direct synchronization (e.g., a script that reads from SQL Server and writes to Elasticsearch) couples the two systems tightly:

```
SQL Server → [sync script] → Elasticsearch
```

Problems:
- If Elasticsearch is down, the script must buffer or fail
- Only one consumer: adding MongoDB requires a separate script reading the source again
- No replay: if the Elasticsearch index is corrupted, you cannot re-derive it without re-reading the source DB
- Backpressure does not propagate: a slow Elasticsearch import can overload the sync script

Kafka solves all of these:

```
SQL Server → [Debezium] → Kafka topic → [consumer A] → Elasticsearch
                                      → [consumer B] → MongoDB
                                      → [consumer C] → BigQuery
```

Each consumer reads independently at its own pace. Kafka retains messages for a configurable period (e.g., 7 days). A consumer can be reset to offset 0 to replay all history.

### 16.2.3 Northwind Pipeline: Orders to MongoDB and Elasticsearch

**Use case:**
- The Northwind OLTP database (SQL Server) is the system of record
- The analytics team needs a MongoDB collection for flexible document queries
- The search team needs an Elasticsearch index for full-text order search

**Architecture:**

```
┌─────────────────────────────────────────────────────────────┐
│                     Northwind Pipeline                       │
│                                                             │
│  SQL Server ──[Debezium]──► Kafka: northwind.dbo.Orders     │
│                                        │                    │
│                         ┌─────────────┴──────────────┐     │
│                         ▼                            ▼     │
│              MongoDB Sink Connector       Elasticsearch     │
│              (northwind.orders           Sink Connector     │
│               collection)               (orders-idx)       │
└─────────────────────────────────────────────────────────────┘
```

**Inspect the topic (using Kafka CLI):**

```bash
# List topics
kafka-topics.sh --bootstrap-server localhost:9092 --list

# Tail the orders topic
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic northwind.dbo.Orders \
  --from-beginning \
  --max-messages 5 | jq .
```

A Debezium change event for an INSERT looks like:

```json
{
  "op": "c",
  "ts_ms": 1714000000000,
  "before": null,
  "after": {
    "OrderID": 10248,
    "CustomerID": "VINET",
    "EmployeeID": 5,
    "OrderDate": "1996-07-04",
    "ShipCountry": "France"
  }
}
```

The `op` field values: `c` = create (INSERT), `u` = update, `d` = delete, `r` = read (initial snapshot).

### 16.2.4 Consumer Groups and Offset Management

```bash
# List consumer groups
kafka-consumer-groups.sh --bootstrap-server localhost:9092 --list

# Show lag (how far behind a consumer is)
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group mongodb-sink-group
```

If the MongoDB sink falls behind (high lag), Kafka applies **backpressure** naturally: the messages pile up in the topic (within retention limits) and the MongoDB sink catches up at its own pace, without affecting the SQL Server source or the Elasticsearch sink.

## 16.3 Database CI/CD with Flyway

### 16.3.1 The Problem with Manual SQL Scripts

Without a schema migration tool, database deployments look like this:

1. A developer writes `ALTER TABLE Orders ADD COLUMN priority INT`
2. They run it manually in production — and forget to document which environment has it
3. A colleague runs the same script on staging — error: column already exists
4. A rollback is needed — no one knows what state each environment is in

This is the equivalent of deploying application code without version control.

### 16.3.2 Flyway Migration Naming Convention

Flyway enforces a strict naming convention for migration scripts:

```
V{version}__{description}.sql
```

Rules:
- `V` prefix (uppercase) for versioned migrations
- Double underscore `__` separator
- Description uses underscores for spaces
- Files are applied **once**, in ascending version order
- Applied migrations are recorded in a `flyway_schema_history` table

Examples:

```
V1__create_orders_table.sql
V2__add_priority_column_to_orders.sql
V3__create_index_orders_customer.sql
V4__insert_seed_data.sql
```

### 16.3.3 Example Flyway Migrations for Northwind

**V1__create_orders_table.sql**

```sql
CREATE TABLE IF NOT EXISTS orders (
    OrderID      INT          PRIMARY KEY,
    CustomerID   CHAR(5)      NOT NULL,
    EmployeeID   INT,
    OrderDate    DATE,
    ShipCountry  VARCHAR(15),
    Freight      DECIMAL(10,2) DEFAULT 0
);
```

**V2__add_priority_column_to_orders.sql**

```sql
ALTER TABLE orders ADD COLUMN priority INT NOT NULL DEFAULT 0;

COMMENT ON COLUMN orders.priority IS
    '0=normal, 1=high, 2=urgent — added for SLA tracking';
```

**V3__create_index_orders_customer.sql**

```sql
CREATE INDEX IF NOT EXISTS idx_orders_customer
    ON orders (CustomerID, OrderDate DESC);
```

**V4__add_audit_columns.sql**

```sql
ALTER TABLE orders ADD COLUMN created_at  TIMESTAMP DEFAULT NOW();
ALTER TABLE orders ADD COLUMN updated_at  TIMESTAMP DEFAULT NOW();

-- Trigger to auto-update updated_at on PostgreSQL
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER orders_set_updated_at
BEFORE UPDATE ON orders
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

### 16.3.4 Running Flyway

**Command line (SQL Server):**

```bash

  -url="jdbc:sqlserver://localhost:1433;databaseName=Northwind" \
  -user=sa \
  -password=YourStrong!Passw0rd \

```

**Check migration status:**


### 16.3.5 Flyway vs Ad-Hoc ALTER TABLE

| Dimension | Ad-hoc ALTER TABLE | Flyway |
|---|---|---|
| Version control | None | Every migration is a file in Git |
| Reproducibility | Different environments drift | All environments get identical migrations |
| Rollback | Manual, error-prone | Undo migrations (Flyway Teams) or compensating V{n} |
| CI/CD integration | Impossible to automate safely | `flyway migrate` in the pipeline |
| Audit trail | Non-existent | `flyway_schema_history` table |
| Conflict detection | Run it and hope | Checksum mismatch fails fast |

## 16.4 The ACID Test Lab

### 16.4.1 What the Script Does

The `scripts/acid_test.py` script creates 10 bank accounts, each with a balance of 1,000 currency units (total: 10,000). It then spawns 20 concurrent threads. Each thread:

1. Picks two random account IDs
2. Begins a database transaction
3. Reads the source account balance
4. Sleeps 0–50 ms to simulate processing and amplify race conditions
5. Transfers a random amount (1–100) from source to destination if funds are sufficient
6. Commits the transaction

The fundamental **ACID invariant** is that the total balance must always equal 10,000, regardless of concurrency.

### 16.4.2 Running the Lab

**Default run (SQLite, SERIALIZABLE):**

```bash
cd northwind-control-center
python3 scripts/acid_test.py
```

Expected output:

```

  ACID Concurrent Bank Transfer Test
============================================================
  Backend  : SQLITE
  DB file  : /tmp/acid_test.db
  Threads  : 20
  Accounts : 10  x  balance 1000

[1/3] Setting up schema ...
  Schema ready: 10 accounts x 1000 = 10000 total

[2/3] Spawning 20 concurrent transfer threads ...
  All threads finished in 0.84s

[3/3] Verifying total balance ...

============================================================
  ACID Test Results
============================================================
  Database        : SQLITE
  Threads         : 20
  Accounts        : 10
  Initial total   : 10000
  Elapsed         : 0.84s

  Successful xfers: 15
  Insuf. funds    : 3
  Failures        : 2
  Deadlock retries: 4

  Final total bal : 10000
  Expected total  : 10000

  Verdict         : PASS
============================================================
```

**PostgreSQL run with READ UNCOMMITTED:**

```bash
python3 scripts/acid_test.py \
  --db postgresql \
  --host localhost \
  --db-name northwind \
  --user postgres \
  --password secret \
  --isolation READ_UNCOMMITTED
```

### 16.4.3 What to Observe

**With default isolation (SERIALIZABLE / READ COMMITTED):**

The total balance is always 10,000. The database engine ensures that concurrent reads and writes are properly ordered, either by locking (pessimistic) or by detecting conflicts and aborting (optimistic, as in PostgreSQL's SERIALIZABLE).

**With READ UNCOMMITTED:**

A thread can read a balance that has been modified by another transaction that has not yet committed — a **dirty read**. This can lead to:

- Thread A reads balance=500, decides to transfer 200
- Thread B (not yet committed) has subtracted 300 from the same account
- Thread A sees the uncommitted balance and makes a decision based on phantom data
- Thread B rolls back — the account balance is restored, but Thread A's decision was based on a value that never existed

In practice, READ UNCOMMITTED on a transfer workload can produce a final total that does not equal 10,000, demonstrating an **isolation violation**.

Run the comparison:

```bash
# PASS: proper isolation
python3 scripts/acid_test.py --db sqlite

# Observe READ_UNCOMMITTED behavior (SQLite ignores this for writes,
# but PostgreSQL/MySQL will show dirty reads)
python3 scripts/acid_test.py --db postgresql --isolation READ_UNCOMMITTED
```

### 16.4.4 The Isolation Level Ladder

| Isolation Level | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| READ UNCOMMITTED | Possible | Possible | Possible |
| READ COMMITTED | Prevented | Possible | Possible |
| REPEATABLE READ | Prevented | Prevented | Possible |
| SERIALIZABLE | Prevented | Prevented | Prevented |

The higher the isolation level, the more the database must coordinate between concurrent transactions — which increases latency and lock contention. Most production systems use READ COMMITTED (PostgreSQL default) or REPEATABLE READ (MySQL InnoDB default) as a balance.
