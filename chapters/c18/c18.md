# Chapter 18 — Cloud Database Ecosystems: GCP, AWS, and Azure

The era of renting rack space and hand-patching database servers is over for most teams. Managed cloud databases emerged because the operational burden of running a database — applying security patches, scheduling backups, resizing disks, failover testing — has little to do with the business problem a team is trying to solve. Cloud providers absorb that burden in exchange for a subscription fee and, in some cases, vendor lock-in. The key conceptual shift is from "run a database server" to "choose the right managed service for the workload."

Managed services offer four capabilities that are difficult or expensive to replicate on-premises:

- **No-DBA patching and backups.** The provider handles OS patches, engine upgrades (with maintenance windows), and automated backups with point-in-time recovery.
- **Elastic scaling.** Storage autogrows; compute can be resized with a few minutes of downtime (or none, on serverless tiers).
- **Global distribution.** Read replicas, multi-region writes, and automatic failover are first-class features, not hand-rolled Pacemaker configurations.
- **SLA-backed availability.** Providers guarantee 99.95%–99.999% uptime, with financial credits for violations. On-premises SLAs are only as good as the operations team's pager rotation.

This chapter surveys the major managed database services across Google Cloud Platform (GCP), Amazon Web Services (AWS), and Microsoft Azure, organized by workload type.

## 18.1 Managed Relational (OLTP)

The most common database workload is a standard relational database powering a web application: transactional writes, indexed reads, foreign-key constraints, and SQL joins. All three major clouds offer managed equivalents of the databases teams already know.

| Feature | GCP Cloud SQL | AWS RDS | Azure SQL Database |
|---|---|---|---|
| Engines | PostgreSQL, MySQL, SQL Server | PostgreSQL, MySQL, MariaDB, Oracle, SQL Server | SQL Server, PostgreSQL, MySQL |
| Max storage | 64 TB | 64 TB | 4 TB (General Purpose) |
| Read replicas | Yes | Yes | Active geo-replication |
| Auto backups | Yes (PITR 7–35 days) | Yes (automated snapshots) | Yes (automated) |
| Serverless option | No | Aurora Serverless v2 | Serverless tier |
| Best for | Standard web apps, GCP-native | AWS-first workloads | Azure-first workloads |

### Connection Pooling

A naive application opens one database connection per HTTP request. At 500 concurrent users, that is 500 connections — each consuming ~5 MB of RAM on the database server and a file descriptor. Managed databases are not immune to this problem.

- **PgBouncer** (GCP/AWS): a lightweight connection pooler that sits between the application and PostgreSQL. It maintains a small pool of long-lived server connections and multiplexes client connections onto them. Transaction-mode pooling allows the pool to be much smaller than the number of application threads.
- **RDS Proxy** (AWS): a fully managed connection pooler native to RDS and Aurora. It also integrates with AWS IAM for credential rotation without connection drops, which is useful in Lambda environments where connections are opened and closed frequently.

### Parameter Groups and Cloud SQL Flags

On-premises DBAs tune `postgresql.conf` directly. In managed services, engine parameters are controlled through:

- **RDS parameter groups**: a named set of engine parameters attached to an RDS instance or cluster. Changes to static parameters require a reboot; dynamic parameters apply immediately.
- **Cloud SQL flags**: per-instance flags set through the Cloud SQL API or Console. The same static/dynamic distinction applies.
- **Azure SQL**: some parameters are fixed by the service tier; others are exposed as server-level settings or database-scoped configurations.

### Maintenance Windows

Managed services apply patches and minor version upgrades during a configurable maintenance window — typically a 30-minute slot during off-peak hours. For production systems:

1. Set the window to the lowest-traffic period (e.g., Sunday 03:00–03:30 local time).
2. Enable notifications so on-call engineers are aware.
3. Test the next minor version on a staging replica before the window arrives.

## 18.2 Globally Distributed Relational

Standard managed relational databases are single-region at heart. Read replicas help with read scaling but writes still go to one primary. For truly global applications — where users on four continents write data concurrently — a new category of database is required.

| Feature | GCP Cloud Spanner | AWS Aurora Global | Azure Cosmos DB (SQL API) |
|---|---|---|---|
| Consistency | External (true linearizability) | Eventual / strongly consistent (per region) | Tunable (5 levels) |
| Multi-region writes | Yes | Primary + read replicas | Yes (multi-master) |
| SQL support | ANSI SQL with extensions | MySQL/PostgreSQL dialect | SQL API (subset) |
| CAP position | CP | CP | AP (default) / CP (strong) |
| Price model | Per node + I/O | Per ACU | Per RU (Request Unit) |

### Spanner TrueTime

Cloud Spanner achieves external consistency — a property stronger than serializability — without a global lock. The mechanism is TrueTime, a timekeeping API built on GPS receivers and atomic clocks deployed in every Google data center.

TrueTime exposes not a single timestamp but an interval `[earliest, latest]` that is guaranteed to contain the true wall-clock time. The uncertainty (typically 1–7 ms) is bounded because GPS and atomic clocks drift slowly and are cross-checked continuously.

When Spanner assigns a commit timestamp to a transaction, it waits out the TrueTime uncertainty interval before declaring the commit complete. This "commit wait" means that by the time a transaction is visible to readers, its timestamp is provably in the past — no other transaction with a later timestamp can have been committed first. External consistency follows without any inter-node locking.

If a GPS clock on a Spanner node drifts beyond the uncertainty bound, TrueTime widens the interval, which increases commit-wait latency. The system remains correct at the cost of throughput until the clock is resynchronized. In practice, Google reports that the 99th percentile uncertainty is under 7 ms.

### Aurora Global Database

Aurora Global consists of one primary region (read/write) and up to five secondary regions (read-only). Replication uses a dedicated log-shipping layer with typical lag under one second. In a disaster recovery scenario, a secondary region can be promoted to primary in under one minute. Aurora does not offer true multi-region writes in the way Spanner does; the primary region is still the single write authority.

### Cosmos DB Consistency Levels

Azure Cosmos DB offers five tunable consistency levels on a spectrum from strong to eventual:

1. **Strong** — linearizable, like Spanner. Reads always see the latest committed write.
2. **Bounded staleness** — reads lag by at most K versions or T seconds.
3. **Session** — within a single client session, reads are monotonic; strong consistency for the session, eventual for others.
4. **Consistent prefix** — reads never see out-of-order writes; no gaps, but may be stale.
5. **Eventual** — lowest latency, no ordering guarantee.

Most multi-region Cosmos DB deployments use Session consistency, which covers the common case (a user always sees their own writes) without the latency cost of Strong.

## 18.3 NoSQL Document Stores

Document databases store JSON-like objects with flexible schemas. They excel when the data model evolves rapidly, when objects have variable shapes, or when the application needs to retrieve an entire entity in a single read without joins.

| Feature | GCP Firestore | AWS DynamoDB | Azure Cosmos DB (NoSQL API) |
|---|---|---|---|
| Model | Document (JSON) | Key-value + document | Document (JSON) |
| Query | Collection queries | Primary key + GSI | SQL-like NoSQL query |
| Real-time sync | Yes (listeners) | No (DynamoDB Streams) | No |
| Serverless | Yes | Yes (on-demand) | Yes (serverless) |
| Offline sync | Yes (mobile SDK) | No | No |
| Best for | Mobile/web real-time apps | High-throughput K/V | Multi-model, Azure-native |

### Firestore Python SDK

```python
from google.cloud import firestore

db = firestore.Client(project="northwind-project")

# Add a document to the 'orders' collection
order_ref = db.collection("orders").document()
order_ref.set({
    "customer_id": "ALFKI",
    "order_date": firestore.SERVER_TIMESTAMP,
    "items": [
        {"product_id": 11, "unit_price": 14.0, "quantity": 12},
        {"product_id": 42, "unit_price": 9.8, "quantity": 10},
    ],
    "shipped": False,
})
print(f"Order written: {order_ref.id}")

# Real-time listener — fires on every change to this document
def on_snapshot(doc_snapshot, changes, read_time):
    for doc in doc_snapshot:
        print(f"Order status updated: {doc.to_dict()}")

order_ref.on_snapshot(on_snapshot)
# The listener runs in a background thread; keep the process alive
import time
time.sleep(30)
```

Real-time listeners are a differentiating feature of Firestore. A mobile app can subscribe to a document or collection query, and the SDK pushes updates over a persistent gRPC stream without the app polling. This makes Firestore the natural choice for chat applications, live dashboards, and collaborative editing where the server must push changes to clients.

### DynamoDB boto3

```python
import boto3
from boto3.dynamodb.conditions import Key

dynamodb = boto3.resource("dynamodb", region_name="us-east-1")
table = dynamodb.Table("NorthwindOrders")

# Write an item
table.put_item(Item={
    "CustomerId": "ALFKI",          # partition key
    "OrderId": "ORD-10248",         # sort key
    "OrderDate": "1996-07-04",
    "TotalAmount": 440.0,
    "Shipped": False,
})

# Query all orders for a customer
response = table.query(
    KeyConditionExpression=Key("CustomerId").eq("ALFKI")
)
for item in response["Items"]:
    print(item["OrderId"], item["TotalAmount"])

# Query with sort key condition (orders placed in 1996)
response = table.query(
    KeyConditionExpression=(
        Key("CustomerId").eq("ALFKI") &
        Key("OrderId").begins_with("ORD-1996")
    )
)
```

DynamoDB's data model requires choosing a partition key and an optional sort key upfront. All queries must include the partition key; sort key conditions and filters are secondary. Global Secondary Indexes (GSIs) allow querying on alternate key combinations, but each GSI is a separate copy of the data, charged separately.

## 18.4 Wide-Column / Time-Series

Wide-column databases store data in rows identified by a row key, with columns grouped into column families. Unlike relational tables, not every row needs the same columns. This model is optimal for time-series data (sensor readings, log events) where each row is a unique timestamp-keyed event and columns are metric names.

| Feature | GCP Bigtable | AWS Keyspaces (Cassandra) | Azure Cosmos DB (Cassandra API) |
|---|---|---|---|
| Model | Wide-column (proprietary) | Wide-column (CQL compatible) | Wide-column (CQL compatible) |
| Latency | < 10 ms p99 | < 10 ms p99 | < 10 ms p99 |
| Replication | Cross-region cluster | Multi-region via Cassandra | Global distribution |
| Min cost | ~$0.65/hr/node | Serverless (per RCU/WCU) | Serverless |
| Best for | IoT sensor streams, ML feature store | Existing Cassandra migration | Azure-native Cassandra |

### Bigtable Row Key Design

The row key is the only indexed field in Bigtable. Every read and write is keyed by row key, so its design determines whether load is evenly distributed across nodes (tablets) or concentrated on a single node (a hotspot).

**Anti-pattern — monotonic timestamp prefix:**

```
# BAD: all writes go to the same tablet at the leading edge of the keyspace
row_key = f"{timestamp_ms}#sensor_{sensor_id}"
# Example: "1714694400000#sensor_42"
```

Because Bigtable stores rows in lexicographic order by row key, a monotonically increasing timestamp prefix means all new writes land on the last tablet. That tablet becomes a hotspot while all others sit idle.

**Pattern 1 — Reverse timestamp:**

```
# GOOD: newest data sorts first; avoids leading-edge hotspot
MAX_TIMESTAMP = 9999999999999
row_key = f"{MAX_TIMESTAMP - timestamp_ms}#sensor_{sensor_id}"
```

This evenly distributes sequential writes across the keyspace because the reversed timestamp decreases monotonically, spreading load to the first tablet rather than the last. It has the added benefit that a scan from row 0 returns the most recent data first.

**Pattern 2 — Salted prefix:**

```python
import hashlib

def make_row_key(sensor_id: str, timestamp_ms: int, buckets: int = 100) -> str:
    salt = int(hashlib.md5(sensor_id.encode()).hexdigest(), 16) % buckets
    return f"{salt:03d}#{sensor_id}#{timestamp_ms}"
# Example: "042#sensor_42#1714694400000"
```

The salt (a hash of the sensor ID modulo the number of buckets) distributes sensors across 100 evenly spaced row key prefixes. This spreads write load across many tablets. The tradeoff: to retrieve all readings for a sensor, you must query all 100 buckets and merge the results.

## 18.5 Data Warehouses (OLAP)

Online Analytical Processing (OLAP) workloads are the opposite of OLTP: few writes, massive reads spanning billions of rows, aggregations over entire column ranges. Columnar storage is optimal here because a `SELECT SUM(amount)` query only reads the `amount` column bytes, skipping all other columns entirely.

| Feature | GCP BigQuery | AWS Redshift | Azure Synapse Analytics |
|---|---|---|---|
| Architecture | Serverless, columnar | Cluster-based (RA3), serverless option | Dedicated pool + serverless |
| Pricing | $5/TB scanned | ~$0.25/hr/node | Per DWU-hour |
| Partitioning | By date/range | Distribution keys | Hash / round-robin |
| ML integration | BigQuery ML (SQL) | Redshift ML (SageMaker) | Azure ML |
| Streaming ingest | Pub/Sub → BigQuery | Kinesis → Redshift | Event Hubs → Synapse |
| Best for | Ad-hoc analytics, petabyte scale | AWS-native analytics | Azure BI workloads |

### BigQuery ML

BigQuery ML allows training and scoring machine learning models using standard SQL syntax, without moving data out of the warehouse.

```sql
-- Train a linear regression model to predict order revenue
CREATE OR REPLACE MODEL northwind.revenue_model
OPTIONS (
    model_type = 'linear_reg',
    input_label_cols = ['total_revenue']
) AS
SELECT
    c.country,
    p.category_id,
    EXTRACT(MONTH FROM o.order_date) AS order_month,
    SUM(od.unit_price * od.quantity * (1 - od.discount)) AS total_revenue
FROM
    northwind.orders o
    JOIN northwind.order_details od ON o.order_id = od.order_id
    JOIN northwind.customers c ON o.customer_id = c.customer_id
    JOIN northwind.products p ON od.product_id = p.product_id
GROUP BY 1, 2, 3;

-- Score new data using the trained model
SELECT *
FROM ML.PREDICT(
    MODEL northwind.revenue_model,
    (
        SELECT 'Germany' AS country, 1 AS category_id, 12 AS order_month
    )
);
```

### Cost Optimization with Partition Filters

BigQuery charges per byte scanned. On a 500 GB orders table with no partition filter, a query costs approximately $2.50 (500 GB × $0.005/GB). Adding a partition filter that limits the scan to the current month (5 GB) reduces the cost to $0.025 — a 100× reduction.

```sql
-- Expensive: scans entire table (500 GB)
SELECT customer_id, SUM(freight) AS total_freight
FROM northwind.orders
GROUP BY customer_id;

-- Cheap: partition filter limits scan to ~5 GB
SELECT customer_id, SUM(freight) AS total_freight
FROM northwind.orders
WHERE order_date >= DATE_TRUNC(CURRENT_DATE(), MONTH)
GROUP BY customer_id;
```

Always check the estimated bytes scanned in the BigQuery UI before running ad-hoc queries on large tables.

## 18.6 Serverless and Auto-scaling

The serverless model eliminates capacity planning: there are no servers to provision, no instance sizes to choose, and no charges when the service is idle. The tradeoff is reduced control over performance characteristics and the risk of cold starts.

- **Firestore**: scales to zero when idle. Pricing is per document read ($0.06/100K), write ($0.18/100K), and delete ($0.02/100K), plus storage at $0.18/GB/month. A Northwind-scale app with 10,000 reads/day costs pennies.
- **DynamoDB on-demand**: pricing per Write Capacity Unit (WCU) and Read Capacity Unit (RCU) consumed. Auto-scales in milliseconds when traffic spikes. No idle charge. Suitable when traffic is spiky and unpredictable.
- **Aurora Serverless v2**: scales Aurora Capacity Units (ACUs) from 0.5 to 128 in increments of 0.5 ACU. Each ACU is approximately 2 GB of RAM. In pause mode, the cluster can scale to 0 ACUs and incur only storage costs, but resuming from pause takes approximately 30 seconds.
- **BigQuery**: no server concept at all. Charges are based purely on bytes scanned by queries, plus storage at $0.02/GB/month. Slots (compute units) are allocated and released automatically per query.
- **Azure Cosmos DB serverless**: priced per Request Unit (RU) consumed. A single-point read of a 1 KB document costs 1 RU. Maximum burst is 5,000 RU/s per container.

### Cold Start Implications

Aurora Serverless pause mode introduces a 25–35 second resume latency. For a low-traffic production service (e.g., a nightly batch job), this is acceptable. For a web application where a user triggers the first request of the day, it is not. Design connection retry logic with exponential backoff:

```python
import time
import psycopg2

def connect_with_retry(dsn: str, max_attempts: int = 5) -> psycopg2.extensions.connection:
    delay = 1.0
    for attempt in range(max_attempts):
        try:
            return psycopg2.connect(dsn)
        except psycopg2.OperationalError as e:
            if attempt == max_attempts - 1:
                raise
            print(f"Connection failed (attempt {attempt + 1}): {e}. Retrying in {delay:.1f}s...")
            time.sleep(delay)
            delay = min(delay * 2, 30.0)  # cap at 30 seconds
```

Alternatively, disable pause mode and keep a minimum of 0.5 ACU allocated at all times. The cost is approximately $43/month — cheap insurance against cold-start complaints.

## 18.7 HTAP — Hybrid Transactional and Analytical Processing

### The Problem

OLTP databases are optimized for row-based storage: B-tree indexes, small random reads, and short transactions. OLAP databases are optimized for columnar scan: no indexes, sequential I/O over gigabytes of data, and long-running aggregations. Traditionally, data was moved from the OLTP system to the OLAP warehouse via an ETL pipeline that ran every hour or every night, introducing lag of hours or days before analytical results reflected the latest transactions.

HTAP (Hybrid Transactional + Analytical Processing) eliminates this lag by maintaining both a row store and a columnar store within the same database engine, automatically routing queries to the appropriate store.

### HTAP Systems

- **TiDB**: a distributed NewSQL database with two storage layers — TiKV (OLTP row store, based on RocksDB) and TiFlash (real-time columnar replica). The TiDB query planner automatically routes OLTP queries to TiKV and OLAP queries to TiFlash. No ETL pipeline; TiFlash is updated in real time via Raft replication.
- **SingleStore (formerly MemSQL)**: an in-memory database with both a rowstore (for OLTP) and a columnstore (for OLAP) in the same engine. Rows migrate from the in-memory rowstore to the disk-based columnstore after a configurable threshold.
- **AlloyDB for PostgreSQL** (GCP): a PostgreSQL-compatible managed database with an adaptive columnar engine. OLAP queries are transparently accelerated by a columnar cache layer without requiring a separate data warehouse. AlloyDB maintains full PostgreSQL wire protocol compatibility.
- **SQL Server 2019 HTAP**: columnstore indexes can be created alongside traditional B-tree indexes on the same table. The query optimizer chooses the columnstore for aggregate queries automatically. Accelerated Database Recovery (ADR) reduces recovery time after a crash, which is critical in mixed HTAP workloads.

### Northwind HTAP Example

On a large Northwind dataset (millions of orders), a revenue-by-category aggregation runs on a columnstore index with sub-second response time even while concurrent order inserts are in flight:

```sql
-- This query is routed to the columnstore index (or TiFlash) automatically
SELECT
    cat.category_name,
    SUM(od.unit_price * od.quantity * (1 - od.discount)) AS total_revenue,
    COUNT(DISTINCT o.order_id) AS order_count
FROM
    dbo.Orders o
    JOIN dbo.[Order Details] od ON o.OrderID = od.OrderID
    JOIN dbo.Products p ON od.ProductID = p.ProductID
    JOIN dbo.Categories cat ON p.CategoryID = cat.CategoryID
GROUP BY
    cat.category_name
ORDER BY
    total_revenue DESC;
```

In SQL Server, create the columnstore index alongside the primary clustered index:

```sql
CREATE NONCLUSTERED COLUMNSTORE INDEX ncci_OrderDetails
ON dbo.[Order Details] (OrderID, ProductID, UnitPrice, Quantity, Discount);
```

The query optimizer uses the columnstore for the aggregation and the clustered B-tree index for individual order lookups — both workloads are served optimally from the same table.

## 18.8 Immutable Ledger Databases

### Concept

An immutable ledger database is append-only: records can be inserted but never updated or deleted. Every insert is cryptographically chained to the previous one, creating a tamper-evident history. If any historical record is altered, the chain of hashes breaks, making the tampering detectable.

Use cases: financial audit trails (every debit and credit, forever), supply chain provenance (every handoff of a physical good), and regulatory compliance (evidence that records were not modified after the fact).

Immutable ledgers are distinct from blockchain in that they are centrally managed by a single authority (the cloud provider or the database engine). They do not require consensus across untrusted nodes, which is why their throughput is orders of magnitude higher than public blockchains.

| Feature | AWS QLDB | SQL Server Ledger Tables | Blockchain (Bitcoin) |
|---|---|---|---|
| Centralized | Yes (AWS-managed) | Yes (on-prem/cloud) | No (decentralized) |
| Consensus | Single authority | Single authority | PoW / PoS |
| Throughput | ~1,000 TPS | Full SQL Server TPS | ~7 TPS |
| Queryable | Yes (PartiQL) | Yes (T-SQL) | No (need indexer) |
| Immutability proof | Cryptographic digest | SHA-256 hash chain | Merkle tree |

### SQL Server Ledger Tables

SQL Server 2022 and Azure SQL Database support ledger tables natively. When a table is created as a ledger table, SQL Server maintains a system-generated, append-only ledger view that records every insert, update, and delete as a sequence of immutable rows with SHA-256 hashes linking each transaction block.

```sql
-- Create a ledger-enabled transactions table
CREATE TABLE dbo.AccountTransactions (
    TransactionId   INT IDENTITY PRIMARY KEY,
    AccountId       INT NOT NULL,
    Amount          DECIMAL(18, 2) NOT NULL,
    Description     NVARCHAR(255),
    TransactionDate DATETIME2 DEFAULT GETDATE()
) WITH (SYSTEM_VERSIONING = ON, LEDGER = ON);

-- SQL Server automatically creates a ledger view:
-- dbo.AccountTransactions_Ledger
-- Query the full immutable history of an account
SELECT
    lt.ledger_transaction_id,
    lt.commit_time,
    lt.principal_name,
    t.AccountId,
    t.Amount,
    t.Description,
    lt.ledger_operation_type_desc  -- INSERT / UPDATE / DELETE
FROM
    dbo.AccountTransactions_Ledger lt
    JOIN sys.database_ledger_transactions dlt
        ON lt.ledger_transaction_id = dlt.transaction_id
WHERE
    lt.AccountId = 1001
ORDER BY
    lt.commit_time;

-- Generate a cryptographic digest for audit submission
EXECUTE sp_generate_database_ledger_digest;
```

The digest is a JSON document containing the Merkle root of all ledger blocks up to the current time. An auditor can store this digest independently and later verify that the database has not been tampered with by running `sp_verify_database_ledger`.

## 18.9 Multi-Cloud Strategy and Vendor Lock-in

Choosing a cloud-native managed service always involves a lock-in tradeoff: the more deeply integrated the service, the harder and more expensive it is to leave. Teams should evaluate portability risk before committing to a proprietary data store.

| Technology | Portability risk | Open alternative |
|---|---|---|
| BigQuery | High (proprietary API) | Apache Spark on any cloud |
| DynamoDB | High (proprietary SDK) | Apache Cassandra |
| Spanner | High (no open equivalent) | CockroachDB |
| RDS PostgreSQL | Low (standard PostgreSQL) | Any PostgreSQL host |
| Cosmos DB | Medium (SQL API is subset) | MongoDB + Cosmos migration |

### Cost Gotchas

Managed databases are often cheaper than they appear on the pricing page once the following charges are factored in:

- **Egress fees**: data transferred out of a cloud region or to the internet typically costs $0.08–0.12/GB. A data warehouse that exports 10 TB/month to an on-premises BI tool incurs $800–$1,200/month in egress alone.
- **Cross-region replication charges**: replicating data between regions is charged separately from storage. Cosmos DB multi-master replication is priced per region added.
- **Reserved Instance vs on-demand tradeoff**: reserved instances (1- or 3-year commitments) reduce compute costs by 30–60% compared to on-demand pricing. Teams that run stable, predictable workloads should evaluate RIs early.
- **Read replica costs**: each read replica is a full copy of the primary instance, billed at the same instance rate. Five read replicas cost six times the primary.

### Portability Strategies

1. **Use standard engines where possible.** RDS PostgreSQL is dramatically more portable than DynamoDB. The PostgreSQL wire protocol is supported by every major cloud and dozens of third-party providers.
2. **Abstract behind an interface.** A thin data access layer that hides DynamoDB-specific query patterns behind a repository interface can be swapped out if migration becomes necessary.
3. **Export data regularly.** Even if you cannot easily migrate the live service, a daily export to cloud storage (Parquet or CSV) ensures data is never hostage to the vendor.

## 18.10 Choosing the Right Service

No single database service is optimal for every workload. The decision tree below provides a starting point; the right answer always depends on team expertise, existing cloud commitments, and cost constraints.

```
Need SQL?
  → Global multi-region writes required?
      YES → Spanner / Aurora Global (read-scale) / Cosmos DB SQL API (multi-master)
      NO  → Single-region OLTP?
              YES → Cloud SQL / RDS / Azure SQL Database
              NO  → Analytical queries on petabyte scale?
                      YES → BigQuery / Redshift / Azure Synapse Analytics
Need NoSQL?
  → Real-time mobile/web push sync?
      YES → Firestore
      NO  → High-throughput key-value / document?
              YES → DynamoDB (AWS-native) / Cosmos DB NoSQL API (Azure-native)
              NO  → Graph traversal?
                      YES → Neptune / Cosmos DB Gremlin API / Neo4j AuraDB
                      NO  → Time-series / IoT sensor streams?
                              YES → Bigtable / AWS Keyspaces / InfluxDB Cloud
Need HTAP (OLTP + OLAP, no ETL lag)?
  → TiDB / AlloyDB / SingleStore / SQL Server columnstore
Need immutable audit trail?
  → AWS QLDB / SQL Server Ledger Tables
```

### Service Selection Summary

| Workload | GCP | AWS | Azure |
|---|---|---|---|

| Global writes | Cloud Spanner | Aurora Global + QLDB | Cosmos DB |
