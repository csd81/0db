# 18. fejezet — Felhőalapú adatbázis-ökoszisztémák: GCP, AWS és Azure

A szerverparkbérlés és a kézzel végzett adatbázis-szerverjavítás kora véget ért a legtöbb csapat számára. A felügyelt felhőalapú adatbázisok azért jöttek létre, mert az adatbázis üzemeltetésének terhei — biztonsági javítások alkalmazása, biztonsági mentések ütemezése, lemezek átméretezése, feladatátvételi tesztek — kevés köze van ahhoz az üzleti problémához, amelyet a csapat meg szeretne oldani. A felhőszolgáltatók vállalják ezt a terhet előfizetési díj és egyes esetekben szállítói kötöttség ellenében. A legfontosabb szemléletváltás: az „adatbázis-szervert üzemeltetünk" gondolkodástól az „a munkaterhelésnek megfelelő felügyelt szolgáltatást választjuk" gondolkodás felé kell elmozdulni.

A felügyelt szolgáltatások négy olyan képességet nyújtanak, amelyeket helyszíni (on-premises) megoldásokkal nehéz vagy drága megvalósítani:

- **DBA nélküli javítás és biztonsági mentés.** A szolgáltató gondoskodik az operációs rendszer javításairól, a motorfrissítésekről (karbantartási ablakokkal) és az automatikus biztonsági mentésekről, időalapú visszaállítással (PITR).
- **Rugalmas skálázás.** A tárhely automatikusan növekszik; a számítási kapacitás néhány perc leállással átméretezhető (egyes szerver nélküli rétegeken leállás nélkül).
- **Globális terjesztés.** Az olvasási replikák, a több régióba irányuló írások és az automatikus feladatátvétel elsőrangú funkciók, nem kézzel összerakott Pacemaker-konfigurációk.
- **SLA által garantált rendelkezésre állás.** A szolgáltatók 99,95%–99,999%-os üzemidőt garantálnak, pénzügyi jóváírással a megsértések esetén. A helyszíni SLA csak annyira jó, amennyire az üzemeltetési csapat ügyeletési rendszere megbízható.

Ez a fejezet áttekinti a legfontosabb felügyelt adatbázis-szolgáltatásokat a Google Cloud Platform (GCP), az Amazon Web Services (AWS) és a Microsoft Azure platformokon, munkaterhelés-típusonként szervezve.

## 18.1 Felügyelt relációs adatbázisok (OLTP)

A leggyakoribb adatbázis-munkaterhelés egy webalkalmazást kiszolgáló standard relációs adatbázis: tranzakciós írások, indexelt olvasások, idegen kulcsos korlátok és SQL-összekapcsolások. Mindhárom nagy felhőszolgáltató felügyelt megfelelőjét kínálja a csapatok által már ismert adatbázisoknak.

| Funkció | GCP Cloud SQL | AWS RDS | Azure SQL Database |
|---|---|---|---|
| Motorok | PostgreSQL, MySQL, SQL Server | PostgreSQL, MySQL, MariaDB, Oracle, SQL Server | SQL Server, PostgreSQL, MySQL |
| Maximális tárhely | 64 TB | 64 TB | 4 TB (általános célú) |
| Olvasási replikák | Igen | Igen | Aktív geo-replikáció |
| Automatikus biztonsági mentés | Igen (PITR 7–35 nap) | Igen (automatikus pillanatképek) | Igen (automatikus) |
| Szerver nélküli lehetőség | Nem | Aurora Serverless v2 | Serverless csomag |
| Legjobb alkalmazás | Standard webalkalmazások, GCP-natív | AWS-elsőként tervezett munkaterhelések | Azure-elsőként tervezett munkaterhelések |

### Kapcsolatkészletezés

Egy naiv alkalmazás HTTP-kérésenként egy adatbázis-kapcsolatot nyit. 500 egyidejű felhasználó esetén ez 500 kapcsolat — mindegyik ~5 MB RAM-ot és egy fájlleírót emészt az adatbázis-szerveren. A felügyelt adatbázisokat sem kerüli el ez a probléma.

- **PgBouncer** (GCP/AWS): egy könnyűsúlyú kapcsolatkészletező, amely az alkalmazás és a PostgreSQL között helyezkedik el. Kisebb hosszú életű szerverkapcsolat-készletet tart fenn, és erre vetíti a kliens-kapcsolatokat. Tranzakció-módú készletezéssel a készlet mérete jóval kisebb lehet, mint az alkalmazásszálak száma.
- **RDS Proxy** (AWS): az RDS és Aurora számára natív, teljes körűen felügyelt kapcsolatkészletező. AWS IAM-integrációt is kínál a hitelesítési adatok megújításához kapcsolatbontás nélkül, ami Lambda-környezetekben különösen hasznos, ahol a kapcsolatok sűrűn nyílnak és záródnak.

### Paraméterfájlok és Cloud SQL jelzők

A helyszíni DBA-k közvetlenül szerkesztik a `postgresql.conf`-ot. A felügyelt szolgáltatásokban a motor paramétereit az alábbiak vezérlik:

- **RDS paraméterfájlok**: egy névvel ellátott paraméterkészlet, amelyet egy RDS-példányhoz vagy klaszterhez csatolnak. A statikus paraméterek módosítása újraindítást igényel; a dinamikus paraméterek azonnal érvényesülnek.
- **Cloud SQL jelzők**: példányonkénti jelzők, amelyeket a Cloud SQL API-n vagy konzolán keresztül állítanak be. Ugyanaz a statikus/dinamikus megkülönböztetés érvényes.
- **Azure SQL**: egyes paramétereket a szolgáltatáscsomag rögzít; mások kiszolgálói szintű beállításként vagy adatbázis-hatályú konfigurációként érhetők el.

### Karbantartási ablakok

A felügyelt szolgáltatások javításokat és kisebb verziófrissítéseket alkalmaznak egy konfigurálható karbantartási ablak során — általában egy 30 perces sáv az alacsony forgalmú időszakban. Éles rendszereknél:

1. Az ablakot a legkisebb forgalmú időszakra állítsd be (pl. vasárnap 03:00–03:30 helyi idő szerint).
2. Engedélyezz értesítéseket, hogy az ügyeletes mérnökök tájékozódhassanak.
3. Teszteld a következő kisebb verziót egy átmeneti replikán, mielőtt az ablak megérkezne.

## 18.2 Globálisan elosztott relációs adatbázisok

A standard felügyelt relációs adatbázisok lényegüknél fogva egyetlen régióhoz kötöttek. Az olvasási replikák segítenek az olvasási skálázhatóságon, de az írások egyetlen elsődleges csomóponthoz kerülnek. Valóban globális alkalmazásokhoz — ahol négy kontinens felhasználói egyidejűleg írnak adatokat — egy új kategóriájú adatbázisra van szükség.

| Funkció | GCP Cloud Spanner | AWS Aurora Global | Azure Cosmos DB (SQL API) |
|---|---|---|---|
| Konzisztencia | Külső (valódi linearizálhatóság) | Végleges / erős konzisztencia (régiónként) | Hangolható (5 szint) |
| Több régióba irányuló írások | Igen | Elsődleges + olvasási replikák | Igen (több mester) |
| SQL-támogatás | ANSI SQL bővítményekkel | MySQL/PostgreSQL dialektus | SQL API (részhalmaz) |
| CAP-pozíció | CP | CP | AP (alapért.) / CP (erős) |
| Ármodell | Csomópontonként + I/O | ACU-nként | RU (Request Unit) alapon |

### Spanner TrueTime

A Cloud Spanner külső konzisztenciát ér el — ez az erős sorrendezhetőségnél is erősebb tulajdonság — globális zárolás nélkül. A mechanizmus a TrueTime: egy időmérési API, amelyet GPS-vevők és atomi órák támasztanak alá minden Google-adatközpontban.

A TrueTime nem egyetlen időbélyeget, hanem egy `[legkorábbi, legkésőbbi]` intervallumot ad vissza, amely garantáltan tartalmazza az igazi falióra-időt. A bizonytalanság (jellemzően 1–7 ms) korlátozott, mert a GPS és az atomi óra lassan sodródik, és folyamatosan keresztellenőrzött.

Amikor a Spanner egy tranzakcióhoz commit-időbélyeget rendel, kirakatja a TrueTime bizonytalansági intervallumát, mielőtt a commit teljesnek minősítené. Ez a „commit wait" azt jelenti, hogy mire egy tranzakció láthatóvá válik az olvasók számára, az időbélyege bizonyíthatóan a múltban van — semelyik, késóbbi időbélyegű tranzakció sem kerülhetett korábban commitálásra. A külső konzisztencia ebből következik, csomópontok közötti zárolás nélkül.

Ha egy Spanner-csomópont GPS-órája a bizonytalansági határon túl sodródik, a TrueTime szélesíti az intervallumot, ami növeli a commit-wait késést. A rendszer helyesen működik marad, de az áteresztőképesség csökken, amíg az óra újra nem szinkronizálódik. A Google adatai szerint a 99. percentilis bizonytalanság 7 ms alatt van.

### Aurora Global adatbázis

Az Aurora Global egy elsődleges régióból (olvasás-írás) és legfeljebb öt másodlagos régióból (csak olvasás) áll. A replikáció dedikált naplóküldési réteget használ, jellemzően 1 másodpercnél kisebb késéssel. Katasztrófa-helyreállítási helyzetben egy másodlagos régió egy percen belül elő léphet elsődleges régióvá. Az Aurora nem kínál valódi több-régióba irányuló írásokat a Spanner módjára; az elsődleges régió marad az egyetlen írási jogosultság.

### Cosmos DB konzisztenciaszintek

Az Azure Cosmos DB öt hangolható konzisztenciaszintet kínál az erőstől a véglegesig:

1. **Erős** — linearizálható, mint a Spanner. Az olvasások mindig a legfrissebb commitált írást látják.
2. **Korlátozott elavulás** — az olvasások legfeljebb K verziót vagy T másodpercet késnek.
3. **Munkamenet** — egyetlen kliens-munkameneten belül az olvasások monotonok; a munkamenet szempontjából erős konzisztencia, mások számára végleges.
4. **Konzisztens előtag** — az olvasások soha nem látnak felcserélt sorrendű írásokat; nincs hiány, de lehet elavult.
5. **Végleges** — a legkisebb késés, nincs sorrendbiztosítás.

A legtöbb több-régiós Cosmos DB-telepítés Munkamenet konzisztenciát használ, amely lefedi a leggyakoribb esetet (a felhasználó mindig látja a saját írásait) az Erős konzisztencia késési költsége nélkül.

## 18.3 NoSQL dokumentumtárak

A dokumentum-adatbázisok JSON-szerű objektumokat tárolnak rugalmas sémával. Akkor jeleskednek, ha az adatmodell gyorsan változik, ha az objektumok változó szerkezetűek, vagy ha az alkalmazásnak egy teljes entitást kell egyetlen olvasásban lekérni összekapcsolások nélkül.

| Funkció | GCP Firestore | AWS DynamoDB | Azure Cosmos DB (NoSQL API) |
|---|---|---|---|
| Modell | Dokumentum (JSON) | Kulcs-érték + dokumentum | Dokumentum (JSON) |
| Lekérdezés | Gyűjteményes lekérdezések | Elsődleges kulcs + GSI | SQL-szerű NoSQL lekérdezés |
| Valós idejű szinkronizáció | Igen (listenerek) | Nem (DynamoDB Streams) | Nem |
| Szerver nélküli | Igen | Igen (igény szerinti) | Igen (serverless) |
| Offline szinkronizáció | Igen (mobil SDK) | Nem | Nem |
| Legjobb alkalmazás | Mobil/webes valós idejű alkalmazások | Nagy áteresztőképességű K/V | Több modell, Azure-natív |

### Firestore Python SDK

```python
from google.cloud import firestore

db = firestore.Client(project="northwind-project")

# Dokumentum hozzáadása a 'orders' gyűjteményhez
order_ref = db.collection("orders").document()
order_ref.set({
    "customer_id": "ALFKI",
    "order_date": firestore.SERVER_TIMESTAMP,
    "items": [
        {"product_id": 11, "unit_price": 14.0, "quantity": 12},
        {"product_id": 42, "unit_price": 9.8, "quantity": 10},
    ],
    "shipped": False,
})
print(f"Rendelés mentve: {order_ref.id}")

# Valós idejű listener — minden dokumentumváltozáskor lefut
def on_snapshot(doc_snapshot, changes, read_time):
    for doc in doc_snapshot:
        print(f"Rendelés állapota frissült: {doc.to_dict()}")

order_ref.on_snapshot(on_snapshot)
# A listener háttérszálban fut; tartsuk életben a folyamatot
import time
time.sleep(30)
```

A valós idejű listenerek a Firestore megkülönböztető funkciói. Egy mobilalkalmazás feliratkozhat egy dokumentumra vagy gyűjteményes lekérdezésre, és az SDK frissítéseket küld egy tartós gRPC-adatfolyamon keresztül anélkül, hogy az alkalmazás lekérdezne. Ez teszi a Firestore-t természetes választássá csevegőalkalmazásokhoz, élő irányítópultokhoz és kollaboratív szerkesztéshez, ahol a szervernek változásokat kell küldenie a klienseknek.

### DynamoDB boto3

```python
import boto3
from boto3.dynamodb.conditions import Key

dynamodb = boto3.resource("dynamodb", region_name="us-east-1")
table = dynamodb.Table("NorthwindOrders")

# Elem írása
table.put_item(Item={
    "CustomerId": "ALFKI",          # partíció kulcs
    "OrderId": "ORD-10248",         # rendezési kulcs
    "OrderDate": "1996-07-04",
    "TotalAmount": 440.0,
    "Shipped": False,
})

# Egy ügyfél összes rendelésének lekérdezése
response = table.query(
    KeyConditionExpression=Key("CustomerId").eq("ALFKI")
)
for item in response["Items"]:
    print(item["OrderId"], item["TotalAmount"])

# Lekérdezés rendezési kulcs feltétellel (1996-ban leadott rendelések)
response = table.query(
    KeyConditionExpression=(
        Key("CustomerId").eq("ALFKI") &
        Key("OrderId").begins_with("ORD-1996")
    )
)
```

A DynamoDB adatmodellje megköveteli a partíciókulcs és az opcionális rendezési kulcs előzetes meghatározását. Minden lekérdezésnek tartalmaznia kell a partíciókulcsot; a rendezési kulcs feltételei és szűrői másodlagosak. A globális másodlagos indexek (GSI) alternatív kulcskombinációkon való lekérdezést tesznek lehetővé, de minden GSI az adatok külön másolata, amelyet külön számláznak.

## 18.4 Széles oszlopos / idősor-adatbázisok

A széles oszlopos adatbázisok sorkulccsal azonosított sorokban tárolják az adatokat, oszlopcsaládokba csoportosítva. A relációs táblákkal ellentétben nem minden sornak kell azonos oszlopokkal rendelkeznie. Ez a modell optimális az idősor-adatokhoz (szenzorleolvasások, naplóesemények), ahol minden sor egyedi időbélyeg-kulcsú esemény, az oszlopok pedig metrikanevek.

| Funkció | GCP Bigtable | AWS Keyspaces (Cassandra) | Azure Cosmos DB (Cassandra API) |
|---|---|---|---|
| Modell | Széles oszlopos (saját fejlesztésű) | Széles oszlopos (CQL kompatibilis) | Széles oszlopos (CQL kompatibilis) |
| Késés | < 10 ms p99 | < 10 ms p99 | < 10 ms p99 |
| Replikáció | Régiók közötti klaszter | Több régió Cassandran keresztül | Globális terjesztés |
| Minimális ár | ~0,65 $/óra/csomópont | Serverless (RCU/WCU-nként) | Serverless |
| Legjobb alkalmazás | IoT szenzorfolyamok, ML jellemzőtár | Meglévő Cassandra-migráció | Azure-natív Cassandra |

### Bigtable sorkulcs-tervezés

A sorkulcs a Bigtable egyetlen indexelt mezője. Minden olvasás és írás sorkulccsal kereshető, így tervezése határozza meg, hogy a terhelés egyenletesen oszlik-e el a csomópontok (tabletok) között, vagy egyetlen csomóponton összpontosul (hotspot).

**Antipattern — monoton időbélyeg-előtag:**

```
# ROSSZ: minden írás ugyanarra a tabletre kerül a kulcstér vezető szélén
row_key = f"{timestamp_ms}#sensor_{sensor_id}"
# Példa: "1714694400000#sensor_42"
```

Mivel a Bigtable lexikográfiai sorrendben tárolja a sorokat sorkulcs szerint, egy monoton növekvő időbélyeg-előtag azt jelenti, hogy minden új írás az utolsó tabletre kerül. Az a tablet hotspottá válik, míg a többi tétlen marad.

**1. minta — Fordított időbélyeg:**

```
# JÓ: a legújabb adat kerül előre; elkerüli a vezető-széli hotspotot
MAX_TIMESTAMP = 9999999999999
row_key = f"{MAX_TIMESTAMP - timestamp_ms}#sensor_{sensor_id}"
```

Ez egyenletesen osztja el a szekvenciális írásokat a kulcstérben, mert a fordított időbélyeg monoton csökken, az első tabletre terítve a terhelést az utolsó helyett. Az a járulékos előnye is megvan, hogy a 0. sortól való szkennelés a legfrissebb adatot adja vissza először.

**2. minta — Sózott előtag:**

```python
import hashlib

def make_row_key(sensor_id: str, timestamp_ms: int, buckets: int = 100) -> str:
    salt = int(hashlib.md5(sensor_id.encode()).hexdigest(), 16) % buckets
    return f"{salt:03d}#{sensor_id}#{timestamp_ms}"
# Példa: "042#sensor_42#1714694400000"
```

A só (a szenzor-azonosító hashe moduló a vödrök száma) 100 egyenletesen elosztott sorkulcs-előtag közé osztja a szenzorokat. Ez az írási terhelést sok tabletre osztja szét. A kompromisszum: egy szenzor összes leolvasásának lekéréséhez mind a 100 vödröt le kell kérdezni és az eredményeket össze kell olvasztani.

## 18.5 Adattárházak (OLAP)

Az online analitikai feldolgozás (OLAP) munkaterhelése az OLTP ellentéte: kevés írás, milliárd sorokat átfogó óriási olvasások, egész oszloptartományokra kiterjedő aggregációk. Az oszlopos tárolás itt optimális, mert a `SELECT SUM(amount)` lekérdezés csak az `amount` oszlop bájtjait olvassa, az összes többi oszlopot kihagyva.

| Funkció | GCP BigQuery | AWS Redshift | Azure Synapse Analytics |
|---|---|---|---|
| Architektúra | Szerver nélküli, oszlopos | Klaszteres (RA3), serverless lehetőséggel | Dedikált készlet + serverless |
| Árképzés | 5 $/beolvasott TB | ~0,25 $/óra/csomópont | DWU-óránként |
| Particionálás | Dátum/tartomány szerint | Disztribúciós kulcsok | Hash / körhöz rendelés |
| ML-integráció | BigQuery ML (SQL) | Redshift ML (SageMaker) | Azure ML |
| Streaming-betöltés | Pub/Sub → BigQuery | Kinesis → Redshift | Event Hubs → Synapse |
| Legjobb alkalmazás | Ad-hoc analitika, petabájtos skála | AWS-natív analitika | Azure BI-munkaterhelések |

### BigQuery ML

A BigQuery ML standard SQL-szintaxissal teszi lehetővé gépi tanulási modellek betanítását és pontozását anélkül, hogy az adatokat ki kellene mozgatni az adattárházból.

```sql
-- Lineáris regressziós modell betanítása a rendelési bevétel előrejelzéséhez
CREATE OR REPLACE MODEL northwind.revenue_model
OPTIONS (
    model_type = 'linear_reg',
    input_label_cols = ['total_revenue']
) AS
SELECT
    c.country,
    p.category_id,
    EXTRACT(MONTH FROM o.order_date) AS order_month,
    SUM(od.unit_price * od.quantity * (1 - od.discount)) AS total_revenue
FROM
    northwind.orders o
    JOIN northwind.order_details od ON o.order_id = od.order_id
    JOIN northwind.customers c ON o.customer_id = c.customer_id
    JOIN northwind.products p ON od.product_id = p.product_id
GROUP BY 1, 2, 3;

-- Új adatok pontozása a betanított modellel
SELECT *
FROM ML.PREDICT(
    MODEL northwind.revenue_model,
    (
        SELECT 'Germany' AS country, 1 AS category_id, 12 AS order_month
    )
);
```

### Költségoptimalizálás partíciószűrőkkel

A BigQuery beolvasott bájtok alapján számláz. Egy 500 GB-os rendeléstáblán partíciószűrő nélkül egy lekérdezés körülbelül 2,50 dollárba kerül (500 GB × 0,005 $/GB). Egy partíciószűrő hozzáadása, amely a vizsgálatot az aktuális hónapra (5 GB) korlátozza, a költséget 0,025 dollárra csökkenti — 100-szoros megtakarítás.

```sql
-- Drága: az egész táblát beolvassa (500 GB)
SELECT customer_id, SUM(freight) AS total_freight
FROM northwind.orders
GROUP BY customer_id;

-- Olcsó: a partíciószűrő ~5 GB-ra korlátozza a beolvasást
SELECT customer_id, SUM(freight) AS total_freight
FROM northwind.orders
WHERE order_date >= DATE_TRUNC(CURRENT_DATE(), MONTH)
GROUP BY customer_id;
```

Nagy táblákon végzett ad-hoc lekérdezések futtatása előtt mindig ellenőrizd a BigQuery felületen a becsült beolvasott bájtokat.

## 18.6 Szerver nélküli és automatikus skálázás

A szerver nélküli (serverless) modell kiküszöböli a kapacitástervezést: nincsenek kiosztandó szerverek, nincsenek választandó példányméret, és nincs díj, amikor a szolgáltatás üresjáratban van. A kompromisszum a teljesítményjellemzők feletti csökkentett irányítás és a hideg indítás (cold start) kockázata.

- **Firestore**: üresjáratban nullára skálázódik. Az árképzés dokumentumolvasásonként (0,06 $/100 K), írásonként (0,18 $/100 K) és törlésenként (0,02 $/100 K) történik, plusz tárhely 0,18 $/GB/hó. Egy Northwind-méretű alkalmazás napi 10 000 olvasással filléreket költ.
- **DynamoDB igény szerint**: az árképzés felhasznált írási kapacitásegységenként (WCU) és olvasási kapacitásegységenként (RCU) történik. Forgalmi csúcsok esetén ezredmásodpercek alatt automatikusan skálázódik. Nincs üresjárati díj. Alkalmas csúcsos, kiszámíthatatlan forgalom esetén.
- **Aurora Serverless v2**: az Aurora Kapacitásegységeket (ACU) 0,5-től 128-ig, 0,5 ACU-s lépésekben skálázza. Minden ACU körülbelül 2 GB RAM-nak felel meg. Szüneteltetési módban a klaszter 0 ACU-ra skálázódhat és csak tárhelyköltséget von maga után, de a szünetből való visszaállás körülbelül 30 másodpercet vesz igénybe.
- **BigQuery**: nincs szerver fogalom. A díjak kizárólag a lekérdezések által beolvasott bájtokra vonatkoznak, plusz 0,02 $/GB/hó tárhely. A slotokat (számítási egységeket) a rendszer lekérdezésenként automatikusan foglalja és szabadítja fel.
- **Azure Cosmos DB serverless**: felhasznált kérésenységenként (RU) számlázódik. Egy 1 KB-os dokumentum egypont-olvasása 1 RU-ba kerül. A maximális kitörés tárolónként 5 000 RU/s.

### Hideg indítás hatásai

Az Aurora Serverless szüneteltetési módja 25–35 másodperces folytatási késést vezet be. Kis forgalmú éles szolgáltatásnál (pl. éjszakai kötegelt feladat) ez elfogadható. Olyan webalkalmazásnál, ahol egy felhasználó indítja a nap első kérését, nem az. Tervezz kapcsolat-újrapróbálkozási logikát exponenciális visszalépéssel:

```python
import time
import psycopg2

def connect_with_retry(dsn: str, max_attempts: int = 5) -> psycopg2.extensions.connection:
    delay = 1.0
    for attempt in range(max_attempts):
        try:
            return psycopg2.connect(dsn)
        except psycopg2.OperationalError as e:
            if attempt == max_attempts - 1:
                raise
            print(f"Kapcsolat sikertelen ({attempt + 1}. kísérlet): {e}. Újrapróbálkozás {delay:.1f}s múlva...")
            time.sleep(delay)
            delay = min(delay * 2, 30.0)  # maximálás 30 másodpercnél
```

Alternatívaként tiltsd le a szüneteltetési módot, és tartsd fenn a minimum 0,5 ACU-t. A költség körülbelül havi 43 dollár — olcsó biztosíték a hideg indítási panaszok ellen.

## 18.7 HTAP — Hibrid tranzakciós és analitikai feldolgozás

### A probléma

Az OLTP adatbázisok soralapú tárolásra vannak optimalizálva: B-fa indexek, kis véletlenszerű olvasások és rövid tranzakciók. Az OLAP adatbázisok oszlopos szkennelésre optimalizáltak: nincsenek indexek, szekvenciális I/O gigabájtokon át, és hosszú futású aggregációk. Hagyományosan az adatokat az OLTP rendszerből az OLAP adattárházba ETL-folyamattal mozgatták, amely óránként vagy éjszakánként futott, így az analitikai eredmények órákat vagy napokat késtek a legfrissebb tranzakcióktól.

A HTAP (Hibrid Tranzakciós + Analitikai Feldolgozás) ezt a késést küszöböli ki azzal, hogy ugyanazon az adatbázismotoron belül sor tárat és oszlopos tárat is fenntart, a lekérdezéseket automatikusan a megfelelő tárolóhoz irányítva.

### HTAP rendszerek

- **TiDB**: elosztott NewSQL adatbázis két tárolóréteggel — TiKV (OLTP sortár, RocksDB alapú) és TiFlash (valós idejű oszlopos replika). A TiDB lekérdezéstervező automatikusan irányítja az OLTP lekérdezéseket a TiKV-re és az OLAP lekérdezéseket a TiFlash-re. Nincs ETL-folyamat; a TiFlash valós időben frissül Raft-replikációval.
- **SingleStore (korábban MemSQL)**: memóriában lévő adatbázis sor-tárolóval (OLTP-hez) és oszloptárolóval (OLAP-hoz) ugyanazon a motoron belül. A sorok egy konfigurálható küszöb után a memóriában lévő sortárból a lemezes oszloptárba kerülnek.
- **AlloyDB for PostgreSQL** (GCP): PostgreSQL-kompatibilis felügyelt adatbázis adaptív oszlopos motorral. Az OLAP lekérdezések átlátszóan gyorsulnak egy oszlopos gyorsítótár réteggel, külön adattárház nélkül. Az AlloyDB megőrzi a teljes PostgreSQL wire-protokoll kompatibilitást.
- **SQL Server 2019 HTAP**: az oszloptar-indexek ugyanazon a táblán hozhatók létre a hagyományos B-fa indexek mellett. A lekérdezésoptimalizáló automatikusan az oszloptart választja az aggregációs lekérdezésekhez. A gyorsított adatbázis-helyreállítás (ADR) csökkenti az összeomlás utáni helyreállítási időt, ami vegyes HTAP-munkaterheléseknél kritikus.

### Northwind HTAP példa

Egy nagy Northwind-adatkészleten (millió rendelés) a kategória szerinti bevétel-aggregáció oszloptar-indexen fut ezredmásodperces válaszidővel még akkor is, ha egyidejű rendelési írások folynak:

```sql
-- Ez a lekérdezés automatikusan az oszloptár-indexre (vagy TiFlash-re) kerül
SELECT
    cat.category_name,
    SUM(od.unit_price * od.quantity * (1 - od.discount)) AS total_revenue,
    COUNT(DISTINCT o.order_id) AS order_count
FROM
    dbo.Orders o
    JOIN dbo.[Order Details] od ON o.OrderID = od.OrderID
    JOIN dbo.Products p ON od.ProductID = p.ProductID
    JOIN dbo.Categories cat ON p.CategoryID = cat.CategoryID
GROUP BY
    cat.category_name
ORDER BY
    total_revenue DESC;
```

SQL Serverben az oszloptár-index létrehozása az elsődleges fürtözött index mellé:

```sql
CREATE NONCLUSTERED COLUMNSTORE INDEX ncci_OrderDetails
ON dbo.[Order Details] (OrderID, ProductID, UnitPrice, Quantity, Discount);
```

A lekérdezésoptimalizáló az aggregációhoz az oszloptárat, az egyedi rendelési keresésekhez a fürtözött B-fa indexet használja — mindkét munkaterhelés optimálisan kiszolgált ugyanabból a táblából.

## 18.8 Megváltoztathatatlan főkönyvi adatbázisok

### Fogalom

A megváltoztathatatlan (immutable) főkönyvi adatbázis csak hozzáfűzéses: rekordok beszúrhatók, de soha nem frissíthetők vagy törölhetők. Minden egyes beszúrás kriptográfiailag láncolódik az előzőhöz, hamisítás-nyilvánvaló előzményt alkotva. Ha bármely korábbi rekordot megváltoztatják, a hashek lánca megszakad, így a hamisítás felderíthető.

Felhasználási területek: pénzügyi könyvvizsgálati nyomvonalak (minden terhelés és jóváírás, örökre), ellátási lánc eredetigazolás (egy fizikai áru minden átadása) és szabályozási megfelelőség (bizonyíték arra, hogy a rekordokat nem módosították utólag).

A megváltoztathatatlan főkönyvek különböznek a blokkláncától abban, hogy egy egyedüli hatóság (a felhőszolgáltató vagy az adatbázismotor) centrálisan kezeli őket. Nem igényelnek konszenzust nem megbízható csomópontok között, ezért az áteresztőképességük nagyságrendekkel nagyobb, mint a nyilvános blokkláncoknak.

| Funkció | AWS QLDB | SQL Server Ledger Tables | Blokklánc (Bitcoin) |
|---|---|---|---|
| Centralizált | Igen (AWS által felügyelt) | Igen (helyszíni/felhő) | Nem (decentralizált) |
| Konszenzus | Egyedüli hatóság | Egyedüli hatóság | PoW / PoS |
| Áteresztőképesség | ~1 000 TPS | Teljes SQL Server TPS | ~7 TPS |
| Lekérdezhető | Igen (PartiQL) | Igen (T-SQL) | Nem (szükséges indexelő) |
| Megváltoztathatatlanság bizonyítéka | Kriptográfiai kivonat | SHA-256 hash-lánc | Merkle-fa |

### SQL Server főkönyvi táblák

Az SQL Server 2022 és az Azure SQL Database natívan támogatja a főkönyvi táblákat. Amikor egy táblát főkönyvi táblaként hoznak létre, az SQL Server fenntart egy rendszer által generált, csak hozzáfűzéses főkönyv-nézetet, amely minden egyes beszúrást, frissítést és törlést megváltoztathatatlan sorok sorozataként rögzít, SHA-256 hashekkel összeláncolva az egyes tranzakcióblokkokat.

```sql
-- Főkönyv-engedélyezett tranzakciótábla létrehozása
CREATE TABLE dbo.AccountTransactions (
    TransactionId   INT IDENTITY PRIMARY KEY,
    AccountId       INT NOT NULL,
    Amount          DECIMAL(18, 2) NOT NULL,
    Description     NVARCHAR(255),
    TransactionDate DATETIME2 DEFAULT GETDATE()
) WITH (SYSTEM_VERSIONING = ON, LEDGER = ON);

-- Az SQL Server automatikusan létrehozza a főkönyv-nézetet:
-- dbo.AccountTransactions_Ledger
-- Egy számla teljes megváltoztathatatlan előzményének lekérdezése
SELECT
    lt.ledger_transaction_id,
    lt.commit_time,
    lt.principal_name,
    t.AccountId,
    t.Amount,
    t.Description,
    lt.ledger_operation_type_desc  -- INSERT / UPDATE / DELETE
FROM
    dbo.AccountTransactions_Ledger lt
    JOIN sys.database_ledger_transactions dlt
        ON lt.ledger_transaction_id = dlt.transaction_id
WHERE
    lt.AccountId = 1001
ORDER BY
    lt.commit_time;

-- Kriptográfiai kivonat generálása könyvvizsgálati célra
EXECUTE sp_generate_database_ledger_digest;
```

A kivonat egy JSON-dokumentum, amely tartalmazza az összes főkönyvblokk Merkle-gyökerét az aktuális időpontig. Egy könyvvizsgáló tárolhatja ezt a kivonatot függetlenül, és később ellenőrizheti, hogy az adatbázist nem hamisították-e meg az `sp_verify_database_ledger` futtatásával.

## 18.9 Több-felhős stratégia és szállítói kötöttség

Egy felhőnatív felügyelt szolgáltatás választása mindig kötöttségi kompromisszumot jelent: minél mélyebben integrált a szolgáltatás, annál nehezebb és drágább elhagyni. A csapatoknak értékelniük kell a hordozhatósági kockázatot, mielőtt egy saját fejlesztésű adattárra kötelezik el magukat.

| Technológia | Hordozhatósági kockázat | Nyílt alternatíva |
|---|---|---|
| BigQuery | Magas (saját fejlesztésű API) | Apache Spark bármelyik felhőn |
| DynamoDB | Magas (saját fejlesztésű SDK) | Apache Cassandra |
| Spanner | Magas (nincs nyílt megfelelője) | CockroachDB |
| RDS PostgreSQL | Alacsony (standard PostgreSQL) | Bármelyik PostgreSQL-hoszt |
| Cosmos DB | Közepes (az SQL API részhalmaz) | MongoDB + Cosmos-migráció |

### Rejtett költségek

A felügyelt adatbázisok az árlistán szereplőnél gyakran többe kerülnek, ha a következő díjakat is figyelembe vesszük:

- **Kimenő forgalom díjai**: a felhőrégióból vagy az internetre irányuló adatátvitel jellemzően 0,08–0,12 $/GB. Egy adattárház, amely havi 10 TB-ot exportál egy helyszíni BI-eszközbe, önmagában 800–1 200 dollár kimenő forgalmi díjat okoz havonta.
- **Régiók közötti replikációs díjak**: az adatok régiók közötti replikálása a tárhelydíjon felül számolódik. A Cosmos DB több-mesteres replikációja régiónként hozzáadva kerül díjazásra.
- **Fenntartott példány versus igény szerinti kompromisszum**: a fenntartott példányok (1 vagy 3 éves kötelezettségek) az igény szerinti árképzéshez képest 30–60%-kal csökkentik a számítási költségeket. A stabil, kiszámítható munkaterheléseket futtató csapatoknak érdemes korán értékelniük a fenntartott példányokat.
- **Olvasási replika költségei**: minden olvasási replika az elsődleges példány teljes másolata, azonos példányárral számlázva. Öt olvasási replika hatszor annyiba kerül, mint az elsődleges.

### Hordozhatósági stratégiák

1. **Ahol lehetséges, standard motorokat használj.** Az RDS PostgreSQL lényegesen hordozhatóbb, mint a DynamoDB. A PostgreSQL wire-protokollt minden nagy felhő és tucatnyi harmadik féltől származó szolgáltató támogatja.
2. **Interfész mögé rejtsd.** Egy vékony adathozzáférési réteg, amely egy repository-interfész mögé rejti a DynamoDB-specifikus lekérdezési mintákat, felváltható, ha szükségessé válik a migráció.
3. **Rendszeresen exportáld az adatokat.** Még ha az élő szolgáltatást nem lehet könnyen migrálni, egy napi export felhőtárhelyre (Parquet vagy CSV formátumban) biztosítja, hogy az adatok soha ne legyenek a szállítónak kiszolgáltatva.

## 18.11 Feladatok

1. **BigQuery költségbecslő.** A BigQuery árképzési kalkulátorával (vagy a 5 $/beolvasott TB képlettel) becsüld meg egy 500 GB-os Northwind-méretű rendeléstáblán napi 1 000 lekérdezés futtatásának havi költségét, feltételezve, hogy nincs partíciószűrő. Ezután becsüld meg újra egy `WHERE order_date >= DATE_TRUNC(CURRENT_DATE(), MONTH)` partíciószűrővel, amely lekérdezésenként 5 GB-ra csökkenti a beolvasott adatot. Számítsd ki a havi megtakarítást és a megtérülési pontot, ha a particionálás megvalósítása 2 óra mérnöki munkát igényel 150 $/óra áron.

2. **Spanner TrueTime.** Magyarázd el saját szavaiddal, miért tud a Spanner globális zárolás nélkül külső konzisztenciát garantálni. A válaszod terjedjen ki: (a) mit tesz közzé a TrueTime API-ként, (b) mit jelent a „commit wait" és miért szükséges, (c) mi történik a commit késésével, ha egy Spanner-csomópont GPS-órája a bizonytalansági határon túl sodródik.

3. **DynamoDB forró partíció.** Egy autómegosztó alkalmazáshoz tervezel DynamoDB-táblát. A sofőrök 5 másodpercenként frissítik GPS-pozíciójukat. A naiv tervezés a `driver_id`-t használja partíciókulcsként. Magyarázd el, milyen probléma lép fel csúcsidőben 10 000 aktív, egyidejűleg író sofőrnél. Javasolj két alternatív partíciókulcs-tervezési megközelítést, amelyek egyenletesebben osztják el az írási terhelést. Mindkét megközelítésnél írd le az olvasási lekérdezésekre bevezetett kompromisszumot.

4. **Szállítói migráció.** A céged a Northwind rendeléskezelő rendszert DynamoDB-n futtatja, egytáblás tervezéssel (rendelések, rendelési tételek, termékek és ügyfelek mind egy táblában, `PK`/`SK` mintával megkülönböztetve). A cég úgy dönt, hogy PostgreSQL-re migrál helyszínre a felhőköltségek csökkentése érdekében. Írd le a teljes migrációs tervet: (a) hogyan terveznéd újra a sémát az egytáblás DynamoDB-től a normalizált relációs táblákig, (b) hogyan exportálnád a DynamoDB-adatokat (formátum és eszközök), (c) milyen ETL-transzformációk szükségesek a `PK`/`SK` összetett kulcsok idegen kulcsokká alakításához, (d) mi az átállási stratégiád a minimális állásidő érdekében, (e) legalább két képességet, amelyet az új sémával nyersz, és kettőt, amelyet elveszítesz a DynamoDB-tervezéshez képest.
