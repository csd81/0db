# Chapter 17 — Advanced MongoDB Patterns: Schema Validation, CQRS, and Semantic Search

This chapter covers three production-grade topics that extend the Northwind Control Center beyond basic CRUD: enforcing data contracts with JSON Schema validation, separating read and write workloads with the CQRS pattern, and enabling semantic product search with pgvector.

---

## 17.1 Enforcing Schema in MongoDB

MongoDB's schema-less nature is its greatest flexibility and its most common footgun. In a relational database, inserting a row without a required column is a compile-time error. In MongoDB, inserting a document without `unit_price` succeeds silently and corrupts every aggregation that depends on that field. JSON Schema validation, introduced in MongoDB 3.6, closes this gap without sacrificing the document model's flexibility.

### The `$jsonSchema` Operator

A validator is attached to a collection via the `collMod` command:

```javascript
db.runCommand({
  collMod: 'products',
  validator: {
    $jsonSchema: {
      bsonType: 'object',
      required: ['product_id', 'product_name', 'unit_price'],
      properties: {
        product_id:     { bsonType: 'int' },
        product_name:   { bsonType: 'string' },
        unit_price:     { bsonType: 'double', minimum: 0 },
        units_in_stock: { bsonType: 'int' },
        discontinued:   { bsonType: 'bool' }
      }
    }
  },
  validationLevel:  'moderate',
  validationAction: 'warn'
})
```

Key `bsonType` values: `string`, `int`, `long`, `double`, `bool`, `array`, `object`, `date`, `objectId`. Unlike JSON Schema's `number`, BSON distinguishes integers from doubles, which matters when aggregations multiply quantities by prices.

### `validationLevel`: moderate vs strict

| Level | Behavior |
|---|---|
| `strict` (default) | All inserts AND updates must satisfy the schema. |
| `moderate` | Only new documents and updates to already-valid documents are checked. Existing invalid documents are untouched. |

Use `moderate` when retrofitting a validator onto a collection that already contains data — this avoids blocking legitimate update operations on legacy documents while still protecting new writes.

### `validationAction`: error vs warn

| Action | Behavior |
|---|---|
| `error` | The write is rejected; the driver receives a `WriteError`. |
| `warn` | The write is allowed but the violation is logged to the MongoDB server log. |

`warn` is the right choice during a migration or audit phase. You can scan the log to see which application paths produce invalid data before switching to `error` mode and breaking them.

### Comparison with SQL Constraints

SQL constraints (`NOT NULL`, `CHECK`, `FOREIGN KEY`) and MongoDB `$jsonSchema` serve the same purpose but at different layers. SQL constraints are enforced by the storage engine and are always on. MongoDB validators are optional, per-collection, and configurable at runtime — you can tighten them as your data model matures. The tradeoff: SQL constraints are validated before the data is ever written; MongoDB schema validation happens after the BSON is parsed, so malformed BSON still produces a parse error before validation runs.

### The Northwind Control Center Schema Validation Module

The CC's `services/mongo_service.py` ships two preset validators: one for `products` (requires `product_id`, `product_name`, `unit_price`) and one for `orders` (requires `customer_id`, `order_date`). The "Schema Validation" card in the MongoDB Explorer lets you apply either preset with one click, view the current validator JSON, and dry-run any document against the schema before inserting it.

## 17.2 The CQRS Pattern

### The Problem

A single MongoDB replica set node serving both the Northwind web shop and the analytics dashboard creates contention. An analyst running a 30-second aggregation over three million order documents acquires collection-level locks (on older storage engines) or saturates CPU and I/O (on WiredTiger), degrading response times for the checkout flow. The OLTP and OLAP workloads have opposing characteristics: OLTP favors small, indexed reads with low latency; OLAP favors sequential, full-collection scans with high throughput.

### The Solution: Command Query Responsibility Segregation

CQRS separates **commands** (writes: insert, update, delete) from **queries** (reads: find, aggregate). In a MongoDB deployment:

- **Commands** are always routed to the **primary** node. This ensures write durability and immediate consistency.
- **Queries** are routed to a **secondary** (replica) node using a read preference of `secondaryPreferred` or `nearest`.

The replica continuously replicates the oplog from the primary with a typical lag of under one second on a healthy network. Analytics queries operate on slightly stale data — a known and acceptable trade-off.

```python
# Command path — write to primary
primary_client = MongoClient(uri, readPreference='primary')
primary_client['northwind']['orders'].insert_one(order_doc)

# Query path — read from replica
replica_client = MongoClient(uri, readPreference='secondaryPreferred')
results = replica_client['northwind']['orders'].aggregate(pipeline)
```

### The Northwind CC CQRS Module

The CC's CQRS tab (in the MongoDB Explorer) lets you select separate command and query connections, fire a write against the primary, and immediately run the revenue-by-category aggregation against the secondary. A replication lag indicator shows how many seconds behind the secondary is by comparing `rs.status()` timestamps.

### Trade-offs

| Concern | Detail |
|---|---|
| Eventual consistency | A document inserted via the command path may not appear in a query path result for up to a few seconds. |
| Connection overhead | Two separate connection pools must be managed, each with its own auth credentials and TLS configuration. |
| Operational complexity | Replica set configuration, failover monitoring, and oplog sizing become the application team's concern. |
| Read scale | Multiple secondaries can be added independently; read throughput scales horizontally while write throughput remains on the primary. |

For Northwind-scale data (830 orders, 77 products), CQRS is over-engineering. It becomes essential once order volume reaches tens of thousands per hour and analytics dashboards are refreshed every 30 seconds.

## 17.3 pgvector and Semantic Search

### Limits of Traditional Indexing

A B-tree index on `product_name` supports exact matches and prefix searches efficiently. `LIKE '%beverage%'` forces a sequential scan. Neither approach captures meaning: a user searching for "chai" in a text box should find "Chai", "Chang", and "Ipoh Coffee" — all beverages — not only documents that literally contain the word "chai".

Semantic search solves this by representing both documents and queries as high-dimensional vectors (embeddings) where similar meanings map to geometrically close points.

### Embeddings and Vector Indexes

An embedding model (such as OpenAI's `text-embedding-3-small` or a local `sentence-transformers` model) converts a text string into a vector of 768 or 1536 floating-point numbers. The embedding for "chai" and the embedding for "Ipoh Coffee" will have a high cosine similarity because both appear in similar contexts in the training corpus.

pgvector adds a `vector` column type to PostgreSQL and three distance operators:

| Operator | Distance Metric | Use Case |
|---|---|---|
| `<=>` | Cosine distance | Semantic similarity (normalized embeddings) |
| `<->` | Euclidean (L2) distance | Absolute magnitude matters |
| `<#>` | Negative inner product | Maximum inner product search |

An approximate nearest-neighbor index (HNSW or IVFFlat) turns a full-scan cosine computation over millions of rows into a sub-millisecond lookup:

```sql
CREATE INDEX ON products USING hnsw (embedding vector_cosine_ops);
```

### Northwind Lab: Finding Beverages by Meaning

After embedding all 77 Northwind product names into a `products.embedding vector(1536)` column, the query:

```sql
SELECT product_name, category_name,
       1 - (embedding <=> query_embedding) AS similarity
FROM   products
ORDER  BY embedding <=> query_embedding
LIMIT  5;
```

— where `query_embedding` is the embedding of the string "chai" — returns:

| product_name | category_name | similarity |
|---|---|---|
| Chai | Beverages | 0.98 |
| Chang | Beverages | 0.87 |
| Ipoh Coffee | Beverages | 0.83 |
| Chartreuse verte | Beverages | 0.79 |
| Steeleye Stout | Beverages | 0.77 |

All top results are beverages even though none of them contain "chai" in their name (except the first).

### Comparison: SQL LIKE vs pgvector

| Dimension | `LIKE '%chai%'` | pgvector cosine search |
|---|---|---|
| Match basis | Literal substring | Semantic meaning |
| Index support | None (sequential scan) | HNSW / IVFFlat ANN index |
| Query latency | O(n) rows | O(log n) approximate |
| Recall | Exact (no false negatives for substring) | Approximate (tunable recall/speed trade-off) |
| Setup cost | Zero | Embedding model + pgvector extension |
| Language-aware | No | Yes (multilingual embedding models) |

pgvector does not replace B-tree indexes — it complements them. A production Northwind search would combine a trigram index for exact prefix completion and a vector index for semantic fallback.

## 17.4 Exercises

1. **Schema enforcement audit.** Apply the `orders` validator to your MongoDB `orders` collection with `validationLevel: moderate` and `validationAction: warn`. Then insert a document missing `order_date`. Check the MongoDB server log (or `db.adminCommand({getLog:'global'})`) to find the warning entry. Repeat with `validationAction: error` and observe the `WriteError` returned to the driver.

2. **CQRS replication lag.** On a local replica set with one primary and one secondary, insert 10,000 documents in a tight loop on the primary. Query `rs.status()` on the secondary every 100 milliseconds and record the `optimeDate` lag. Plot lag vs insert rate. At what inserts-per-second does the lag exceed one second?

3. **Custom validator for customers.** Write a `$jsonSchema` validator for the Northwind `customers` collection that requires `customer_id` (string, maxLength 5) and `company` (string). Apply it with `validationLevel: strict` and `validationAction: error`. Try inserting a customer with a numeric `customer_id` and capture the error message.

4. **pgvector semantic search lab.** Install the `pgvector` PostgreSQL extension (`CREATE EXTENSION vector`). Add an `embedding vector(384)` column to a local copy of the Northwind `products` table. Use the `sentence-transformers/all-MiniLM-L6-v2` model to generate embeddings for each `product_name`. Create an HNSW index. Run semantic queries for "sweet drink", "cured meat", and "grain staple". Compare the top-5 results against what a `LIKE` query would return for the same terms.
